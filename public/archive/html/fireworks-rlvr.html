    1. 为什么要从 SFT 走向 RLVR？

    SFT 的全称是 Supervised Fine-Tuning，中文是监督微调。 它是后训练的第一步，也是最稳定、最容易理解的一步。

    SFT 的基本形式是： 

    输入 prompt：
请计算 1497 × 63。

标准回答 response：
1497 × 63 = 94311。
所以最终答案是 94311。

    模型在训练时学习的是： 

    给定 prompt，尽量生成和标准 response 相似的 token 序列。
    一句话理解： SFT 本质上是模仿学习。它让模型模仿高质量示范，但不一定让模型真正学会试错和探索。

    1.1 SFT 的优点

    SFT 优点 解释
    训练稳定 就是普通 next-token prediction，工程上最成熟。
    容易启动 只要有 prompt-response 数据就能训练。
    适合学格式 比如 Chat Template、JSON、工具调用格式、回答风格。
    适合冷启动 让 base model 先学会像助手一样回答。

    1.2 SFT 的局限

    SFT 的问题也很明显：它依赖标准答案。 如果每一道数学题、每一个代码任务、每一个工具调用轨迹都要人工写完整示范，成本会非常高。

    任务 SFT 的困难 可能更适合什么
    数学推理 中间解法很多，人工写完整推理成本高。 用最终答案是否正确作为 reward。
    代码生成 同一个需求可以有很多实现方式。 用单元测试是否通过作为 reward。
    工具调用 轨迹多样，人工标注每一步成本高。 用工具名、参数、执行结果作为 reward。
    SQL Agent SQL 写法不唯一，但查询结果可验证。 用执行结果是否匹配作为 reward。
    网页 Agent 操作路径不唯一。 用最终页面状态是否达成目标作为 reward。
    关键问题： SFT 需要“标准答案”；但很多任务不需要唯一标准答案，只需要判断结果是否成功。 这就是 RLVR 出现的根本原因。

    1.3 从“模仿答案”到“验证结果”

    如果一个任务可以被自动验证，我们就可以把训练目标从“模仿一个固定答案”改成“最大化任务成功率”。 

    SFT：
人写出标准答案 → 模型模仿这个答案

RLVR：
模型自己生成答案 → 验证器判断是否正确 → 正确就奖励，错误就惩罚

    这种变化非常重要。它意味着模型不一定只能学习人类已经写好的推理路径， 也可以通过多次生成、自动检查、奖励反馈，逐渐提高自己在可验证任务上的成功率。

    2. Verifiable Reward：什么叫可验证奖励？

    Verifiable Reward 可以翻译成可验证奖励。 它不是一个抽象口号，而是一个可以执行的函数。

    最简单的形式是： 

    输入：
标准答案 reference_answer
模型输出 model_response

输出：
reward 分数

    伪代码如下： 

    def verifiable_reward_function(reference_answer, model_response):
    if extract_answer(model_response) == reference_answer:
        return 1.0
    else:
        return 0.0
    一句话理解： 可验证奖励就是：不用人主观评价，而是用规则、程序、测试器、执行结果来判断模型输出是否成功。

    2.1 可验证奖励的三种典型来源

    来源 例子 适合任务
    规则函数 正则抽取答案、字符串匹配、格式检查。 数学答案、分类标签、JSON 格式。
    测试器 / 执行器 单元测试、编译器、SQL 执行、schema validator。 代码生成、SQL Agent、工具调用。
    环境反馈 网页状态变化、任务是否完成、API 是否返回成功。 Web Agent、工具 Agent、多步任务。

    2.2 为什么它叫“Verifiable”？

    因为它判断的是可以被验证的结果。 比如：

    • 数学题最终答案是否等于标准答案；
    • 代码是否通过单元测试；
    • 函数调用名称是否正确；
    • 函数参数是否完整；
    • JSON 是否能被解析；
    • SQL 是否能执行并返回正确结果；
    • 网页任务是否进入目标状态。

    这些任务和“回答是否优美”“语气是否舒服”不一样。 它们有相对明确的成功标准。

    主观偏好任务 可验证任务
    回答是否优雅。 答案是否正确。
    解释是否让人舒服。 代码是否通过测试。
    语气是否足够礼貌。 JSON 是否合法。
    内容是否有启发性。 工具调用是否成功。
    实践判断： 一个任务能不能做 RLVR，先问一句： “我能不能写一个程序，自动判断这次输出是否成功？” 如果能，就有 RLVR 潜力。

    2.3 可验证奖励不一定只有 0 和 1

    最简单的 reward 是二值奖励： 

    正确：1
错误：0

    但真实训练中，reward 通常可以更细。 例如工具调用任务： 

    JSON 格式合法：+0.2
工具名称正确：+0.5
必要参数完整：+0.3
参数值正确：+0.5
工具执行成功：+1.0
无意义调用工具：-0.5
格式完全错误：-1.0

    这类奖励比单纯的 0/1 更细，可以指导模型学会更稳定的行为。

    注意： reward 越复杂，不一定越好。复杂 reward 如果设计不好，模型可能会钻漏洞。 真正重要的是 reward 能稳定地区分“好行为”和“坏行为”。

    3. RLVR 和 RLHF / DPO 有什么区别？

    RLVR、RLHF、DPO 都属于后训练，但它们使用的训练信号不同。

    方法 训练信号 是否需要人工偏好 适合任务
    SFT 标准答案 通常需要人工或强模型写答案 格式学习、基础指令遵循、冷启动。
    DPO chosen / rejected 偏好对 需要偏好数据 回答风格、偏好对齐、减少坏回答。
    RLHF Reward Model 分数 需要人类偏好数据训练 Reward Model 主观偏好、帮助性、安全性、表达质量。
    RLVR 规则 / 测试器 / 验证器奖励 不一定需要人工偏好 数学、代码、工具调用、SQL、格式校验。

    3.1 RLHF：让模型符合人类偏好

    RLHF 适合处理主观评价问题。 比如：

    • 哪个回答更有帮助；
    • 哪个回答更礼貌；
    • 哪个回答更安全；
    • 哪个回答更符合用户意图。

    这些问题很难写成简单规则。 所以 RLHF 通常需要人类偏好数据，再训练 Reward Model。

    3.2 RLVR：让模型提高可验证成功率

    RLVR 更适合客观任务。 比如：

    • 答案是否正确；
    • 代码是否能跑；
    • 工具是否选对；
    • 参数是否完整；
    • 任务是否完成。

    这些问题可以用规则或程序判断。 因此 RLVR 的成本可能更低，也更容易规模化。

    核心区别： RLHF 优化“人类觉得好不好”；RLVR 优化“任务是否被验证成功”。

    4. GRPO：为什么这篇文章重点讲它？

    GRPO 的全称是 Group Relative Policy Optimization， 中文可以译为组相对策略优化。

    Fireworks AI 原文把 GRPO 放在重要位置，是因为它和 RLVR 很搭。 可验证任务通常可以对同一个 prompt 生成多个答案，然后用 reward function 给每个答案打分。 GRPO 正好利用这一组答案内部的相对分数来更新模型。

    4.1 PPO 为什么重？

    PPO，全称 Proximal Policy Optimization，中文是近端策略优化。 它是经典 RLHF 中经常使用的算法。

    但 PPO 在大模型后训练中比较重，主要因为它通常需要：

    • Policy Model：正在训练的模型；
    • Reference Model：用于 KL 约束，防止模型偏离太远；
    • Reward Model：给模型输出打分；
    • Value Model：估计 baseline，用于计算 advantage；
    • GAE：Generalized Advantage Estimation，广义优势估计。
    PPO 组件 中文解释 为什么带来负担
    Value Model 价值模型 通常和 Policy Model 尺寸接近，显存和算力压力大。
    Token-level value 逐 token 价值 每个 token 都要估计 value，中间状态更多。
    GAE 广义优势估计 计算复杂，工程实现细节多。
    参数搜索 训练超参调节 组件越多，越难稳定训练。

    4.2 GRPO 做了什么简化？

    GRPO 最关键的简化是： 去掉 Value Model，用同一个 prompt 下多个生成结果的组内相对 reward 来估计 advantage。

    直观流程如下： 

    同一个 prompt
  ↓
模型生成多个回答
  ↓
reward function 给每个回答打分
  ↓
计算这一组回答的平均分
  ↓
高于平均分的回答被鼓励
  ↓
低于平均分的回答被削弱

    4.3 一个简单例子

    问题：
1497 × 63 = ?

模型生成 4 个回答：

回答 A：94311 → reward = 1
回答 B：94211 → reward = 0
回答 C：94301 → reward = 0
回答 D：94311 → reward = 1

组内平均 reward = 0.5

回答 A advantage = 1 - 0.5 = 0.5
回答 B advantage = 0 - 0.5 = -0.5
回答 C advantage = 0 - 0.5 = -0.5
回答 D advantage = 1 - 0.5 = 0.5

    这样模型就会提高生成 A、D 这类正确答案的概率，降低生成 B、C 这类错误答案的概率。

    一句话理解 GRPO： 同一道题让模型做多次，谁在同组里表现更好就鼓励谁，谁表现更差就压低谁。

    4.4 PPO 和 GRPO 对比

    比较维度 PPO GRPO
    是否需要 Value Model 通常需要。 通常不需要。
    Advantage 来源 reward + value baseline + GAE。 同一 prompt 下多个回答的相对 reward。
    奖励粒度 可以 token-level，也可以 sequence-level。 常见为 sequence-level。
    工程复杂度 较高。 相对较低。
    适合场景 通用 RLHF、复杂偏好优化。 数学、代码、reasoning、可验证任务。

    5. PPO 与 GRPO 图解

    Fireworks AI 原文引用了一张 PPO 与 GRPO 的对比图。 这张图很重要，因为它把两者的工程差别讲得很直观。

    图 1：PPO 与 GRPO 的结构对比。来源：Fireworks AI 博客《Reinforcement Learning With Verifiable Reward》。 上半部分是 PPO，需要 Value Model 参与 advantage 估计；下半部分是 GRPO，同一个 prompt 生成多个回答， 用组内 reward 的相对关系估计 advantage。 如果图片不显示，可以打开原文： https://fireworks.ai/blog/reinforcement-learning-with-verifiable-reward

    5.1 这张图上半部分：PPO

    PPO 的流程可以理解成： 

    Prompt
  ↓
Policy Model 生成回答
  ↓
Reward Model 给回答打分
  ↓
Value Model 估计 baseline
  ↓
GAE 计算 advantage
  ↓
通过 PPO loss 更新 Policy Model

    PPO 的优点是成熟、通用、理论和实践都比较丰富。 但它的缺点是重：多一个 Value Model，就多一套训练、显存、同步、调参负担。

    5.2 这张图下半部分：GRPO

    GRPO 的流程可以理解成： 

    Prompt
  ↓
Policy Model 生成多个回答
  ↓
Reward Function / Reward Model 给每个回答打分
  ↓
Group Computation 计算组内相对 advantage
  ↓
更新 Policy Model

    GRPO 不再依赖单独的 Value Model，而是把同一 prompt 下多个回答放在一起比较。 如果一个回答比同组其他回答得分更高，就说明它相对更好。

    这一节你要记住： GRPO 的核心不是“没有 reward”，而是“没有单独的 Value Model”。 它仍然需要 reward，只是 advantage 的 baseline 来自组内比较。

    6. RLVR：强化学习中的 reward 可以不是神经网络

    Fireworks AI 原文中有一个很关键的判断： 强化学习算法本身只要求有一个东西能给生成结果打分， 它并没有规定这个打分器必须是一个深度学习模型。

    因此 reward 可以来自：

    Reward 来源 说明 例子
    深度学习模型 用一个 Reward Model 给回答打分。 RLHF 中的偏好模型。
    硬编码函数 用规则直接判断输出。 答案是否正确、格式是否合规。
    两者结合 主观任务用模型，客观任务用规则。 reasoning 用 rule reward，通用对话用 RM。

    6.1 为什么这很重要？

    因为如果 reward 必须来自一个大 Reward Model，那么 RLHF 的成本会很高。 你需要：

    • 收集偏好数据；
    • 训练 Reward Model；
    • 维护 Reward Model；
    • 防止模型攻击 Reward Model；
    • 处理 Reward Model 的偏差。

    但如果任务可以由规则验证，就可以直接写 reward function。 例如： 

    数学题：
最终答案正确 → reward = 1
最终答案错误 → reward = 0

代码题：
单元测试通过 → reward = 1
运行报错 → reward = -1

工具调用：
工具名正确 → reward += 0.5
参数完整 → reward += 0.3
执行成功 → reward += 1
    核心转变： 从“训练一个模型来判断回答好不好”，转向“用规则直接判断任务有没有完成”。

    6.2 DeepSeek-R1-Zero 和 rule-based reward

    Fireworks AI 原文把 DeepSeek-R1-Zero 作为背景来讲。 它强调 DeepSeek-R1-Zero 没有先经过传统 SFT，而是直接通过大规模 RL 训练， 并使用规则化奖励来推动 reasoning 能力出现。

    这给后训练学习一个非常重要的启发：

    启发： 对数学、代码、逻辑推理这类可验证任务来说， 模型不一定只能通过模仿人类写好的推理链变强。 它也可能通过“生成 → 验证 → 奖励 → 更新”的循环逐渐提升推理能力。

    7. 实验一：四位数乘两位数

    Fireworks AI 的第一个实验非常简单：训练模型做四位数乘两位数。 它看起来像小学数学题，但非常适合说明 RLVR 的基本逻辑。

    7.1 任务格式

    原文中的任务大致是： 

    Please answer the following question.
You may write out your argument before stating your final answer X in a new line,
following the format:

"Final Answer: the final answer is X. I hope it is correct."

1497 * 63

    模型需要最终输出： 

    Final Answer: the final answer is 94311. I hope it is correct.

    7.2 为什么这个任务适合 RLVR？

    条件 说明
    答案明确 1497 × 63 的答案是确定的。
    容易抽取 可以用正则表达式抽取 final answer。
    容易验证 抽取出的数字和标准答案比较即可。
    不需要人类偏好 不需要人判断“哪个回答更好”，只要看是否正确。

    7.3 Reward Function 示例

    import re

def extract_answer(model_output):
    pattern = r"Final Answer: the final answer is (\d+). I hope it is correct."
    match = re.search(pattern, model_output)

    if match is None:
        return None

    return int(match.group(1))

def reward_fn(model_output, reference_answer):
    pred = extract_answer(model_output)

    if pred is None:
        return 0.0

    if pred == reference_answer:
        return 1.0

    return 0.0

    这个 reward function 同时检查了两个东西：

    • 模型有没有按照指定格式输出；
    • 模型最终答案是否正确。

    7.4 实验结果怎么理解？

    原文中，模型一开始在这个任务上的正确率大约是 25%，训练后逐渐提升到大约 70%。 这个实验的意义不是说“四位数乘法本身多重要”，而是说明：

    实验意义： 只要任务结果可以被自动验证，即使 reward 很简单，也可以通过 RL 让模型提高成功率。

    这对 reasoning 后训练很关键。 因为很多时候我们并不想人工写每道题的完整 CoT。 我们真正需要的是：模型最终能不能做对。 如果最终结果可验证，reward 就可以来自验证器。

    8. 实验二：Function Picking，函数选择

    第二个实验更接近 Agent。 Function Picking 的意思是：给模型一个用户请求和一组可用函数， 让模型判断是否应该调用函数，以及应该调用哪个函数。

    这比普通数学题更接近真实工具调用场景。

    8.1 原文中的任务例子

    用户请求大致是： 

    Hi, I need to know the distance between New York and Los Angeles.

    9. RLVR 对 Agent 后训练的启发

    RLVR 不只是数学推理模型的训练方法。 它对 Agent 后训练更重要，因为 Agent 的很多行为天然可以被验证。

    Agent 类型 可验证信号 Reward 示例
    Tool-use Agent 工具名、参数、调用时机。 工具正确 +0.5，参数缺失 -0.5。
    Code Agent 代码是否运行，测试是否通过。 测试通过 +1，运行报错 -1。
    SQL Agent SQL 是否能执行，结果是否正确。 结果匹配 +1，语法错误 -1。
    RAG Agent 回答是否引用证据，证据是否支持答案。 引用命中 +0.3，编造引用 -1。
    Web Agent 网页状态是否达到目标。 成功提交表单 +1，误点页面 -0.5。

    9.1 Agent 后训练不是只训练最终回答

    普通聊天模型主要优化最终回答。 但 Agent 需要优化的是整条轨迹： 

    User Request 用户请求
  ↓
Thought / Plan 思考或计划
  ↓
Action 工具调用 / 代码执行 / SQL 查询
  ↓
Observation 工具返回 / 环境反馈
  ↓
Next Action 下一步动作
  ↓
Final Answer 最终回答

    所以 Agent 的 reward 不应该只看最终文字，还应该看：

    • 有没有选对工具；
    • 参数有没有填对；
    • 工具调用格式是否合法；
    • 有没有根据工具返回结果继续推理；
    • 有没有在该停止的时候停止；
    • 有没有无意义地反复调用工具。

    9.2 一个 Agent Reward 模板

    def agent_reward(output, ground_truth, execution_result):
    reward = 0.0

    # 1. 格式是否可解析
    if is_valid_schema(output):
        reward += 0.2
    else:
        reward -= 0.5

    # 2. 工具名称是否正确
    if tool_name(output) == ground_truth.tool_name:
        reward += 0.5
    else:
        reward -= 0.5

    # 3. 参数是否完整
    if required_args_complete(output):
        reward += 0.3
    else:
        reward -= 0.3

    # 4. 参数值是否正确
    if args_match(output, ground_truth.args):
        reward += 0.5

    # 5. 工具执行是否成功
    if execution_result.success:
        reward += 1.0

    # 6. 是否过度调用工具
    if unnecessary_tool_call(output):
        reward -= 0.4

    return reward
    Agent 后训练核心： 普通后训练优化的是回答；Agent 后训练优化的是轨迹。 RLVR 的价值就在于，它可以把轨迹中的很多环节转化为可验证奖励。

    10. 实践路线：怎么做一个最小 RLVR / GRPO 小实验？

    如果你想真正动手，不要一开始就做复杂 Agent。 建议从最简单的可验证任务开始。

    10.1 第一步：选择可验证任务

    任务 难度 为什么适合入门
    数学答案抽取 低 答案容易比较，reward 容易写。
    JSON 格式校验 低 能否解析非常明确。
    Function Picking 中 接近工具调用 Agent，但还不需要真实执行工具。
    代码单元测试 中高 reward 清楚，但执行环境更复杂。
    多步 Agent 高 轨迹长、reward 稀疏、工程复杂。

    10.2 第二步：写 Reward Function

    先不要追求复杂，先保证 reward 稳定。 例如数学任务： 

    def reward(output, answer):
    pred = extract_answer(output)

    if pred is None:
        return -0.2

    if pred == answer:
        return 1.0

    return 0.0

    再比如 JSON 任务： 

    def json_reward(output):
    try:
        obj = json.loads(output)
    except:
        return -1.0

    if "action" not in obj:
        return -0.5

    if "arguments" not in obj:
        return -0.5

    return 1.0

    10.3 第三步：同一个 prompt 多采样

    GRPO 需要 group。 所以每个 prompt 不要只生成一个回答，而要生成多个回答： 

    prompt_1 → response_1, response_2, response_3, response_4
prompt_2 → response_1, response_2, response_3, response_4
prompt_3 → response_1, response_2, response_3, response_4

    然后分别计算 reward，再做组内比较。

    10.4 第四步：计算组内 advantage

    rewards = [1, 0, 0, 1]

mean_reward = sum(rewards) / len(rewards)

advantages = [
    r - mean_reward
    for r in rewards
]

    得到： 

    advantages = [0.5, -0.5, -0.5, 0.5]

    高 advantage 的回答被强化，低 advantage 的回答被削弱。

    实践建议： 你可以先做一个 toy 版本，不必马上跑大模型训练。 先用 Python 模拟： prompt → 多个 response → reward → group advantage。 把这个逻辑跑通，再去看 verl、OpenRLHF、TRL 等工程框架。

    11. 常见坑：RLVR 不是万能药

    11.1 Reward Function 太表面

    如果只奖励格式，模型可能只学会套格式。 

    错误 reward：
只要输出 JSON 就给 1 分。

模型可能学到：
输出格式正确但内容胡写的 JSON。
    解决办法： reward 不要只检查格式，还要检查内容、参数、执行结果。

    11.2 只看最终答案，忽略过程质量

    数学题只看最终答案很方便，但复杂任务中可能不够。 模型可能碰巧猜对答案，却没有形成可靠推理。

    对 Agent 来说，如果只看最终任务是否完成，模型可能学会：

    • 疯狂调用工具；
    • 反复尝试直到成功；
    • 忽略成本；
    • 使用不安全路径。
    解决办法： 对 Agent 任务，reward 最好同时包含任务成功、工具成本、步骤数量、安全约束。

    11.3 任务太简单或太难

    如果任务太简单，所有回答都得 1 分，组内没有差异。 如果任务太难，所有回答都得 0 分，也没有差异。

    任务难度 训练信号 问题
    太简单 reward 全是 1 模型没有进一步学习信号。
    太难 reward 全是 0 模型不知道什么行为更好。
    适中 有对有错 最适合 GRPO 组内比较。

    11.4 Reward Hacking

    模型会优化 reward，但不一定优化你的真实目标。 如果 reward 有漏洞，模型会利用漏洞。

    真实目标 错误 reward 模型可能学到什么
    写出正确代码 只要通过少量测试就给满分。 硬编码测试样例。
    准确调用工具 只奖励最终成功。 反复乱调用工具直到成功。
    简洁推理 回答越长分越高。 输出冗长无用推理。
    格式稳定 只奖励标签出现。 标签正确但内容错误。
    重要警告： RLVR 的难点不是“写一个 reward function”，而是写一个不会轻易被模型钻空子的 reward function。

    12. 本篇术语表

    英文术语 中文翻译 解释
    RLVR 基于可验证奖励的强化学习 用规则、测试器、执行结果等可验证信号作为 reward 来训练模型。
    Verifiable Reward 可验证奖励 可以由程序自动判断的奖励，比如答案正确、测试通过、格式合法。
    Verifiable Reward Function 可验证奖励函数 接收模型输出并返回 reward 的函数，通常由规则或验证器实现。

    PPO 近端策略优化 经典强化学习算法，RLHF 中常用，但通常需要 Value Model。
    Value Model 价值模型 估计当前状态或 token 的预期价值，用于计算 advantage。
    Advantage 优势值 表示某个回答比 baseline 好多少。
    Reward Model 奖励模型 用模型给回答打分，常见于 RLHF。
    Outcome Reward 结果奖励 只根据最终结果给奖励，例如最终答案是否正确。

    14. DeepSeek-R1：为什么它是 RLVR / GRPO 的代表案例？

    学完 Fireworks AI 这篇文章之后，下一步最自然要读的是 DeepSeek-R1 Technical Report。 因为 DeepSeek-R1 系列基本把前面这些关键词都串起来了： 

    Reasoning Model 推理模型
  ↓
Rule-based Reward 规则奖励
  ↓
GRPO 组相对策略优化
  ↓
RLVR 可验证奖励强化学习
  ↓
Cold-start Data 冷启动数据
  ↓
Multi-stage Training 多阶段训练
    一句话理解： Fireworks AI 这篇文章是在解释“为什么 RLVR 可行”；DeepSeek-R1 报告则是一个更完整的案例， 展示 reasoning model 如何通过规则奖励和强化学习获得推理能力。

    14.1 DeepSeek-R1-Zero：不经过 SFT 的纯 RL 尝试

    DeepSeek-R1-Zero 最值得关注的一点是：它没有先经过传统 SFT 冷启动，而是直接对 base model 做大规模强化学习。 这说明一个很重要的趋势： 在某些可验证 reasoning 任务中，模型可能不完全依赖人类写好的推理链，而是可以通过奖励反馈自己探索。

    模型 训练特点 意义
    DeepSeek-R1-Zero 直接使用大规模 RL，没有先做 SFT 冷启动。 验证纯 RL 是否能激发 reasoning 能力。
    DeepSeek-R1 加入 cold-start data 和多阶段训练。 改善 R1-Zero 的可读性、语言混杂和格式问题。
    Distilled Models 从 DeepSeek-R1 蒸馏到较小模型。 把强模型推理能力迁移到更小模型。

    14.2 为什么 R1-Zero 会出现“可读性差”和“语言混杂”？

    这和纯 RL 的特点有关。 如果模型一开始没有经过 SFT 学习稳定的回答格式，那么它在 RL 过程中会主要追求 reward。 只要答案能拿高分，它未必会主动保持清楚、整洁、统一语言的表达。

    所以 R1-Zero 虽然展示了推理能力的 emergence，但也暴露出一些问题：

    • 推理过程可读性不稳定；
    • 中英文或多语言混杂；
    • 回答格式不够统一；
    • 可能出现很长的自我反思过程；
    • 不一定符合普通用户的阅读习惯。
    关键理解： RL 能提高任务 reward，但不一定自动带来好格式、好表达、好可读性。 所以真实后训练通常不能只靠 RL，还要结合 SFT、格式数据、偏好优化和多阶段训练。

    14.3 DeepSeek-R1 为什么加入 Cold-start Data？

    Cold-start Data 可以理解为给模型一个比较好的起点。 在 reasoning 后训练里，它通常包含一些高质量的推理样本，让模型先学会：

    • 如何展开推理；
    • 如何保持语言一致；
    • 如何输出最终答案；
    • 如何控制格式；
    • 如何避免混乱的中间过程。

    也就是说，Cold-start Data 不是最终能力的全部来源，而是给 RL 一个更稳定的起点。

    没有 Cold-start Data 加入 Cold-start Data
    模型可能从混乱格式开始探索。 模型先学到基本推理格式。
    RL 过程更不稳定。 RL 更容易在合理行为附近优化。
    容易出现语言混杂、可读性差。 可读性和格式更好。
    更像纯粹探索实验。 更适合实际可用模型训练。
    一句话理解： R1-Zero 证明“纯 RL 可以激发推理”；R1 证明“纯 RL 之后还需要冷启动、多阶段和格式约束，才能变成更可用的模型”。

    14.4 Rule-based Reward：准确性奖励与格式奖励

    DeepSeek-R1 报告中最重要的一个点是 rule-based reward。 它主要包括两类：

    奖励类型 中文解释 作用
    Accuracy Reward 准确性奖励 检查最终答案是否正确，例如数学答案是否匹配、代码是否通过测试。
    Format Reward 格式奖励 检查模型是否按照要求格式输出，例如是否有特定标记、是否能抽取最终答案。

    这和 Fireworks AI 文章中的 Verifiable Reward 是同一条逻辑： 如果最终结果可以被验证，就不一定需要一个神经网络 Reward Model 来主观打分。 

    Accuracy Reward：
答案正确 → reward = 1
答案错误 → reward = 0

Format Reward：
格式符合要求 → reward += 0.2
格式不符合要求 → reward -= 0.2
    实践启发： 如果你要做一个自己的 reasoning 小实验，不要一开始设计复杂 Reward Model。 先从两类 rule reward 开始：
    1. 答案是否正确；
    2. 格式是否符合要求。
    这就是最小 RLVR 实验的基本骨架。

    15. verl：从 RLVR / GRPO 理论走向工程实现

    理解 RLVR 和 GRPO 之后，下一步要看工程框架。 其中一个很重要的框架是 verl，全称可以理解为 Volcano Engine Reinforcement Learning for LLMs。

    verl 的意义在于：它不是单纯讲理论，而是把大模型 RL 后训练中最麻烦的工程环节组织起来。 比如 rollout、reward、advantage、PPO / GRPO、分布式训练、推理引擎对接等。

    一句话理解： Fireworks AI 和 DeepSeek-R1 帮你理解“为什么要做 RLVR / GRPO”； verl 帮你理解“这套东西工程上怎么跑起来”。

    15.1 为什么 RL 后训练需要专门框架？

    SFT 很像普通微调，只要准备数据、算 loss、反向传播就可以。 但 Online RL / GRPO 要复杂得多。 

    SFT 训练循环：
读取固定数据
  ↓
模型 forward
  ↓
计算 next-token loss
  ↓
反向传播更新

RL / GRPO 训练循环：
当前模型生成回答
  ↓
reward function 打分
  ↓
计算 advantage
  ↓
加入 KL 约束
  ↓
更新模型
  ↓
再用新模型生成新回答
    工程环节 为什么麻烦
    Rollout 模型需要不断生成新回答，自回归生成速度慢。
    Reward 每个回答都要计算 reward，可能涉及代码执行、工具调用、格式检查。
    Advantage 需要把 reward 转成策略更新信号。
    KL 控制 防止模型偏离原始模型太远。
    分布式调度 训练模型、推理模型、reward 计算可能在不同设备上。
    推理引擎对接 常常要对接 vLLM、SGLang、FSDP、Megatron 等组件。

    15.2 verl 里你应该重点看什么？

    初学者不要一上来就改框架源码。 更好的方式是先看它的几个核心概念：

    模块 你要理解什么
    Rollout Worker 当前 policy 如何生成回答或轨迹。
    Reward Function reward 是如何计算的，是规则、模型还是执行环境。
    Advantage Estimator PPO / GRPO 如何把 reward 转成 advantage。
    Trainer 训练循环如何组织 rollout、reward、update。
    Reference Policy KL 约束如何防止模型漂移。
    Config batch size、rollout 数量、采样温度、KL 系数等参数如何设置。
    学习 verl 的正确顺序：
    1. 先看官方 README，知道它支持哪些算法；
    2. 再看 GRPO / PPO 的 example 配置；
    3. 然后看 reward function 怎么接入；
    4. 最后再看 trainer 和 rollout worker 的实现。
    不要一上来就从底层分布式代码看起。

    15.3 verl 和你前面学的概念如何对应？

    前面学的概念 在 verl 工程里对应什么
    Prompt 训练任务输入。
    Response / Rollout 模型生成的回答或轨迹。
    Reward Function 规则函数、奖励模型或环境反馈。
    Group Reward 同一 prompt 下多个 response 的 reward 列表。
    Advantage 策略更新的权重信号。
    Policy Model 正在训练的 LLM。
    Reference Model 用于 KL 约束的参考模型。
    Trainer 整体训练循环。
    一句话理解： verl 不是一个新的理论概念，而是把你学过的 RL 后训练概念工程化。 你要把代码里的 rollout、reward、advantage、trainer 和前面的理论一一对应起来。

    16. OpenRLHF / TRL：另外两个实践入口

    除了 verl，后训练实践里还经常见到两个入口： OpenRLHF 和 Hugging Face TRL。 它们的定位不完全一样，但都适合作为学习 RLHF / DPO / PPO / GRPO 的工程参考。

    16.1 OpenRLHF：偏大规模 RLHF / RL 训练框架

    OpenRLHF 的重点是易用、可扩展、高性能。 它支持 PPO、DPO、Rejection Sampling、GRPO、REINFORCE++ 等多种后训练方法。

    特点 解释

    工程组件丰富 可对接 Ray、vLLM、DeepSpeed 等。

    17. 最小 RLVR 实验：从数学答案抽取开始

    学 RLVR 最怕一直停留在概念。 最小实验不需要复杂框架，你可以先用 Python 模拟一个完整流程： 

    准备 prompt
  ↓
生成多个 response
  ↓
抽取最终答案
  ↓
计算 reward
  ↓
做 group reward normalization
  ↓
得到 advantage

    17.1 准备数据

    先准备几道简单数学题： 

    dataset = [
    {
        "prompt": "1497 * 63 = ?",
        "answer": 94311
    },
    {
        "prompt": "2356 * 42 = ?",
        "answer": 98952
    },
    {
        "prompt": "8765 * 18 = ?",
        "answer": 157770
    }
]

    17.2 规定输出格式

    为了方便抽取答案，需要规定模型输出格式： 

    Final Answer: the final answer is X. I hope it is correct.

    如果模型没有按这个格式输出，reward 可以直接给 0 或负分。

    17.3 写答案抽取器

    import re

def extract_answer(output):
    pattern = r"Final Answer: the final answer is (-?\d+). I hope it is correct."
    match = re.search(pattern, output)

    if match is None:
        return None

    return int(match.group(1))

    17.4 写 Reward Function

    def reward_fn(output, reference_answer):
    pred = extract_answer(output)

    if pred is None:
        return -0.2

    if pred == reference_answer:
        return 1.0

    return 0.0

    17.5 模拟同一 prompt 多采样

    responses = [
    "Final Answer: the final answer is 94311. I hope it is correct.",
    "Final Answer: the final answer is 94211. I hope it is correct.",
    "I think the answer is 94311.",
    "Final Answer: the final answer is 94301. I hope it is correct."
]

rewards = [
    reward_fn(r, 94311)
    for r in responses
]

print(rewards)

    可能得到： 

    [1.0, 0.0, -0.2, 0.0]

    17.6 计算组内 advantage

    def compute_group_advantages(rewards):
    mean_reward = sum(rewards) / len(rewards)
    return [
        r - mean_reward
        for r in rewards
    ]

advantages = compute_group_advantages(rewards)

print(advantages)

    这一步就是 GRPO 直觉的最小版本。 你先不用训练模型，只要理解：

    • reward 高于组内平均值 → 正 advantage；
    • reward 低于组内平均值 → 负 advantage；
    • 模型会被训练成更倾向于生成正 advantage 的回答。
    最小实验目标： 先不要急着跑大模型训练。 你先把 prompt、response、reward、advantage 这个链条用 Python 跑通。 这一步跑通后，再看 GRPO Trainer 或 verl 代码就不会迷路。

    18. Agent RLVR 数据设计：从 Function Picking 到 Tool-use

    Function Picking 是 Agent RLVR 的入门任务。 它不要求模型真的执行工具，只要求模型判断：

    • 需不需要调用工具；
    • 应该调用哪个工具；
    • 参数是否足够；
    • 如果不能调用，是否应该拒绝或追问。

    18.1 一个 Function Picking 样本

    {
  "user": "请帮我查一下明天柏林的天气。",
  "tools": [
    {
      "name": "get_weather",
      "arguments": {
        "location": "string",
        "date": "string"
      }
    }
  ],
  "expected": "<functioncall> get_weather"
}

    这个样本中，用户给出了地点“柏林”和时间“明天”，所以可以调用天气工具。

    18.2 一个不应该调用工具的样本

    {
  "user": "请计算纽约到洛杉矶的距离。",
  "tools": [
    {
      "name": "calculate_distance",
      "arguments": {
        "start_coordinates": "string",
        "end_coordinates": "string"
      }
    }
  ],
  "expected": "<no_functioncall>"
}

    虽然用户问的是距离，但工具需要坐标。 用户只给了城市名，没有提供经纬度。 所以模型不应该直接调用工具。

    关键点： 工具名看起来相关，不代表可以调用。 Agent 必须判断参数是否足够。 这正是 Function Picking 任务的价值。

    18.3 Function Picking 的 reward

    def function_picking_reward(output, expected_label):
    pred = extract_label(output)

    if pred == expected_label:
        return 1.0

    return 0.0

    更细一点可以这样写： 

    def function_picking_reward(output, expected):
    reward = 0.0

    if format_valid(output):
        reward += 0.2
    else:
        reward -= 0.5

    if label_correct(output, expected):
        reward += 1.0

    if calls_tool_when_should_not(output, expected):
        reward -= 0.5

    if refuses_when_should_call(output, expected):
        reward -= 0.5

    return reward

    18.4 从 Function Picking 扩展到 Tool-use

    Function Picking 只判断是否调用工具。 真正的 Tool-use 还要判断参数和执行结果。

    阶段 模型要学什么 reward 怎么给
    Function Picking 是否调用工具、调用哪个工具。 标签正确给分。
    Argument Filling 参数字段和值是否正确。 字段完整、值正确给分。
    Tool Execution 工具调用是否能执行成功。 执行成功给分，报错扣分。
    Observation Use 是否根据工具结果回答。 引用工具结果给分，忽略结果扣分。
    Multi-step Agent 多步调用是否合理。 任务完成、步骤少、安全合规给分。
    Agent 实践路线： 不要一开始做完整多步 Agent。 建议按这个顺序：
    1. Function Picking：判断是否调用工具；
    2. Argument Filling：填对参数；
    3. Tool Execution：让工具能跑通；
    4. Observation Use：根据工具返回回答；
    5. Multi-step Agent：多步任务轨迹优化。

    21. DeepSeek-R1-Zero：纯 RL 路线到底说明了什么？

    DeepSeek-R1-Zero 是报告中最值得关注的部分。 它的关键特征是：没有先经过传统 SFT 冷启动，而是直接在 base model 上进行大规模强化学习。

    这和传统后训练路线不太一样。 传统路线通常是： 

    Base Model
  ↓
SFT
  ↓
DPO / RLHF
  ↓
Online RL

    而 R1-Zero 的尝试更接近： 

    Base Model
  ↓
Rule-based Reward
  ↓
GRPO / RL
  ↓
Reasoning Ability Emergence
    一句话理解： R1-Zero 想验证的是：如果任务奖励足够清楚，模型能不能不先模仿人类推理样本，而是直接通过 RL 学出推理能力。

    21.1 为什么这件事重要？

    因为过去很多人默认：模型要会推理，必须先看大量人类写好的 CoT 数据。 但 R1-Zero 提供了另一种思路： 

    不一定先给模型完整推理过程
  ↓
只要最终答案可以验证
  ↓
模型就可以通过生成、验证、奖励、更新
  ↓
逐渐提高推理成功率

    这意味着，reasoning 后训练不只是“模仿人类推理链”，也可以是“通过奖励反馈搜索更好的推理策略”。

    21.2 R1-Zero 可能学到了什么？

    从训练现象看，R1-Zero 可能逐渐学到几类行为：

    行为 解释
    更长的思考 模型为了提高正确率，会生成更长的推理过程。
    自我检查 模型会重新检查前面的推理是否有错。
    反思与修正 模型可能意识到前一步错误，然后换一种方法。
    答案格式意识 为了拿到 format reward，模型会学会输出可抽取的最终答案。
    探索多种路径 模型可能尝试不同解法，只要最后答案正确即可。
    实践启发： 在 reasoning 任务中，不一定要人工标注每一步。 如果最终答案可靠可验证，模型可能通过 outcome reward 学到部分中间策略。

    21.3 纯 RL 的问题

    但是 R1-Zero 也说明：纯 RL 不是万能的。 如果只靠 reward，模型可能会优先追求“拿分”，而不是追求“用户容易阅读”。

    问题 原因 后果
    可读性差 reward 主要奖励正确性，不一定奖励表达清楚。 推理过程可能很乱。
    语言混杂 模型没有被强约束使用统一语言。 中英文或多语言混在一起。
    格式不稳定 格式 reward 不足以覆盖所有表达细节。 最终答案抽取可能不稳定。
    过长推理 模型发现长推理有时更容易拿高分。 输出成本上升，用户体验下降。
    关键警告： RL 会优化 reward，但 reward 没有写进去的东西，模型不一定会自动学好。 如果 reward 只奖励正确率，它不一定会自动学会清晰表达。

    21.4 这对我们做小实验有什么启发？

    如果你要做自己的 R1-Zero-like 小实验，不能只看最终答案 reward。 至少要同时设计： 

    Accuracy Reward：答案是否正确
Format Reward：格式是否符合要求
Length Penalty：是否过度冗长
Language Constraint：是否保持目标语言
Safety / Validity：是否出现非法输出

    也就是说，一个看似简单的 reward function，背后其实决定了模型会往哪里发展。

    22. DeepSeek-R1：为什么需要多阶段训练？

    DeepSeek-R1 相比 R1-Zero，更接近实际可用的 reasoning model。 它不是简单地“纯 RL 到底”，而是引入了 cold-start data 和多阶段训练。

    这说明一个重要事实：

    一句话理解： R1-Zero 证明了纯 RL 可以激发推理；DeepSeek-R1 则说明，要做成真正可用的模型，还需要 SFT、RL、数据筛选、通用能力对齐等多阶段组合。

    22.1 为什么要 Cold-start？

    Cold-start data 可以理解为“给模型一个比较好的初始推理状态”。 它不是为了替代 RL，而是为了让 RL 不要从完全混乱的状态开始。

    没有 Cold-start 加入 Cold-start
    模型可能不知道怎么组织推理。 模型先学到基本推理格式。
    语言风格不稳定。 语言更统一，可读性更好。
    最终答案格式可能混乱。 最终答案更容易被抽取。
    RL 初期探索成本高。 RL 从更合理的行为区域开始。

    这和我们前面学的 SFT 并不矛盾。 SFT 的作用是让模型先学会基础行为；RL 的作用是进一步用 reward 提高能力上限。

    22.2 多阶段训练大致解决哪些问题？

    阶段 主要目标 解决什么问题
    Cold-start SFT 建立较好的推理格式和表达方式。 可读性差、格式混乱。
    Reasoning RL 提升数学、代码、逻辑推理能力。 正确率和复杂推理能力。
    Rejection Sampling 筛选高质量推理样本。 构造更好的后续训练数据。
    General SFT 恢复或增强通用能力。 避免模型只会做推理题。
    Final RL / Alignment 进一步对齐帮助性和安全性。 提升实际助手体验。
    学习重点： DeepSeek-R1 的价值不只是“用了 GRPO”，而是展示了一个多阶段 recipe： 先给格式，再用 RL 提高推理，再用数据筛选和通用训练修复可用性。

    22.3 为什么不能只做 reasoning RL？

    如果只做 reasoning RL，模型可能在数学和代码题上变强，但在普通对话、写作、安全回答、信息组织上退化。 这就是所谓能力分布偏移。

    只做 reasoning RL 可能带来的问题 解释
    通用对话能力下降 训练过度集中在数学和代码题。
    回答风格变奇怪 模型习惯输出很长推理过程。
    安全性不足 reward 主要关注正确率，未必覆盖安全边界。
    用户体验下降 所有问题都进入长思考模式，效率低。
    重要理解： Reasoning 后训练不是让模型只会“长推理”。 真正好的 reasoning model 应该知道什么时候需要深度推理，什么时候应该直接回答。

    23. 蒸馏：为什么小模型也能学 reasoning？

    DeepSeek-R1 报告还有一个重要部分：蒸馏。 它把 DeepSeek-R1 的推理能力蒸馏到一些较小的 dense models 上，例如基于 Qwen 和 Llama 的模型。

    蒸馏的基本思想是： 

    大 reasoning model 作为 teacher
  ↓
生成高质量 reasoning traces
  ↓
小模型作为 student
  ↓
用这些数据做 SFT
  ↓
小模型学习大模型的推理模式
    一句话理解： RL 训练大模型很贵；但一旦有了强 reasoning model，就可以用它生成数据，再通过 SFT 把能力迁移给小模型。

    23.1 蒸馏和 RL 的区别

    比较项 RL 训练 蒸馏训练
    训练信号 reward。 teacher model 生成的高质量回答。
    训练方式 强化学习，成本高。 SFT，成本相对低。
    是否需要 rollout 需要。 通常不需要在线 rollout。
    目标 让模型通过奖励探索能力。 让小模型模仿强模型能力。
    适合对象 大模型或需要进一步突破的模型。 中小模型、部署模型。

    23.2 蒸馏数据长什么样？

    一个 reasoning distillation 样本大致是： 

    Prompt:
Solve the problem: ...

Teacher Response:
We need to reason step by step.
...
Therefore, the final answer is X.

Final Answer:
X

    小模型训练时不是直接得到 reward，而是学习 teacher response 的 token。 因此它本质上仍然是 SFT。

    实践启发： 如果你暂时跑不起 RL，可以先做 distillation。 比如用强 reasoning model 生成高质量数学 / 代码 / 工具调用样本， 再用这些样本 SFT 一个小模型。 这比直接从零做 RL 更适合入门。

    23.3 蒸馏适合哪些任务？

    任务 是否适合蒸馏 原因
    数学推理 适合 强模型能生成清楚的解题过程。
    代码生成 适合 强模型可生成代码和解释，测试器可筛选正确样本。
    工具调用 适合 强模型可生成规范 tool-use 轨迹。
    复杂主观写作 部分适合 可以模仿风格，但质量评价仍较主观。
    网页操作 Agent 较难 环境交互复杂，轨迹质量难自动保证。

    23.4 蒸馏的风险

    蒸馏不是万能的。 如果 teacher 的输出有错误，小模型会照着学。 如果 teacher 的推理很啰嗦，小模型也可能变啰嗦。

    风险 表现 解决办法
    错误蒸馏 teacher 答错，小模型也学错。 用验证器筛掉错误样本。
    风格污染 teacher 过度冗长，小模型也冗长。 筛选更简洁的 reasoning traces。
    能力上限受限 小模型只能模仿已有数据。 后续再结合 DPO 或 RL。
    数据分布偏 只会某类题，不会泛化。 增加任务类型和难度层次。
    注意： 蒸馏不是“把大模型能力完整复制给小模型”。 它更像是把大模型的一部分行为模式迁移给小模型，效果仍受模型容量和数据质量限制。

    24. 玩具版 R1-Zero：如何复现一个最小 reasoning RL 流程？

    真正复现 DeepSeek-R1 不现实，因为它需要巨大算力、数据和工程系统。 但我们可以复现一个玩具版 R1-Zero-like 流程，用来理解它的核心思想。

    目标不是复现性能，而是复现机制： Base model 生成多个答案 → rule reward 打分 → GRPO 组内比较 → 更新模型。

    24.1 任务选择

    最小任务建议选择数学题或格式任务。 不要一开始做复杂代码 Agent。

    任务 推荐程度 原因
    两位数加减乘除 高 答案容易验证，任务简单。
    四位数乘两位数 高 和 Fireworks AI 实验接近。
    JSON 格式输出 高 格式是否正确容易判断。
    Function Picking 中 更接近 Agent，但数据设计稍复杂。
    代码生成 中高 reward 清楚，但执行环境更复杂。

  "answer": "94311"

    24.3 Reward 设计

    def reward(output, answer):
    reward = 0.0

    # 格式奖励
    if "Final Answer:" in output:
        reward += 0.2
    else:
        reward -= 0.2

    # 答案抽取
    pred = extract_final_answer(output)

    # 准确性奖励
    if pred == answer:
        reward += 1.0

    # 长度惩罚
    if len(output) > 1000:
        reward -= 0.1

    return reward

    24.4 GRPO 训练直觉

    for prompt in batch:
    responses = model.generate(prompt, num_return_sequences=8)

    rewards = [
        reward(response, answer)
        for response in responses
    ]

    advantages = normalize_within_group(rewards)

    update_model_with_grpo(prompt, responses, advantages)

    这里最重要的是： 每个 prompt 生成多个 response。 如果只有一个 response，就没有 group comparison。

    24.5 观察什么指标？

    指标 说明
    Accuracy 最终答案正确率是否提升。
    Format Pass Rate 最终答案格式是否越来越稳定。
    Average Length 推理是否越来越冗长。
    Reward Mean 平均 reward 是否上升。
    Reward Variance 组内是否还有区分度。
    Error Type 错误是算错、格式错，还是抽取失败。
    实践建议： 玩具实验中不要只画 reward 曲线。 还要保存训练前后模型的真实输出，观察模型行为是否变化： 有没有更会检查？ 有没有更稳定输出 Final Answer？ 有没有变得过度啰嗦？
