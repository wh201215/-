强化学习大模型路线｜Stage 6：Agent RL / Tool-use RL / 多轮环境强化学习

学习定位：Stage 5 讲了 GRPO / RLVR，重点是单轮可验证任务：模型生成一个回答，验证器给 reward。Stage 6 进入 Agent RL：模型不只生成答案，还要在环境中多轮行动、调用工具、接收观察、继续决策。

这一阶段的核心变化是：

\[ \text{single response} \quad\Longrightarrow\quad \text{multi-turn action trajectory} \]

也就是从：

Prompt
  ↓
Response
  ↓
Reward

变成：

User Task
  ↓
Plan / Thought
  ↓
Tool Call
  ↓
Observation
  ↓
Next Action
  ↓
More Observations
  ↓
Final Answer
  ↓
Trajectory Reward

Stage 6 目标：

  1. 理解 LLM Agent 如何写成 MDP / POMDP；
  2. 理解 Agent 的状态、动作、观察、奖励分别是什么；
  3. 掌握多轮工具调用轨迹的概率分解；
  4. 理解 trajectory-level reward 和 step-level reward；
  5. 理解为什么 Agent RL 比单轮 RLVR 更难；
  6. 学会设计 tool-use、search、code、SQL Agent 的 reward；
  7. 能设计一个最小可复现的 Tool-use Agent RL toy project。

1. 为什么 GRPO / RLVR 之后要学 Agent RL？

Stage 5 的 RLVR 适合这种形式：

\[ x \rightarrow y \rightarrow R(x,y) \]

也就是 prompt \(x\)，模型输出回答 \(y\)，验证器给奖励 \(R(x,y)\)。

但很多真实任务不是一次回答能完成的，而是要多步交互。

任务 是否单轮足够 为什么需要 Agent
简单数学题 通常够 直接输出答案即可
复杂代码调试 不够 需要运行代码、看报错、修改、再运行
搜索问答 不够 需要检索、阅读、整合证据
SQL 问答 不够 需要查 schema、写 SQL、执行、修正
网页任务 不够 需要点击、输入、观察页面变化
工具调用任务 不够 需要判断何时调用工具、如何用结果继续推理
一句话： RLVR 解决“回答是否正确”；Agent RL 解决“行动过程是否能完成任务”。

2. 什么是 LLM Agent

LLM Agent 可以理解为：

以大语言模型为策略核心，能够根据任务状态选择动作、调用工具、读取观察结果，并持续更新行动的系统。

2.1 普通 LLM 和 Agent 的区别

普通 LLM LLM Agent
输入 prompt，输出文本 输入任务，输出动作序列和最终答案
主要依靠模型内部知识 可以调用外部工具和环境
一般单轮生成 可以多轮交互
错误后难以自我修正 可以根据 observation 修正下一步动作

2.2 Agent 的典型循环

while not done:
    1. 读取当前上下文和历史 observation
    2. 模型生成下一步 action
    3. 环境执行 action
    4. 返回 observation
    5. 模型根据 observation 决定下一步

数学上可以写成：

\[ a_t \sim \pi_\theta(\cdot \mid h_t) \]
\[ o_{t+1} \sim \mathcal{E}(\cdot \mid h_t,a_t) \]

其中 \(h_t\) 是到第 \(t\) 步为止的历史，\(\mathcal{E}\) 是环境或工具系统。

关键变化： Agent 的状态通常不是单个 \(s_t\)，而是一整段历史 \(h_t\)：任务、过去动作、工具返回、当前中间结果。

3. 把 Agent 写成 MDP / POMDP

在 Stage 1 里，MDP 是：

\[ \mathcal{M}=(\mathcal{S},\mathcal{A},p,r,\gamma) \]

Agent 也可以写成类似结构，但实际更接近 POMDP，因为模型通常看不到完整真实环境状态，只看到 observation。

3.1 MDP 视角

MDP 元素 Agent 中对应什么
状态 \(s_t\) 任务上下文、工具历史、环境状态
动作 \(a_t\) 文本回复、工具调用、搜索查询、代码执行、停止
转移 \(p(s_{t+1}\mid s_t,a_t)\) 执行动作后环境变化
奖励 \(r_t\) 工具调用是否正确、任务是否完成、成本是否低
折扣 \(\gamma\) 控制远期成功和短期成本权衡

3.2 POMDP 视角

在很多 Agent 任务中，真实状态 \(s_t\) 不完全可见。

例如网页 Agent：

  • 真实状态包括完整网页 DOM、登录状态、后端数据库状态；
  • 模型看到的只是当前页面截图、文本摘要或部分 DOM；
  • 所以模型根据 observation 决策，而不是完整状态。

POMDP 可以写成：

\[ o_t \sim O(\cdot \mid s_t) \]

模型的策略通常依赖历史：

\[ a_t \sim \pi_\theta(a_t\mid h_t) \]

其中：

\[ h_t=(x,a_0,o_1,a_1,o_2,\ldots,a_{t-1},o_t) \]
一句话： Agent 的“状态”常常就是历史上下文。模型通过历史近似理解真实环境状态。

4. 多轮轨迹 Trajectory

单轮 RLVR 的轨迹很短：

\[ \tau=(x,y,R) \]

Agent 的轨迹更长：

\[ \tau= (x,a_0,o_1,a_1,o_2,\ldots,a_{T-1},o_T,y_{\mathrm{final}},R) \]

其中：

符号 含义
\(x\) 用户任务
\(a_t\) 第 \(t\) 步动作
\(o_{t+1}\) 执行动作后得到的观察
\(y_{\mathrm{final}}\) 最终回答
\(R\) 整条轨迹的最终奖励

4.1 轨迹概率分解

给定任务 \(x\)，Agent 轨迹概率可以写成：

\[ p_\theta(\tau\mid x) = \prod_{t=0}^{T-1} \pi_\theta(a_t\mid h_t) \, \mathcal{E}(o_{t+1}\mid h_t,a_t) \]

这里：

部分 含义
\(\pi_\theta(a_t\mid h_t)\) 模型在历史 \(h_t\) 下选择动作 \(a_t\) 的概率
\(\mathcal{E}(o_{t+1}\mid h_t,a_t)\) 环境执行动作后返回 observation 的概率

对 \(\theta\) 求导时，环境项通常不含模型参数，所以：

\[ \nabla_\theta \log p_\theta(\tau\mid x) = \sum_{t=0}^{T-1} \nabla_\theta \log \pi_\theta(a_t\mid h_t) \]
这和 Stage 2 的 Policy Gradient 完全对应： 只不过这里的 action 不一定是 token，也可能是一整个工具调用。

5. Agent 的动作空间

Agent 的动作比普通 LLM 更丰富。

5.1 常见动作类型

动作类型 例子 是否可验证
直接回答 Final Answer: ... 可以用答案验证器验证
搜索 search(query="...") 可以检查 query 质量和证据命中
计算器 calculator(expression="1497*63") 可以检查表达式和结果
Python 执行 python(code="...") 可以看运行结果和测试
SQL 查询 sql(query="SELECT ...") 可以检查执行结果
API 调用 get_weather(city="北京") 可以检查工具名和参数
停止动作 finish(answer="...") 可以检查是否该停止

5.2 动作可以是结构化的

工具调用通常不是普通文本，而是结构化动作： 

{
  "action_type": "tool_call",
  "tool_name": "calculator",
  "arguments": {
    "expression": "1497 * 63"
  }
}

数学上可以把动作分解为：

\[ a_t=(c_t,u_t,\eta_t) \]
符号 含义
\(c_t\) 动作类别，例如 answer / tool_call / search / finish
\(u_t\) 具体工具名或动作名
\(\eta_t\) 动作参数，例如 JSON arguments

5.3 动作概率怎么写

如果动作是结构化的，可以分解为：

\[ \pi_\theta(a_t\mid h_t) = \pi_\theta(c_t\mid h_t) \, \pi_\theta(u_t\mid h_t,c_t) \, \pi_\theta(\eta_t\mid h_t,c_t,u_t) \]

在实际 LLM 中，这些结构通常还是通过 token 序列生成出来。

实践难点： 结构化动作看起来清楚，但模型输出时可能 JSON 格式错误、参数缺失、工具名拼错，所以 format reward 很重要。

6. Observation：工具返回如何进入状态

Agent 调用工具后，环境会返回 observation。 

Action:
calculator("1497 * 63")

Observation:
94311

下一步模型的输入就变成： 

User task:
请计算 1497 × 63。

Action history:
calculator("1497 * 63")

Observation:
94311

Next:
请给出最终回答。

6.1 历史状态更新

如果当前历史是 \(h_t\)，执行动作 \(a_t\)，得到 observation \(o_{t+1}\)，则新历史是：

\[ h_{t+1} = (h_t,a_t,o_{t+1}) \]

下一步策略是：

\[ a_{t+1}\sim\pi_\theta(\cdot\mid h_{t+1}) \]

6.2 Observation 的质量很关键

Observation 类型 例子 风险
数值结果 94311 模型可能忽略或抄错
搜索结果 网页片段 证据可能噪声大
代码报错 Traceback 模型需要理解报错并修正
SQL 执行结果 表格行 模型需要整合结果
网页状态 按钮、表单、页面文本 可能只看到部分页面
Agent 能力的一半来自 action，一半来自能不能正确利用 observation。

7. Agent RL 目标函数

Agent 的目标仍然是最大化期望回报：

\[ J(\theta) = \mathbb{E}_{\tau\sim p_\theta(\tau\mid x)} [ R(\tau) ] \]

如果有 prompt 分布：

\[ x\sim \mathcal{D} \]

则：

\[ J(\theta) = \mathbb{E}_{x\sim\mathcal{D}} \mathbb{E}_{\tau\sim p_\theta(\tau\mid x)} [ R(x,\tau) ] \]

7.1 Policy Gradient 形式

根据 Stage 2：

\[ \nabla_\theta J(\theta) = \mathbb{E} \left[ \sum_{t=0}^{T-1} \nabla_\theta \log \pi_\theta(a_t\mid h_t) \, \hat{A}_t \right] \]

这里和普通 RL 的区别是：

普通 RL Agent RL
\(s_t\) \(h_t\)，包含历史动作和 observation
\(a_t\) 工具调用、搜索、代码执行、回答、停止
\(\hat{A}_t\) 该步动作对最终任务成功的贡献估计

7.2 轨迹级奖励

很多 Agent 任务只有最终奖励：

\[ R(\tau)= \begin{cases} 1, & \text{任务完成}\\ 0, & \text{任务失败} \end{cases} \]

这会导致长轨迹 credit assignment 很难。

Agent RL 的难点： 最终成功或失败可能由很早之前某一步工具调用决定，但奖励只在最后出现。

8. 多轮 Agent Reward 怎么设计

Agent reward 可以分成四类：

Reward 类型 检查什么 例子
Outcome Reward 最终任务是否完成 答案正确、网页任务成功
Step Reward 每一步动作是否合理 工具名正确、参数完整
Cost Reward 是否节省成本 少调用工具、少 token、少搜索
Safety Reward 是否遵守约束 不泄露信息、不执行危险代码

8.1 总 reward 形式

\[ R(\tau) = R_{\mathrm{outcome}}(\tau) + \alpha R_{\mathrm{step}}(\tau) - \lambda C(\tau) - \mu R_{\mathrm{risk}}(\tau) \]
项 含义
\(R_{\mathrm{outcome}}\) 最终任务成功奖励
\(R_{\mathrm{step}}\) 中间步骤质量奖励
\(C(\tau)\) 成本，例如工具次数、token 数、时间
\(R_{\mathrm{risk}}\) 风险或违规惩罚
\(\alpha,\lambda,\mu\) 不同奖励项的权重

8.2 成本惩罚

如果没有成本惩罚，Agent 可能疯狂调用工具。

\[ C(\tau) = c_{\mathrm{tool}}N_{\mathrm{tool}} + c_{\mathrm{token}}N_{\mathrm{token}} + c_{\mathrm{time}}T \]

例如： 

reward = task_success
reward -= 0.02 * num_tool_calls
reward -= 0.001 * num_generated_tokens
关键： Agent reward 不能只看成功，还要看成本和行为质量。

9. 长轨迹 Credit Assignment

假设一个 Agent 任务轨迹如下： 

a0: search("DeepSeek R1 date")
o1: 搜索结果错误
a1: read(result 1)
o2: 信息不相关
a2: final answer
R: 0

最终失败了，但到底是哪一步错了？

  • 是搜索 query 不好？
  • 是读错了结果？
  • 是过早停止？
  • 是最终回答整合错误？

9.1 只用最终奖励的问题

\[ R_T=0 \]

如果把这个 0 分平均归因给所有动作，模型很难知道具体改哪里。

9.2 Step-level reward

可以给每一步单独奖励：

\[ R(\tau)= \sum_{t=0}^{T} \gamma^t r_t \]

例如：

步骤 奖励
搜索 query 命中关键实体 +0.2
读取相关文档 +0.2
引用证据支持答案 +0.3
最终答案正确 +1.0
编造证据 -1.0

9.3 Advantage 的时间分配

Agent 中可以用 return-to-go：

\[ G_t= \sum_{k=t}^{T} \gamma^{k-t}r_k \]

然后：

\[ \hat{A}_t=G_t-V(h_t) \]

或者像 GRPO 那样，用同任务多轨迹的相对 reward：

\[ \hat{A}_i = \frac{ R(\tau_i)-\bar{R} }{ \sigma_R+\epsilon } \]
长轨迹的核心问题： 最终 reward 给的是整条轨迹的评价，但训练要更新每一步 action 的概率。

10. Tool-use Reward 设计

工具调用任务是 Agent RL 最好的入门场景。

10.1 工具调用动作

{
  "tool": "calculator",
  "arguments": {
    "expression": "1497 * 63"
  }
}

10.2 Reward 拆解

检查项 奖励 说明
JSON 可解析 +0.1 格式基础
工具名正确 +0.3 选择了正确工具
参数完整 +0.2 没有漏填必要参数
参数值正确 +0.3 参数语义正确
工具执行成功 +0.3 调用能跑通
不该调用时调用 -0.5 惩罚乱调用
工具返回后回答正确 +0.5 利用 observation 完成任务

10.3 Reward Function 示例

def tool_use_reward(action, observation, final_answer, gold):
    reward = 0.0

    if not is_valid_json(action):
        return -1.0

    reward += 0.1

    if action["tool"] == gold["tool"]:
        reward += 0.3
    else:
        reward -= 0.3

    if required_args_complete(action, gold):
        reward += 0.2

    if args_match(action, gold):
        reward += 0.3

    if observation_success(observation):
        reward += 0.3

    if final_answer_correct(final_answer, gold["answer"]):
        reward += 0.5

    return reward
注意： 不要只奖励工具名正确。否则模型可能学会“工具名对，但参数胡写”。

11. Search Agent RL

Search Agent 的动作通常是： 

search(query="...")

环境返回搜索结果： 

Observation:
1. title ...
2. snippet ...
3. source ...

11.1 Search Agent 的 reward

奖励项 说明
Query relevance 搜索 query 是否包含关键实体和问题意图
Evidence hit 搜索结果是否包含答案证据
Source quality 是否使用高质量来源
Answer correctness 最终回答是否正确
Citation support 引用是否支持回答
Search cost 搜索次数是否过多

11.2 Search trajectory

\[ \tau=(x,q_1,o_1,q_2,o_2,y_{\mathrm{final}}) \]

其中 \(q_i\) 是第 \(i\) 次搜索 query。

11.3 Search reward 示例

reward = 0.0

if query_contains_key_entities(query):
    reward += 0.2

if retrieved_evidence_contains_answer(observation):
    reward += 0.4

if final_answer_correct(answer):
    reward += 1.0

if cites_unsupported_source(answer):
    reward -= 0.5

reward -= 0.05 * num_searches
Search Agent 的难点： 搜索 query 本身就是动作。模型不仅要会回答，还要学会问出好 query。

12. Code Agent RL

Code Agent 不只是生成代码，还会运行代码、看错误、修改代码。 

User task
  ↓
write code
  ↓
run tests
  ↓
observe error
  ↓
edit code
  ↓
run tests again
  ↓
final answer

12.1 Code Agent 状态

\[ h_t=(x,\text{current code},\text{test results},\text{error messages},\text{edit history}) \]

12.2 Code Agent 动作

动作 例子
write_code 生成初始代码
run_tests 执行单元测试
edit_code 根据错误修改代码
explain 解释解决方案
finish 提交最终代码

12.3 Code Agent reward

\[ R(\tau) = \frac{\#\text{passed tests}}{\#\text{total tests}} - \lambda N_{\mathrm{runs}} - \mu N_{\mathrm{tokens}} \]

如果代码无法运行，可以给负奖励：

\[ R_{\mathrm{compile}}= \begin{cases} 0, & \text{can compile}\\ -1, & \text{compile error} \end{cases} \]
Code Agent 很适合 RL： 因为环境反馈非常明确：代码能不能运行、测试能不能通过。

13. SQL / Table Agent RL

SQL Agent 的任务是根据自然语言问题查询数据库。 

Question:
哪个部门的平均工资最高？

Agent:
1. inspect schema
2. write SQL
3. execute SQL
4. read result
5. answer

13.1 SQL Agent 动作

动作 例子
inspect_schema 查看表结构
write_sql 生成 SQL 查询
execute_sql 执行 SQL
revise_sql 根据报错修改 SQL
finish 输出最终答案

13.2 SQL reward

检查项 奖励
SQL 语法正确 +0.2
执行成功 +0.3
结果匹配标准答案 +1.0
查询无关表 -0.2
执行次数过多 -0.05 每次

13.3 SQL reward 公式

\[ R(\tau) = \mathbb{I}[\text{result correct}] + 0.3\mathbb{I}[\text{execution success}] + 0.2\mathbb{I}[\text{syntax valid}] - 0.05N_{\mathrm{exec}} \]
SQL Agent 的核心： 最终 SQL 写法不唯一，但执行结果可以验证，所以天然适合 RLVR / Agent RL。

14. ReAct 轨迹与 RL 训练

ReAct 是 Reasoning + Acting 的组合格式。 

Thought: 我需要先查找相关信息。
Action: search("...")
Observation: ...
Thought: 现在我知道了关键事实。
Action: calculator("...")
Observation: ...
Final Answer: ...

14.1 ReAct 中哪些部分是动作？

部分 是否动作 说明
Thought 可以看作隐式动作或中间文本 影响后续 action，但通常不直接执行
Action 是 环境会执行
Observation 不是模型动作 环境返回
Final Answer 是停止 / 回答动作 结束任务

14.2 ReAct 轨迹概率

如果把 Thought 和 Action 都看作模型输出，则：

\[ \log\pi_\theta(\tau\mid x) = \sum_{t} \log\pi_\theta(\text{Thought}_t\mid h_t) + \sum_t \log\pi_\theta(\text{Action}_t\mid h_t,\text{Thought}_t) \]

如果只训练工具 Action，也可以只对 Action tokens 计算 RL loss。

实践选择： 是否训练 Thought tokens 是一个重要设计。训练全部 tokens 简单，但可能强化冗长推理；只训练 Action tokens 更聚焦，但实现更复杂。

15. Agent 中如何用 GRPO

Agent 也可以用 GRPO 思路：对同一个任务采样多条轨迹。

\[ \tau_1,\tau_2,\ldots,\tau_G \sim \pi_{\theta_{\mathrm{old}}}(\cdot\mid x) \]

每条轨迹得到 reward：

\[ R_i=R(x,\tau_i) \]

组内 advantage：

\[ \hat{A}_i = \frac{ R_i-\bar{R} }{ \sigma_R+\epsilon } \]

然后对轨迹中每个模型动作 token 或 action 计算 policy ratio。

15.1 Trajectory-level GRPO

\[ L_{\mathrm{Agent-GRPO}} = \mathbb{E} \left[ \frac{1}{G} \sum_{i=1}^{G} \sum_{t=0}^{T_i-1} \min \left( r_{i,t}(\theta)\hat{A}_i, \mathrm{clip}(r_{i,t}(\theta),1-\epsilon,1+\epsilon)\hat{A}_i \right) \right] \]

其中：

\[ r_{i,t}(\theta) = \frac{ \pi_\theta(a_{i,t}\mid h_{i,t}) }{ \pi_{\theta_{\mathrm{old}}}(a_{i,t}\mid h_{i,t}) } \]

15.2 和单轮 GRPO 的区别

单轮 GRPO Agent-GRPO
一个 response 一条多步轨迹
reward 看最终回答 reward 看任务完成、工具调用、成本
token 序列较短 多轮 action + observation
credit assignment 相对简单 长轨迹归因更难
一句话： Agent-GRPO 把“同题多回答比较”扩展成“同任务多轨迹比较”。

16. Agent SFT 与 Agent RL 的关系

Agent 通常不能直接从 RL 开始。更稳的路线是： 

Agent SFT
  ↓
Agent Rejection Sampling
  ↓
Agent RL / GRPO

16.1 Agent SFT 学什么？

能力 例子
工具调用格式 JSON schema、函数名、参数格式
基本规划 先查 schema，再写 SQL
观察结果利用 根据报错修改代码
停止行为 知道何时 finish

16.2 Agent RL 学什么？

能力 例子
更优工具选择 少用无关工具
更强纠错 根据失败 observation 调整
更高任务成功率 最终结果正确
成本控制 用更少步骤完成任务
关系： SFT 负责让 Agent “能跑起来”；RL 负责让 Agent “跑得更好”。

17. 稳定性问题与约束

Agent RL 比单轮 RL 更容易不稳定。

17.1 常见问题

问题 表现 原因
工具滥用 反复调用搜索或计算器 没有成本惩罚
过早停止 还没查完就 final answer 停止动作 reward 设计不当
格式崩坏 JSON 无法解析 format 约束不足
Observation 忽略 工具结果正确但回答仍错 没有奖励 observation utilization
长轨迹退化 越做越绕 没有长度和步骤惩罚
环境漏洞利用 钻测试器漏洞 reward function 不严密

17.2 常用约束

  • 最大工具调用次数；
  • 最大轨迹长度；
  • JSON schema 强校验；
  • 危险工具白名单；
  • 失败动作立即终止；
  • 重复 action 惩罚；
  • KL penalty 限制偏离 reference；
  • trajectory cost penalty。

17.3 KL 约束

\[ L(\theta) = L_{\mathrm{RL}}(\theta) - \beta \mathrm{KL} \left( \pi_\theta \| \pi_{\mathrm{ref}} \right) \]

在实现中，也可以对每个动作 token 加 KL penalty：

\[ r_t^{\mathrm{KL}} = -\beta \left[ \log\pi_\theta(a_t\mid h_t) - \log\pi_{\mathrm{ref}}(a_t\mid h_t) \right] \]
关键： Agent RL 中 KL 不只是防止语言变怪，也防止工具调用策略突然偏离 SFT 学到的基本格式。

18. 最小 Tool-use Agent Toy 实验

建议你第一个 Agent RL 项目不要做网页或复杂搜索，而是做一个最小工具选择任务。

18.1 任务设定

给模型一个用户请求和三个工具： 

tools = [
    "calculator(expression)",
    "search(query)",
    "translator(text, target_language)"
]

任务样例：

User Request 正确工具 正确参数
计算 1497 * 63 calculator {"expression": "1497 * 63"}
把 hello 翻译成中文 translator {"text": "hello", "target_language": "zh"}
查一下强化学习是什么 search {"query": "强化学习 是什么"}

18.2 模型输出格式

{
  "tool": "...",
  "arguments": {
    ...
  }
}

18.3 Reward Function

def reward(output, gold):
    obj = parse_json(output)
    if obj is None:
        return -1.0

    reward = 0.0
    reward += 0.2

    if obj["tool"] == gold["tool"]:
        reward += 0.4
    else:
        reward -= 0.4

    if required_args_complete(obj, gold):
        reward += 0.2

    if args_match(obj, gold):
        reward += 0.4

    return reward

18.4 如何做 GRPO

for each prompt:
    sample G tool-call outputs

    for each output:
        compute reward

    compute group advantage

    update policy with GRPO objective
这个 toy 项目的意义： 它能让你把 Stage 5 的 GRPO 和 Stage 6 的 Tool-use Agent 连接起来，而且 reward 很容易写。

20. 一页总结

概念 核心公式或理解 你要掌握什么
Agent 模型通过动作与环境多轮交互 不只是生成文本，而是行动
历史状态 \(h_t=(x,a_0,o_1,\ldots,a_{t-1},o_t)\) Agent 常用历史近似状态
动作 \(a_t\sim\pi_\theta(\cdot\mid h_t)\) 动作可以是工具调用、搜索、代码执行、回答
Observation \(o_{t+1}\sim\mathcal{E}(\cdot\mid h_t,a_t)\) 环境返回结果进入下一步状态
轨迹 \(\tau=(x,a_0,o_1,\ldots,y_{\mathrm{final}})\) 训练对象是一整条行动链
Agent RL 目标 \(J(\theta)=\mathbb{E}_{\tau}[R(\tau)]\) 最大化任务轨迹 reward
Policy Gradient \(\nabla J=\mathbb{E}[\sum_t\nabla\log\pi_\theta(a_t\mid h_t)\hat{A}_t]\) 更新每一步动作概率
Tool Reward 格式、工具名、参数、执行结果、最终答案 工具调用可分项验证
Cost Penalty \(-\lambda C(\tau)\) 防止工具滥用和长轨迹
Agent-GRPO 同任务多轨迹比较 用组内 reward 训练多轮 Agent
Stage 6 结束标准： 你应该能解释清楚：
  1. Agent 的状态为什么常写成历史 \(h_t\)；
  2. 工具调用为什么可以看成动作；
  3. Observation 如何影响下一步决策；
  4. Agent 轨迹概率如何分解；
  5. 为什么最终 reward 会带来 credit assignment 问题；
  6. 如何设计 tool-use reward；
  7. 如何把 GRPO 扩展到同任务多轨迹 Agent 训练。

21. 本阶段练习

21.1 手推练习

  1. 把一个工具调用任务写成 \(h_t,a_t,o_{t+1}\) 形式。
  2. 写出 Agent 轨迹概率 \(p_\theta(\tau\mid x)\)。
  3. 推导 \(\nabla_\theta\log p_\theta(\tau\mid x)=\sum_t\nabla_\theta\log\pi_\theta(a_t\mid h_t)\)。
  4. 给一个三步工具调用轨迹设计 reward。
  5. 解释为什么同任务多轨迹可以用 GRPO。

21.2 编程练习

实现一个最小 Tool-use Agent 环境：

1. 定义 3 个工具：
   calculator / translator / search

2. 准备 30 条用户请求。

3. 为每条请求写 gold tool 和 gold arguments。

4. 模型或模拟器生成 4 个候选工具调用。

5. 写 reward_fn：
   - JSON 是否合法
   - 工具名是否正确
   - 参数是否完整
   - 参数是否正确

6. 计算 group advantage。

7. 输出哪些候选动作应该被提高概率。

21.3 研究思考题

  1. 如果 Agent 会反复调用 search，应该怎么设计成本惩罚？
  2. 如果工具返回正确结果，但模型最终回答错，reward 应该怎么分配？
  3. 如果模型为了高 reward 学会不输出 Thought，只输出工具调用，这一定是坏事吗？
  4. 数学 Agent 中，什么时候应该让模型自己算，什么时候应该调用 Python？
  5. 如何设计一个既能鼓励正确答案，又能鼓励短轨迹的 reward？
