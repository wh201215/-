    1. DeepSeek-R1 解决的核心问题是什么？

    DeepSeek-R1 报告的核心问题不是“怎么做一个聊天模型”，而是： 如何通过强化学习激发大模型的推理能力。

    这里的 reasoning 不是普通聊天中的“解释一下”，而是更接近：

    • 数学竞赛题的多步推理；
    • 代码题的逻辑拆解；
    • 复杂问题的自我检查；
    • 多路径尝试和策略调整；
    • 最终答案可验证的任务求解。

    1.1 为什么 reasoning 后训练不能只靠 SFT？

    SFT 可以让模型学习人类写好的推理过程。 但是它有一个问题：推理路径太多，人工标注成本太高。

    任务 为什么 SFT 成本高 为什么适合 RL
    数学题 每道题都要写完整解法。 最终答案可以自动验证。
    代码题 实现方式很多，人工示范不唯一。 代码能否通过测试可以验证。
    逻辑推理 中间思路多样。 最终选项或答案可以验证。
    工具调用 轨迹多样，标注每一步很贵。 工具是否调用成功可以验证。
    关键句： SFT 需要人类告诉模型“正确推理过程长什么样”；RL 只需要有办法判断“最后有没有做对”。

    1.2 DeepSeek-R1 的大方向

    DeepSeek-R1 的技术路线可以看成三层：

    层次 内容 作用
    第一层：纯 RL 探索 DeepSeek-R1-Zero 直接在 base model 上做 RL。 验证 reasoning 能否通过 RL 被激发。
    第二层：可用性修复 DeepSeek-R1 加入 cold-start data 和多阶段训练。 解决可读性差、语言混杂、格式不稳定等问题。
    第三层：能力迁移 把 DeepSeek-R1 的 reasoning 能力蒸馏到小模型。 降低部署成本，让小模型也能获得推理能力。

    这三层正好对应后训练学习中的三个问题： 

    能力从哪里来？
  → 通过 RL 和可验证奖励激发。

可用性怎么保证？
  → 通过 cold-start、多阶段训练和格式约束修复。

能力怎么扩散？
  → 通过蒸馏迁移到小模型。

    2. DeepSeek-R1-Zero：不经过 SFT 的纯 RL 尝试

    DeepSeek-R1-Zero 是整篇报告最有冲击力的部分。 它的核心特点是： 不先进行监督微调，而是直接对 base model 做大规模强化学习。

    传统后训练路线通常是： 

    Base Model
  ↓
SFT：先让模型学会像助手一样回答
  ↓
DPO / RLHF：再做偏好优化
  ↓
Online RL：最后继续强化能力

    R1-Zero 的路线更激进： 

    Base Model
  ↓
Rule-based Reward
  ↓
GRPO / RL
  ↓
Reasoning Ability Emergence
    一句话理解： R1-Zero 想验证的是：如果任务答案可以被验证，模型能不能不先模仿人类推理链，而是直接通过强化学习学会推理。

    2.1 为什么 R1-Zero 可以跳过 SFT？

    因为它不是在训练普通聊天偏好，而是在训练可验证 reasoning 任务。 对于这类任务，训练信号不一定来自人工示范，而可以来自规则奖励。

    普通聊天任务 Reasoning 可验证任务
    回答是否好，很大程度依赖人类偏好。 最终答案是否正确，可以自动验证。
    需要教模型如何礼貌、自然、有帮助。 更关心模型能不能做对题。
    Reward 往往需要 Reward Model。 Reward 可以来自规则函数。
    SFT 很重要。 可以尝试直接 RL。

    所以 R1-Zero 的核心不是“所有任务都不需要 SFT”，而是： 在可验证 reasoning 任务上，纯 RL 有可能成为激发能力的有效方式。

    2.2 R1-Zero 训练中模型可能发生了什么？

    从训练逻辑看，模型会经历这样一个过程： 

    一开始：
模型生成一些不稳定的答案。

经过 reward 筛选：
正确答案获得更高奖励，错误答案获得低奖励。

经过多轮更新：
模型更倾向于生成能拿高 reward 的推理和答案。

逐渐出现：
更长推理、自我检查、反思、重新计算、策略调整等行为。
    出现的行为 中文解释 为什么会出现
    Long CoT 更长的思考链。 模型发现复杂题需要更多中间步骤。
    Self-reflection 自我反思。 模型尝试检查前面推理是否出错。
    Verification 验证。 模型重新计算或验证答案。
    Strategy Adaptation 策略调整。 一种方法不行时换另一种方法。
    后训练启发： Reasoning 能力不一定只来自人工标注的 CoT。 如果 reward 设计得好，模型可能通过试错逐渐形成更有效的推理行为。

    3. Rule-based Reward：R1-Zero 的奖励信号从哪里来？

    R1-Zero 的关键不是单纯“做了 RL”，而是它使用了适合 reasoning 任务的规则奖励。 这种奖励不依赖一个神经网络 Reward Model，而是由规则函数判断。

    报告中最核心的是两类 reward：

    Reward 类型 中文解释 作用
    Accuracy Reward 准确性奖励 判断最终答案是否正确。
    Format Reward 格式奖励 判断模型是否按照规定格式输出。
    一句话理解： Accuracy Reward 让模型追求做对；Format Reward 让模型学会把答案放在可解析的位置。

    3.1 Accuracy Reward：准确性奖励

    准确性奖励是 reasoning RL 中最核心的 reward。 它关心最终答案是否正确。

    例如数学题： 

    问题：
1497 × 63 = ?

标准答案：
94311

模型输出：
Final Answer: 94311

reward = 1

    如果模型输出： 

    Final Answer: 94211

reward = 0

    这就是最朴素的 outcome reward。 它不一定检查模型中间推理是否漂亮，只检查最终结果是否对。

    优点 缺点
    简单、客观、容易规模化。 不直接监督中间推理过程。
    不需要人工偏好标注。 模型可能碰巧猜对答案。
    适合数学、代码、逻辑题。 对不可验证任务不适用。

    3.2 Format Reward：格式奖励

    Format Reward 的作用是让模型按照指定格式输出。 这非常重要，因为如果模型答案无法被抽取，accuracy reward 就无法稳定计算。

    例如要求模型使用： 

    <think>
这里是推理过程
</think>

<answer>
最终答案
</answer>

    那么 format reward 可以检查：

    • 是否有 <think> 标签；
    • 是否有 </think> 标签；
    • 是否有 <answer> 标签；
    • 最终答案是否放在正确位置；
    • 格式是否能被解析器稳定抽取。
    def format_reward(output):
    if "<think>" in output and "</think>" in output and "<answer>" in output:
        return 0.2
    else:
        return 0.0
    注意： Format Reward 只能保证模型按格式输出，不保证答案正确。 所以它必须和 Accuracy Reward 配合使用。

    3.3 为什么不用 Reward Model？

    在 RLHF 中，reward 通常来自 Reward Model。 但在数学、代码、逻辑推理这类任务中，最终结果可以直接验证。 所以没必要总是训练一个模型来判断。

    Reward Model Rule-based Reward
    适合主观偏好。 适合客观可验证任务。
    需要偏好数据训练。 通常只需要规则或测试器。
    可能有模型偏差。 规则明确，但覆盖范围有限。
    适合帮助性、安全性、表达质量。 适合数学答案、代码测试、格式校验。
    实践建议： 做自己的 reasoning 小实验时，先不要训练 Reward Model。 先写两个 rule reward：
    1. 答案是否正确；
    2. 格式是否可解析。
    这就是最小 R1-Zero-like 实验的起点。

    4. GRPO：为什么 R1-Zero 适合用组相对策略优化？

    GRPO 的全称是 Group Relative Policy Optimization， 中文可以译为组相对策略优化。

    它之所以适合 R1-Zero 这类 reasoning RL，是因为 reasoning 任务天然适合“同题多采样”： 同一道题，模型可以生成多个不同解法。 有的答案正确，有的答案错误。 然后 reward function 可以分别打分。 

    同一道题 q
  ↓
生成 response 1
生成 response 2
生成 response 3
生成 response 4
  ↓
分别计算 reward
  ↓
组内比较谁更好
  ↓
更新模型

    4.1 为什么不用 PPO 的 Value Model？

    PPO 通常需要 Value Model 来估计 baseline，再计算 advantage。 但是在大模型训练中，Value Model 会带来很大成本：

    • 额外占用显存；
    • 需要训练和同步；
    • 可能和 policy model 一样大；
    • 训练细节复杂；
    • 容易引入不稳定因素。

    GRPO 的做法是：不用单独 Value Model，而是用同组回答的平均 reward 作为 baseline。 

    rewards = [1, 0, 0, 1]

mean_reward = 0.5

advantages = [
  1 - 0.5,
  0 - 0.5,
  0 - 0.5,
  1 - 0.5
]

advantages = [0.5, -0.5, -0.5, 0.5]
    一句话理解： GRPO 不问“这个回答绝对值是多少”，而问“这个回答在同一组里是不是比其他回答更好”。

    4.2 GRPO 在 reasoning 任务中的优势

    优势 解释
    不需要 Value Model 减少显存和工程复杂度。
    适合同题多解 同一道题可以生成多个推理路径。
    适合 outcome reward 每个回答都可以根据最终答案打分。
    和可验证奖励天然结合 答案正确与否可以自动计算 reward。

    4.3 GRPO 不是魔法

    GRPO 只是更轻量的策略优化方法，并不自动保证成功。 它仍然依赖：

    • 任务是否适合可验证 reward；
    • reward function 是否准确；
    • 同组采样是否有足够差异；
    • 训练数据难度是否适中；
    • KL、长度、格式等约束是否合理。
    关键警告： GRPO 解决的是 PPO 中 Value Model 较重的问题。 但它不解决 reward 设计错误、任务不可验证、样本全对或全错这些问题。

    5. R1-Zero 的能力涌现：模型到底学到了什么？

    R1-Zero 最吸引人的地方，是它在大规模 RL 过程中出现了一些 reasoning 行为。 这些行为不是人工逐条标注出来的，而是在奖励驱动下逐渐形成的。

    5.1 长推理：Long Chain-of-Thought

    模型发现：对于复杂题，直接给答案容易错；展开更多中间步骤，有时更容易得到正确答案。 于是模型逐渐倾向于生成更长的推理过程。

    训练早期 训练后期
    回答短，容易直接猜答案。 回答更长，包含更多中间计算和检查。
    推理链不稳定。 推理结构逐渐清楚。
    最终答案容易错。 复杂题正确率提高。

    5.2 自我反思：Self-reflection

    有些输出会出现类似“等一下，我需要重新检查”这样的行为。 这说明模型开始尝试检测自己的中间错误。 

    初始思路：
我先用方法 A 计算。

发现问题：
Wait, this may be wrong.

重新检查：
Let me verify the previous step.

修正：
Actually, the correct calculation should be ...
    为什么这重要： 自我反思不是简单变长，而是模型开始把“检查自己”当成提高正确率的策略。

    5.3 策略调整：Dynamic Strategy Adaptation

    复杂题往往有多种解法。 模型在 RL 过程中可能会学到： 当一种方法走不通时，尝试另一种方法。

    行为 例子
    换公式 代数方法不方便，改用几何或数值方法。
    重算 发现计算结果不合理，重新计算。
    验证答案 把结果代回原题检查。
    拆分问题 把复杂问题拆成多个子问题。
    后训练启发： 这类行为说明，RL 不只是提高最终答案概率，也可能改变模型的中间解题策略。 这正是 reasoning RL 最有价值的地方。

    6. R1-Zero 的问题：为什么纯 RL 不够？

    虽然 R1-Zero 展示了很强的推理潜力，但它也暴露了明显问题。 报告中提到的主要问题包括：可读性差、语言混杂、重复、格式不稳定等。

    6.1 可读性差

    纯 RL 主要追求 reward。 如果 reward 主要奖励最终答案正确，那么模型不一定会主动保持表达清楚。 它可能输出很长、很乱、难以阅读的推理过程。

    原因 表现
    reward 更关心正确率。 中间过程可能不够清楚。
    没有足够 SFT 格式约束。 回答结构不稳定。
    长推理可能提高正确率。 模型越来越啰嗦。

    6.2 语言混杂

    R1-Zero 还可能出现 language mixing，也就是同一个推理过程中混用多种语言。 这通常是因为 reward 没有明确惩罚语言混杂。 

    例子：
We need to solve this equation.
首先，我们计算左边。
Then we check the answer.
所以 final answer is ...
    关键理解： 模型不会自动优化 reward 之外的目标。 如果 reward 没有要求语言统一，它就不一定会保持语言统一。

    6.3 重复和过度思考

    纯 RL 还可能让模型出现 endless repetition 或 overthinking。 因为模型发现长推理有时能提高得分，就可能过度延长推理过程。

    问题 后果
    重复检查 同一个步骤反复验证。
    推理过长 token 成本升高。
    答案延迟 用户要等很久才看到结果。
    格式漂移 长输出中更容易破坏格式。
    一句话理解： R1-Zero 证明纯 RL 有能力，但也证明纯 RL 不够可控。 真正可用的 reasoning model 还需要冷启动数据、格式约束和多阶段训练。

    7. DeepSeek-R1：为什么要加入 Cold-start Data？

    为了修复 R1-Zero 的问题，DeepSeek-R1 引入了 cold-start data。 Cold-start data 可以理解为“在正式 RL 之前，先给模型一小批高质量推理样本”。

    它的作用不是替代 RL，而是给 RL 一个更好的起点。 

    没有 cold-start：
模型从混乱状态直接做 RL。

加入 cold-start：
模型先学会基本推理格式、语言风格和最终答案格式，
再进入 RL。

    7.1 Cold-start Data 解决什么问题？

    R1-Zero 的问题 Cold-start Data 的作用
    可读性差 提供清楚、结构化的推理样本。
    语言混杂 提供统一语言风格的样本。
    格式不稳定 提供稳定的 answer format。
    推理表达混乱 让模型先模仿较好的推理组织方式。

    7.2 Cold-start 和 SFT 的关系

    Cold-start 本质上仍然是一种 SFT。 但它不是普通聊天 SFT，而是专门面向 reasoning RL 的启动数据。

    普通 SFT Cold-start Reasoning SFT
    让模型学会基本对话和指令遵循。 让模型学会稳定的推理格式。
    覆盖广泛任务。 更聚焦数学、代码、逻辑推理。
    重点是助手行为。 重点是 reasoning trace 和 final answer。
    用于普通后训练第一步。 用于 reasoning RL 前的稳定起点。
    实践建议： 如果你做自己的小模型 reasoning RL，不建议直接纯 RL。 更稳妥的方式是：先准备少量格式统一、答案正确、推理清楚的 cold-start 样本，再做 GRPO。

    8. DeepSeek-R1 的多阶段训练路线

    DeepSeek-R1 不是简单地“加了 cold-start，然后做一次 RL”。 它更像一个多阶段 recipe。 这个 recipe 的核心思想是： 用不同阶段分别解决不同问题。 

    阶段 1：Cold-start SFT
  ↓
阶段 2：Reasoning-oriented RL
  ↓
阶段 3：Rejection Sampling 生成高质量 reasoning 数据
  ↓
阶段 4：加入通用 SFT 数据，恢复通用能力
  ↓
阶段 5：Final RL，对齐 helpfulness / harmlessness / reasoning

    目标 说明

    9. Rejection Sampling：为什么 RL 之后还要筛数据？

    前面讲到，DeepSeek-R1 的路线不是简单地“做一次 RL 就结束”。 在 reasoning-oriented RL 之后，模型已经能够生成大量带推理过程的回答。 但是这些回答不一定都适合继续训练。

    这时候就需要 Rejection Sampling，中文可以译为拒绝采样。 它的核心逻辑是： 

    模型生成多个候选回答
  ↓
用规则、答案校验、格式校验、人类或模型筛选
  ↓
保留高质量样本
  ↓
丢弃或标记低质量样本
  ↓
把高质量样本加入下一阶段训练数据
    一句话理解： RL 让模型探索出更多候选；Rejection Sampling 负责从这些候选里筛出真正值得继续学习的样本。

    9.1 为什么不能把 RL 生成的所有样本都留下？

    因为 RL 过程中模型会生成大量不同质量的回答。 有些回答答案正确、推理清楚、格式稳定； 有些回答虽然最终答案正确，但中间推理混乱； 有些回答格式正确但答案错误； 还有些回答可能语言混杂、过度冗长，甚至无法抽取最终答案。

    样本类型 是否适合继续训练 原因
    答案正确、推理清楚、格式稳定 适合 可以作为高质量 reasoning SFT 数据。
    答案正确，但推理混乱 谨慎使用 可能让模型学到不好的中间过程。
    答案错误，但格式正确 不适合 SFT 会把错误答案教给模型。
    格式错误，无法抽取答案 不适合 会破坏输出稳定性。
    语言混杂、过度冗长 通常不适合 会影响可读性和用户体验。
    关键警告： RL 生成数据不等于高质量数据。 如果不筛选，错误推理、坏格式、语言混杂都会被模型继续学进去。

    9.2 Rejection Sampling 筛什么？

    对 reasoning model 来说，拒绝采样至少要筛以下几个维度：

    筛选维度 具体判断 为什么重要
    答案正确性 最终答案是否通过验证。 避免模型学习错误答案。
    格式稳定性 是否有明确 final answer，是否能被抽取。 保证后续 reward 能计算。
    推理可读性 中间步骤是否清楚，是否有明显混乱。 提升最终模型可用性。
    语言一致性 是否出现中英文或多语言混杂。 解决 R1-Zero 暴露出的 language mixing 问题。
    长度合理性 是否过度冗长或重复。 避免模型学会无意义长推理。

    9.3 一个简单的筛选器示意

    def accept_reasoning_sample(output, reference_answer):
    pred = extract_final_answer(output)

    if pred != reference_answer:
        return False

    if not format_valid(output):
        return False

    if language_mixing(output):
        return False

    if too_long_or_repetitive(output):
        return False

    if reasoning_is_too_messy(output):
        return False

    return True

    这个筛选器不是训练 reward，而是数据构造阶段的过滤器。 它的作用是保证进入下一阶段 SFT 或 continued training 的数据质量。

    实践启发： 如果你做自己的 R1-like 小实验，可以把 rejection sampling 设计成一个数据清洗脚本。 不要只看答案对不对，还要看格式、语言、长度和可读性。

    9.4 Rejection Sampling 和 SFT / DPO / RL 的关系

    方法 如何使用样本 和 Rejection Sampling 的关系
    SFT 直接学习高质量样本。 拒绝采样可以为 SFT 筛选好样本。
    DPO 学习 chosen / rejected 差异。 被拒绝样本可以转成 rejected。
    RL 根据 reward 更新模型。 RL 生成候选，拒绝采样再筛选数据。
    Distillation 小模型模仿大模型输出。 蒸馏前必须筛掉大模型错误输出。
    一句话理解： Rejection Sampling 是后训练里的“数据质量闸门”。 它不一定是一个新算法，但对最终模型质量非常关键。

    10. Reasoning Data 与 General Data：为什么不能只训练推理？

    DeepSeek-R1 的多阶段训练中，一个非常重要的点是：不仅要有 reasoning data， 还要混入 general data。

    Reasoning data 主要提升模型的数学、代码、逻辑推理能力； General data 则帮助模型保持普通对话、写作、问答、安全、指令遵循等通用能力。

    一句话理解： Reasoning data 让模型更会“想”；General data 让模型仍然会“正常当助手”。

    10.1 如果只训练 Reasoning Data 会发生什么？

    如果模型长期只在数学、代码、逻辑题上训练，它可能变成一个“只会做题”的模型。 它在复杂推理题上变强，但普通任务可能退化。

    只训练 reasoning data 的风险 具体表现
    普通对话能力下降 用户问日常问题，模型也进入很长推理模式。
    写作风格变僵硬 所有回答都像解题过程。
    帮助性下降 模型只关注推理正确，不关注用户是否容易理解。
    安全性覆盖不足 数学 reward 不会自动训练安全边界。
    指令遵循退化 用户要求简短回答，模型仍然长篇推理。
    重要理解： Reasoning model 不是所有问题都要长思考。 真正好的模型应该知道什么时候深度推理，什么时候直接回答。

    10.2 General Data 解决什么问题？

    General data 可以理解为通用能力数据。 它帮助模型保持基础助手能力。

    General Data 类型 作用
    普通问答 保持基础知识问答能力。
    写作与润色 保持表达和组织能力。
    角色扮演 保持对不同场景指令的适应能力。
    多轮对话 保持上下文跟踪能力。
    安全数据 保持拒答、边界和风险识别能力。
    格式遵循 保持 JSON、表格、列表、摘要等输出能力。

    10.3 Reasoning Data 和 General Data 的分工

    数据类型 主要训练什么 如果缺失会怎样
    Reasoning Data 数学、代码、逻辑、多步推理。 模型推理能力不足。
    General Data 普通对话、写作、帮助性、安全性。 模型只会做题，不像助手。
    Format Data 稳定输出结构。 答案抽取和工具调用不稳定。
    Preference Data 回答风格和用户偏好。 模型可能正确但不好用。
    实践建议： 如果你训练自己的小 reasoning model，不要只准备数学题。 至少要混入一部分普通指令数据，比如解释、总结、短回答、格式遵循。 否则模型容易变成“只会长推理”的偏科模型。

    10.4 对 Agent 后训练的启发

    Agent 训练也有同样问题。 如果只训练工具调用轨迹，模型可能过度调用工具。 如果只训练最终回答，模型又不会正确使用工具。 所以 Agent 数据也要混合：

    Agent 数据类型 作用
    Tool-use Data 学习什么时候调用工具、调用哪个工具。
    No-tool Data 学习什么时候不需要工具，直接回答。
    Error Recovery Data 学习工具失败后如何修正。
    Final Answer Data 学习如何基于工具结果给用户最终回答。
    Safety Data 学习危险工具或敏感任务的边界。
    一句话迁移： Reasoning model 不能只会推理；Agent model 也不能只会调用工具。 数据混合决定了模型最终是不是“可用助手”。

    11. Distillation：为什么 DeepSeek-R1 还要做蒸馏？

    DeepSeek-R1 报告中另一个重要部分是 Distillation，中文是蒸馏。 DeepSeek 官方开源了基于 Qwen 和 Llama 的多个 dense distilled models， 包括 1.5B、7B、8B、14B、32B、70B 等规模。

    这说明 DeepSeek-R1 路线不只是训练一个大模型，还要把大模型的 reasoning 能力迁移到更小、更容易部署的模型上。

    一句话理解： RL 训练大 reasoning model 很贵；蒸馏可以让小模型通过模仿大模型的高质量推理样本，获得一部分 reasoning 能力。

    11.1 蒸馏的基本流程

    Teacher Model：DeepSeek-R1
  ↓
生成高质量 reasoning traces
  ↓
筛选答案正确、格式稳定、推理清楚的数据
  ↓
Student Model：Qwen / Llama 等小模型
  ↓
用这些数据做 SFT
  ↓
小模型学到 teacher 的推理风格和解题过程

    蒸馏本质上不是 RL，而是 SFT。 它用 teacher model 生成的数据作为训练样本，让 student model 去模仿。

    角色 对应对象 作用
    Teacher Model DeepSeek-R1 生成高质量推理过程和答案。
    Student Model Qwen / Llama 小模型 学习 teacher 的推理模式。
    Distillation Data 筛选后的 reasoning traces 作为小模型 SFT 数据。

    11.2 为什么不用小模型直接做 RL？

    小模型当然也可以做 RL，但直接做 RL 成本和稳定性都更难控制。 蒸馏的好处是更简单、更稳定。

    路线 优点 缺点
    小模型直接 RL 有机会探索自己的策略。 训练不稳定，reward 稀疏，能力上限受模型容量限制。
    大模型 RL 后蒸馏给小模型 训练更稳定，数据质量更高，适合部署。 小模型主要模仿 teacher，上限受 teacher 数据和自身容量限制。
    实践建议： 如果你算力有限，不要一开始就做小模型 RL。 更现实的路线是：用强模型生成 reasoning 数据，先做 SFT 蒸馏，再考虑 DPO 或 GRPO 微调。

    11.3 蒸馏数据为什么也要筛选？

    Teacher model 生成的内容不一定全对。 如果不筛选，student model 会学习错误推理。

    筛选项 原因
    答案正确 避免错误知识和错误解题过程进入小模型。
    格式稳定 让小模型学会可解析输出。
    推理清楚 避免学习混乱推理。
    长度适中 避免小模型学会过度冗长。
    语言一致 避免中英文混杂。

    11.4 蒸馏和 Rejection Sampling 的关系

    蒸馏通常离不开 Rejection Sampling。 因为 teacher model 可以为同一道题生成多个候选推理， 然后用验证器筛出正确、清楚、格式好的样本。 

    Prompt
  ↓
Teacher 生成多个 reasoning traces
  ↓
Answer verifier 检查答案
  ↓
Format checker 检查格式
  ↓
Quality filter 检查长度和语言
  ↓
保留高质量 trace
  ↓
用于 Student SFT
    一句话理解： 蒸馏负责“能力迁移”，拒绝采样负责“数据筛选”。 二者结合，才能把大模型的 reasoning 能力比较干净地传给小模型。

    11.5 蒸馏模型的意义

    DeepSeek-R1 开源蒸馏模型的意义在于： 更多人可以在较低成本下使用 reasoning model。 不是所有场景都能部署巨大模型，蒸馏模型让 reasoning 能力更容易进入实际应用。

    模型规模 适合场景
    1.5B / 7B 学习实验、端侧尝试、轻量服务。
    8B / 14B 个人 GPU 或小规模部署。
    32B 较强 reasoning 需求，服务器部署。
    70B 更强性能，但部署成本更高。
    注意： 蒸馏模型不是完整复制 DeepSeek-R1。 它们学到的是 teacher 的一部分行为模式，最终能力仍然受到模型规模、数据质量和训练方式限制。

    Fireworks AI RLVR DeepSeek-R1

    DeepSeek-R1 主要讲 reasoning model，但它对 Agent 后训练也很有启发。 因为 Agent 的很多任务也可以被验证。

    把 R1 的思路迁移到 Agent，可以得到这样一条路线： 

    Agent Base Model
  ↓
Cold-start Tool-use SFT
  ↓
Rule-based Reward / Environment Reward
  ↓
GRPO / RL
  ↓
Rejection Sampling 筛选高质量轨迹
  ↓
General Instruction Data 混合
  ↓
Final Alignment
  ↓
Tool-use / Code / SQL / Web Agent
    一句话理解： R1 把“最终答案是否正确”变成 reward； Agent 可以把“工具调用是否成功、任务是否完成”变成 reward。

    13.1 Agent 的 Accuracy Reward 是什么？

    在数学 reasoning 中，accuracy reward 是最终答案是否正确。 在 Agent 中，它可以对应任务是否完成。

    Reasoning Model Agent Model
    数学答案是否正确。 任务是否完成。
    代码是否通过测试。 工具是否执行成功。
    最终答案是否可抽取。 最终状态是否达成。
    推理格式是否正确。 工具调用格式是否正确。

    13.2 Agent 的 Format Reward 是什么？

    Agent 的 format reward 更重要，因为工具调用必须机器可解析。 如果模型输出格式错了，工具根本无法执行。 

    正确工具调用格式示例：

{
  "tool": "get_weather",
  "arguments": {
    "location": "Berlin",
    "date": "tomorrow"
  }
}

    Format Reward 可以检查：

    • 是否是合法 JSON；
    • 是否包含 tool 字段；
    • 是否包含 arguments 字段；
    • 字段名是否符合 schema；
    • 参数类型是否正确；
    • 是否多输出了无法解析的内容。

    13.3 Agent 的 Cold-start Data 应该长什么样？

    Agent cold-start data 不应该只是普通问答，而应该是高质量工具调用轨迹。 

    User:
请查一下明天柏林的天气。

Assistant:
Thought: 用户需要实时天气信息，需要调用天气工具。
Action:
{
  "tool": "get_weather",
  "arguments": {
    "location": "Berlin",
    "date": "tomorrow"
  }
}

Observation:
{"weather": "cloudy", "high": "18°C", "low": "11°C"}

Assistant:

明天柏林多云，最高气温约 18°C，最低气温约 11°C。
    Agent cold-start 数据要素 说明
    任务意图清楚 用户请求必须明确。
    工具选择正确 不能乱调用工具。
    参数完整合法 JSON 字段必须符合 schema。
    利用 Observation 最终回答必须基于工具返回。
    知道何时停止 不能无限调用工具。
    实践建议： 如果你要做 Agent R1-like 训练，先不要直接 RL。 先准备几百条高质量 tool-use cold-start 轨迹，让模型学会基本格式和动作结构。

    13.4 Agent 的 Rejection Sampling 怎么做？

    Agent 轨迹也可以做拒绝采样。 同一个任务可以让模型生成多条轨迹，然后根据工具执行结果筛选。

    轨迹类型 是否保留 原因
    工具正确、参数正确、任务完成 保留 可作为高质量 SFT 或蒸馏数据。
    工具正确但参数缺失 拒绝或作为负样本 说明模型还不会填参数。
    工具错误 拒绝 会误导模型。

    def agent_reward(trajectory, ground_truth):
    reward = 0.0

    # 1. 工具调用格式
    if valid_tool_call_schema(trajectory):
        reward += 0.2
    else:
        reward -= 0.5

    # 2. 工具名称
    if selected_tool(trajectory) == ground_truth.tool:
        reward += 0.5
    else:
        reward -= 0.5

    # 3. 参数完整性
    if arguments_complete(trajectory):
        reward += 0.3
    else:
        reward -= 0.3

    # 4. 参数正确性
    if arguments_match(trajectory, ground_truth.arguments):
        reward += 0.5

    # 5. 工具执行结果
    if tool_execution_success(trajectory):
        reward += 1.0

    # 6. 最终回答是否基于 observation
    if final_answer_uses_observation(trajectory):
        reward += 0.5

    # 7. 轨迹成本
    if too_many_tool_calls(trajectory):
        reward -= 0.4

    return reward

    14.3 Reward Function 设计

    最小 reward 可以包含三个部分：

    Reward 项 作用 示例
    Accuracy Reward 奖励最终答案正确。 答案正确 +1，错误 +0。
    Format Reward 奖励输出格式可解析。 有 answer 标签 +0.2。
    Length Penalty 防止模型过度冗长。 超过长度阈值 -0.1。 
    import re

def extract_answer(output):
    pattern = r"<answer>\s*(-?\d+)\s*</answer>"
    match = re.search(pattern, output)

    if match is None:
        return None

    return match.group(1)

def reward_fn(output, reference_answer):
    reward = 0.0

    # Format Reward
    if "<answer>" in output and "</answer>" in output:
        reward += 0.2
    else:
        reward -= 0.2

    # Accuracy Reward
    pred = extract_answer(output)
    if pred == reference_answer:
        reward += 1.0

    # Length Penalty
    if len(output) > 1200:
        reward -= 0.1

    return reward

    14.5 组内 advantage 计算

    def group_advantages(rewards):
    mean_reward = sum(rewards) / len(rewards)

    advantages = [
        r - mean_reward
        for r in rewards
    ]

    return advantages

    如果得到： 

    rewards = [1.2, 0.2, 0.0, 1.2, -0.2, 0.2, 1.2, 0.0]

    那么组内平均值大约是： 

    mean_reward = 0.475

    高于 0.475 的回答会得到正 advantage，低于 0.475 的回答会得到负 advantage。 这就是 GRPO 的直观核心。

    一句话理解： GRPO 不是只看“这个回答几分”，而是看“这个回答在同一组回答里是不是相对更好”。

    15. 常见误解：不要把 DeepSeek-R1 简化成几个口号

    DeepSeek-R1 很火，所以也很容易被误读。 下面这些误解尤其常见。

    15.1 误解一：R1 证明 SFT 没用了

    这是错误的。 R1-Zero 展示了纯 RL 的潜力，但 DeepSeek-R1 本身又加入了 cold-start data 和多阶段训练。 这说明 SFT 不是没用，而是位置发生了变化。

    错误理解 更准确理解
    SFT 没用了。 纯 RL 可以激发推理，但可用模型仍然需要 cold-start 和 SFT 类数据。
    只要 RL 就够了。 RL 提升能力，但格式、可读性、通用能力仍需要数据和对齐。
    所有任务都可以跳过 SFT。 只有可验证 reasoning 任务更适合尝试直接 RL。
    关键句： R1-Zero 是探索；DeepSeek-R1 是可用 recipe。 不要把探索阶段的结论简单套到所有模型训练上。

    15.2 误解二：GRPO 是 R1 成功的全部原因

    GRPO 很重要，但不是全部。 DeepSeek-R1 的效果来自一整套组合：

    • base model 能力；
    • rule-based reward；
    • GRPO / RL；
    • cold-start data；
    • rejection sampling；
    • general data mixing；
    • final RL；
    • distillation。
    只看 GRPO 的问题 为什么不够
    忽略数据质量 reward 和训练数据决定了模型学什么。
    忽略 cold-start 没有好起点，RL 容易混乱。
    忽略多阶段训练 单一 RL 阶段难以同时兼顾能力和可用性。
    忽略蒸馏 R1 影响力很大一部分来自能力迁移到小模型。
    一句话理解： GRPO 是重要算法组件，但 DeepSeek-R1 的核心是完整后训练 recipe。

    15.3 误解三：只要答案正确，推理过程无所谓

    如果只是训练一个判题模型，最终答案可能足够。 但如果要做用户可用的 reasoning assistant，推理过程很重要。

    为什么推理过程重要 说明
    用户需要理解 只给答案不利于学习和信任。
    便于检查错误 中间步骤清楚，错误更容易定位。
    提升可控性 可读推理更容易约束和改进。
    适合教学场景 数学、代码、学术任务都需要过程解释。
    注意： Outcome Reward 很有用，但不能替代所有过程质量控制。 实际模型还需要可读性、格式、长度和安全约束。

    15.4 误解四：RLVR 可以解决所有任务

    RLVR 只适合可验证任务。 如果任务是主观写作、文学评价、情感安慰、审美判断，就很难写一个可靠验证器。

    适合 RLVR 不太适合纯 RLVR
    数学答案。 文学风格好不好。
    代码测试。 情绪安慰是否到位。
    SQL 结果。 观点是否有启发。
    工具调用。 表达是否优美。
    格式校验。 复杂价值判断。

    15.5 误解五：蒸馏小模型等于复制大模型

    蒸馏不是完整复制。 小模型只是学习 teacher model 生成的数据分布。 最终效果仍然受小模型容量、蒸馏数据质量、训练方式限制。

    准确理解： 蒸馏是能力迁移，不是能力克隆。 它能让小模型学到一部分 reasoning 行为，但不能保证完全达到 teacher model 水平。
