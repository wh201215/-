    Agent RLVR 中文精读
    从 Function Picking 到 Tool-use Agent 后训练

    本篇定位：这是后训练学习路线中的 Agent 专题。前面你已经理解了 SFT、DPO、PPO、GRPO、RLVR 和 DeepSeek-R1 式 reasoning RL；这一篇要做的事情，是把 reasoning RL 的思路迁移到 Tool-use Agent。

    核心迁移：

    Reasoning RLVR：
答案是否正确 → reward

Agent RLVR：
工具是否选对 / 参数是否正确 / 工具是否执行成功 / 是否利用 Observation / 最终任务是否完成 → reward

    适合读者：想继续学习 agentic RL、tool-use、Web Agent、Code Agent、RAG Agent、SQL Agent、浏览器 Agent、自动化办公 Agent 的人。

    0. Agent 后训练和普通后训练有什么区别？

    普通 LLM 后训练，很多时候优化的是一段回答。用户问一个问题，模型输出一段文本，训练系统判断这段文本好不好。Agent 后训练不一样，它优化的不是一个静态回答，而是一条可以和环境交互的轨迹。

    图 1：普通后训练与 Agent 后训练的差别。图源：本文自绘，参考 ReAct 的 Reasoning + Acting 框架、Toolformer 的 API 调用思想，以及 OpenAI Function Calling 文档。

    0.1 普通后训练：优化 response

    普通聊天模型的训练对象通常是一段 response。例如：

    User: 解释什么是 DPO。
Assistant: DPO 是一种直接偏好优化方法……

    奖励可以直接看最终回答：是否准确、是否清楚、是否遵守格式、是否安全。

    0.2 Agent 后训练：优化 trajectory

    Agent 的输出不是一段话，而是一串交互过程：

    User task
  ↓
Assistant decides tool
  ↓
Tool call with arguments
  ↓
Tool returns observation
  ↓
Assistant decides next step
  ↓
Maybe call another tool
  ↓
Final answer

    这条链路里，每一步都可能出错。因此 Agent RLVR 不能只看最后一句 answer，它要看整条轨迹。

    比较项普通后训练Agent 后训练
    训练对象回答文本 response。工具调用轨迹 trajectory。
    动作 action生成 token。生成 token + 选择工具 + 填参数 + 决定是否继续。
    环境 environment通常是评估器或用户。工具系统、数据库、浏览器、代码执行器、搜索引擎、文件系统。
    奖励 reward回答好坏。工具选择、参数合法、执行结果、Observation 利用、任务完成。
    主要风险幻觉、啰嗦、偏好错位。调用错工具、参数错、忽略工具结果、循环调用、危险操作。
    适合算法SFT、DPO、PPO、GRPO。Tool-use SFT、轨迹 DPO、Agentic GRPO、RLVR、环境反馈 RL。
    这一节要记住：Agent 后训练的关键不是让模型“更会说”，而是让模型“更会做”。会做的意思包括：知道什么时候不用工具、什么时候要用工具、用哪个工具、参数怎么填、工具失败后如何恢复、什么时候停止。

    0.3 为什么 Agent 特别适合 RLVR？

    RLVR 的全称可以理解为 Reinforcement Learning with Verifiable Rewards，即“基于可验证奖励的强化学习”。Agent 任务很多地方天然可验证：

    Agent 子能力可验证信号能否自动打分
    工具选择是否调用目标工具；是否调用了无关工具。可以。
    JSON 格式是否能被解析；是否符合 schema。可以。
    参数完整性必填字段是否齐全；类型是否正确。可以。
    执行成功工具是否返回成功状态；测试是否通过。可以。
    Observation 使用最终回答是否引用/利用工具返回。部分可以，需要规则或 judge。
    最终任务完成数据库状态、网页状态、代码测试、答案匹配。很多场景可以。

    这就是 Agent RLVR 的价值：它把“主观觉得 Agent 好不好”拆成许多可以自动检查的小指标。

    1. Function Picking 是什么？

    Function Picking 可以翻译成“函数选择”或“工具选择”。它是 Tool-use Agent 的第一层能力：面对一个任务，模型先判断应该调用哪个工具。

    一句话理解：Function Picking 不是让模型直接解决问题，而是让模型先判断“这个问题应该交给哪个工具处理”。

    1.1 Function Picking 的最小例子

    User: 计算 573 * 89。

可选工具：
1. search_web(query)
2. calculator(expression)
3. translate(text, target_language)

正确选择：
calculator(expression="573 * 89")

    在这个例子中，模型不需要自己心算。它真正要学的是：这是一个精确计算任务，因此应该调用 calculator，而不是 search_web，也不是 translate。

    1.2 Function Picking 和 Function Calling 的区别

    概念中文解释关注点
    Function Picking从多个工具中选择一个或多个合适工具。选哪个工具。
    Function Calling按照工具 schema 生成可执行调用。怎么调用工具。
    Tool Execution外部系统真正执行函数。工具是否执行成功。
    Observation Integration模型读取工具返回并继续推理。是否利用返回结果。

    1.3 Function Picking 的常见错误

    错误类型例子为什么严重
    不用工具需要查实时信息，却直接凭记忆回答。容易产生幻觉。
    错选工具计算任务调用 search_web。工具链从第一步就错了。
    过度调用工具用户问“你好”，模型调用 search。成本高、延迟高、体验差。
    相似工具混淆把 search_docs 和 search_web 混用。在企业知识库/RAG 中尤其危险。
    多工具顺序错误还没查询数据就先总结。会导致无依据回答。

    1.4 Function Picking 的训练目标

    如果把 Function Picking 单独拿出来训练，它可以是一个分类问题：

    输入：user query + tool descriptions
输出：tool_name

    也可以是一个生成问题：

    {
  "tool_name": "calculator",
  "arguments": {
    "expression": "573 * 89"
  }
}
    实践理解：Agent 后训练不要一开始就追求复杂自主规划。先让模型稳定学会 Function Picking，是非常重要的第一步。因为只要第一步选错工具，后面的参数、执行、Observation 使用都会跟着错。

    2. Tool-use SFT 数据长什么样？

    在进入 RLVR 之前，通常先做 Tool-use SFT。原因很简单：如果模型还不会基本的工具调用格式，直接做 RL 会非常不稳定。SFT 的作用是让模型先学会“像一个工具 Agent 一样行动”。

    2.1 普通 SFT 数据

    {
  "messages": [
    {"role": "system", "content": "你是一个有帮助的助手。"},
    {"role": "user", "content": "法国首都是哪里？"},
    {"role": "assistant", "content": "法国首都是巴黎。"}
  ]
}

    2.2 Tool-use SFT 数据

    Tool-use SFT 不只包含 user 和 assistant，还要包含工具定义、工具调用、工具返回结果。

    {
  "tools": [
    {
      "name": "calculator",
      "description": "执行数学表达式计算。",
      "parameters": {
        "type": "object",
        "properties": {
          "expression": {"type": "string"}
        },
        "required": ["expression"]
      }
    }
  ],
  "messages": [
    {"role": "system", "content": "你可以在需要时调用工具。"},
    {"role": "user", "content": "计算 573 * 89。"},
    {
      "role": "assistant",
      "tool_call": {
        "name": "calculator",
        "arguments": {"expression": "573 * 89"}
      }
    },
    {"role": "tool", "name": "calculator", "content": "50997"},
    {"role": "assistant", "content": "573 * 89 = 50997。"}
  ]
}

    2.3 Tool-use SFT 到底训练哪些 token？

    这一点非常重要。不是所有内容都应该计算 loss。

    数据部分是否作为输入是否计算 loss原因
    system prompt是否提供规则，不是模型要学习生成的目标。
    user message是否提供任务，不是模型要生成的内容。
    tool definitions是否提供工具 schema，帮助模型决定调用。
    assistant tool_call是是模型需要学习选择工具和生成参数。
    tool observation是否这是环境返回，不是模型生成。
    assistant final answer是是模型需要学习如何基于工具结果回答。
    常见坑：不要让模型对 tool observation 计算 loss。Observation 是环境返回，不是 assistant 应该生成的内容。如果把工具返回也当成模型输出训练，模型可能学会“伪造工具返回”。

    2.4 三类最适合入门的 Tool-use SFT 数据

    数据类型工具为什么适合入门
    数学计算calculator / python答案容易验证，参数简单。
    文档问答search_doc能训练“不知道就查”的习惯。
    代码执行run_python / unit_test能训练执行反馈、错误修复和测试驱动。

    2.5 一条高质量 Tool-use SFT 样本应该包含什么？

    1. 明确任务：用户到底要什么
2. 工具列表：模型可用哪些工具
3. 工具描述：每个工具适合什么场景
4. 参数 schema：每个字段类型、含义、是否必填
5. 正确工具调用：tool_name + arguments
6. 工具返回：真实 observation
7. 最终回答：必须基于 observation
8. 可选解释：为什么调用这个工具
    实践建议：刚开始做 Agent SFT，不要准备“复杂多工具长轨迹”。先准备 300 条单工具样本：100 条 calculator、100 条 search_doc、100 条 run_python。让模型先稳定掌握工具调用格式，再逐步加入多工具、多轮、错误恢复。

    3. Action / Observation / Final Answer 格式

    Agent 训练的核心格式，可以用三个词概括：

    Action：模型做了什么
Observation：环境返回了什么
Final Answer：模型什么时候结束并回答用户

    3.1 ReAct 风格格式

    ReAct 的经典思路是把 reasoning trace 和 action 交织起来：

    Thought: 我需要计算这个乘法。
Action: calculator({"expression": "573 * 89"})
Observation: 50997
Thought: 工具已经返回结果，可以回答。
Final Answer: 573 * 89 = 50997。

    这个格式的优点是非常直观，适合教学和调试。缺点是如果直接用自然语言解析 Action，工程上容易出错。因此真实系统通常更偏向结构化 tool_call。

    3.2 Function Calling 风格格式

    {
  "role": "assistant",
  "tool_calls": [
    {
      "type": "function",
      "function": {
        "name": "calculator",
        "arguments": "{\"expression\": \"573 * 89\"}"
      }
    }
  ]
}

    这个格式更适合工程系统，因为它可以直接被程序解析并执行。

    3.3 为什么格式比你想象的重要？

    格式问题后果
    Action 不是合法 JSON工具执行系统无法解析。
    字段名错误schema 校验失败。
    参数类型错误工具运行时报错。
    Observation 和 Final Answer 混在一起模型可能忽略工具返回。
    没有 stop 标记模型可能无限调用工具。

    3.4 推荐的教学格式

    如果你是为了学习 Agent RLVR，我建议先用下面这种简化格式：

    <tool_call>
{"name": "calculator", "arguments": {"expression": "573 * 89"}}
</tool_call>

<observation>
50997
</observation>

<final_answer>
573 * 89 = 50997。
</final_answer>

    这个格式的好处是：既像自然语言轨迹，又容易用正则或 JSON parser 提取。

    4. 工具调用 reward 怎么设计？

    到了 Agent RLVR，最核心的问题是：如何把一条工具调用轨迹打成 reward？

    核心原则：不要只奖励最终答案。Agent 的错误往往发生在中间过程：工具选错、参数错、执行失败、忽略 Observation、调用太多次。只看最终答案会让训练信号太稀疏，也很难定位模型到底错在哪里。

    4.1 一个完整的 Agent reward 可以拆成 7 个部分

    R_total =
  R_tool_choice
+ R_json_schema
+ R_arg_complete
+ R_execution
+ R_observation_use
+ R_final_success
- R_trajectory_cost
- R_safety_violation
    奖励项中文判断什么例子
    R_tool_choice工具选择奖励是否选对工具。数学题调用 calculator。
    R_json_schemaJSON 格式奖励输出是否可解析、符合 schema。arguments 是合法 JSON。
    R_arg_complete参数完整性奖励必填字段是否齐全。search_doc 有 query 字段。
    R_execution执行成功奖励工具是否真正运行成功。代码测试通过、SQL 查询成功。
    R_observation_useObservation 使用奖励最终回答是否利用工具返回。答案包含搜索到的事实。
    R_final_success最终任务奖励用户任务是否完成。最终状态符合目标。
    R_trajectory_cost轨迹成本惩罚是否调用太多工具、太多 token。每次工具调用扣 0.05。
    R_safety_violation安全惩罚是否越权、执行危险操作。试图删除文件、泄露密钥。

    4.2 为什么 reward 要拆细？

    假设一个 Agent 最终任务失败，有很多可能原因：

    失败结果：用户没有得到正确答案

可能原因 A：模型没有调用工具
可能原因 B：模型调用了错工具
可能原因 C：工具对了，但参数错了
可能原因 D：参数对了，但工具执行失败
可能原因 E：工具返回正确，但模型没读
可能原因 F：模型回答对了，但过程调用 20 次工具，成本太高

    如果只给一个 final_reward = 0，模型不知道自己错在什么地方。拆成细粒度 reward 后，模型至少能得到更明确的优化方向。

    不要过度迷信最终答案奖励：在 Agent 任务里，最终答案正确不代表轨迹健康。模型可能通过错误工具碰巧得到答案，也可能用极高成本完成任务。Agent reward 必须同时关心“做成了没有”和“怎么做成的”。

    4.3 推荐的入门 reward 权重

    奖励项建议分值说明
    工具选择正确+0.2鼓励先选对工具。
    JSON 合法+0.15格式稳定非常重要。
    参数完整+0.15保证工具能运行。
    执行成功+0.2工具调用必须真实有效。
    利用 Observation+0.15防止调用工具后仍然胡编。
    最终答案正确+0.3任务完成仍然是最重要目标。
    每多调用一次工具-0.03 到 -0.1防止无意义循环。
    危险/越权调用-1.0 或直接终止安全边界必须强约束。

    注意：这些权重不是标准答案，只是入门教学用。真实任务中要根据工具成本、任务难度、错误代价调权重。

    5. JSON schema reward

    JSON schema reward 是 Agent RLVR 里最容易做、最应该先做的 reward。它判断模型输出的工具调用是否符合结构化格式。

    5.1 为什么 JSON schema reward 很重要？

    工具调用不是普通文本。普通文本写错一个标点，人还能看懂；JSON 工具调用错一个括号，程序可能直接无法执行。

    输出人类能否理解系统能否执行
    calculator({expression: 573*89})能不一定，非标准 JSON。
    {"name": "calculator", "arguments": {"expression": "573 * 89"}}能能。
    {"tool": "calculator", "expr": "573 * 89"}能猜可能失败，因为字段名不符合 schema。

    5.2 JSON schema reward 的检查层次

    Level 1：能否提取 tool_call 块
Level 2：是否为合法 JSON
Level 3：是否包含 name / arguments
Level 4：name 是否在工具列表中
Level 5：arguments 是否符合该工具 schema
Level 6：是否没有多余危险字段

    5.3 示例代码

    import json

TOOLS = {
    "calculator": {
        "required": {"expression": str},
        "allowed_keys": {"expression"},
    }
}

def json_schema_reward(text: str) -> float:
    try:
        obj = json.loads(text)
    except Exception:
        return 0.0

    if not isinstance(obj, dict):
        return 0.0
    if "name" not in obj or "arguments" not in obj:
        return 0.1

    name = obj["name"]
    args = obj["arguments"]
    if name not in TOOLS:
        return 0.2
    if not isinstance(args, dict):
        return 0.3

    schema = TOOLS[name]
    for key, typ in schema["required"].items():
        if key not in args:
            return 0.45
        if not isinstance(args[key], typ):
            return 0.55

    if set(args.keys()) - schema["allowed_keys"]:
        return 0.75

    return 1.0
    实践建议：JSON schema reward 是最适合放在 Agent RLVR 第一阶段的奖励。它不会让模型立刻变聪明，但会让模型变“可执行”。Agent 一旦不可执行，后面的 reward 都没有意义。

    6. 参数完整性 reward

    参数完整性 reward 判断的是：模型选了正确工具以后，参数是否足够让工具运行。

    6.1 例子：search_doc

    工具定义：
search_doc(query: string, top_k: int = 5)

用户任务：
总结文档中 DPO 和 PPO 的区别。

正确调用：
{"name": "search_doc", "arguments": {"query": "DPO PPO 区别", "top_k": 5}}

错误调用 1：
{"name": "search_doc", "arguments": {}}

错误调用 2：
{"name": "search_doc", "arguments": {"keyword": "DPO PPO 区别"}}

错误调用 3：
{"name": "search_doc", "arguments": {"query": 123}}

    6.2 参数完整性不等于 JSON 合法

    下面这个输出是合法 JSON，但参数不完整：

    {"name": "search_doc", "arguments": {}}

    因此 JSON schema reward 和参数完整性 reward 要分开设计。

    检查项例子reward
    必填字段存在query 存在。+0.3
    字段类型正确query 是 string。+0.3
    字段语义相关query 和用户问题相关。+0.3
    可选参数合理top_k 不过大。+0.1

    6.3 参数语义相关性怎么判断？

    字段是否存在、类型是否正确很容易判断；语义相关性更难。可以用三种方式：

    方式做法适合场景
    关键词规则query 中必须包含任务关键词。文档问答、小数据集。
    embedding 相似度计算用户任务与 query 的相似度。RAG、搜索任务。
    LLM judge让评审模型判断参数是否相关。复杂工具、多字段参数。
    注意：不要只奖励“参数非空”。模型可能学会填一堆无意义参数。参数 reward 必须同时看完整性、类型、语义相关性和成本。

    7. 执行成功 reward

    执行成功 reward 判断的是：工具调用是否真的跑通。

    7.1 执行成功比格式正确更进一步

    阶段判断问题例子
    格式正确JSON 是否能解析。有 name 和 arguments。
    参数正确arguments 是否符合 schema。expression 是字符串。
    执行成功工具是否返回成功。calculator 返回数字；Python 没报错；SQL 有结果。

    7.2 不同工具的执行成功信号

    工具执行成功信号失败例子
    calculator返回可解析数值。表达式非法、除零。
    search_doc返回非空检索结果。query 太泛、无结果。
    run_python代码运行无异常。SyntaxError、Timeout。
    unit_test测试通过率。测试失败。
    sql_querySQL 执行成功且结果合理。表名错、字段错、权限错。
    web_browser页面状态达到目标。点击错按钮、登录失败。

    7.3 示例：Python 执行 reward

    import subprocess
import tempfile
from pathlib import Path


def python_execution_reward(code: str, timeout: int = 3) -> float:
    with tempfile.TemporaryDirectory() as tmp:
        path = Path(tmp) / "main.py"
        path.write_text(code, encoding="utf-8")
        try:
            result = subprocess.run(
                ["python", str(path)],
                capture_output=True,
                text=True,
                timeout=timeout,
            )
        except subprocess.TimeoutExpired:
            return -0.2

        if result.returncode != 0:
            return 0.0
        return 1.0
    实践启发：代码 Agent、SQL Agent、网页 Agent 都特别适合执行成功 reward。因为它们不一定有唯一标准答案，但可以通过测试、数据库状态或页面状态判断是否成功。

    8. Observation 使用 reward

    很多 Agent 会出现一个很典型的问题：工具调用了，Observation 也返回了，但最终回答没有使用 Observation，而是继续凭空编。

    8.1 例子：调用工具但忽略结果

    User: 查找文档中 DPO 的定义。

Assistant tool_call:
{"name": "search_doc", "arguments": {"query": "DPO definition"}}

Observation:
DPO，全称 Direct Preference Optimization，是一种直接利用偏好数据优化模型的方法。

Bad Final Answer:
DPO 是一种强化学习算法，主要依靠环境交互进行探索。

Good Final Answer:
DPO 是 Direct Preference Optimization，中文可译为直接偏好优化。它直接利用 chosen / rejected 偏好对优化模型，而不需要显式训练 reward model。

    8.2 Observation 使用 reward 怎么设计？

    方法判断方式优点缺点
    关键词重叠Final Answer 是否包含 Observation 关键词。简单。容易被表面匹配骗过。
    引用检查要求回答带来源编号或片段引用。适合 RAG。需要结构化检索结果。
    事实一致性 judge判断答案是否被 Observation 支持。更准确。成本高，也可能有 judge 偏差。
    答案提取匹配从 Observation 和答案中提取关键字段比较。适合表格、数据库、票务任务。需要任务定制。

    8.3 RAG Agent 的 Observation 使用规则

    如果任务是文档问答：
1. 必须调用 search_doc 或 retrieve
2. Final Answer 必须基于检索片段
3. 不能出现 Observation 中没有的信息
4. 最好带 chunk id / page id / source title
5. 如果检索为空，应该说明没有找到，而不是编造
    重要风险：Observation 使用 reward 不要只看“有没有复制工具返回”。模型可能学会机械复制。更好的 reward 是判断：最终回答是否真正被 Observation 支持，是否删除了无关信息，是否完成了用户任务。

    9. 轨迹长度惩罚

    Agent 的一个常见坏习惯是：为了提高成功率，疯狂调用工具。最终答案可能对，但成本极高。

    9.1 为什么需要轨迹长度惩罚？

    没有长度惩罚时可能学到的坏行为
    只奖励最终成功模型反复搜索，直到碰巧找到答案。
    只奖励执行成功模型调用很多低风险工具刷成功。
    只奖励 Observation 使用模型检索大量内容后拼接答案。
    不惩罚 token 成本模型生成冗长 Thought 和无意义中间步骤。

    9.2 常见惩罚方式

    R_cost = - alpha * num_tool_calls
R_cost = - beta * total_tokens
R_cost = - gamma * num_failed_calls
R_cost = - delta * trajectory_seconds
    惩罚项适合控制什么
    num_tool_calls工具调用次数。
    total_tokens思考和回答长度。
    num_failed_calls错误试探。
    trajectory_seconds真实系统延迟。
    tool_cost昂贵 API 或付费工具。

    9.3 不要过度惩罚长度

    轨迹长度惩罚也有副作用。如果惩罚太强，模型可能学会不调用工具，直接猜答案。

    平衡原则：先奖励成功，再轻微惩罚成本。不要让成本惩罚压过任务完成奖励。 
    推荐入门形式：
R_total = R_success + R_format + R_execution - 0.05 * max(0, num_tool_calls - expected_calls)

    10. Rejection Sampling 筛选 Agent 轨迹

    Rejection Sampling 在 Agent 后训练里非常有用。它的作用是：让模型或强模型对同一个任务生成多条轨迹，然后筛选成功、短、稳定、安全的轨迹作为训练数据。

    10.1 Agent 版拒绝采样流程

    同一个任务 prompt
  ↓
采样 N 条 Agent trajectories
  ↓
执行每条轨迹中的工具调用
  ↓
记录 tool choice / arguments / observation / final answer
  ↓
用 reward function 打分
  ↓
保留高分轨迹作为 SFT chosen
  ↓
低分轨迹可作为 DPO rejected
  ↓
进入下一轮训练
    图 4：Agent Rejection Sampling 流程。图源：本文自绘，参考 Rejection Sampling、Toolformer 的自监督筛选思路，以及 Agent RL 中通过执行反馈筛选轨迹的做法。

    10.2 什么样的轨迹应该保留？

    轨迹类型是否保留原因
    任务成功、工具正确、参数正确、步骤短保留为 chosen / SFT。高质量示范。
    任务成功但步骤过长可保留，也可作为次优 rejected。适合训练效率偏好。
    工具正确但参数错作为 rejected。训练参数偏好。
    工具选错作为 rejected。训练 Function Picking。
    JSON 解析失败丢弃或作为格式 rejected。取决于 DPO 是否需要负样本。
    出现安全越权丢弃并加入安全负样本。安全优先。

    10.3 拒绝采样后的数据可以怎么用？

    高分轨迹：
  → 加入 SFT 数据

一高一低：
  → 构造成 DPO pair

同一任务多条轨迹 + reward：
  → 用 GRPO 做组内相对优化
    实践建议：Agent 数据最贵的不是 prompt，而是可执行轨迹。你可以先用强模型生成多条轨迹，再用规则 reward 筛选。这样比人工从零写轨迹更便宜，也比直接拿所有轨迹训练更安全。

    11. DPO 比较工具调用轨迹

    DPO 在 Agent 后训练里非常自然。普通 DPO 比较的是两个回答；Agent DPO 比较的是两条轨迹。

    11.1 普通 DPO

    prompt + chosen_response + rejected_response

    11.2 Agent DPO

    task + chosen_trajectory + rejected_trajectory

    11.3 三类最容易构造的 Agent DPO 数据

    类型一：正确工具 vs 错误工具

    Task: 计算 573 * 89。

Chosen:
{"name": "calculator", "arguments": {"expression": "573 * 89"}}
Observation: 50997
Final Answer: 573 * 89 = 50997。

Rejected:
{"name": "search_web", "arguments": {"query": "573 * 89"}}
Observation: 搜索结果若干
Final Answer: 573 * 89 大约是 51000。

    类型二：合法参数 vs 非法参数

    Task: 搜索文档中关于 RLHF 的定义。

Chosen:
{"name": "search_doc", "arguments": {"query": "RLHF definition"}}

Rejected:
{"name": "search_doc", "arguments": {"keyword": "RLHF definition"}}

    类型三：短成功轨迹 vs 长低效轨迹

    Task: 总结文档中 DPO 和 PPO 的区别。

Chosen:
1. search_doc({"query": "DPO PPO 区别"})
2. Final Answer

Rejected:
1. search_doc({"query": "DPO"})
2. search_doc({"query": "PPO"})
3. search_doc({"query": "RLHF"})
4. search_web({"query": "OpenAI PPO"})
5. Final Answer

    11.4 Agent DPO 的优点和局限

    优点局限
    比在线 RL 稳定。不主动探索新策略。
    可以利用 rejected 负轨迹。依赖 pair 数据质量。
    适合纠正工具选择和参数错误。复杂多轮任务构造 pair 成本高。
    工程成本低于 PPO/GRPO。无法直接利用环境实时反馈。
    阶段理解：Agent DPO 不是最终形态，但非常适合作为 SFT 后的第二阶段。它可以把“坏轨迹”变成有用训练信号，尤其适合训练工具选择、参数合法性、轨迹效率。

    12. GRPO 优化多条 Agent 轨迹

    GRPO 的核心直觉是：对同一个 prompt 生成一组回答，用组内平均 reward 做相对比较。迁移到 Agent 后，就是对同一个任务生成多条工具调用轨迹，然后比较哪条更好。

    12.1 Reasoning GRPO

    同一道数学题
  ↓
生成 8 个解答
  ↓
每个解答打 reward
  ↓
比组内平均分高的，advantage 为正
  ↓
比组内平均分低的，advantage 为负

    12.2 Agent GRPO

    同一个 Agent 任务
  ↓
生成 8 条工具调用轨迹
  ↓
每条轨迹执行工具并打 reward
  ↓
比较工具选择、参数、执行、最终任务
  ↓
高分轨迹被强化
  ↓
低分轨迹被压低

    12.3 Agent 轨迹的 reward 示例

    轨迹工具选择JSON执行最终答案成本总分
    A正确合法成功正确1 次调用0.95
    B正确合法成功错误1 次调用0.55
    C错误合法成功不准确2 次调用0.25
    D正确非法失败无0 次成功0.10

    如果组内平均分是 0.46，那么 A 和 B 的 advantage 为正，C 和 D 为负。模型会更倾向学习 A、B 中的行为，减少 C、D 的行为。更精细时，A 的优势会明显大于 B。

    12.4 Agent GRPO 的关键难点

    难点解释
    多轮 rollout 慢每条轨迹可能要多次调用工具，训练吞吐下降。
    reward 延迟最终任务成功可能要到最后才知道。
    credit assignment一条轨迹失败，究竟是哪一步导致失败？
    环境不稳定搜索、网页、API 可能返回变化结果。
    安全边界RL 可能鼓励模型钻工具漏洞。

    12.5 用 TRL / verl 怎么理解 Agent GRPO？

    框架概念Agent 中的对应
    prompt用户任务 + 工具列表 + 系统规则。
    completion一条完整工具调用轨迹。
    reward_func解析轨迹、执行工具、计算多项 reward。
    num_generations同一个任务采样多少条轨迹。
    advantage该轨迹相对同组其他轨迹好多少。
    KL防止模型偏离参考模型太远。
    核心理解：Reasoning GRPO 优化的是“哪条解题路径更好”；Agent GRPO 优化的是“哪条工具使用轨迹更好”。这就是从 DeepSeek-R1 迁移到 Tool-use Agent 的关键。

    13. Mini Tool-use Agent 小实验

    现在做一个最小实验，不依赖真实大模型训练，先用 Python 模拟 Agent RLVR 的核心链路：

    任务 → 生成候选工具调用 → 执行工具 → 计算 reward → 选择高分轨迹

    13.1 定义工具

    import json
import re


def calculator(expression: str) -> str:
    # 教学示例：真实系统不要直接 eval 用户输入
    if not re.fullmatch(r"[0-9+\-*/ ().]+", expression):
        raise ValueError("unsafe expression")
    return str(eval(expression))

TOOLS = {
    "calculator": calculator,
}

    13.2 定义候选轨迹

    candidates = [
    {
        "name": "calculator",
        "arguments": {"expression": "573 * 89"},
        "final_answer": "573 * 89 = 50997。"
    },
    {
        "name": "search_web",
        "arguments": {"query": "573 * 89"},
        "final_answer": "大约是 51000。"
    },
    {
        "name": "calculator",
        "arguments": {"expr": "573 * 89"},
        "final_answer": "50997。"
    },
    {
        "name": "calculator",
        "arguments": {"expression": "573 **"},
        "final_answer": "无法计算。"
    },
]

    13.3 定义 reward function

    def score_trajectory(traj):
    score = 0.0

    # 1. 工具选择
    if traj.get("name") == "calculator":
        score += 0.2
    else:
        return score

    # 2. 参数完整性
    args = traj.get("arguments", {})
    if isinstance(args, dict) and "expression" in args:
        score += 0.2
    else:
        return score

    # 3. 执行成功
    try:
        obs = calculator(args["expression"])
        score += 0.25
    except Exception:
        return score

    # 4. Observation 使用
    if obs in traj.get("final_answer", ""):
        score += 0.2

    # 5. 最终答案正确
    if obs == "50997":
        score += 0.25

    return min(score, 1.0)

    13.4 筛选高分轨迹

    scores = [(score_trajectory(t), t) for t in candidates]
scores = sorted(scores, key=lambda x: x[0], reverse=True)

for score, traj in scores:
    print(score, traj)

best = scores[0][1]
print("chosen trajectory:", best)

    Toy 代码真实 Agent RLVR
    candidates同一 prompt 下模型采样出的多条 trajectories。
    calculator真实工具、API、数据库、浏览器、代码环境。
    score_trajectoryreward function / reward manager。
    scores 排序Rejection Sampling 或 GRPO 组内 reward。
    bestSFT chosen 轨迹或 DPO chosen。
    你下一步可以扩展：
    1. 把 calculator 扩展成 search_doc。
    2. 让每个任务采样 4 条候选轨迹。
    3. 把最高分作为 chosen，最低分作为 rejected。
    4. 构造一个 Agent DPO 数据集。
    5. 再进入 TRL / verl 做真正训练。

    14. 总结：从 Function Picking 到 Agent RLVR

    这篇的主线可以压缩成一句话：

    Agent RLVR 的本质：把 Agent 的工具使用过程拆成可验证环节，并把这些可验证环节变成 reward，让模型从“会回答”进化到“会行动”。

    14.1 完整路线

    Tool-use SFT
  ↓
让模型学会工具调用格式

Function Picking
  ↓
让模型知道该选哪个工具

JSON / 参数 reward
  ↓
让工具调用变得可解析、可执行

Execution reward
  ↓
让模型关注工具是否真的成功

Observation reward
  ↓
让模型学会读取和利用工具返回

Trajectory penalty
  ↓
控制成本、防止循环调用

Rejection Sampling
  ↓
筛选高质量 Agent 轨迹

Agent DPO
  ↓
比较好轨迹和坏轨迹

Agent GRPO / RLVR
  ↓
同任务多轨迹采样，根据组内 reward 优化

    14.2 和 DeepSeek-R1 的迁移关系

    DeepSeek-R1 / Reasoning RLAgent RLVR
    题目 prompt。用户任务 + 工具列表。
    模型生成多条 reasoning answer。模型生成多条 tool-use trajectories。
    检查最终答案是否正确。检查工具、参数、执行、Observation、最终状态。
    组内比较 reasoning paths。组内比较 Agent trajectories。
    优化推理能力。优化行动能力。

    14.3 这篇之后应该怎么学？

    下一步学习内容目的
    读 ReActReasoning + Acting 交替格式。理解 Agent 轨迹结构。
    读 Toolformer模型如何自监督学习调用 API。理解工具调用数据生成。
    读 ToolRL工具学习中的 reward 设计。理解为什么细粒度 reward 重要。
    看 OpenAI Function CallingJSON schema / tool schema。理解真实工具调用接口。

    最后一句话：Agent 后训练不是“给模型加工具”这么简单。真正的关键是：工具调用轨迹能不能被记录、执行、验证、打分、筛选、比较和优化。只要这条链路打通，你就从普通后训练进入了 Agentic RL 的核心区域。

    15. 工程落地：从数据到训练的完整管线

    如果你后面真的要把 Agent RLVR 跑起来，不要直接从 GRPO 开始。更稳的路线是先做数据闭环，再做训练闭环。

    15.1 第一阶段：离线数据闭环

    定义工具 schema
  ↓
构造任务 prompts
  ↓
用强模型或当前模型生成 tool-use trajectories
  ↓
真实执行工具
  ↓
记录 observation
  ↓
用 reward function 打分
  ↓
筛选 chosen / rejected
  ↓
得到 SFT / DPO 数据
    步骤你要产出什么最低要求
    定义工具tool name、description、parameters。每个字段有类型、解释、必填说明。
    构造任务user prompts。每个 prompt 能判断是否完成。
    生成轨迹assistant tool_call + final answer。每条轨迹能被 parser 提取。
    执行工具observation。工具结果真实，不要伪造。
    打分reward breakdown。不要只保存总分，要保存每项分。
    筛选chosen / rejected。高分轨迹和低分轨迹原因明确。

    15.2 第二阶段：SFT 冷启动

    SFT 的目标不是让模型变成最强 Agent，而是让它先稳定输出工具调用格式。

    训练目标：
1. 会读工具列表
2. 会选择工具
3. 会生成合法 JSON
4. 会等待 Observation
5. 会基于 Observation 回答
6. 会在不需要工具时直接回答
    SFT 评价指标：这一阶段不要只看最终任务成功率。更应该看 tool_call parse rate、schema pass rate、required argument pass rate、unnecessary tool call rate、final answer grounding rate。

    15.3 第三阶段：DPO 纠偏

    DPO 适合解决 SFT 后仍然存在的偏好问题：

    SFT 后问题DPO pair 构造方式
    工具经常选错。chosen=正确工具；rejected=错误工具。
    参数字段经常错。chosen=合法参数；rejected=字段名错误/缺字段。
    调用工具太多。chosen=短成功轨迹；rejected=长成功轨迹。
    忽略 Observation。chosen=基于 Observation 回答；rejected=凭空回答。
    工具失败后不会恢复。chosen=换策略；rejected=重复同样错误。

    15.4 第四阶段：GRPO / RLVR 提升上限

    当模型已经能稳定调用工具后，再进入 GRPO / RLVR。这个阶段的目标不是“学格式”，而是“通过探索找到更好的工具使用策略”。

    同一个 task prompt
  ↓
采样 K 条轨迹
  ↓
每条轨迹执行工具
  ↓
计算 reward breakdown
  ↓
组内标准化 reward
  ↓
更新 policy model

    16. 常见失败模式与排查表

    Agent RLVR 最容易失败的原因，不是算法名选错，而是数据、环境、reward、parser 其中一个环节不稳定。

    16.1 模型不调用工具

    可能原因解决方式
    SFT 数据里无工具回答太多。提高需要工具的样本比例。
    工具描述不清楚。重写 tool description，明确适用场景。
    工具调用惩罚太强。降低 cost penalty。
    最终答案 reward 太容易拿。加入“未调用必要工具则不给最终分”的规则。

    16.2 模型疯狂调用工具

    可能原因解决方式
    只奖励执行成功，不惩罚成本。加入 tool_call penalty。
    没有 stop / final answer 训练。增加结束轨迹样本。
    Observation 使用 reward 设计错误。不要让复制大量 Observation 获得高分。
    工具失败后无终止规则。设置最大调用次数和失败终止。

    16.3 模型输出 JSON 经常坏

    可能原因解决方式
    训练格式不统一。统一 tool_call 模板。
    schema 太复杂。先从单工具、少字段开始。
    response 中混入解释文字。明确要求 tool_call 块内只输出 JSON。
    parser 太脆弱。使用边界标签提取 JSON，如 <tool_call>...</tool_call>。

    16.4 reward 越训越高，但真实效果没变好

    可能原因表现解决方式
    reward hacking模型学会迎合规则漏洞。加入人工抽查和隐藏测试。
    最终答案匹配太宽松错误答案也拿高分。改成严格解析或执行验证。
    Observation reward 太表面模型复制工具返回但不总结。加入事实一致性和任务完成检查。
    训练环境和测试环境不同训练集好，真实任务差。增加 held-out tools / held-out tasks。
    最重要的排查习惯：每次训练后随机抽 20 条轨迹人工看。不要只看 reward 曲线。Agent 的 reward 曲线可能很好看，但轨迹已经学歪了。

    17. 可直接复用的数据模板

    下面给一个你后面可以直接复制改造的 Agent 数据模板。

    17.1 单工具 SFT 模板

    {
  "id": "calc_sft_0001",
  "tools": [
    {
      "name": "calculator",
      "description": "执行精确数学计算。适用于加减乘除、括号表达式、小数计算。",
      "parameters": {
        "type": "object",
        "properties": {
          "expression": {
            "type": "string",
            "description": "需要计算的数学表达式"
          }
        },
        "required": ["expression"]
      }
    }
  ],
  "messages": [
    {"role": "user", "content": "计算 573 * 89。"},
    {
      "role": "assistant",
      "content": "{\"name\":\"calculator\",\"arguments\":{\"expression\":\"573 * 89\"}}"
    },
    {"role": "tool", "name": "calculator", "content": "50997"},
    {"role": "assistant", "content": "573 * 89 = 50997。"}
  ]
}

    17.2 Agent DPO 模板

    {
  "id": "agent_dpo_0001",
  "prompt": "计算 573 * 89。",
  "tools": ["calculator", "search_web", "translate"],
  "chosen": [
    "{\"name\":\"calculator\",\"arguments\":{\"expression\":\"573 * 89\"}}",
    "50997",
    "573 * 89 = 50997。"
  ],
  "rejected": [
    "{\"name\":\"search_web\",\"arguments\":{\"query\":\"573 * 89\"}}",
    "搜索结果若干",
    "结果大约是 51000。"
  ],
  "preference_reason": "chosen 使用了正确计算工具，rejected 选择了不必要的搜索工具且答案不精确。"
}

    17.3 GRPO / RLVR 轨迹模板

    {
  "task_id": "agent_rlvr_0001",
  "prompt": "计算 573 * 89。",
  "samples": [
    {
      "sample_id": "s1",
      "trajectory": [
        "{\"name\":\"calculator\",\"arguments\":{\"expression\":\"573 * 89\"}}",
        "50997",
        "573 * 89 = 50997。"
      ],
      "reward": 1.0
    },
    {
      "sample_id": "s2",
      "trajectory": [
        "{\"name\":\"calculator\",\"arguments\":{\"expr\":\"573 * 89\"}}",
        "50997。"
      ],
      "reward": 0.35
    }
  ]
}
    你真正要积累的是这三类数据：SFT 让模型学格式，DPO 让模型学偏好，GRPO/RLVR 让模型从多条轨迹中学会探索和优化。Agent 后训练不是一次完成，而是这三类数据循环迭代。
