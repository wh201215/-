本文定位：这是一篇非常适合学习 Agent 基础结构的博客。它把 LLM Agent 拆成三个核心组件：Planning 规划、Memory 记忆、Tool Use 工具使用，并进一步讨论了 ReAct、Reflexion、HuggingGPT、API-Bank、Generative Agents 等代表性工作。

总判断： 这篇文章适合当作“后训练 Agent”的第一篇系统精读材料。它虽然不是专门讲 SFT、DPO、GRPO 或 RLVR 的训练论文，但它把 Agent 的能力拆成了可以训练的模块：任务分解、动作选择、工具调用、记忆检索、反馈修正和长期规划。真正做 Agent 后训练时，训练目标并不是“让模型回答得更好”，而是让模型在多步轨迹中更稳定地选择正确动作。

一、Agent System Overview：Agent 系统总览

原文开头的核心判断是：当 LLM 作为 autonomous agent 的核心控制器时，它不只是一个文本生成器，而可以被理解为一个 general problem solver，即通用问题求解器。这个求解器要真正变成 Agent，不能只有语言生成能力，还需要规划、记忆和工具使用能力。

图 1：LLM-powered autonomous agent system 总览图。原图来源：Lilian Weng 博客。 打开原图

图片逐项解读

图中元素 中文解释 作用 后训练视角
Agent 智能体 中心控制单元，负责根据目标、记忆、工具和环境反馈决定下一步。 训练核心是让模型从当前状态中选择合适动作，而不是只生成最终答案。
Planning 规划 把复杂任务拆解为子目标，并决定执行顺序。 可以训练 task decomposition，即任务分解轨迹。
Memory 记忆 保存短期上下文与长期信息，使 Agent 能够利用历史经验。 训练模型何时写入、检索、压缩和使用记忆。
Tools 工具 连接日历、计算器、代码解释器、搜索等外部能力。 训练工具选择、参数生成、工具返回结果理解。
Action 动作 Agent 对环境采取的具体操作。 强化学习中的 action，可以用任务完成率或工具反馈评估。
Reflection / Self-critics / Chain of thoughts 反思 / 自我批评 / 思维链 帮助 Agent 检查过去行为、调整计划、改进下一步。 可作为 DPO 或 RLVR 中的中间监督信号。

中文总结

这张图是整篇文章最重要的入口。它说明：LLM Agent 不是“一个模型 + 一个 prompt”，而是一个由多个模块共同组成的系统。LLM 是 brain，即大脑；Planning 负责把目标拆成可操作步骤；Memory 负责存储和召回信息；Tools 负责让模型能够行动；Action 是模型对环境的具体操作。

如果从后训练角度看，这张图直接提示了训练对象：不要只训练“最终回答”，而要训练 Agent 在每一步如何规划、如何查找记忆、如何选择工具、如何根据反馈更新行动。

后训练启发： 这张图可以拆成五类训练任务：
  1. 规划训练：给定目标，输出子目标和执行顺序。
  2. 工具调用训练：给定上下文，选择合适工具并生成参数。
  3. 记忆检索训练：给定当前任务，从长期记忆中检索相关内容。
  4. 反思训练：给定失败轨迹，生成改进建议。
  5. 动作选择训练：给定 observation，预测下一步 action。

二、Planning：规划能力

原文认为，一个复杂任务通常包含许多步骤，Agent 必须知道这些步骤是什么，并提前规划。Planning 主要包含两个方向：Task Decomposition 任务分解，以及 Self-Reflection 自我反思。

1. Task Decomposition：任务分解

Task Decomposition 指的是把大任务拆成多个更小、更容易处理的子任务。原文提到 Chain of Thought，简称 CoT，即“思维链”，通过提示模型 step by step thinking，让模型把复杂问题分解成多个中间步骤。

Tree of Thoughts，简称 ToT，即“思维树”，则比 CoT 更进一步。CoT 通常是一条推理链，而 ToT 会在每一步生成多个可能的 thought，再通过 BFS 或 DFS 搜索不同路径。它的核心思想是：复杂任务不一定只有一条推理路径，模型应该探索多个候选路径。

方法 中文 核心思想 对 Agent 的意义
CoT, Chain of Thought 思维链 让模型一步一步推理。 适合线性分解问题。
ToT, Tree of Thoughts 思维树 每一步探索多个候选想法，形成树状搜索。 适合有多个可能路径的复杂任务。
LLM+P LLM 加规划器 让 LLM 把问题转成 PDDL，再交给传统规划器。 适合机器人、规划问题等有明确规划语言的任务。

2. 中文总结

规划能力的关键，不是模型能不能“说出计划”，而是它能不能把计划转成可以执行、可以检查、可以修正的步骤。很多 Agent Demo 失败，并不是因为模型语言能力不够，而是因为它把任务拆错了，或者无法根据失败反馈修改计划。

后训练启发： Planning 可以被训练成一种轨迹任务。数据格式可以是： 
用户目标 → 子任务列表 → 每个子任务需要的工具 → 执行顺序 → 完成标准
后训练时，可以用好计划和坏计划构造 preference data：合理拆解为 chosen，不合理拆解为 rejected。

三、ReAct：推理与行动结合

ReAct 是 Reasoning and Acting 的缩写，即“推理与行动结合”。原文指出，ReAct 把语言空间中的 reasoning traces 和任务空间中的 discrete actions 结合起来，使模型能够一边思考，一边和环境互动。

图 2：ReAct 在知识密集型任务和决策任务中的推理轨迹示例。原图来源：Lilian Weng 博客引用 Yao et al. 2023。 打开原图

图片解读

这张图展示了 ReAct 的基本工作方式：模型不是一次性回答问题，而是反复执行 Thought、Action、Observation 三个步骤。 

Thought: 当前我应该思考什么？
Action: 我应该调用什么动作或工具？
Observation: 环境或工具返回了什么？
Thought: 根据新观察，我下一步应该做什么？
Action: ...
Observation: ...
ReAct 元素 中文解释 功能
Thought 思考 模型对当前状态的分析、假设和下一步策略。
Action 动作 模型对环境采取的操作，例如搜索、点击、调用 API。
Observation 观察 环境或工具返回的信息。

中文总结

ReAct 的重要性在于，它把 Agent 的推理过程从“封闭的文本生成”变成“开放的环境交互”。普通 CoT 只是在模型内部推理，而 ReAct 允许模型用外部反馈修正推理。对于 Agent 来说，这比单纯写长思维链更接近真实任务。

后训练启发： ReAct 是 Agent 后训练最基础的数据格式之一。你可以把训练样本组织成： 
User Query
Thought
Action
Observation
Thought
Action
Observation
Final Answer
其中 Action 是否正确、Observation 是否被正确利用、Final Answer 是否完成任务，都可以作为评测点。

四、Reflexion：自我反思与动态记忆

原文把 Self-Reflection 称为自主 Agent 迭代改进的重要能力。Reflexion 是一个代表性框架，它为 Agent 加入动态记忆和自我反思机制，使 Agent 能从失败轨迹中总结经验，并改进下一轮行动。

图 3：Reflexion 框架。原图来源：Lilian Weng 博客引用 Shinn & Labash 2023。 打开原图

图片逐项解释

图中元素 中文解释 作用
Query 任务输入 用户或环境给定的问题。
LLM 大语言模型 根据任务和记忆生成动作。
Action 动作 Agent 执行的具体操作。
Environment 环境 Agent 所处任务环境，例如问答环境、游戏环境、网页环境。
Reward 奖励 任务成功或失败的反馈。
Heuristic 启发式判断 判断轨迹是否低效、是否幻觉、是否应该停止。
Reflection 反思 LLM 根据失败轨迹生成改进建议，并写入记忆。

核心理解

Reflexion 的关键不是“模型自言自语”，而是把失败经验转成可复用记忆。也就是说，Agent 在一次任务失败后，不只是丢掉失败轨迹，而是把失败原因总结成一段 reflection，在后续尝试时作为上下文使用。

图 4：Reflexion 在 AlfWorld 和 HotpotQA 上的实验结果。原图来源：Lilian Weng 博客引用 Shinn & Labash 2023。 打开原图

实验图解读

图中可以看到，加入 Reflection 后，Agent 的成功率随 trial 增加而提高。左图 AlfWorld 中，带有反思机制的成功率明显上升；右图 HotpotQA 中，without reflection 的成功率基本停滞，而 with reflection 持续改善。这说明反思机制的价值在于让 Agent 利用过去失败经验，避免重复错误。

后训练启发： Reflexion 可以转化为三类训练数据：
  1. 失败轨迹 → 反思：训练模型总结失败原因。
  2. 反思 + 新尝试 → 更好动作：训练模型利用 reflection 改进动作。
  3. 好反思 vs 坏反思：构造 DPO 数据，让模型偏向更有效的反思。

五、Memory：记忆机制

原文把 Memory 作为 Agent 的第二个核心组件。记忆不仅是聊天历史，也包括短期上下文、长期知识、经验轨迹和外部向量数据库。

图 5：人类记忆分类图。原图来源：Lilian Weng 博客。 打开原图

图片逐项解释

记忆类型 中文翻译 在人类认知中的含义 映射到 LLM Agent
Sensory Memory 感觉记忆 短暂保存视觉、听觉、触觉等感官印象。 可类比为模型对原始输入的 embedding 表征。
Short-term Memory / Working Memory 短期记忆 / 工作记忆 保存当前正在处理的信息。 对应上下文窗口中的信息，即 in-context learning。
Long-term Memory 长期记忆 长期保存事实、经验、技能。 对应外部向量数据库、知识库、文件系统等。
Explicit / Declarative Memory 显性记忆 / 陈述性记忆 可以被有意识回忆的事实和事件。 用户偏好、历史对话、任务记录、事实知识。
Implicit / Procedural Memory 隐性记忆 / 程序性记忆 技能和习惯，比如骑车、打字。 模型参数中习得的模式，或者工具使用习惯。

中文总结

Agent 的记忆系统不能简单理解为“把所有历史都塞进 prompt”。上下文窗口有限，信息越多，噪声越多。因此记忆系统的关键是：什么信息值得保存，什么信息应该检索，什么信息需要压缩，什么信息应该遗忘。

后训练启发： Memory 相关训练至少有四类：
  1. 写入判断：当前信息是否值得存入长期记忆？
  2. 检索判断：当前任务需要检索哪些历史信息？
  3. 记忆压缩：把长历史压缩成短摘要。
  4. 抗干扰训练：避免错误记忆、过期信息和无关信息影响决策。

六、MIPS：向量检索与长期记忆

原文指出，外部记忆可以缓解 Transformer 上下文长度有限的问题。常见做法是把信息编码成 embedding，然后存入 vector store，查询时通过 Maximum Inner Product Search，简称 MIPS，即最大内积搜索，找出最相关的记忆。

图 6：MIPS 算法速度与召回率对比。原图来源：Lilian Weng 博客引用 Google Blog 2020。 打开原图

图片解读

这张图的横轴是 Accuracy，也就是召回准确率；纵轴是 Speed，即每秒查询次数。图中不同曲线代表不同近似最近邻搜索算法。核心信息是：检索系统通常需要在速度和准确率之间做权衡。

算法 中文说明 作用
LSH 局部敏感哈希 把相似向量映射到同一桶中，加快搜索。
ANNOY 近似最近邻搜索库 使用随机投影树进行快速近似搜索。
HNSW 分层可导航小世界图 通过多层图结构快速找到近邻。
FAISS Facebook AI Similarity Search 常用高性能向量检索库。
ScaNN Scalable Nearest Neighbors Google 提出的高效向量检索方法。

中文总结

长期记忆不是“无限上下文”，而是“外部存储 + 高效检索”。Agent 要想利用长期记忆，就必须解决三个问题：存什么、怎么检索、怎么使用检索结果。很多 Agent 出错不是因为没有记忆，而是因为检索到了错误或无关记忆。

后训练启发： 可以训练模型生成更好的检索 query，也可以训练模型判断检索结果是否可用。比如： 
任务 → 生成检索关键词 → 返回候选记忆 → 判断相关性 → 整合进下一步行动
这类数据非常适合做 RAG-Agent 的 SFT 或偏好优化。

七、Tool Use：工具使用

原文把 Tool Use 作为 Agent 的第三个核心组件。工具使用让 LLM 能够突破模型参数本身的限制，例如获取最新信息、运行代码、访问私有数据、调用计算器、调用专业模型等。

为什么工具重要？

大模型的知识主要来自训练数据，训练后模型参数很难频繁更新。因此，当前信息、实时数据、精确计算、数据库查询、代码执行等任务，不能只靠模型内部知识完成。工具调用让模型从“会说”变成“会做”。

工具类型 中文 例子 训练难点
Search API 搜索工具 网页搜索、文档搜索 什么时候搜索、搜索什么关键词。
Calculator 计算器 数学计算、单位换算 参数提取必须准确。
Code Interpreter 代码解释器 运行 Python、分析数据 要能读懂报错并修正。
Database API 数据库接口 查订单、查用户、查论文 字段、权限、查询条件要准确。
Specialist Model 专家模型 图像识别、语音识别、翻译模型 需要选择合适模型并解释结果。

MRKL、TALM、Toolformer 的区别

方法 中文解释 核心思想
MRKL 模块化推理、知识与语言系统 LLM 作为路由器，把任务分配给神经模块或符号模块。
TALM 工具增强语言模型 微调语言模型，让它学会调用外部工具 API。
Toolformer 工具自学习模型 模型通过判断 API 调用是否能提升输出质量，自动扩展工具调用数据。
后训练启发： 工具调用训练最关键的不是让模型知道“有工具”，而是让模型学会：
  1. 判断是否需要工具；
  2. 选择哪个工具；
  3. 生成合法参数；
  4. 理解工具返回结果；
  5. 根据结果决定是否再次调用工具。

八、HuggingGPT：LLM 作为任务控制器

HuggingGPT 是原文介绍的典型工具使用案例。它让 ChatGPT 作为 task planner，即任务规划器，从 HuggingFace 平台选择合适模型完成任务。

图 7：HuggingGPT 工作流程。原图来源：Lilian Weng 博客引用 Shen et al. 2023。 打开原图

图片逐项解释

阶段 英文 中文解释 作用
I Task Planning 任务规划 LLM 把用户请求拆成多个子任务。
II Model Selection 模型选择 LLM 从候选模型中选择适合每个子任务的模型。
III Task Execution 任务执行 被选中的专家模型执行具体任务。
IV Response Generation 回答生成 LLM 汇总各模型结果，给用户最终回答。

中文总结

HuggingGPT 说明了 Agent 的一种典型模式：LLM 不一定亲自完成所有任务，而是作为 controller，即控制器，负责理解任务、拆解任务、选择工具、整合结果。这个模式和现在的多工具 Agent 很接近。

后训练启发： HuggingGPT 可以拆成四个训练任务：
  1. 任务解析：用户请求 → 子任务 JSON。
  2. 依赖判断：判断子任务之间的先后关系。
  3. 模型选择：子任务 → 最合适工具或模型。
  4. 结果汇总：工具结果 → 面向用户的最终回答。

九、API-Bank：工具调用评测

API-Bank 是用于评测 tool-augmented LLM 的 benchmark。它关注模型是否能判断需要 API、找到合适 API、阅读文档、生成 API 调用，并根据 API 返回结果回答用户。

图 8：API-Bank 中 LLM 调用 API 的伪代码流程。原图来源：Lilian Weng 博客引用 Li et al. 2023。 打开原图

图片解读

这张图描述了模型调用 API 的完整流程。首先判断是否需要 API；如果需要，就搜索 API；找到 API 后读取文档；然后生成 API call；执行后检查返回结果；如果不满意，继续修改调用；最后基于 API 返回生成用户回答。

API-Bank 的三层能力

层级 英文 中文解释 能力要求
Level 1 Call API 调用 API 给定 API 描述，模型判断是否调用、如何调用、如何回应。
Level 2 Retrieve API 检索 API 模型需要搜索可能可用的 API，并阅读文档学习使用。
Level 3 Plan API Calls 规划 API 调用 面对模糊任务，模型需要多步调用多个 API。

中文总结

API-Bank 的价值在于，它把工具调用能力拆成了可评测的步骤。Agent 不是只要最后回答正确就行，还要看中间决策是否正确：是否该调用 API？调用哪个 API？参数是否正确？返回结果是否被正确使用？

后训练启发： API-Bank 的流程可以直接转成 reward 设计：
  1. 是否需要 API：二分类奖励；
  2. API 选择是否正确：工具选择奖励；
  3. 参数是否正确：schema 奖励；
  4. 是否根据结果再次调用：多轮策略奖励；
  5. 最终回答是否满足用户需求：任务完成奖励。

十、Generative Agents：生成式智能体仿真

Generative Agents 是一个非常有代表性的 Agent 案例。它让 25 个由 LLM 控制的虚拟角色生活在沙盒环境中，并通过记忆、规划和反思机制产生类似人类的社会行为。

图 9：Generative Agents 架构。原图来源：Lilian Weng 博客引用 Park et al. 2023。 打开原图

图片逐项解释

图中元素 中文解释 作用
Perceive 感知 Agent 接收环境中的新事件和观察。
Memory Stream 记忆流 长期保存 Agent 的观察、事件和经验。
Retrieve 检索 从记忆流中取回与当前情境相关的记忆。
Retrieved Memories 被检索出的记忆 作为当前行动的上下文。
Reflect 反思 把低层事件总结成高层理解。
Plan 规划 根据记忆和当前情境制定行动计划。
Act 行动 Agent 在环境中执行具体行为。

中文总结

Generative Agents 的重点是：记忆不是简单记录，而是要参与行为生成。Agent 会感知环境，把观察写入记忆流；再根据相关性、时间近近度和重要性检索记忆；然后反思形成更高层理解；最后根据这些信息规划和行动。

后训练启发： 这类系统适合训练“长期任务中的行为一致性”。比如：
  1. 给定角色历史记忆，预测下一步行动；
  2. 给定一批观察，生成高层 reflection；
  3. 给定当前情境，检索相关记忆；
  4. 根据长期记忆保持角色行为一致。

十一、Challenges：Agent 的主要挑战

原文最后总结了 LLM-centered agents 的几个常见限制。这部分对后训练 Agent 特别重要，因为它指出了 Agent 失败的根源。

挑战 中文解释 表现 后训练方向
Finite context length 有限上下文长度 历史信息、工具返回、API 文档、长任务状态无法全部放进 prompt。 训练压缩、检索、记忆选择。
Long-term planning 长期规划困难 任务变长后，模型难以持续维护计划，也难以处理意外错误。 训练多步轨迹、失败恢复、计划修正。
Reliability of natural language interface 自然语言接口可靠性不足 模型输出格式错误、工具参数错误、无法稳定解析。 训练结构化输出、schema 遵循、工具调用格式。
关键判断： 很多 Agent 系统的难点不是“模型不会说”，而是“模型输出不能稳定变成可执行动作”。因此，后训练 Agent 时要特别关注结构化输出、工具参数、错误恢复和停止条件。

十二、后训练 Agent 的学习路线

根据这篇文章，可以把后训练 Agent 的路线拆成六个阶段。

阶段 训练目标 数据形式 适合方法
1. 工具调用格式 让模型输出合法工具调用。 任务 → 工具名 + 参数 JSON SFT
2. 工具选择 让模型判断何时用哪个工具。 上下文 → 正确工具 / 错误工具 SFT / DPO
3. 任务分解 让模型把复杂目标拆成子任务。 用户目标 → 子目标列表 SFT / DPO
4. 多步 ReAct 轨迹 训练模型根据 Observation 继续行动。 Thought / Action / Observation 序列 SFT / GRPO / PPO
5. 自我反思 训练模型从失败轨迹中生成改进建议。 失败轨迹 → Reflection SFT / DPO
6. 可验证奖励优化 用任务完成结果优化模型。 轨迹 → reward RLVR / GRPO

一个可复现的小项目设计

项目名称：Mini Tool-use Agent 后训练

  1. 准备三个工具：calculator、search_doc、run_python。
  2. 构造 200 条 SFT 数据，让模型学习什么时候调用哪个工具。
  3. 构造 chosen / rejected 数据，例如正确参数 vs 错误参数。
  4. 用 DPO 优化工具调用偏好。
  5. 加入简单可验证 reward，例如计算结果正确给 1，JSON 格式错误给 0。
  6. 最后训练多步 ReAct 轨迹，让模型根据工具返回继续行动。

十三、术语中英对照

英文术语 中文译法 解释
Autonomous Agent 自主智能体 能够根据目标、环境反馈和内部状态自主行动的系统。
Agent Brain 智能体大脑 在 LLM Agent 中，LLM 通常作为核心控制器或决策器。
Planning 规划 把复杂任务拆成多个子目标，并确定执行顺序。
Task Decomposition 任务分解 把大任务拆成更小、更容易处理的子任务。
Self-Reflection 自我反思 根据过去行动和错误生成改进建议。
Memory 记忆 保存短期上下文、长期经验和外部知识。
Short-term Memory 短期记忆 当前上下文窗口中的信息。
Long-term Memory 长期记忆 外部向量库、文件系统、数据库等长期存储。
Tool Use 工具使用 模型调用外部工具、API、代码执行器或专业模型。

RLVR 可验证奖励强化学习 使用可自动验证的结果作为奖励信号，例如测试通过、答案正确。

十四、最终总结

Lilian Weng 这篇文章的核心价值在于，它把 LLM Agent 从一个模糊概念拆成了清晰结构：LLM 作为大脑，Planning 负责规划，Memory 负责经验保存与召回，Tool Use 负责外部行动，Reflection 负责从错误中学习。

从后训练 Agent 的角度看，这篇文章的启发是：我们不应该只训练模型生成最终答案，而应该训练模型在整个任务轨迹中的行为能力。也就是训练它如何分解任务、如何选择工具、如何解释工具返回、如何反思失败、如何更新计划、如何判断停止。

一句话概括：后训练 Agent 的核心不是让模型“更会说”，而是让模型在“观察—行动—反馈”的循环中更稳定地做出正确决策。
