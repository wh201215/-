学习定位： 这篇文章适合作为“后训练路线”的第一篇。它不是讲 Agent 架构，也不是只讲某一个算法， 而是从工程学习角度解释：模型为什么需要 post-training，post-training 数据是什么样， SFT、DPO、PPO、GRPO 各自在路线中解决什么问题。

0. 这篇博客应该怎么读？

这篇 PyTorch 博客的重点不是让你立刻推公式，而是先建立一条后训练主线： 

Base Model 基座模型
  ↓
Post-training Data Format 后训练数据格式
  ↓
SFT 监督微调
  ↓
Preference Data 偏好数据
  ↓
DPO / RLHF 偏好优化
  ↓
Online RL / PPO 在线强化学习
  ↓
Reward Pipeline 奖励管线
  ↓
Reasoning / GRPO 推理后训练
一句话理解： 预训练让模型“会补全文本”；后训练让模型“会按照人的任务方式做事”。

0.1 为什么这篇适合做后训练路线第一篇？

学习问题 这篇博客能解决什么
Post-training 到底是什么？ 它把后训练解释为预训练之后，让模型学会对话、推理、规划和交互的阶段。
SFT 和预训练有什么区别？ 它解释了 SFT 仍然是 next-token prediction，但只对 assistant response 计算 loss。
DPO 和 RLHF 是什么关系？ 它把 DPO 放在偏好优化路线中讲，说明为什么它比 PPO 更容易上手。
PPO 为什么复杂？ 它从基础 policy gradient 讲到 advantage、ratio、clip 和 trust region。
GRPO 为什么火？ 它解释了 GRPO 如何用组内相对奖励替代 critic。
怎么接 Agent 后训练？ 它的 reward pipeline 思路可以直接迁移到工具调用、代码 Agent 和多步任务轨迹。

1. 后训练数据格式：模型到底看到了什么？

很多人学后训练会直接跳到 SFT、DPO、PPO，但其实第一步应该先理解数据格式。 因为 LLM 的底层接口没有变：模型接收一串 token，然后预测下一个 token。 所谓“聊天模型”，本质上是通过特殊格式把 system、user、assistant 等角色写进同一段 token 序列中。

图 1：聊天界面示例。来源：PyTorch 官方博客《A Primer on LLM Post-Training》。 如果图片不显示，请打开原图： https://pytorch.org/wp-content/uploads/2025/08/1-3.png

1.1 这张图表面上在讲什么？

图中是一个普通聊天界面：用户问问题，助手回答；用户继续追问，助手继续回答。 但原文真正想说明的是：聊天界面只是人看到的 UI，模型实际看到的是一段被特殊 token 包装过的文本。

你在界面中看到的 模型实际需要看到的 为什么重要
用户说：“What is the capital of France?” 一段带有 user role 标记的 token。 模型必须知道这是用户输入，不是助手要生成的内容。
助手回答：“Paris.” 一段带有 assistant role 标记的 token。 模型只应该学习生成 assistant 部分。
多轮对话 system + user + assistant + user + assistant 的完整序列。 模型根据历史上下文继续生成，而不是孤立回答。
助手停止回答 end-of-turn token。 模型必须学会在合适的位置停止。

1.3 这一节和后训练的关系

后训练第一步不是算法，而是格式。如果格式不稳定，后面 SFT、DPO、RL 都会乱。 对于普通聊天模型，格式决定谁是 system、谁是 user、谁是 assistant； 对于 Agent 模型，格式还要决定谁是 tool、谁是 environment、谁是 observation。

普通聊天后训练 Agent 后训练
system / user / assistant system / user / assistant / tool / environment
训练 assistant 回答 训练 assistant 的 Thought、Action、Final Answer
用户消息不计算 loss 工具返回 Observation 也不应该计算 loss
模型学习什么时候停止 模型学习什么时候继续调用工具、什么时候 finish
这一节你要记住： 后训练不是直接把问答扔给模型。你必须先设计数据格式。 格式设计得清楚，模型才知道“谁说了什么”“自己应该生成什么”“什么时候停止”“什么时候调用工具”。
