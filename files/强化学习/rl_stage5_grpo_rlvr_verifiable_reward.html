强化学习大模型路线｜Stage 5：GRPO / RLVR / Verifiable Reward 数学推导

学习定位：Stage 4 讲了 DPO，也就是用 chosen / rejected 偏好数据直接优化语言模型。Stage 5 进入 GRPO / RLVR，也就是基于可验证奖励的强化学习。

这一阶段和前面最大的区别是：

\[ \text{偏好数据} \quad\Longrightarrow\quad \text{可验证任务奖励} \]

在数学、代码、工具调用、SQL、表格问答、Agent 任务中，很多时候不需要人类主观判断哪个回答更好，而是可以用程序自动判断：

  • 最终答案是否正确；
  • 代码是否通过测试；
  • JSON 格式是否可解析；
  • 工具名称是否正确；
  • 参数是否完整；
  • SQL 执行结果是否匹配；
  • 网页 Agent 是否完成目标。

Stage 5 目标：

  1. 理解 RLVR 和 RLHF / DPO 的区别；
  2. 掌握 Verifiable Reward Function 的设计方式；
  3. 理解 GRPO 为什么用 group 内相对奖励替代 Value Model；
  4. 推导 group relative advantage；
  5. 理解 GRPO objective 与 PPO objective 的关系；
  6. 学会设计数学、代码、工具调用任务的 reward；
  7. 为 Stage 6 的多轮 Agent RL 做准备。

1. 为什么 DPO 之后要学 GRPO / RLVR？

DPO 适合这样的数据：

\[ (x,y_w,y_l) \]

也就是 prompt、chosen response、rejected response。

但很多推理任务的训练信号不是“哪个回答更受偏好”，而是“这个回答是否真的正确”。

任务 DPO 的训练信号 RLVR 的训练信号
开放式问答 人类更喜欢哪个回答 不一定容易验证
数学推理 可以比较两个解答好坏 最终答案可以自动验证
代码生成 可以比较代码可读性 单元测试是否通过
工具调用 可以比较回答是否合适 工具名、参数、执行结果是否正确
SQL Agent 可以比较解释质量 SQL 执行结果是否匹配
一句话： DPO 适合“偏好优化”；GRPO / RLVR 更适合“可验证任务优化”。

1.1 为什么可验证任务很重要？

因为它可以降低人工标注成本。 

传统偏好优化：
模型生成回答 A / B
  ↓
人类标注哪个更好
  ↓
训练 DPO / Reward Model

可验证奖励强化学习：
模型生成多个回答
  ↓
程序自动检查答案是否正确
  ↓
直接给 reward
  ↓
用 GRPO / RL 更新模型
关键转变： 从“让人判断好坏”转为“让验证器判断成败”。

2. RLVR：什么是可验证奖励

RLVR 全称：

\[ \text{Reinforcement Learning with Verifiable Reward} \]

中文可以理解为：

基于可验证奖励的强化学习。

它的核心不是某个特定算法，而是一种训练信号设计方式。

2.1 Verifiable Reward 的定义

给定 prompt \(x\)，模型生成回答 \(y\)，我们写一个验证函数：

\[ R(x,y)\in \mathbb{R} \]

它自动输出 reward。

例如数学题：

\[ R(x,y)= \begin{cases} 1, & \text{最终答案正确}\\ 0, & \text{最终答案错误} \end{cases} \]

代码题：

\[ R(x,y)= \begin{cases} 1, & \text{全部单元测试通过}\\ 0, & \text{部分或全部测试失败}\\ -1, & \text{代码无法运行} \end{cases} \]
判断一个任务能不能做 RLVR： 问自己一句：我能不能写一个程序，自动判断模型输出是否成功？

2.2 可验证奖励和 Reward Model 的区别

维度 Reward Model Verifiable Reward
来源 神经网络模型 规则、测试器、执行器、验证器
训练成本 需要偏好数据训练 不一定需要人工偏好
适合任务 主观偏好、帮助性、安全性 数学、代码、工具调用、SQL
风险 Reward Model 偏差和 reward hacking 规则漏洞和 reward hacking
注意： 可验证奖励不是万能的。如果验证器设计得很粗糙，模型也会钻漏洞。

3. Verifiable Reward Function 怎么设计

一个 reward function 通常包括三层：

层级 检查什么 例子
格式层 输出是否可解析 是否包含 Final Answer，JSON 是否合法
内容层 答案或参数是否正确 数字答案、工具参数、SQL 结果
行为层 是否符合任务策略 是否少调用工具、是否没有编造

3.1 数学题 reward

def math_reward(output, reference_answer):
    pred = extract_final_answer(output)

    if pred is None:
        return -0.2

    if normalize(pred) == normalize(reference_answer):
        return 1.0

    return 0.0

3.2 代码题 reward

def code_reward(generated_code, tests):
    if not can_compile(generated_code):
        return -1.0

    passed, total = run_tests(generated_code, tests)

    return passed / total

3.3 工具调用 reward

def tool_reward(output, ground_truth):
    reward = 0.0

    if is_valid_json(output):
        reward += 0.2
    else:
        return -1.0

    if tool_name(output) == ground_truth.tool_name:
        reward += 0.4

    if required_args_complete(output):
        reward += 0.2

    if args_match(output, ground_truth.args):
        reward += 0.4

    return reward
实践原则： reward 先简单、稳定、可解释，再逐步加细。不要一开始设计十几个复杂奖励项。

4. 从 PPO 到 GRPO：为什么要去掉 Value Model

Stage 2 里 PPO 需要 advantage：

\[ \hat{A}_t \]

通常用 Value Model / Critic 来估计：

\[ V_\phi(s_t) \]

然后：

\[ \hat{A}_t = \hat{R}_t - V_\phi(s_t) \]

4.1 大模型 PPO 的负担

组件 负担
Policy Model 正在训练的大模型
Reference Model 计算 KL，需要额外显存或推理
Reward Model / Reward Function 给 response 打分
Value Model / Value Head 估计每个 token 的 value
GAE 需要 token-level value 和 reward
问题： 在大模型 reasoning 任务中，训练一个稳定的 Value Model 很重，而且 value 估计不准会直接影响 PPO。

4.2 GRPO 的核心简化

GRPO 的核心想法：

同一个 prompt 生成多个 response，用这一组 response 的相对 reward 来估计 advantage，不再依赖单独的 Value Model。 
同一道题 x
  ↓
生成 G 个回答 y1, y2, ..., yG
  ↓
验证器分别给 reward R1, R2, ..., RG
  ↓
计算组内平均和标准差
  ↓
把每个 reward 转成组内相对 advantage
  ↓
更新模型

5. Group Sampling：同一题多采样

GRPO 里的 Group 指的是：对同一个 prompt 采样多个回答。

\[ y_1,y_2,\ldots,y_G \sim \pi_{\theta_{\mathrm{old}}}(\cdot\mid x) \]

其中 \(G\) 是 group size。

5.1 为什么要同一题多采样？

因为我们要在同一道题内部比较谁表现更好。

例如同一道数学题： 

x: 1497 × 63 = ?

模型采样 4 个回答：

y1: 94311      reward = 1
y2: 94211      reward = 0
y3: 94301      reward = 0
y4: 94311      reward = 1

组内平均 reward：

\[ \bar{R} = \frac{1+0+0+1}{4} = 0.5 \]

于是高于平均分的回答是好样本，低于平均分的回答是差样本。

5.2 Group 的意义

如果只采一个回答 如果采多个回答
只知道这个回答得多少分 知道同一道题下哪些回答相对更好
需要 Value Model 做 baseline 可以用组内平均作为 baseline
reward 方差可能大 组内标准化后更稳定
一句话： GRPO 用同题多采样制造一个“小型比较环境”，用组内相对表现替代 value baseline。

6. Group Relative Advantage 逐步推导

假设对同一个 prompt \(x\)，采样 \(G\) 个回答：

\[ \{y_i\}_{i=1}^{G} \]

每个回答得到一个 reward：

\[ R_i=R(x,y_i) \]

6.1 组内平均 reward

\[ \bar{R} = \frac{1}{G} \sum_{i=1}^{G}R_i \]

它相当于这一组回答的 baseline。

6.2 最简单的相对 advantage

\[ \hat{A}_i = R_i-\bar{R} \]
情况 含义
\(\hat{A}_i \gt 0\) 回答 \(y_i\) 比同组平均更好，应该提高概率
\(\hat{A}_i = 0\) 回答 \(y_i\) 等于同组平均，更新信号弱
\(\hat{A}_i \lt 0\) 回答 \(y_i\) 比同组平均更差，应该降低概率

6.3 标准化 advantage

实践中常用标准化：

\[ \hat{A}_i = \frac{ R_i-\bar{R} }{ \sigma_R+\epsilon } \]

其中：

\[ \sigma_R = \sqrt{ \frac{1}{G} \sum_{i=1}^{G} (R_i-\bar{R})^2 } \]

\(\epsilon\) 是一个很小的数，防止除以 0。

6.4 为什么要除以标准差？

不标准化 标准化后
reward 尺度影响梯度大小 不同 prompt 的 reward 尺度更可比
有些题 reward 差异大，更新过猛 更新更稳定
有些题 reward 差异小，学习信号弱 相对差异被放大
注意： 如果同组 reward 全都一样，标准差接近 0，advantage 会没有有效区分。比如全对或全错，都不适合提供组内学习信号。

6.5 一个完整数字例子

假设：

\[ R=[1,0,0,1] \]

平均值：

\[ \bar{R}=0.5 \]

不标准化 advantage：

\[ A=[0.5,-0.5,-0.5,0.5] \]

如果 reward 是：

\[ R=[1.0,0.2,0.0,0.8] \]

平均值：

\[ \bar{R}=0.5 \]

不标准化 advantage：

\[ A=[0.5,-0.3,-0.5,0.3] \]
GRPO 的 advantage 本质： 不是问这个回答绝对好不好，而是问它在同一道题的多个回答中相对好不好。

7. GRPO Objective 完整解释

GRPO 的 policy objective 和 PPO 很像，也使用新旧策略概率比。

7.1 PPO 的 ratio

\[ r_i(\theta) = \frac{ \pi_\theta(y_i\mid x) }{ \pi_{\theta_{\mathrm{old}}}(y_i\mid x) } \]

如果按 token 级别写：

\[ r_{i,t}(\theta) = \frac{ \pi_\theta(y_{i,t}\mid x,y_{i,\lt t}) }{ \pi_{\theta_{\mathrm{old}}}(y_{i,t}\mid x,y_{i,\lt t}) } \]

7.2 GRPO 的 clipped objective

一种常见形式可以写成：

\[ L_{\mathrm{GRPO}}(\theta) = \mathbb{E}_{x,\{y_i\}} \left[ \frac{1}{G} \sum_{i=1}^{G} \frac{1}{|y_i|} \sum_{t=1}^{|y_i|} \min \left( r_{i,t}(\theta)\hat{A}_i, \mathrm{clip} \left( r_{i,t}(\theta), 1-\epsilon, 1+\epsilon \right) \hat{A}_i \right) \right] \]

其中：

符号 含义
\(G\) 每个 prompt 采样的回答数量
\(y_i\) 第 \(i\) 个回答
\(|y_i|\) 第 \(i\) 个回答的 token 长度
\(r_{i,t}(\theta)\) 新旧策略对第 \(i\) 个回答第 \(t\) 个 token 的概率比
\(\hat{A}_i\) 第 \(i\) 个回答的组内相对 advantage
\(\epsilon\) clip 范围参数

7.3 为什么 \(\hat{A}_i\) 对同一回答的所有 token 相同？

在很多 RLVR 任务中，reward 是 response-level 的。例如一个数学答案最后对不对。

所以对同一个回答 \(y_i\)，它的所有 token 共享同一个 group advantage：

\[ \hat{A}_{i,1} = \hat{A}_{i,2} = \cdots = \hat{A}_{i,|y_i|} = \hat{A}_i \]
这也是局限： 如果只有最终答案奖励，那么模型知道整段回答好或坏，但不知道具体哪个 token 或哪一步推理导致好坏。

7.4 和 PPO 的关系

部分 PPO GRPO
ratio 新旧策略概率比 仍然使用
clip 限制更新幅度 仍然使用
advantage 通常由 Value Model + GAE 得到 由组内 reward 相对比较得到
Value Model 通常需要 通常不需要
一句话： GRPO 可以看成“去掉 Value Model，用 group reward 估计 advantage 的 PPO 类方法”。

8. GRPO 中的 KL 约束

和 RLHF-PPO 一样，GRPO 也需要防止模型偏离 reference model 太远。

8.1 为什么还需要 KL？

如果只最大化可验证 reward，模型可能学会：

  • 输出很奇怪但能骗过格式检查的文本；
  • 过度延长推理过程；
  • 只学会套答案格式；
  • 利用验证器漏洞；
  • 语言能力和对话能力退化。

所以通常要加 reference model 约束：

\[ \mathrm{KL} \left( \pi_\theta \| \pi_{\mathrm{ref}} \right) \]

8.2 token-level KL penalty

对于第 \(i\) 个回答第 \(t\) 个 token：

\[ \mathrm{kl}_{i,t} = \log \pi_\theta(y_{i,t}\mid x,y_{i,\lt t}) - \log \pi_{\mathrm{ref}}(y_{i,t}\mid x,y_{i,\lt t}) \]

可以把 GRPO objective 写成带 KL 惩罚的形式：

\[ L(\theta) = L_{\mathrm{GRPO}}(\theta) - \beta \mathbb{E} \left[ \mathrm{KL} \left( \pi_\theta \| \pi_{\mathrm{ref}} \right) \right] \]

如果深度学习框架做 loss 最小化，则通常写成负的 policy objective 加 KL penalty。

一句话： GRPO 负责朝高可验证 reward 的方向走；KL 负责不让模型走得太偏。

9. 数学题 RLVR 例子

9.1 任务设定

Prompt:
请计算 1497 × 63，并把最终答案写成：
Final Answer: 数字

模型生成多个回答：

回答 最终答案 reward
\(y_1\) 94311 1
\(y_2\) 94211 0
\(y_3\) 94301 0
\(y_4\) 94311 1

9.2 Reward Function

import re

def extract_answer(output):
    match = re.search(r"Final Answer:\s*(-?\d+)", output)
    if match is None:
        return None
    return int(match.group(1))

def reward_fn(output, answer):
    pred = extract_answer(output)
    if pred is None:
        return -0.2
    if pred == answer:
        return 1.0
    return 0.0

9.3 Group Advantage

\[ R=[1,0,0,1] \]
\[ \bar{R}=0.5 \]
\[ \hat{A}=[0.5,-0.5,-0.5,0.5] \]

训练时，\(y_1,y_4\) 的 token 概率会被提高，\(y_2,y_3\) 的 token 概率会被降低。

注意： 这里 reward 只看最终答案，不保证中间推理一定正确。如果需要更高质量推理，后面要引入 process reward 或验证中间步骤。

10. 代码任务 RLVR 例子

10.1 任务设定

Prompt:
写一个 Python 函数，判断一个字符串是否是回文。

模型生成代码后，用测试用例验证： 

tests = [
    ("aba", True),
    ("abc", False),
    ("", True),
    ("a", True),
    ("abba", True)
]

10.2 Reward Function

def code_reward(code, tests):
    try:
        fn = compile_and_load_function(code)
    except Exception:
        return -1.0

    passed = 0
    for inp, expected in tests:
        try:
            pred = fn(inp)
            if pred == expected:
                passed += 1
        except Exception:
            pass

    return passed / len(tests)

10.3 为什么代码任务很适合 RLVR？

特点 说明
答案不唯一 同一个需求可以有多种正确代码
可执行 代码可以直接运行
可验证 单元测试可以自动判断对错
reward 可细分 通过多少测试就给多少分
代码任务的坑： 如果测试用例太少，模型可能学会硬编码测试样例，而不是真正解决问题。

11. 工具调用 / Agent RLVR 例子

11.1 Function Calling 任务

给模型一个用户请求和可用工具，让它判断应该调用哪个工具。 

User:
帮我查一下今天北京天气。

Available tools:
get_weather(city, date)
calculator(expression)
search_web(query)

正确调用应该是： 

{
  "tool": "get_weather",
  "arguments": {
    "city": "北京",
    "date": "today"
  }
}

11.2 工具调用 reward

def tool_call_reward(output, gold):
    reward = 0.0

    obj = parse_json(output)
    if obj is None:
        return -1.0

    reward += 0.2

    if obj.get("tool") == gold["tool"]:
        reward += 0.4

    if has_required_args(obj, gold):
        reward += 0.2

    if args_exact_or_equivalent(obj, gold):
        reward += 0.4

    return reward

11.3 不该调用工具的情况

有时正确行为是“不调用工具”，比如用户信息不足。 

User:
帮我算一下从 A 到 B 的距离。

Tool:
calculate_distance(start_lat, start_lon, end_lat, end_lon)

如果用户没有提供经纬度，正确行为可能是： 

<no_function_call>

这时 reward 要能惩罚“乱调用工具”： 

if should_not_call_tool and model_called_tool:
    reward -= 1.0
Agent RLVR 核心： 工具是否调用、何时调用、调用哪个、参数是否正确、执行结果是否成功，都可以成为可验证 reward。

12. Format Reward 与 Accuracy Reward

RLVR 中常见两类奖励：

奖励类型 中文 检查什么
Format Reward 格式奖励 是否按指定格式输出，是否可解析
Accuracy Reward 准确性奖励 答案是否正确，工具执行是否成功

12.1 Format Reward

例如要求模型输出： 

Final Answer: 94311

如果模型输出： 

答案应该是 94311。

虽然内容对，但格式不符合，抽取器可能失败。因此可以设置：

\[ R_{\mathrm{format}}= \begin{cases} 0.2, & \text{格式正确}\\ -0.2, & \text{格式错误} \end{cases} \]

12.2 Accuracy Reward

\[ R_{\mathrm{accuracy}}= \begin{cases} 1.0, & \text{答案正确}\\ 0.0, & \text{答案错误} \end{cases} \]

12.3 总 reward

\[ R = R_{\mathrm{accuracy}} + \alpha R_{\mathrm{format}} \]

其中 \(\alpha\) 控制格式奖励权重。

注意： 格式奖励不能太大。如果格式奖励超过准确性奖励，模型可能只学会格式，而不是学会解题。

13. Outcome Reward 与 Process Reward

13.1 Outcome Reward

Outcome Reward 只看最终结果。

\[ R_{\mathrm{outcome}}(x,y) = \mathbb{I} [ \text{final answer correct} ] \]

优点是简单，缺点是中间过程没有直接监督。

13.2 Process Reward

Process Reward 会评价中间步骤。

\[ R_{\mathrm{process}} = \sum_{k=1}^{K} r_k \]

其中 \(r_k\) 是第 \(k\) 个推理步骤的奖励。

奖励类型 优点 缺点
Outcome Reward 简单、便宜、容易规模化 信用分配粗糙
Process Reward 能指导中间推理 需要过程标注或过程验证器
Stage 5 先掌握 Outcome Reward。 等进入 Agent 和复杂推理后，再系统学习 Process Reward、step verifier、trajectory reward。

14. PPO / DPO / GRPO 对比

方法 训练信号 是否在线采样 是否需要 Value Model 适合任务
PPO Reward Model 或环境 reward 需要 通常需要 通用 RLHF
DPO chosen / rejected 偏好对 不需要 不需要 离线偏好优化
GRPO 同题多样本 reward 需要 通常不需要 数学、代码、可验证推理

14.1 PPO 和 GRPO 的关键差异

\[ \text{PPO: } \hat{A}_t = \text{GAE}(R_t,V_\phi) \]
\[ \text{GRPO: } \hat{A}_i = \frac{R_i-\bar{R}}{\sigma_R+\epsilon} \]

PPO 用 Value Model 做 baseline；GRPO 用同组平均 reward 做 baseline。

14.2 DPO 和 GRPO 的关键差异

维度 DPO GRPO
数据 已有 chosen / rejected 模型在线生成多个回答
奖励 偏好隐式 reward 验证器显式 reward
探索 较弱，依赖离线数据 更强，模型可采样新解法
典型任务 风格、偏好、对齐 数学、代码、工具调用
一句话： DPO 是“从偏好对学习”；GRPO 是“从同题多次尝试的验证结果中学习”。

15. GRPO / RLVR 算法流程

15.1 输入

  • 一批 prompt：\(\{x_j\}\)
  • 当前 policy model：\(\pi_\theta\)
  • 旧 policy model：\(\pi_{\theta_{\mathrm{old}}}\)
  • reference model：\(\pi_{\mathrm{ref}}\)
  • reward function：\(R(x,y)\)
  • group size：\(G\)

15.2 流程

for each training iteration:

    1. 采样一批 prompts

    2. 对每个 prompt x:
           用 old policy 生成 G 个回答:
           y1, y2, ..., yG

    3. 对每个回答 yi:
           reward_i = R(x, yi)

    4. 对同一个 prompt 的 G 个 rewards:
           mean_reward = mean(rewards)
           std_reward = std(rewards)
           advantage_i = (reward_i - mean_reward) / (std_reward + eps)

    5. 对每个 token 计算:
           ratio = πθ(token | context) / πold(token | context)

    6. 使用 PPO-like clipped objective 更新 policy

    7. 加入 KL penalty，限制 policy 偏离 reference model

15.3 数学形式简化版

\[ y_i\sim \pi_{\theta_{\mathrm{old}}}(\cdot\mid x), \quad i=1,\ldots,G \]
\[ R_i=R(x,y_i) \]
\[ \hat{A}_i = \frac{ R_i-\frac{1}{G}\sum_{j=1}^{G}R_j }{ \sqrt{\frac{1}{G}\sum_{j=1}^{G}(R_j-\bar{R})^2}+\epsilon } \]
\[ r_{i,t}(\theta) = \frac{ \pi_\theta(y_{i,t}\mid x,y_{i,\lt t}) }{ \pi_{\theta_{\mathrm{old}}}(y_{i,t}\mid x,y_{i,\lt t}) } \]
\[ L_{\mathrm{clip}} = \mathbb{E} \left[ \min ( r_{i,t}\hat{A}_i, \mathrm{clip}(r_{i,t},1-\epsilon,1+\epsilon)\hat{A}_i ) \right] \]
注意： 不同框架对 GRPO objective 的细节可能略有不同，但“同 prompt 多采样、组内 reward 相对 advantage、PPO-like ratio 更新”是核心结构。

16. 常见坑：Reward Hacking、全 0 奖励、长度偏置

16.1 Reward Hacking

模型会优化 reward，但不一定优化真实目标。

真实目标 错误 reward 模型可能学到
正确解题 只检查是否出现数字 随便输出一个数字
正确代码 测试太少 硬编码测试样例
正确工具调用 只检查工具名 工具名对但参数乱写
格式稳定 格式奖励太高 只学格式，不学内容

16.2 全 0 或全 1 奖励

GRPO 依赖组内差异。

reward 情况 问题
全是 0 没有哪个回答相对更好
全是 1 没有进一步学习信号
有 0 有 1 最适合组内比较
任务难度要适中： 太难全错，太简单全对，都会让 GRPO 缺少有效信号。

16.3 长度偏置

如果 reward 不惩罚冗长，模型可能输出越来越长的推理。

解决方式包括：

  • 加入长度惩罚；
  • 设置最大生成长度；
  • 对过长无效推理扣分；
  • 区分有用推理和重复废话。

16.4 格式过拟合

如果验证器只会抽取固定格式，模型可能过度学习模板。

16.5 数据污染

数学题或代码题如果训练集和评测集泄漏，reward 提升不代表真实泛化能力提升。

17. 最小 Toy 实验

先不用训练大模型，可以用 Python 模拟 GRPO 的核心逻辑。

17.1 输入

responses = [
    "Final Answer: 94311",
    "Final Answer: 94211",
    "I think it is 94311",
    "Final Answer: 94301"
]

answer = 94311

17.2 Reward

def reward_fn(output, answer):
    pred = extract_answer(output)
    if pred is None:
        return -0.2
    if pred == answer:
        return 1.0
    return 0.0

rewards = [reward_fn(r, answer) for r in responses]

17.3 Advantage

import numpy as np

rewards = np.array(rewards)
mean = rewards.mean()
std = rewards.std()

advantages = (rewards - mean) / (std + 1e-8)

17.4 Policy 更新的直觉

for response, reward, adv in zip(responses, rewards, advantages):
    if adv > 0:
        print("提高这个回答的概率", response)
    elif adv < 0:
        print("降低这个回答的概率", response)
    else:
        print("几乎不更新", response)
你先把这个 toy 逻辑跑通，再去看 TRL、verl、OpenRLHF 里的 GRPO Trainer。

19. 一页总结

概念 核心公式或理解 你要掌握什么
RLVR \(R(x,y)\) 由验证器给出 用规则、测试、执行结果自动给 reward
Group Sampling \(y_1,\ldots,y_G\sim\pi_{\theta_{\mathrm{old}}}(\cdot\mid x)\) 同一个 prompt 多采样
Group Mean \(\bar{R}=\frac{1}{G}\sum_i R_i\) 作为组内 baseline
Group Advantage \(\hat{A}_i=\frac{R_i-\bar{R}}{\sigma_R+\epsilon}\) 回答相对同组平均好多少
GRPO Ratio \(\frac{\pi_\theta(y_{i,t}\mid x,y_{i,\lt t})}{\pi_{\theta_{\mathrm{old}}}(y_{i,t}\mid x,y_{i,\lt t})}\) 新旧策略 token 概率比
Clip Objective \(\min(r\hat{A},\mathrm{clip}(r,1-\epsilon,1+\epsilon)\hat{A})\) 限制更新幅度
KL Penalty \(\mathrm{KL}(\pi_\theta\|\pi_{\mathrm{ref}})\) 防止模型偏离 reference
Format Reward 检查输出格式 保证答案可解析
Accuracy Reward 检查答案是否正确 提供核心任务信号
Outcome Reward 只看最终结果 简单但信用分配粗糙
Process Reward 评价中间步骤 更细但更难设计
进入 Stage 6 前的标准： 你应该能解释清楚：
  1. RLVR 和 RLHF / DPO 的区别；
  2. 为什么 GRPO 不需要 Value Model；
  3. 为什么同一个 prompt 要生成多个回答；
  4. group relative advantage 怎么计算；
  5. format reward 和 accuracy reward 的区别；
  6. 为什么全 0 或全 1 reward 都不适合 GRPO；
  7. 为什么 Agent RL 比单轮 RLVR 更难。
