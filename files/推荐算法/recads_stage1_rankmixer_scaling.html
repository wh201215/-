推荐广告大模型新范式 Stage 1：RankMixer 与 Ranking Model Scaling

RecommendationAds RankingScalingRankMixer工业推荐大模型

学习定位：这篇不是传统推荐基础，也不是重新讲 FM、DeepFM、DIN、DCN。这里默认你已经理解召回、粗排、精排、CTR/CVR、多任务排序、用户行为序列建模。本文直接进入新的工业趋势：推荐广告 Ranking Model 的大模型化与 Scaling。

这一阶段的核心问题是：

\[ \text{推荐广告排序模型能不能像 LLM 一样，通过扩大模型规模持续提升效果？} \]

但推荐系统的 scaling 不是简单把模型做大，而是要在高 QPS、低延迟、强实时和多目标约束下，把模型有效容量、特征交互能力、序列建模能力和硬件利用率一起做上去。

1. 为什么推荐广告也开始讲 Scaling

LLM 的成功给推荐系统一个启发：如果模型结构、数据规模、计算量可以稳定扩大，模型能力可能会持续提升。

LLM scaling:
更多参数
更多数据
更多计算
  ↓
更低 loss
更强能力

推荐系统过去的提升方式更像工程模块堆叠：

新特征
新交叉模块
新行为序列模块
新多任务结构
新样本加权
新 loss 设计

典型工业 ranker 可能长这样：

Sparse ID Embedding
        ↓
User Behavior Sequence Module
        ↓
Feature Crossing Module
        ↓
Multi-task Tower
        ↓
CTR / CVR / WatchTime / GMV
推荐广告开始讲 Scaling，本质是在问：
能不能减少手工模块拼接，让排序模型进入统一架构、统一放大、统一优化的阶段？

2. 推荐 Scaling 和 LLM Scaling 的区别

2.1 LLM Scaling 主要看什么

维度含义
参数量模型有多少参数
训练 token 数训练语料规模
FLOPs训练计算量
Loss语言建模损失
下游能力推理、代码、问答、对齐能力

LLM 的目标相对统一：

\[ \min_\theta -\sum_t\log p_\theta(x_t\mid x_{\lt t}) \]

2.2 推荐 Scaling 多了工业约束

约束说明
高 QPS每秒请求量极高
低延迟精排通常有严格毫秒级预算
大规模稀疏特征user、item、author、category、device、context
强实时用户行为和广告状态快速变化
多目标CTR、CVR、GMV、留存、时长、广告收入
商业约束广告预算、出价、ROI、pacing
在线 A/B离线指标提升不等于线上收益
成本约束模型再大，也要能服务全流量
推荐 Scaling 的核心不是“大”，而是“可服务的大”。

3. 传统 Ranker 为什么难继续做大

3.1 模块碎片化

传统工业排序模型常由多个模块拼接：

DIN / SIM
DCN / CrossNet
FM / DeepFM
MMoE / PLE
MLP
Auxiliary Loss

这种结构工程上灵活，但不容易像 Transformer 那样统一加深、统一加宽、统一扩参数。

3.2 特征交互模块未必 GPU 友好

很多 feature crossing 模块来自传统机器学习或早期深度推荐阶段，本身有效，但不一定充分利用现代 GPU 的大规模矩阵计算能力。RankMixer 把低 MFU、低可扩展性视为工业 ranker scaling 的关键障碍之一。

3.3 推荐特征高度异质

自然语言 token 比较同质：

我 / 喜欢 / 这个 / 视频

推荐特征高度异质：

user_id
item_id
author_id
city
device
gender
category
history_item
query
time
price

因此，推荐模型既要做跨特征交互，又要保留不同特征域的独立建模能力。

3.4 行为序列和非序列特征分离

用户行为序列 → sequence module
非序列稀疏特征 → feature interaction module
二者最后再拼接

这种方式会限制序列特征和非序列特征之间的早期信息交换，也使得整体结构难以统一 scaling。

4. RankMixer 想解决什么

RankMixer 的定位可以概括为：

\[ \text{面向工业推荐排序的硬件友好、统一、可扩展 feature-interaction 架构} \]

它主要解决三个问题。

4.1 低 MFU

MFU 是 Model FLOPs Utilization，表示模型理论计算量中有多少真正被硬件有效利用。传统手工 feature-crossing 模块可能存在“理论有计算，但 GPU 利用率低”的问题。

4.2 难以 Scaling

如果一个结构不能稳定加深、加宽、加专家，它就很难成为大模型 backbone。RankMixer 试图把 ranker 变成：

RankMixer Block × N

而不是：

DIN + DCN + MLP + 各种手工模块

4.3 Latency 约束

工业推荐不能只看离线效果。模型扩大后必须满足线上延迟、QPS 和服务成本约束。RankMixer 的贡献之一，是在近似相同 inference latency 下扩大模型规模。

5. RankMixer 核心结构

可以把 RankMixer 抽象成：

Feature Tokens
    ↓
RankMixer Block × N
    ↓
Prediction Head
    ↓
CTR / CVR / WatchTime / Ads Score

一个 RankMixer Block 可以粗略理解为：

Input feature tokens
    ↓
Token Mixing
    ↓
Per-token FFN
    ↓
Residual / Normalization
    ↓
Output feature tokens
组件作用
Feature Tokenization把推荐特征组织成 token 序列
Token Mixing让不同特征 token 交换信息
Per-token FFN保留不同特征子空间的独立非线性建模
Sparse-MoE扩大参数容量，但每个样本只激活部分专家
Dynamic Routing缓解专家训练不充分和负载不均衡
一句话：RankMixer 不是把 LLM 原样搬进推荐，而是把推荐特征交互模块改造成更适合 scaling 的 Mixer/Transformer-like backbone。

6. Feature Tokenization：推荐特征如何 token 化

6.1 传统拼接方式

传统排序模型通常会把特征 embedding 拼接：

\[ \mathbf{x}=[\mathbf{e}_1;\mathbf{e}_2;\cdots;\mathbf{e}_n] \]

然后送进 MLP 或交叉网络：

\[ \hat{y}=f_\theta(\mathbf{x}) \]

6.2 Token 化方式

Token 化后，每个特征不是简单拼进一个长向量，而是成为一个 token：

\[ \mathbf{E}=[\mathbf{e}_1,\mathbf{e}_2,\ldots,\mathbf{e}_n]\in\mathbb{R}^{n\times d} \]
符号含义
\(n\)特征 token 数
\(d\)每个 token 的 embedding 维度
\(\mathbf{e}_i\)第 \(i\) 个特征 token 表示

例如：

e_1 = user_id embedding
e_2 = item_id embedding
e_3 = author_id embedding
e_4 = category embedding
e_5 = device embedding
e_6 = city embedding
e_7 = user_history representation

这样做的好处是：保留每个特征的独立位置，方便在 token 维度上做 mixing，也方便堆叠统一 block。

7. Token Mixing：为什么不用标准 Self-Attention

7.1 Self-Attention 的形式

\[ \mathrm{Attention}(Q,K,V)=\mathrm{softmax}\left(\frac{QK^\top}{\sqrt{d}}\right)V \]

其中：

\[ Q=XW_Q,\quad K=XW_K,\quad V=XW_V \]

复杂度通常和 token 数平方相关：

\[ O(n^2d) \]

7.2 推荐排序里为什么不一定需要标准 attention

工业推荐场景中，虽然特征 token 数可能没有文本长，但 batch 很大、QPS 很高、延迟预算严格。如果每一层都用标准 self-attention，可能带来不必要的开销。

7.3 Token Mixing 的直觉

Token Mixing 的目的不是精细计算所有 token 两两注意力，而是用更便宜、更并行的方式，让不同特征 token 交换信息。

\[ \mathbf{E}'=\mathrm{Mix}(\mathbf{E}),\quad \mathbf{E}\in\mathbb{R}^{n\times d} \]

它希望第 \(i\) 个 token 的新表示包含其他 token 的信息：

\[ \mathbf{e}'_i=g_i(\mathbf{e}_1,\mathbf{e}_2,\ldots,\mathbf{e}_n) \]
Token Mixing 的本质：用更适合工业 serving 的方式做 feature interaction。

8. Per-token FFN：为什么推荐特征需要自己的子空间

推荐特征不是语言 token，而是异质字段。user_id、item_id、city、device、category、history sequence 的语义空间不同。

如果所有 token 完全共享同一个 FFN，可能无法充分表达不同特征域的差异。因此 RankMixer 使用 Per-token FFN 来保留不同特征子空间的建模能力。

Token Mixing：负责跨特征交流
Per-token FFN：负责每类特征自己的非线性变换

可以抽象写成：

\[ \mathbf{e}'_i=\mathrm{FFN}_i(\mathbf{e}_i) \]

如果进一步引入专家，可以写成：

\[ \mathbf{e}'_i=\sum_{k\in\mathcal{K}_i}g_{i,k}(\mathbf{e}_i)\mathrm{Expert}_{i,k}(\mathbf{e}_i) \]

9. Sparse-MoE 与 Dynamic Routing

9.1 为什么大 Ranker 需要稀疏专家

如果模型所有参数都是 dense 激活，那么参数量扩大通常会带来推理 FLOPs 和延迟上升。Sparse-MoE 的思想是：

\[ \text{总参数很多，但每个样本只激活其中一小部分。} \]

例如：

总共有 64 个专家
每个 token 只选择 top-2 专家

MoE 可以写成：

\[ \mathrm{MoE}(x)=\sum_{k\in\mathrm{TopK}(g(x))}g_k(x)E_k(x) \]

9.2 Dynamic Routing 解决什么

MoE 的常见问题是专家负载不均衡。

某些专家被大量样本选择
某些专家很少被选择
  ↓
热门专家过载
冷门专家训练不足
  ↓
模型容量没有被充分利用

推荐系统的数据高度长尾，专家训练不均衡会更明显。因此 dynamic routing 或 load balancing loss 的目标是让专家既能分工，又不严重失衡。

10. MFU 与硬件效率

MFU 是 Model FLOPs Utilization，衡量模型理论计算量中有多少真正转化为硬件有效计算。

\[ \mathrm{MFU}=\frac{\text{实际有效模型计算吞吐}}{\text{硬件理论峰值计算吞吐}} \]
低 MFU 问题影响
GPU 利用率低服务成本高
模块碎片化kernel 调度和内存访问效率差
计算不规则难以获得大矩阵乘法优势
模型扩大后吞吐下降难以全流量上线
RankMixer 的重要性：它把推荐模型 scaling 从“只看模型结构”推进到“模型结构 + 硬件效率 + 线上服务”的统一问题。

12. 这条线和 OneRec 的区别

OneRec 是生成式推荐路线；RankMixer 是排序模型 scaling 路线。

方向核心问题代表
Ranking Model Scaling如何把精排模型做大、做统一、做高效Wukong、RankMixer、TokenMixer-Large、OneTrans
Generative Recommendation能否直接生成推荐 item / 推荐列表TIGER、OneRec、OpenOneRec
RankMixer:
手工模块拼接 ranker
  ↓
可 scaling 的大 ranking model

OneRec:
retrieve + rank 级联
  ↓
生成式统一推荐
一句话：RankMixer 关心“排序模型如何变大”；OneRec 关心“推荐范式能否生成式统一”。

先做一个小型 toy ranker，不追求工业规模，只验证结构思想。

输入：
user_id、item_id、category、device、city、history embedding

模型：
feature tokenization
  ↓
token mixing block
  ↓
per-token FFN
  ↓
pooling / flatten

模型对比点

14.3 观察指标

  • AUC / Logloss 是否提升；
  • 参数量增加后是否稳定提升；
  • 推理延迟是否明显增加；
  • 不同 token 数下效果如何；
  • MoE 是否出现专家负载不均衡。
注意：toy 项目不可能复现字节工业效果。它的意义是帮助你理解 RankMixer 的结构动机。

15. 一页总结

概念核心理解
Ranking Model Scaling在推荐广告低延迟、高 QPS 约束下扩大 ranker 的有效容量
传统 Ranker 问题模块拼接、低 MFU、难统一放大、特征异质
Feature Tokenization把推荐字段组织成 token 序列，而不是简单拼接长向量
Token Mixing用更高效的方式做跨特征交互，替代昂贵或不必要的标准 attention
Per-token FFN保留不同特征子空间的独立建模能力
Sparse-MoE扩大总参数，但每个样本只激活部分专家
Dynamic Routing缓解专家训练不充分和负载不均衡
MFU衡量模型计算是否真正有效利用硬件
RankMixer面向工业推荐排序的硬件友好可扩展 feature-interaction backbone
TokenMixer-Large进一步把 RankMixer / TokenMixer 思路推到 billion-level ranker
OneTrans统一用户行为序列建模和非序列特征交互
OneRec生成式推荐路线，和 RankMixer 解决不同层面的问题
Stage 1 结束标准：
  1. 能讲清推荐广告 scaling 和 LLM scaling 的区别；
  2. 能解释为什么传统 ranker 不容易像 Transformer 一样做大；
  3. 能解释 RankMixer 为什么强调 MFU 和 hardware-aware；
  4. 能解释 feature tokenization、token mixing、per-token FFN 分别解决什么；
  5. 能区分 RankMixer、TokenMixer-Large、OneTrans、OneRec。

16. 阅读顺序

  1. Wukong: Towards a Scaling Law for Large-Scale Recommendation：先理解推荐 scaling law 为什么是问题。
  2. RankMixer: Scaling Up Ranking Models in Industrial Recommenders：重点理解 feature interaction backbone、token mixing、per-token FFN、Sparse-MoE、MFU。
  3. TokenMixer-Large: Scaling Up Large Ranking Models in Industrial Recommenders：重点看 RankMixer 继续放大后，残差、梯度、MoE、部署遇到什么问题。
  4. OneTrans: Unified Feature Interaction and Sequence Modeling with One Transformer in Industrial Recommender：重点看 sequence modeling 和 feature interaction 如何统一到一个 Transformer backbone。
  5. OneRec / TIGER / OpenOneRec：放到下一阶段看，进入生成式推荐路线。
