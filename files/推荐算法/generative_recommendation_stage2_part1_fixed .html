学习定位：这篇是 Stage 2 的第一部分。它不急着讲 OneRec 的工业系统，也不急着做大模型后训练，而是先解决一个更基础的问题：

\[ \text{推荐系统为什么可以从“给候选 item 打分排序”，变成“直接生成 item”？} \]

这条线的核心入口是 TIGER / Recommender Systems with Generative Retrieval。它提出用 Semantic ID 表示 item，再训练 Transformer 自回归生成下一个 item 的 Semantic ID。理解这个范式后，后面再看 OneRec、OpenOneRec、推荐后训练、推荐 RL，就不会只停留在“推荐模型变成 LLM”这种表面理解。

2. 生成式推荐到底是什么

很多人第一次听“生成式推荐”，会误解成：让 LLM 生成一句推荐理由，或者让 ChatGPT 回答“我推荐你买这个”。这只是生成式推荐的一小部分。更关键的是：推荐任务本身被改写成生成任务。

路线生成对象典型任务是否是本文重点
文本生成式推荐推荐理由、商品描述、对话回复Explainable Rec、Conversational Rec不是
ID 生成式推荐item_id、Semantic ID、Textual IDNext-item recommendation、generative retrieval是
列表生成式推荐一整个推荐 session / listOneRec session-wise generation后续重点
指令式生成推荐按用户指令生成符合条件的推荐Instruction-following RecOpenOneRec 阶段讲
论文视角：LLM-based generative recommendation 的综述把问题拆成三问：什么是生成式推荐、为什么推荐系统要走向生成式推荐、以及如何用 LLM 实现不同推荐任务。该综述还明确指出，与其把推荐过程拆成打分、重排等阶段，不如用 LLM 从完整 item 池中直接生成推荐结果。[Li et al., 2024]
一句话：生成式推荐的关键不是“用了 LLM”，而是把推荐输出从 score/list 改成 token sequence。

3. 从 Retrieve + Rank 到 Generate

传统推荐系统通常是多阶段级联。这个结构非常成熟，所以不要说它“过时”。更准确的说法是：它在大规模工业场景里非常可靠，但存在目标割裂和工程碎片化。

图 2：传统 Retrieve + Rank 与生成式推荐的差别。
注意：生成式推荐不是说召回、排序、重排以后都不重要了。真正的变化是：模型从“判别式打分器”变成了“可生成 item token 的序列模型”。工业上仍然要考虑约束、过滤、去重、多目标和安全策略。

4. 推荐任务的数学改写

4.1 传统排序建模

传统 ranker 的基本形式是给用户和候选 item 打分：

\[ s(u,i)=f_\theta(u,i) \]

在线上，模型不是面对全量 item，而是面对召回阶段给出的候选集合 \(\mathcal{C}_u\)：

\[ \operatorname{TopK}(u)=\operatorname{argsort}_{i\in\mathcal{C}_u} s(u,i) \]

这带来一个天然限制：如果召回阶段没有把某个 item 放进候选集，后面的 ranker 再强也选不到它。

4.2 生成式推荐建模

生成式推荐希望直接建模：

\[ p(i \mid H_u) \]

其中 \(H_u=(i_1,i_2,\ldots,i_n)\) 是用户历史交互序列。为了让模型能够“生成” item，需要把 item 表示成 token 序列：

\[ i \rightarrow \operatorname{SID}(i)=(c_1,c_2,\ldots,c_m) \]

于是 next-item prediction 被改写为：

\[ p(\operatorname{SID}(i_{n+1})\mid H_u) =\prod_{t=1}^{m}p(c_t\mid c_{\lt t},H_u) \]

这就是生成式推荐最重要的形式：把 item 变成 token，把推荐变成 next-token prediction。

User history:  item A, item B, item C
Semantic IDs:  (5,23,55), (5,25,78), (8,10,41)
Model output:  (8,10,66)
Lookup table:  (8,10,66) → item D

5. 为什么不能直接生成原始 item_id

最朴素的生成式推荐是直接生成 item_id，例如让模型输出 item_928391。这个想法直观，但在推荐系统里通常不是好选择。

问题为什么严重后果
原始 ID 无语义item_123 和 item_124 不一定相似模型无法通过 ID 结构共享知识
词表巨大百万、千万、亿级 item 会造成巨大输出空间softmax、embedding、训练和推理成本高
长尾严重大量 item 交互极少低频 item 学不到稳定表示
冷启动困难新 item 没有历史交互基于历史 ID 的模型很难推荐新 item
平台迁移差不同平台的 item_id 完全不共享难做跨域、零样本、foundation model
TIGER 的关键回答：不用随机原子 item ID，而是用 item 内容信息生成语义上有意义的 codeword tuple，作为每个 item 的 Semantic ID；然后训练 Transformer-based sequence-to-sequence 模型预测用户下一个交互 item 的 Semantic ID。[Rajput et al., 2023]

6. Semantic ID 是什么

Semantic ID 可以理解为 item 的“离散语义坐标”。它不是随机编号，而是通过内容或多模态 embedding 量化得到的一串 token。

图 3：Semantic ID 的基本构造流程。图为自绘示意，不复制论文原图。

一个简单例子：

Nike 跑鞋 A      → (5, 23, 88)
Nike 跑鞋 B      → (5, 23, 91)
Adidas 跑鞋 C    → (5, 24, 17)
厨房置物架 D     → (17, 4, 203)

这里的直觉是：第一个 token 可能表示粗粒度大类，第二个 token 可能表示更细的语义簇，后面的 token 进一步区分具体 item。TIGER 在 Amazon Beauty 上的定性分析也观察到 RQ-VAE Semantic ID 具有层级性质：第一个 semantic token 对应粗级类别，第二/第三 token 对应更细粒度类别。[Rajput et al., 2023]

你要记住的点：Semantic ID 同时解决两件事：
1. 让 item 能作为 token 被生成；
2. 让相似 item 共享部分 token，从而增强长尾和冷启动泛化。

7. RQ-VAE / Quantization 生成 Semantic ID

Semantic ID 的关键技术是 连续向量离散化。给定 item 的语义向量 \(x_i\in\mathbb{R}^d\)，我们要把它变成一个离散 code 序列。

7.1 基本符号

\[ E(i) = x_i \in \mathbb{R}^d \]

其中 \(E\) 是内容编码器，例如 Sentence-T5、BERT、LLM embedding、VLM embedding。然后 tokenizer / quantizer 把 \(x_i\) 映射成：

\[ \operatorname{SID}(i)=Q(x_i)=(c_i^1,c_i^2,\ldots,c_i^m) \]

如果每一层 codebook 大小是 \(K\)，SID 长度是 \(m\)，理论可表示空间是：

\[ K^m \]

例如 \(K=256,m=4\)，理论组合空间是 \(256^4\)，约 4.29B。TIGER 论文中使用 length-4 Semantic ID，每个 codeword cardinality 为 256；虽然有效 item 数远小于全部组合空间，但模型在 Top-10 预测里 invalid ID 比例很低。[Rajput et al., 2023]

7.2 RQ-VAE 的直觉

RQ-VAE 的核心是 残差量化：第一层 codebook 先近似原始向量，第二层 codebook 再近似第一层没表示好的残差，第三层继续近似剩余残差。

图 4：残差量化的直觉。每一层 codebook 负责解释上一层剩下的误差。
\[ r_0=x, \quad c_l=\arg\min_k \|r_{l-1}-e_l(k)\|_2, \quad r_l=r_{l-1}-e_l(c_l) \]
\[ \hat{x}=\sum_{l=1}^{m}e_l(c_l), \quad \operatorname{SID}(i)=(c_1,c_2,\ldots,c_m) \]

7.3 TIGER 里的具体设置

组件TIGER 中的做法理解
内容特征title、price、brand、category 等构造成文本让 item 有语义输入
文本编码器Sentence-T5，得到 768 维 semantic embedding把自然语言商品信息转为连续向量
RQ-VAEDNN encoder + residual quantizer + DNN decoder把连续向量压缩成离散 code
量化层数3 层 residual quantization得到 3-tuple Semantic ID
codebook每层 cardinality 256，code vector 维度 32总词表可控，同时组合空间很大
碰撞处理追加第 4 个唯一化 code保证每个 item 有唯一 SID

TIGER 论文给出的实现细节是：Sentence-T5 生成 768 维 item embedding，RQ-VAE 的 encoder 经过 512、256、128 隐层并映射到 32 维 latent，三层残差量化，每层 codebook cardinality 为 256，训练后得到 3-tuple，再追加第 4 个 token 解决 SID 碰撞。[Rajput et al., 2023]

8. TIGER 论文精读

论文：Recommender Systems with Generative Retrieval，NeurIPS 2023。TIGER 全称是 Transformer Index for GEnerative Recommenders。论文的核心是：用 Transformer 的参数记忆作为端到端 retrieval index，让模型直接生成候选 item 的 Semantic ID。[Rajput et al., 2023]

8.1 TIGER 要解决什么

传统 retrieval 大多是 dual-encoder + ANN：

user/query tower → query embedding
item tower       → item embedding table / ANN index
query embedding 与 item embedding 做 nearest neighbor search

TIGER 的替代思路是：

用户历史 item 的 Semantic ID 序列
        ↓
Transformer seq2seq
        ↓
直接生成下一个 item 的 Semantic ID
        ↓
查表得到真实 item
图 5：TIGER 的训练与推理流程。图为自绘示意。

8.2 TIGER 的两阶段框架

阶段输入输出目标
Stage A：Semantic ID generationitem 内容特征每个 item 的 Semantic ID把连续语义 embedding 离散化
Stage B：Generative recommendation用户历史 SID 序列下一 item 的 SID训练 Transformer 生成推荐候选

论文将 item 序列按时间排序，然后把每个 item 替换为 Semantic ID token 序列，训练 seq2seq 模型预测下一个 item 的 SID。[Rajput et al., 2023]

8.3 为什么叫 Transformer Index

传统 ANN retrieval 的“索引”在外部向量库里：

item embeddings → ANN index

TIGER 的思想更像 DSI / generative retrieval：把可检索对象的标识符压进 Transformer 参数记忆里，推理时模型直接生成目标 ID：

Transformer parameters ≈ differentiable retrieval index

这不是说真的不需要 item lookup table。模型生成 SID 后，仍然需要用 SID → item 的映射表查回真实 item。区别在于候选生成本身不是 ANN 近邻搜索，而是 token decoding。

9. 训练目标与样本构造

9.1 样本构造

给定用户历史：

\[ H_u=(i_1,i_2,\ldots,i_n) \]

每个 item 都有 SID：

\[ \operatorname{SID}(i_j)=(c_{j,1},c_{j,2},\ldots,c_{j,m}) \]

输入序列可以拼成：

[USER_u], c_1,1, c_1,2, ..., c_1,m,
          c_2,1, c_2,2, ..., c_2,m,
          ...,
          c_n,1, c_n,2, ..., c_n,m

输出目标是：

c_{n+1,1}, c_{n+1,2}, ..., c_{n+1,m}

9.2 Loss

训练目标就是标准自回归交叉熵：

\[ \mathcal{L}_{gen} = -\sum_{t=1}^{m}\log p_\theta(c_{n+1,t}\mid c_{n+1,\lt t},H_u) \]

直觉上，模型不是一次性分类到某个 item，而是一步一步生成这个 item 的语义坐标。

9.3 TIGER 的模型实现细节

项目TIGER 中的设置为什么重要
框架T5X 实现 Transformer encoder-decoder把推荐转换为 text-to-text / seq2seq 风格
SID 词表1024 个 semantic codeword token，即 256 × 4比直接把所有 item 作为词表更可控
用户 token添加 2000 个 user-specific tokens，用 hashing trick 映射原始 user ID增强个性化
模型规模encoder/decoder 各 4 层，6 heads，约 13M 参数说明 TIGER 不是靠超大参数取胜
训练步数Beauty、Sports 训练 200k steps，Toys 训练 100k steps论文实验级别，不是工业大模型级别

这些设置体现出 TIGER 的定位：它不是一个“工业巨型模型”，而是证明 Semantic ID + generative retrieval 这个范式可行。[Rajput et al., 2023]

10. 推理、Beam Search 与 Invalid ID

训练时模型知道目标 SID；推理时模型要自己生成。

Input:
[USER_u] + history SID tokens

Decoder:
 → c1 → c2 → c3 → c4 → 

Lookup:
(c1,c2,c3,c4) → item

10.1 Top-K 推荐怎么得到

生成式推荐可以通过 beam search 或 sampling 得到多个候选 SID：

Beam 1: (8,10,66,0) → item D
Beam 2: (5,23,91,0) → item E
Beam 3: (17,4,203,0) → item F

每条 beam 对应一个完整 SID，SID 再查表转成 item。这样就能得到 Top-K 候选。

10.2 Invalid ID 问题

由于 \(K^m\) 的组合空间远大于真实 item 数，模型理论上可能生成一个不存在的 SID：

模型生成: (9, 77, 31, 4)
Lookup table: 查不到对应 item
=> Invalid ID

TIGER 论文报告，在其设置下 Top-10 预测中的 invalid ID 比例约为 0.1%–1.6%，并提出可以增大 beam size 后过滤 invalid ID，以保证输出足够数量的有效推荐。[Rajput et al., 2023]

工程注意：invalid ID 是生成式推荐必须处理的问题。工业系统里还要额外处理重复 item、下架 item、已看 item、曝光频控、商业规则、多目标约束等。

11. TIGER 实验该怎么看

读 TIGER 实验，不要只看 “比 baseline 高多少”。你要抓三个问题：

  1. 生成式召回是否真的有效？看 Recall@K、NDCG@K 与 SASRec、BERT4Rec、S3-Rec、P5 等 baseline 对比。
  2. Semantic ID 是否真的比 Random ID 好？看 RQ-VAE SID、LSH SID、Random ID 的消融。
  3. 新能力是否成立？看冷启动 item、长尾 item、多样性控制。

11.1 主实验

论文在 Amazon Reviews 相关数据集上使用 Recall@K 和 NDCG@K 评估。TIGER 对比了 GRU4Rec、Caser、HGN、SASRec、BERT4Rec、FDSA、S3-Rec、P5 等顺序推荐 baseline，在 Sports and Outdoors、Beauty、Toys and Games 上多项指标优于 best baseline。[Rajput et al., 2023]

指标含义生成式推荐里怎么看
Recall@K真实下一个 item 是否出现在 Top-K衡量生成候选是否覆盖正确 item
NDCG@K考虑排序位置的命中质量正确 item 越靠前越好
Invalid ID Rate生成出的 SID 是否能查到 item衡量生成空间约束是否有效
Cold-start Recall新 item / 未见 item 能否被推荐衡量 SID 是否提供内容泛化
Diversity / Entropy推荐结果类别是否多样衡量 sampling temperature 的控制能力

11.2 消融：Random ID vs LSH SID vs RQ-VAE SID

这是 TIGER 最值得细读的实验之一。论文比较了：

ID 类型构造方式问题/优势
Random ID随机分配 codeword唯一但无语义，不能共享知识
LSH SID用随机投影把 embedding hash 成 code有一点 locality，但表达能力有限
RQ-VAE SID用神经量化器学习离散 code能学层级语义，实验效果最好

论文结果显示，在相似 cardinality 条件下，RQ-VAE Semantic ID consistently outperforms LSH 和 Random ID。这说明关键不是“把 item 变成几个数字”就行，而是这些 token 必须携带可学习的语义结构。[Rajput et al., 2023]

读论文抓手：如果一个生成式推荐方法只说“我们生成 item token”，你要追问：这个 token 怎么来的？是否有语义？是否有协同信号？是否会碰撞？是否有 invalid output？是否能处理冷启动？

12. 新能力：冷启动与多样性

12.1 冷启动为什么会改善

传统基于随机 item ID 的模型依赖历史交互学习 item embedding。新 item 没有交互，模型很难推荐它。Semantic ID 的优势在于：只要新 item 有内容信息，就可以通过同一个 encoder + quantizer 得到 SID。

新 item 没有历史点击
但它有 title / brand / category / image
        ↓
Encoder + Quantizer
        ↓
得到 Semantic ID
        ↓
如果模型学过相似语义簇，就可能生成相关 SID 前缀

TIGER 在冷启动分析中使用 RQ-VAE 为 unseen items 也生成 Semantic ID，并把共享前三个 semantic tokens 的 unseen items 加入候选；论文报告其方法在多个 Recall@K 上优于语义空间 KNN baseline。[Rajput et al., 2023]

12.2 多样性为什么可控

因为 SID 是分层 token 序列，解码时可以通过 temperature sampling 控制生成分布：

\[ p_T(c_t\mid \cdot)=\operatorname{softmax}\left(\frac{z_t}{T}\right) \]
Temperature效果风险
低温度更集中、更保守、更像最可能 item多样性不足
高温度更多类别、更丰富候选相关性可能下降

TIGER 论文指出，temperature-based sampling 可以提升推荐结果类别熵；并且由于 RQ-VAE SID 具有层级结构，在第一 token 采样可能影响粗粒度类别，在第二/第三 token 采样可能影响类别内部的细粒度变化。[Rajput et al., 2023]

TIGER 之后，生成式推荐的一个重要分支就变成了：item identifier / item tokenization 到底应该怎么设计？

3. tokenizer 是否应该和推荐模型联合训练？

13.1 Better Generalization with Semantic IDs

这篇工作研究的是工业 ranking model 中随机 hash item IDs 的泛化问题。它指出，随机 hash item ID 有利于记忆热门 item，但会阻碍相似 item 之间泛化，尤其在大规模、长尾、动态变化的 item corpus 中更明显。论文提出用 RQ-VAE 从 frozen content embeddings 学出 compact discrete Semantic IDs，并在 YouTube 推荐 ranking model 中展示其对新/长尾 item 泛化的帮助。[Singh et al., 2024]

13.2 IDGenRec

IDGenRec 的重点不是 RQ-VAE codebook，而是 textual ID learning。它希望每个 item 被表示成唯一、简洁、语义丰富、平台无关的 textual ID，让 LLM 更自然地处理推荐 item。论文进一步探索了跨 19 个数据集训练 foundation recommendation model，并在 6 个未见数据集上做 zero-shot 评估。[Tan et al., 2024]

13.3 LETTER

LETTER 直接针对 item tokenization 的三个痛点：语义不足、协同信号不足、code assignment bias。它把 RQ-VAE 作为 semantic regularization，引入 contrastive alignment loss 进行 collaborative regularization，并用 diversity loss 缓解 code 分配偏置。[Wang et al., 2024]

14. 和 OneRec / OpenOneRec 的关系

这一节先做铺垫，不深入写 OneRec 的全部细节。你可以这样理解：

TIGER：
证明 next-item recommendation 可以通过 Semantic ID generation 来做。

OneRec：
进一步把工业推荐里的 retrieve + rank 统一成一个端到端生成模型。

OpenOneRec：
把生成式推荐扩展成 foundation model + benchmark + training pipeline。

14.1 OneRec V1

OneRec 论文认为，大多数现代推荐系统仍然是 retrieve-and-rank 级联结构，即使引入生成式模型，也往往只把它当作召回阶段的 selector。OneRec 试图用统一生成模型替代级联学习框架，包含 encoder-decoder 架构、Sparse-MoE 扩容、session-wise generation，以及结合 DPO 的 Iterative Preference Alignment；论文报告在快手主场景部署后 watch-time 提升 1.6%。[Deng et al., 2025]

模块OneRec 的作用和 TIGER 的差别
Encoder-Decoder编码用户历史行为序列，逐步解码用户可能感兴趣的视频TIGER 多是 next-item；OneRec 面向更复杂工业生成
Sparse MoE扩大模型容量但不等比例增加 FLOPsTIGER 是小模型验证范式；OneRec 是工业级扩容
Session-wise Generation一次生成更连贯的 session 推荐不是逐点生成后再人工拼接
Iterative Preference Alignment用 reward model + DPO 提升生成质量进入推荐后训练/偏好对齐阶段

14.2 OneRec V2

OneRec-V2 继续解决 V1 的扩展性问题。论文摘要指出，V1 中 97.66% 计算资源消耗在 sequence encoding 而不是 generation；V2 提出 Lazy Decoder-Only Architecture，声称总计算减少 94%、训练资源减少 90%，并扩展到 8B 参数，同时引入 Duration-Aware Reward Shaping 和 Adaptive Ratio Clipping 来利用真实用户反馈进行偏好对齐。[Zhou et al., 2025]

14.3 OpenOneRec

OpenOneRec 是开源框架，目标是连接传统推荐系统与 LLM。其 README 介绍了 RecIF-Bench、OneRec-Foundation Models 和完整训练 pipeline：RecIF-Bench 包含来自短视频、广告、商品三个领域的 100M interactions 和 200k users；模型系列有 1.7B 与 8B，基于 Qwen3 backbone；训练 pipeline 包含数据处理、co-pretraining 和 post-training。[OpenOneRec, 2026]

15. 可复现实验计划

如果你要把 Stage 2 做成真正能讲的项目，第一步不建议直接复现 OneRec。OneRec 工业系统复杂，数据也不容易完整拿到。更现实的路线是先做一个 Toy TIGER / Toy Semantic-ID Generative Recommender。

15.1 公开数据

数据集优点适合做什么

15.2 最小实现流程

Step 1: 读取 item metadata
        title / category / brand / description

Step 2: 用 sentence encoder 得到 item embedding
        MiniLM / BERT / Sentence-T5 / Qwen embedding

Step 3: 训练或近似实现 tokenizer
        简化版：KMeans / Residual KMeans
        完整版：RQ-VAE

Step 4: 生成 Semantic ID
        item_i → (c1,c2,c3,c4)

Step 5: 构造用户序列训练样本
        history SIDs → next item SID

Step 6: 训练小型 Transformer
        encoder-decoder 或 decoder-only

Step 7: 推理
        beam search 生成 SID → lookup item

Step 8: 评估
        Recall@5/10, NDCG@5/10, Invalid ID rate, cold-start recall

15.3 Baseline 设计

模型作用为什么要有
PopRec热门 item baseline最低基线，防止模型无意义
GRU4RecRNN 顺序推荐传统 sequential baseline

16. 一页总结

概念核心理解
生成式推荐把推荐输出建模成 token sequence，而不是只给候选 item 打分
Generative Retrieval模型直接生成可检索对象的 ID，而不是通过向量 ANN 检索
Semantic ID由 item 内容/多模态 embedding 量化得到的离散语义 token 序列
RQ-VAE通过残差量化把连续 embedding 分层压缩成多个 codeword
TIGER用 Semantic ID + Transformer seq2seq 做 next-item generative retrieval
Invalid ID模型生成的 SID 在 lookup table 中不存在，需要 beam/filter/约束解码处理
冷启动新 item 可以通过内容 encoder 得到 SID，从而获得被生成的机会
多样性控制通过 temperature sampling 调节生成分布；SID 层级结构让不同 token 层控制不同语义粒度
OneRec把生成式推荐推进到工业级端到端统一 retrieve + rank，并引入偏好对齐
OpenOneRec把生成式推荐进一步做成 foundation model、benchmark 和 full-stack pipeline
Stage 2 Part 1 结束标准：
  1. 能讲清传统 ranking 和 generative recommendation 的建模差别；
  2. 能解释为什么不能直接生成随机 item_id；
  3. 能用自己的话解释 Semantic ID 和 RQ-VAE；
  4. 能讲出 TIGER 的两阶段流程：SID generation + generative recommendation；
  5. 能说明 TIGER 的实验为什么强调 Random ID、LSH SID、RQ-VAE SID 对比；
  6. 能把 TIGER 和 OneRec 的关系讲清楚：前者证明范式，后者推进工业端到端统一。
