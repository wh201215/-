推荐广告大模型新范式 Stage 4：生成式推荐的训练与后训练

Generative RecommendationPost-trainingDPORec-RLOneRecOpenOneRecReasoning Rec

学习定位：Stage 2 已经讲了 Semantic ID 和 TIGER，Stage 3 讲了 OneRec 如何把 retrieve 和 rank 统一成工业级生成模型。Stage 4 要解决更关键的问题：生成式推荐模型训练完 next-token prediction 之后，怎么继续后训练，让它更像一个真正可用的推荐系统？

这一阶段的核心问题是：

\[ \text{如何把生成模型从“会预测下一个 item”后训练成“会优化用户体验和业务目标”的推荐模型？} \]

这一篇重点看四条线：NTP / SFT、DPO / Preference Alignment、Rec-RL / GRPO、Reasoning-aware Recommendation。

1. 为什么 NTP / SFT 不够

生成式推荐最自然的训练目标是 next-token prediction。把用户历史行为序列作为上下文，把目标 item 的 Semantic ID 或 Itemic Token 作为输出，模型学习：

\[ \mathcal{L}_{\operatorname{NTP}} = -\sum_{t=1}^{T}\log \pi_\theta(y_t \mid x, y_{1:t-1}) \]
符号含义
\(x\)用户历史、上下文、画像、场景特征、指令文本等输入
\(y_t\)第 \(t\) 个目标 token，可以是 Semantic ID token，也可以是自然语言 token
\(\pi_\theta\)当前生成式推荐模型

这个目标让模型学会“历史之后通常出现什么 item”，但推荐系统的真实目标不是只复现历史。真实目标往往包括点击、观看时长、购买、转化、留存、多样性、新颖性、商业约束和安全约束。

关键矛盾：Next-token prediction 学的是历史分布，推荐系统想优化的是未来反馈。历史里出现过的 item 不一定是最优推荐，用户没看见过的 item 也可能更好。

1.1 NTP 的三个局限

问题为什么会发生后训练要解决什么
曝光偏差训练数据来自旧系统曝光，不是完整 item 空间让模型跳出旧系统分布，探索高潜力 item
目标错位NTP 只最大化历史 item 的似然用点击、时长、转化、满意度等反馈重新对齐
列表质量缺失逐 item 预测不直接优化整页推荐从 token 级别优化转向 session/list 级别优化
只做 NTP：
用户历史 → 生成下一个历史 item

做后训练：
用户历史 → 生成更可能带来好反馈的一组 item

3. 训练数据怎么组织

生成式推荐的后训练，本质上是在构造不同形式的监督信号。你可以把数据分成四类。

3.1 监督式 next-item 样本

User history:
  [item_12, item_98, item_35]
Target:
  item_77

如果 item 用 Semantic ID 表示：

Input:
  [SID_12, SID_98, SID_35]
Output:
  SID_77 = (c1, c2, c3)
\[ \mathcal{L}_{\operatorname{item}} = -\sum_{m=1}^{M}\log \pi_\theta(c_m \mid x, c_{1:m-1}) \]

这里 \(M\) 是一个 item 的 token 长度，比如 TIGER 里 Semantic ID 可以是一个 codeword tuple，OpenOneRec 里 itemic tokens 也可以看成压缩后的 item 表示。

3.2 Session-wise 样本

OneRec 的重要变化是从 next-item 走向 session-wise generation。不是生成一个 item，而是生成一个 session 内多个 item：

\[ Y = (i_1,i_2,\ldots,i_K) \]
\[ \mathcal{L}_{\operatorname{session}} = -\sum_{k=1}^{K}\sum_{m=1}^{M}\log \pi_\theta(c_{k,m} \mid x, i_{1:k-1}, c_{k,1:m-1}) \]
为什么 session-wise 重要：推荐页面不是单点预测。一个列表里要考虑重复、互补、多样性、探索、疲劳、商业约束。Session-wise generation 让模型直接学习“整页推荐”的序列结构。

3.3 Pairwise preference 样本

DPO / preference alignment 需要偏好对：

Prompt x:
  用户历史 + 场景 + 候选生成约束
Chosen y+:
  更好的推荐列表 / item 序列
Rejected y-:
  较差的推荐列表 / item 序列

在推荐里，难点是正负样本不总是天然同时出现。用户一次请求通常只看到一个推荐列表，系统不知道“没展示的另一个列表”用户会不会更喜欢。OneRec 因此引入 reward model 和定制采样策略来模拟或构造偏好对。

3.4 On-policy RL 样本

RL 样本来自当前模型自己生成：

同一个用户请求 x
  ↓
当前模型采样 G 个推荐结果
  ↓
用 reward 函数打分
  ↓
相对优势归一化
  ↓
更新模型
\[ \{Y_1,Y_2,\ldots,Y_G\} \sim \pi_\theta(\cdot \mid x) \]

这和 SFT 最大的不同是：SFT 学固定答案，RL 学“在自己会生成的空间里，哪些结果更好”。

4. 预训练与 Co-Pretraining：先让模型认识 item

如果 Stage 2 的重点是“item 怎么变成 token”，那么 Stage 4 的预训练重点是：item token 怎么和自然语言 token、用户行为 token 放进同一个模型空间里。

4.1 Itemic Token / Semantic ID 的作用

原始 item：
视频、广告、商品、作者、类目、价格、文本、图像

压缩后：
Itemic Tokens / Semantic ID

进入 LLM：
用户历史 = item token 序列 + 文本指令 + 上下文

OpenOneRec 把 item 作为一种独立模态处理，用 Itemic Tokens 缩短 item 表示，使用户交互历史可以作为一个统一的长上下文序列输入模型。它的技术报告还强调，层级 token 让相似 item 共享前缀，从而有利于知识迁移。

4.2 Co-Pretraining 的直觉

推荐 foundation model 不能只吃推荐日志，否则容易变成一个“只会 pattern matching 的领域专家”。它也不能只吃通用文本，否则不会理解点击、购买、广告转化这些推荐信号。Co-pretraining 的目标是两边都学：

数据源模型学到什么
用户行为序列协同过滤、兴趣迁移、短期偏好、长期偏好
item metadata标题、类目、品牌、作者、内容语义
通用文本语言理解、指令跟随、常识、推理能力
推荐指令数据把推荐任务表达成问答、解释、条件推荐、交互推荐
论文线索：OpenOneRec 的框架把预训练、混合域 co-pretraining、SFT、on-policy distillation 和 Rec-RL 串起来，目标不是只训练一个 ranker，而是训练一个 recommendation foundation model。

5. SFT：从预测下一个 item 到指令跟随

SFT 不是简单继续 next-item prediction，而是把不同推荐任务包装成统一的 instruction-response 格式。

Instruction:
  根据用户最近观看历史，推荐 10 个短视频。

Input:
  用户历史 itemic tokens + 标签 + 上下文

Response:
  推荐 itemic tokens 列表 + 可选解释

OpenOneRec 的 SFT 阶段混合了推荐任务和通用指令/推理数据，目的有两个：

  1. 增强推荐域任务能力，例如视频推荐、广告推荐、商品推荐、标签条件推荐、交互式推荐；
  2. 防止模型在推荐预训练后遗忘 LLM 的通用指令跟随和推理能力。
\[ \mathcal{L}_{\operatorname{SFT}} = -\mathbb{E}_{(x,y)\sim \mathcal{D}_{\operatorname{SFT}}} \sum_{t=1}^{T}\log \pi_\theta(y_t \mid x,y_{1:t-1}) \]
SFT 的作用：它不是最终优化目标，而是给后面的 DPO / RL 提供一个稳定初始策略。没有 SFT，模型可能连格式、任务、约束、解释方式都不稳定。

6. OneRec 的 DPO / IPA：为什么推荐也需要偏好对齐

OneRec-V1 的一个关键贡献是 Iterative Preference Alignment，也就是把 DPO 引入生成式推荐结果优化中。

在 LLM 里，DPO 通常来自人类偏好：回答 A 比回答 B 好。在推荐里，偏好不是这么直接，因为一次请求通常只展示一个列表，不能同时给同一个用户展示两个不同列表再比较。因此 OneRec 的问题更难：

LLM DPO:
同一个 prompt → 两个回答 → 人类选更好

推荐 DPO:
同一个用户请求 → 系统只展示一个列表 → 只有一个真实反馈

OneRec 的做法可以抽象成：

  1. 先用当前生成式推荐模型生成多个候选结果；
  2. 用 reward model 或反馈估计这些结果质量；
  3. 构造 chosen / rejected 偏好对；
  4. 用 DPO 让模型提高 chosen 的相对概率，降低 rejected 的相对概率；
  5. 迭代更新，形成 preference alignment loop。
图 2：OneRec 里可理解的 IPA / DPO 训练闭环。

7. DPO 数学目标：推荐里到底优化什么

DPO 的核心不是训练一个显式 reward model 再 PPO，而是直接用偏好对优化策略模型。给定同一个输入 \(x\)，有一个较优输出 \(y^+\) 和较差输出 \(y^-\)。

\[ \Delta_\theta(x,y^+,y^-) = \log \pi_\theta(y^+ \mid x)-\log \pi_\theta(y^- \mid x) \]

为了防止模型偏离原始策略太远，DPO 还引入 reference model：

\[ \Delta_{\operatorname{ref}}(x,y^+,y^-) = \log \pi_{\operatorname{ref}}(y^+ \mid x)-\log \pi_{\operatorname{ref}}(y^- \mid x) \]

DPO loss 可以写成：

\[ \mathcal{L}_{\operatorname{DPO}} = -\mathbb{E}\left[\log \sigma\left(\beta\left(\Delta_\theta(x,y^+,y^-)-\Delta_{\operatorname{ref}}(x,y^+,y^-)\right)\right)\right] \]
项直觉
\(\Delta_\theta\)当前模型认为 chosen 比 rejected 高多少
\(\Delta_{\operatorname{ref}}\)参考模型原本的偏好差距
\(\beta\)控制偏好对齐强度；越大，更新越激进
\(\sigma\)sigmoid，把相对优势变成分类概率
推荐视角：DPO 不是让模型“记住 y+”，而是让模型在同一个用户请求下，更偏向生成高反馈列表，而不是低反馈列表。

7.1 Item-level DPO 和 List-level DPO

粒度形式优点问题
Item-level单个 item 做 chosen / rejected构造简单，训练稳定不能直接处理列表多样性和位置关系
List-level整组推荐列表做 chosen / rejected更接近真实推荐页面reward 噪声更大，信用分配更难
Session-level一个 session 的连续推荐结果做偏好能建模上下文连贯性训练成本高，需要更复杂采样

8. OpenOneRec 的 Rec-RL：为什么还要强化学习

DPO 适合有偏好对的场景，但推荐系统还有很多目标不是简单 pairwise 能覆盖的：

  • Recall@K / NDCG@K 这类离散 ranking metric；
  • 整页推荐的多样性、重复率、无效 item、冷启动覆盖；
  • 广告场景里的转化、ROI、预算、pacing；
  • 视频场景里的 watch time、完播率、互动、留存；
  • 通用推荐 foundation model 的 instruction following 和 reasoning 保持。

OpenOneRec 的后训练配方可以概括为三步：

Stage 1: Multi-task SFT
  恢复和增强推荐任务 + 通用指令能力

Stage 2: On-policy Distillation
  用通用 LLM teacher 修复通用推理能力

Stage 3: Rec-RL
  用推荐 reward 直接优化推荐指标

这个设计背后的思想是：推荐模型不能只会推荐，也不能只会聊天。它要同时保留 LLM 的通用能力和推荐系统的目标优化能力。

8.1 On-policy distillation 的意义

OpenOneRec 发现，模型做推荐预训练和 SFT 后，可能会损失通用 reasoning / instruction following 能力。因此它用 teacher model 监督学生模型自己采样出来的轨迹。直觉如下：

学生模型自己回答 general prompt
        ↓
teacher model 评价或提供分布约束
        ↓
学生模型修正自己在通用任务上的漂移
\[ \mathcal{L}_{\operatorname{distill}} = \mathbb{E}_{y\sim \pi_\theta(\cdot \mid x)} D_{\operatorname{KL}}\left(\pi_\theta(\cdot \mid x,y_{1:t-1})\,\Vert\,\pi_{\operatorname{teacher}}(\cdot \mid x,y_{1:t-1})\right) \]

如果模型在通用文本任务里乱生成 itemic token，就要惩罚；否则模型会出现“推荐 token 污染通用语言空间”的问题。

9. GRPO / Group Relative 优化：推荐 RL 的基本形状

OpenOneRec 使用 Group Relative Policy Optimization 思路做 recommendation-oriented RL。核心不是给每个样本训练一个 critic，而是对同一个 prompt 采样一组候选输出，并在组内计算相对优势。

\[ Y_1,Y_2,\ldots,Y_G \sim \pi_\theta(\cdot \mid x) \]
\[ A_i = \frac{r_i-\operatorname{mean}(r_1,r_2,\ldots,r_G)}{\operatorname{std}(r_1,r_2,\ldots,r_G)+\epsilon} \]

其中 \(r_i\) 是第 \(i\) 个生成列表的推荐 reward，\(A_i\) 是组内标准化后的优势。

\[ \rho_i = \frac{\pi_\theta(Y_i \mid x)}{\pi_{\operatorname{old}}(Y_i \mid x)} \]
\[ \mathcal{J}_{\operatorname{GRPO}} = \mathbb{E}\left[\sum_{i=1}^{G}\min\left(\rho_i A_i,\operatorname{clip}(\rho_i,1-\epsilon,1+\epsilon)A_i\right)-\lambda D_{\operatorname{KL}}(\pi_\theta\,\Vert\,\pi_{\operatorname{ref}})\right] \]
注意：推荐 RL 的难点不是公式，而是 reward。Reward 写错，模型就会学会刷指标、重复热门 item、牺牲多样性或过度迎合短期点击。

10. 奖励函数怎么设计：推荐后训练的核心

推荐后训练最重要的不是“用了 DPO 还是 GRPO”，而是 reward 是否代表真实目标。

\[ r(Y,x)=\alpha_1\operatorname{Hit@K}+\alpha_2\operatorname{NDCG@K}+\alpha_3\operatorname{Diversity}-\alpha_4\operatorname{InvalidRate}-\alpha_5\operatorname{DuplicateRate} \]
Reward 项作用可能风险
Hit@K鼓励列表命中用户真实交互过度追热门，忽略排序质量
NDCG@K考虑命中位置，靠前更重要依赖离线 label，仍有曝光偏差
WatchTime / DwellTime贴近视频内容消费质量长视频天然时长高，需要归一化
CTR / CVR适合广告和电商转化容易短期化，可能牺牲长期体验
Diversity减少信息茧房和重复推荐过强会牺牲相关性
InvalidRate惩罚生成不存在或不可展示 item需要和约束解码一起使用
DuplicateRate惩罚列表内部重复需要定义 item、作者、类目多层重复

10.1 为什么 watch time 不能直接当 reward

视频推荐里，watch time 很重要，但 raw watch time 有偏差。长视频天然可能产生更长观看时长，短视频即使用户很满意，绝对时长也较短。因此 OneRec-V2 提到 Duration-Aware Reward Shaping，用视频时长相关的归一化方式缓解这种偏差。

\[ r_{\operatorname{duration}} = \operatorname{normalize}(\operatorname{watch\_time},\operatorname{video\_duration}) \]

直觉是：不要奖励“本来就很长”的内容，而要奖励“在同类时长内容里表现更好”的内容。

10.2 多目标 reward

推荐系统通常不是单目标。一个更真实的 reward 可以写成：

\[ r = w_1 r_{\operatorname{engagement}} + w_2 r_{\operatorname{conversion}} + w_3 r_{\operatorname{retention}} + w_4 r_{\operatorname{diversity}} - w_5 r_{\operatorname{risk}} \]

银行、广告、电商场景尤其要小心风险项。例如金融推荐里不能只优化点击，还要考虑适配性、合规、风险等级、用户承受能力和解释。

11. OneRec-V2：从 reward model 到真实用户反馈

OneRec-V1 的 preference alignment 主要依赖 reward model 构造偏好；OneRec-V2 进一步强调用真实用户交互反馈做后训练，包括 Duration-Aware Reward Shaping 和 Adaptive Ratio Clipping。

版本后训练重点意义
OneRec-V1IPA + DPO，使用 reward model 辅助构造偏好证明生成式推荐可以通过偏好对齐改善结果质量
OneRec-V2真实用户反馈 + reward shaping + ratio clipping减少 proxy reward 偏差，更接近线上目标
OpenOneRecSFT + on-policy distillation + Rec-RL把后训练做成开源 foundation model pipeline
Stage 4 的关键变化：生成式推荐从“离线序列建模”走向“在线反馈驱动的策略优化”。这也是它和传统 sequential recommendation 最大的分水岭。

12. OneRec-Think：推理式推荐从哪里来

OneRec 和 TIGER 这类模型通常是隐式预测器：输入历史，输出 item。它们未必会解释为什么推荐，也不一定能进行显式推理。OneRec-Think 针对这个问题提出 in-text reasoning for generative recommendation，试图把对话、推理和个性化推荐统一起来。

可以把 OneRec-Think 理解成三步：

模块作用通俗理解
Itemic Alignment把 itemic tokens 和文本语义空间对齐让模型知道一个 item token 对应什么内容
Reasoning Activation用 CoT / reasoning SFT 激活推荐推理让模型不仅给 item，还能写出选择逻辑
Reasoning Enhancement用推荐特定 reward 继续优化推理让解释和推荐结果一起对齐用户偏好
图 3：OneRec-Think 的核心直觉。
注意：推理式推荐不是所有场景都需要。短视频 feed 流可能更看重低延迟和反馈优化；金融、教育、医疗、B 端推荐更需要解释、约束和可审计性。

13. 和 LLM 后训练的区别

维度LLM 后训练生成式推荐后训练
输入自然语言 prompt用户历史、上下文、item tokens、业务约束
输出文本回答item 序列、推荐列表、解释文本
偏好来源人类标注、AI feedback、规则点击、观看、购买、转化、留存、负反馈、业务规则
Reward 噪声主观偏好噪声曝光偏差、位置偏差、选择偏差、延迟反馈
解码约束语法上通常自由生成必须生成存在且可展示的 item
线上风险幻觉、不安全回答重复推荐、信息茧房、商业目标冲突、合规风险
一句话：LLM 后训练对齐的是“回答偏好”；生成式推荐后训练对齐的是“用户行为反馈 + 列表质量 + 业务约束”。

14. 可落地小项目：公开数据上的 GenRec 后训练实验

你如果想做一个可以讲的项目，不建议一开始复现 OneRec 工业系统。更合适的是做一个公开数据上的小型生成式推荐后训练实验。

14.1 项目标题

Rec-DPO：面向 Semantic-ID 生成式推荐的偏好对齐后训练方法

14.2 实验任务

模块做法
数据集Amazon Reviews、MovieLens、Yelp 或公开电商序列数据
Tokenizer先用 item metadata embedding + k-means / RQ 得到简化 SID
Base model小型 Transformer / T5 / decoder-only 模型生成 SID
SFTnext-item 或 next-list 生成
Preference data用用户真实后续交互作为 positive，用模型采样未命中 item 作为 negative
DPO让模型提高 positive list 的相对概率
评价Recall@K、NDCG@K、Hit@K、Coverage、Diversity、InvalidRate

14.3 可以写进论文/面试的创新点

  1. 从 pointwise 到 pairwise：不是只训练 next-item likelihood，而是构造推荐偏好对。
  2. 从 token-level 到 list-level：把一个推荐列表作为整体输出，优化列表质量。
  3. 从命中率到多目标：同时考虑 Hit、NDCG、Diversity、InvalidRate。
  4. 从闭集推荐到生成式解码：分析 invalid SID 和 constrained decoding 的影响。
Baseline:
  TIGER-style SFT

Method:
  TIGER-style SFT + Rec-DPO

Ablation:
  item-level DPO
  list-level DPO
  DPO + diversity reward
  constrained decoding vs unconstrained decoding

15. 常见坑：后训练不是万能药

坑表现解决方向
Reward hacking模型学会刷点击或时长加长期指标、多样性、负反馈、人工检查
Invalid generation生成不存在 item 或不可展示 itemSID Trie、constrained decoding、过滤器
热门坍塌推荐越来越集中在热门 itemcoverage / diversity reward，重采样策略
冷启动被压制新 item 很难被生成内容语义 token、探索 reward、冷启动 boost
通用能力遗忘模型只会推荐，不会理解指令混合 SFT、on-policy distillation、KL 约束
离线线上不一致离线 Recall 高，线上体验差更真实反馈、在线 A/B、counterfactual evaluation

16. 阅读顺序

  1. Direct Preference Optimization：先理解 DPO 为什么可以不用显式 reward model 做偏好优化。
  2. OneRec: Unifying Retrieve and Rank with Generative Recommender and Iterative Preference Alignment：重点看 IPA、DPO 和 session-wise generation。
  3. OneRec-V2 Technical Report：重点看真实用户反馈、Duration-Aware Reward Shaping 和 Adaptive Ratio Clipping。
  4. OpenOneRec Technical Report：重点看 SFT、on-policy distillation、Rec-RL、RecIF-Bench 和开源 pipeline。
  5. OneRec-Think：重点看 itemic alignment、reasoning activation 和 recommendation-specific reward。
  6. SAGE：作为进阶，看 list-wise generative recommendation 的 sequence-level RL 优化。

17. 一页总结

概念核心理解
NTP让模型学会生成历史中的下一个 item token，但不等于优化用户体验
SFT把推荐任务包装成 instruction-response，提供稳定后训练起点
DPO用 chosen / rejected 偏好对，让模型提高好结果相对概率
IPAOneRec 的迭代偏好对齐，用 reward model 和采样构造推荐偏好对
Rec-RL用 Hit、NDCG、时长、多样性等 reward 直接优化生成策略
GRPO对同一 prompt 采样多个输出，用组内相对优势更新模型
Reward Shaping把真实反馈改造成更可靠的训练信号，例如时长归一化
On-policy Distillation让模型在自己生成的轨迹上恢复通用能力，防止 item token 污染语言能力
OneRec-Think把 item 生成、文本解释和推理能力结合起来
核心挑战曝光偏差、reward hacking、无效生成、热门坍塌、冷启动、多目标冲突
Stage 4 结束标准：
  1. 能讲清为什么生成式推荐不能只做 NTP / SFT；
  2. 能写出 DPO 在推荐中的 chosen / rejected 形式；
  3. 能解释 OneRec 的 IPA 为什么需要 reward model 和迭代采样；
  4. 能解释 OpenOneRec 为什么要 SFT + on-policy distillation + Rec-RL；
  5. 能设计一个公开数据上的 Rec-DPO 小实验；
  6. 能说明 reward 设计为什么比算法名字更重要。
