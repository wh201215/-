2. 公开数据怎么选

生成式推荐最需要的数据不是“评分表”本身，而是用户行为序列 + item 侧信息。因为 Semantic ID 最好不是纯随机 ID，而是能从 item 内容、类目、文本、图像或行为 embedding 中量化出来。

2.1 数据集选择原则

选择标准为什么重要推荐程度
有用户-物品交互时间next-item prediction 必须构造按时间排序的用户历史必需
有 item metadataSemantic ID 可以从标题、类目、描述、图片等内容生成强烈推荐
item 数不能太小生成式推荐在大 item space 下更能体现优势推荐
有长尾分布可以分析生成模型是否只推荐热门 item推荐
预处理成本可控学生项目不应该被数据清洗拖死必需

2.2 推荐数据集

数据集优点缺点适合做什么
Amazon Reviews 2023规模大；包含 reviews、ratings、text、item metadata、description、price、image、also-bought links 等丰富信息完整数据很大，需要选一个 category 子集最适合做 Semantic ID + 生成式推荐
MovieLens 1M / 20M / latest-small经典、容易处理、社区代码多item 内容较弱，生成 Semantic ID 的优势不明显适合先跑通序列推荐 baseline
RecBole 内置数据格式训练、评估、baseline 统一，指标支持完整生成式推荐需要自己改造模型输出适合做 baseline 对照
OpenOneRec 相关数据/benchmark更接近 recommendation foundation model 和 instruction-following recommender工程复杂度更高适合 Stage 6 以后做 foundation model 实验
建议：第一版项目不要一上来用全量 Amazon Reviews 2023。选择一个中等 category，例如 Beauty、Sports、Toys、Tools、Home 等，过滤出交互次数不少于 5 的用户和物品，先让实验闭环跑通。

2.3 数据规模建议

版本用户数物品数交互数适合阶段
Toy Debug1k - 5k1k - 5k20k - 100k调通 pipeline

如果显存有限，Small Experiment 已经足够做出有说服力的图表。生成式推荐项目的价值不只看绝对指标，而是看你是否能解释：

为什么某种 Semantic ID 更好？
为什么生成模型会产生 invalid SID？
为什么 beam size 增大后 Recall 上升但延迟增加？
为什么冷启动 item 在内容型 Semantic ID 下更容易被召回？

3. 任务定义：Next-item Generative Recommendation

最标准、最容易复现的任务是 next-item prediction。给定用户历史行为序列，预测下一个交互 item。

\[ H_u = \left(i_1, i_2, \ldots, i_n\right) \]
\[ i_{n+1}^{\star} = \text{the next item interacted by user }u \]

传统序列推荐模型通常学习一个打分函数：

\[ s_{u,i}=f_{\theta}\left(H_u,i\right) \]

然后对候选 item 排序：

\[ R_u^K = \operatorname{TopK}_{i\in\mathcal{I}}\; s_{u,i} \]

生成式推荐则先把 item 映射成 Semantic ID：

\[ \operatorname{SID}\left(i\right)=\left(c_{i,1},c_{i,2},\ldots,c_{i,L}\right) \]

模型直接生成目标 item 的 token 序列：

\[ p_{\theta}\left(\operatorname{SID}\left(i_{n+1}^{\star}\right)\mid \operatorname{SID}\left(i_1\right),\ldots,\operatorname{SID}\left(i_n\right)\right) \]

展开成自回归形式：

\[ p_{\theta}\left(c_{\star,1},\ldots,c_{\star,L}\mid H_u\right)= \prod_{\ell=1}^{L}p_{\theta}\left(c_{\star,\ell}\mid c_{\star,\lt \ell},H_u\right) \]
关键变化：传统推荐是“给 item 打分”；生成式推荐是“把 item 变成 token，再预测 token 序列”。

4. 数据切分与样本构造

4.1 用户序列排序

每个用户的交互按时间排序：

user u:
item_3 → item_8 → item_2 → item_15 → item_9

然后构造训练样本：

input:  item_3
label:  item_8

input:  item_3, item_8
label:  item_2

input:  item_3, item_8, item_2
label:  item_15

验证和测试可以采用 leave-one-out：

切分样本用途
训练除最后两个交互之外的滑窗样本训练生成模型
验证倒数第二个 item调参、早停、选 beam size
测试最后一个 item最终报告指标

4.2 滑窗长度

实际训练时不会把无限长历史全部输入，而是截断到最大长度 \(M\)：

\[ H_u^{\left(M\right)}=\left(i_{\max\left(1,n-M+1\right)},\ldots,i_n\right) \]
最大历史长度特点建议
20训练快，适合 debug第一版
50能覆盖多数用户近期兴趣正式小实验
100+更适合大模型和长序列兴趣建模显存充足再做

4.3 样本格式

生成式推荐的训练样本可以组织成文本式 token 序列，也可以直接组织成整数 token 序列。

Input tokens:
<user_hist> 12 87 301 44 <item_sep> 12 88 109 7 <item_sep> 3 91 21 50

Target tokens:
3 91 22 19 <eos>
注意：如果写进 HTML 的代码块里出现尖括号，需要写成 &lt;token&gt;，否则浏览器会把它误解成 HTML 标签。数学公式里也不要直接写 <，要写 \lt。

5. Semantic ID 构造实验

Semantic ID 是整个项目的核心。你不能只说“用了 Semantic ID”，而要证明不同 ID 方案的效果差异。

5.1 至少做四种 ID 对照

ID 方案构造方式作用预期现象
Random ID随机给每个 item 分配 token tuple证明“仅仅生成 ID”不够通常较弱，长尾泛化差
Popularity / Category ID用类目、热门度分桶构造层级 ID简单可解释 baseline粗粒度有效，但细粒度不足
Behavior SID用协同过滤或 LightGCN item embedding 量化捕捉用户共现行为点击预测强，但冷启动弱
Content SID用 title、description、category、image embedding 量化利用 item 内容语义冷启动和语义泛化更好
Hybrid SID行为 embedding + 内容 embedding 拼接或融合后量化结合协同信号和内容语义通常是最值得尝试的主方案

5.2 为什么 Random ID 是必要对照

生成式推荐很容易被误解成“只要 Transformer 足够强，就能记住 item ID”。Random ID 对照就是为了回答这个问题：

\[ \text{性能提升到底来自生成建模，还是来自 Semantic ID 的语义结构？} \]

如果 Content SID 或 Hybrid SID 明显优于 Random ID，就说明模型不是单纯背 ID，而是在利用 item token 的共享结构。

5.3 Semantic ID 的三个评价角度

评价角度怎么观察意义
唯一性多少 item 共享完全相同的 SID重复 SID 会导致生成后无法唯一映射 item
语义聚类共享前缀 token 的 item 是否属于相似类目验证 SID 是否真的有语义
长尾覆盖生成结果中 long-tail item 比例判断模型是否只学热门 item

6. RQ-VAE 训练细节

RQ-VAE 的作用是把连续 item embedding 压缩成多个离散 code。你可以把它理解成“分多层的向量量化”。

6.1 输入 item embedding

先为每个 item 准备一个连续向量：

\[ \mathbf{x}_i \in \mathbb{R}^{d} \]

这个向量可以来自：

来源例子优点
文本 embeddingtitle + description + category冷启动友好
图像 embedding商品主图、视频封面适合电商、短视频
行为 embeddingItem2Vec、LightGCN、SASRec item embedding协同信号强
混合 embedding内容向量和行为向量拼接后投影综合效果更稳

6.2 残差量化

假设有 \(L\) 层 codebook，每层有 \(K\) 个 code vector。第 \(\ell\) 层 codebook 是：

\[ \mathcal{C}^{\left(\ell\right)}=\left\{\mathbf{e}_{1}^{\left(\ell\right)},\mathbf{e}_{2}^{\left(\ell\right)},\ldots,\mathbf{e}_{K}^{\left(\ell\right)}\right\} \]

残差量化过程：

\[ \mathbf{r}_{i}^{\left(0\right)}=\mathbf{z}_i \]
\[ q_{i,\ell}=\operatorname*{arg\,min}_{k\in\left\{1,\ldots,K\right\}} \left\|\mathbf{r}_{i}^{\left(\ell-1\right)}-\mathbf{e}_{k}^{\left(\ell\right)}\right\|_2^2 \]
\[ \mathbf{r}_{i}^{\left(\ell\right)}= \mathbf{r}_{i}^{\left(\ell-1\right)}-\mathbf{e}_{q_{i,\ell}}^{\left(\ell\right)} \]

最终 item 的 Semantic ID 是：

\[ \operatorname{SID}\left(i\right)=\left(q_{i,1},q_{i,2},\ldots,q_{i,L}\right) \]
直觉：第一层 code 往往捕捉粗粒度语义，后几层 code 逐步补充细节。就像先判断“鞋子”，再判断“跑鞋”，再判断“某个品牌/款式”。

6.3 重构损失

RQ-VAE 训练的基本目标是让量化后的向量能重构原始 item 表示：

\[ \hat{\mathbf{x}}_i = D_{\phi}\left(\sum_{\ell=1}^{L}\mathbf{e}_{q_{i,\ell}}^{\left(\ell\right)}\right) \]
\[ \mathcal{L}_{\operatorname{rec}}= \sum_{i\in\mathcal{I}}\left\|\mathbf{x}_i-\hat{\mathbf{x}}_i\right\|_2^2 \]

实际实现里还会加入 commitment loss 或 codebook 更新机制，防止 encoder 输出频繁跳动或 codebook 使用不均衡。

6.4 Codebook collapse 问题

如果大部分 item 都被映射到少数 code，说明 codebook 没有被充分利用。

现象可能原因处理方法
某些 code 从不被选中初始化差、学习率不合适、embedding 分布集中重新初始化冷门 code；加入 usage regularization
大量 item 共享同一 SIDcodebook 太小或层数太少增大 \(K\) 或 \(L\)
SID 无语义聚类item embedding 本身质量差换更好的文本/图像/行为 embedding

7. TIGER-style 生成模型

TIGER-style 模型可以用 encoder-decoder Transformer，也可以用 decoder-only Transformer 简化实现。第一版建议从 encoder-decoder 开始，因为它和原始 TIGER 论文表述更接近。

7.1 输入输出形式

Input:
SID(item_1), SID(item_2), ..., SID(item_n)

Target:
SID(item_{n+1})

如果每个 item 的 SID 长度为 \(L\)，用户历史长度为 \(M\)，那么输入 token 数大约是：

\[ T_{\operatorname{input}} \approx M\cdot L \]

目标 token 数是：

\[ T_{\operatorname{target}}=L+1 \]

其中 \(+1\) 是结束符 \(\operatorname{EOS}\)。

7.2 训练目标

给定用户历史 \(H_u\)，目标 item 的 Semantic ID 为 \(\left(c_{u,1}^{\star},\ldots,c_{u,L}^{\star}\right)\)，训练损失为：

\[ \mathcal{L}_{\operatorname{gen}}=-\sum_{u\in\mathcal{U}} \sum_{\ell=1}^{L} \log p_{\theta}\left(c_{u,\ell}^{\star}\mid c_{u,\lt \ell}^{\star},H_u\right) \]

如果加入 \(\operatorname{EOS}\)，则把 \(L\) 改成 \(L+1\)。

7.3 为什么这是语言模型式训练

它和语言模型的 next-token prediction 非常像：

语言模型生成式推荐
上下文是一段文本上下文是用户历史 item 序列
目标是下一个词 token目标是下一个 item 的 SID token
输出词表是语言 token vocabulary输出词表是 codebook token vocabulary
生成一句话生成一个 item 或一个推荐列表
讲项目时可以这样说：我们不是把 LLM 用来写推荐理由，而是把推荐目标本身转成可生成的离散 token 序列，用 next-token learning 统一建模用户兴趣转移。

8. 推理：Beam Search 与约束解码

训练时模型知道目标 SID；推理时模型要自己生成 SID。因此推理阶段非常关键。

8.1 普通 Beam Search

每一步保留得分最高的 \(B\) 个前缀：

\[ \operatorname{score}\left(c_{1:\ell}\right)= \sum_{t=1}^{\ell}\log p_{\theta}\left(c_t\mid c_{\lt t},H_u\right) \]

最后得到若干完整 SID，再映射回 item。

8.2 为什么需要 constrained decoding

模型可能生成训练集中不存在的 SID：

Generated SID: (12, 87, 999, 44)
但是 item catalog 里没有任何 item 对应这个 code tuple

这就是 invalid SID。解决方法是构建 SID Trie，只允许模型生成真实存在的 SID 前缀。

合法 SID 集合：
(12, 87, 301, 44)
(12, 87, 302, 19)
(12, 90, 11,  5)

当模型已经生成前缀 (12, 87) 时，下一步只允许生成 301 或 302。
\[ \mathcal{A}\left(c_{1:\ell}\right)= \left\{c:\left(c_{1},\ldots,c_{\ell},c\right)\text{ is a valid SID prefix}\right\} \]

约束解码时只在合法集合里归一化：

\[ p_{\theta}^{\operatorname{valid}}\left(c\mid c_{\lt t},H_u\right)= \frac{\exp\left(o_c\right)\cdot \mathbb{I}\left\{c\in\mathcal{A}\left(c_{\lt t}\right)\right\}} {\sum_{c'\in\mathcal{A}\left(c_{\lt t}\right)}\exp\left(o_{c'}\right)} \]

8.3 生成 Top-K 推荐列表

生成式推荐最终也要输出 top-K item：

Beam candidates:
SID_1 → item_1, score_1
SID_2 → item_2, score_2
SID_3 → item_3, score_3
...

After filtering:
Top-K recommended items

过滤规则包括：

过滤原因
去掉用户已交互 item避免重复推荐历史物品
去掉 invalid SID保证能映射回真实 item
去掉不可用 item商品下架、视频删除、广告预算耗尽
业务约束重排多样性、安全、广告 pacing、库存等

9. Baseline 应该怎么设

一个能被认可的项目，不能只有自己的模型。至少要有传统方法、序列推荐方法、生成式方法三类对照。

图推荐 baselineLightGCN对比协同过滤图结构

如果时间紧，至少保留 Popularity、SASRec、Random-ID Generative Transformer 和 Semantic-ID Generative Transformer。

它和主模型唯一差异是 ID 是否有语义：

10. 评估指标与公式

生成式推荐最终输出的是 item 排序列表，所以仍然可以用 top-K recommendation 指标。

10.1 Recall@K / Hit@K

如果每个用户测试集只有一个目标 item，那么 Recall@K 和 Hit@K 基本等价：

\[ \operatorname{Recall@K}=\frac{1}{\left|\mathcal{U}\right|}\sum_{u\in\mathcal{U}} \mathbb{I}\left\{i_u^{\star}\in R_u^K\right\} \]

其中 \(R_u^K\) 是模型给用户 \(u\) 生成的 top-K 推荐列表。

10.2 NDCG@K

NDCG 考虑命中 item 的排名位置。命中越靠前，分数越高。

\[ \operatorname{NDCG@K}=\frac{1}{\left|\mathcal{U}\right|}\sum_{u\in\mathcal{U}} \frac{\mathbb{I}\left\{i_u^{\star}\in R_u^K\right\}} {\log_2\left(\operatorname{rank}_u\left(i_u^{\star}\right)+1\right)} \]

如果没有命中，则该用户贡献为 0。

10.3 MRR@K

\[ \operatorname{MRR@K}=\frac{1}{\left|\mathcal{U}\right|}\sum_{u\in\mathcal{U}} \frac{\mathbb{I}\left\{i_u^{\star}\in R_u^K\right\}} {\operatorname{rank}_u\left(i_u^{\star}\right)} \]

10.4 Coverage / Long-tail

生成式推荐不能只看准确率，还要看是否覆盖更多 item。

\[ \operatorname{Coverage@K}=\frac{\left|\bigcup_{u\in\mathcal{U}}R_u^K\right|}{\left|\mathcal{I}\right|} \]

长尾覆盖可以定义为推荐列表里属于长尾 item 的比例：

\[ \operatorname{TailRatio@K}=\frac{1}{\left|\mathcal{U}\right|K} \sum_{u\in\mathcal{U}}\sum_{i\in R_u^K}\mathbb{I}\left\{i\in\mathcal{I}_{\operatorname{tail}}\right\} \]
评估注意：尽量做 full-catalog evaluation，不要只对 100 个 sampled negatives 排序。采样负例评估容易高估模型效果，尤其不利于比较生成式方法和传统排序方法。

11. 消融实验设计

这部分是项目能不能显得“像研究”的关键。不要只报一张主表，而要让读者看出你的方法为什么有效。

11.1 ID 方案消融

实验变量观察指标要回答的问题
Random ID vs Semantic IDID 是否有语义Recall、NDCG、TailRatioSemantic ID 是否真的带来泛化
Content SID vs Behavior SIDitem embedding 来源冷启动 Recall、总体 Recall内容语义和协同行为哪个更重要
Hybrid SID融合内容和行为总体和长尾指标两种信号是否互补

11.2 Codebook 大小消融

参数候选值影响
SID 长度 \(L\)2 / 4 / 6 / 8越长越能区分 item，但生成更难
每层 codebook 大小 \(K\)64 / 128 / 256 / 512越大表达更细，但稀疏训练更难

历史长度 \(M\)20 / 50 / 100越长能看更多兴趣，但计算更大

11.3 冷启动消融

可以模拟冷启动：把一部分 item 的训练交互减少，只保留 metadata。

Cold-start setting:
1. 按 item 交互次数排序；
2. 选出低频 item 作为 cold-start subset；
3. 比较 Random ID、Behavior SID、Content SID、Hybrid SID 在该 subset 上的 Recall@K。

预期是：Content SID 和 Hybrid SID 在冷启动 subset 上优于 Behavior SID，因为它们能利用 item 内容。

12. 错误分析

错误分析比单纯跑分更能体现你理解生成式推荐。

错误类型表现可能原因改进方向
Invalid SID生成的 code tuple 不存在无约束解码；SID 空间太稀疏Trie constrained decoding；减小 SID 长度
SID collision多个 item 对应同一 SIDcodebook 表达能力不足增大 codebook；加入 disambiguation token
热门偏置推荐列表集中在热门 item训练样本长尾不平衡重采样；tail-aware loss；温度调节
语义漂移生成 item 和用户历史主题不一致SID 聚类差或用户兴趣突变改进 item embedding；加入 session-level intent
重复推荐推荐用户已经交互过的 item缺少 history filter推理后过滤历史 item
冷启动失败新 item 几乎不被生成SID 依赖行为信号加入内容型 SID

14. 如果要落到银行/业务场景

你之前提过“最好能落地银行，但不要太高级”。生成式推荐可以迁移到银行，但不要一上来做风控。更合适的银行场景是：

场景推荐对象为什么适合生成式推荐
手机银行内容推荐理财知识文章、活动页、权益页item 有文本语义，适合 Semantic ID
权益/优惠推荐餐饮券、出行券、支付立减、会员权益类目明确，用户兴趣序列明显
金融产品教育推荐基金科普、保险说明、风险测评引导需要安全解释和合规过滤
App 功能入口推荐转账、缴费、账单、信用卡活动可作为低风险个性化入口排序
不建议第一版做：贷款审批、授信、反欺诈、投资收益承诺。这些场景高风险、高合规，容易把项目复杂度拉爆。

14.1 银行版本怎么讲

可以把 item 从商品换成“内容/权益/服务入口”：

用户历史：
信用卡账单页 → 餐饮优惠页 → 地铁出行券 → 周末活动页

生成目标：
下一个最可能点击的权益/内容/服务入口 Semantic ID

银行版本的重点不是追求极致点击率，而是：

目标说明
可控生成只能生成合规 item catalog 中存在的内容
可解释语义Semantic ID 前缀对应内容类目或业务主题
冷启动友好新活动、新权益可通过文本内容进入推荐池
多样性避免只推信用卡或只推理财

15.1 一个可行的最终项目题目

Hybrid Semantic ID for Controllable Generative Recommendation
中文：面向可控生成式推荐的混合语义标识符构造方法

核心卖点：不是做一个新的大模型，而是研究 item tokenization 如何影响生成式推荐的准确性、冷启动能力、长尾覆盖和可控性。

15.2 最终报告的主结论应该长什么样

1. Semantic ID-based generation 在 next-item recommendation 上可以接近或超过部分序列推荐 baseline；
2. Random ID 明显弱于 Content/Hybrid SID，说明 item token 的语义结构很重要；
3. Content SID 在 cold-start subset 上表现更好；
4. Behavior SID 在高频 item 上更强；
5. Hybrid SID 在总体 Recall 和长尾覆盖之间更平衡；
6. Constrained decoding 能显著降低 invalid SID；
7. Beam size 提升 Recall，但会带来推理成本上升。

16. 一页总结

模块核心理解
任务把 next-item prediction 改写成 Semantic ID sequence generation
数据优先选择有时间序列和 item metadata 的公开数据，例如 Amazon Reviews 2023 子集
Semantic ID项目核心，不同 ID 方案必须做消融
RQ-VAE把连续 item embedding 量化为多层离散 code tuple
生成模型输入用户历史 SID 序列，输出下一个 item 的 SID
推理Beam Search + Trie constrained decoding + item filtering
指标Recall@K、NDCG@K、MRR@K、Coverage@K、TailRatio@K
BaselinePopularity、ItemKNN、SASRec、BERT4Rec、Random-ID GenRec
消融ID 类型、codebook 大小、SID 长度、beam size、history length、cold-start subset
错误分析Invalid SID、SID collision、热门偏置、语义漂移、重复推荐、冷启动失败
项目价值用公开数据证明 Semantic ID 对生成式推荐的泛化、冷启动和可控性有影响

  1. 能说清楚一个生成式推荐实验从数据到评估的完整 pipeline；
