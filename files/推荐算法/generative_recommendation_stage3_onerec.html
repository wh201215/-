推荐广告大模型新范式 Stage 3：OneRec 与工业级生成式推荐

Generative RecommendationOneRecRetrieve & Rank UnificationSemantic IDDPOSparse-MoEOpenOneRec

学习定位：Stage 2 已经讲了 TIGER 和 Semantic ID：推荐可以被改写成“生成目标 item 的离散标识符”。Stage 3 要继续往工业系统走，核心问题变成：

\[ \text{能不能用一个生成式模型，直接替代传统的召回、粗排、精排和重排？} \]

这就是 OneRec 系列最值得学习的地方。它不是单纯把 TIGER 放大，而是把推荐系统从多阶段级联系统，改造成端到端的自回归生成系统。

图 1：Stage 3 在整个学习路线中的位置。本文所有图为学习讲义式自绘示意图。

1. 从 TIGER 到 OneRec：范式升级

Stage 2 的核心是 TIGER。TIGER 的关键贡献是：给每个 item 分配一个由多个 codeword 组成的 Semantic ID，然后训练 Transformer seq2seq 模型根据用户历史行为生成下一个 item 的 Semantic ID。论文摘要强调，模型不是做向量检索，而是自回归解码目标候选 item 的标识符；Semantic ID 又是语义上有意义的 codeword tuple，因此它比随机 item_id 更利于泛化。

阶段核心问题典型形式局限
TIGER / Generative Retrieval能否直接生成目标候选 item？用户历史 → 生成下一个 item 的 Semantic ID更多是召回范式，仍可能接在传统排序系统前面
OneRec-V1能否统一召回和排序？用户历史 → 生成一组视频 / 一页结果Encoder 计算占比高，扩展到更大模型有瓶颈
OneRec-V2能否进一步降低计算、做大模型、用真实反馈对齐？Lazy Decoder-Only + preference alignment更接近工业 foundation recommender，但复现成本高
OpenOneRec / MiniOneRec能否公开复现、评测、后训练？开源数据、benchmark、SID 构造、SFT、RL公开数据和工业数据仍有差距
一句话：TIGER 证明“推荐可以生成 item 标识符”；OneRec 进一步证明“生成式模型可以承担完整推荐系统的一部分核心职责，而不是只做召回候选生成”。

2. 传统 Retrieve + Rank 的系统问题

工业推荐系统通常不是一个模型，而是一个多阶段工程系统：

全量 item 池
  ↓
多路召回 Recall：协同过滤 / 双塔 / 热门 / 规则 / 兴趣簇
  ↓
粗排 Pre-rank：轻量模型快速过滤
  ↓
精排 Rank：复杂模型估计 CTR / WatchTime / CVR
  ↓
重排 Re-rank：多样性、去重、业务约束、探索
  ↓
最终展示列表

这个架构非常成功，但它有几个天然问题：

问题具体含义对生成式推荐的启发
优化割裂召回、粗排、精排、重排分别训练，目标不完全一致如果能端到端生成最终结果，目标会更统一
候选集上限精排只能在召回给出的候选集合里选择生成式模型希望从 item token 空间中直接产生候选
手工规则重多样性、去重、组合策略常需要大量工程规则session-wise generation 希望直接学习一组结果的组合关系
系统碎片化不同阶段使用不同架构和不同 serving 链路OneRec 希望用统一生成模型减少级联复杂度
难利用 LLM 技术Transformer、MoE、DPO、RLHF 等技术不容易直接接入传统 ranker把推荐改成序列生成后，可以借鉴大模型训练和对齐范式
\[ \mathcal{C}_u=\operatorname{Retrieve}(u),\qquad s_i=f_{\theta}(u,i),\qquad \operatorname{TopK}=\operatorname{argsort}_{i\in\mathcal{C}_u}(s_i) \]

传统系统的关键限制在 \(\mathcal{C}_u\)：如果召回阶段没有把某个 item 放进候选集，那么后续精排模型再强也无法选择它。

3. OneRec 的问题定义：不只是生成式召回

OneRec 的标题是 Unifying Retrieve and Rank with Generative Recommender and Iterative Preference Alignment，它的重点是“统一召回和排序”。这和 TIGER 的侧重点不同：TIGER 更像 generative retrieval；OneRec 要处理的是线上推荐主场景中的最终生成质量。

OneRec 的目标：输入用户历史行为序列，模型直接生成用户可能感兴趣的一组 item，而不是先生成候选，再交给传统排序模型。

抽象地看，OneRec 要学习的是：

\[ Y_u=(i_1,i_2,\ldots,i_K),\qquad p_{\theta}(Y_u\mid H_u) \]

其中 \(H_u\) 是用户历史行为，\(Y_u\) 是当前请求要展示的一组推荐结果。每个 item 又可以被表示成若干 token：

\[ i_j\rightarrow (c_{j,1},c_{j,2},\ldots,c_{j,m}) \]

因此整页推荐可以写成一个长序列：

\[ Y_u=(c_{1,1},\ldots,c_{1,m},c_{2,1},\ldots,c_{K,m}) \]

自回归生成目标就是：

\[ p_{\theta}(Y_u\mid H_u)= \prod_{j=1}^{K}\prod_{t=1}^{m} p_{\theta}\left(c_{j,t}\mid H_u,c_{1:m}^{(1:j-1)},c_{j,\lt t}\right) \]
公式注意：这里的 \(c_{j,\lt t}\) 使用了 \(\lt\)，不是 HTML 里的小于号，避免公式在浏览器中被误解析。

4. Session-wise Generation：为什么不是简单 next-item prediction

传统序列推荐常做 next-item prediction：

\[ H_u=(i_1,i_2,\ldots,i_n)\rightarrow i_{n+1} \]

但线上推荐一次请求通常不是只展示一个 item，而是展示一组内容。比如短视频流、信息流广告、电商首页瀑布流，用户看到的是一个 session 或一页结果。OneRec-V1 的一个重要点是 session-wise generation：不是逐点生成一个 item 再用手工规则拼成列表，而是让模型学习整组推荐结果之间的上下文关系。

图 2：point-wise next-item 与 session-wise generation 的区别。
对比Next-item predictionSession-wise generation
训练对象下一个 item一组 item / 一页推荐结果
列表关系通常靠后处理规则模型生成时天然考虑前面已生成内容
多样性需要额外去重、多样性规则可以在解码、奖励、训练数据中学习
更像什么序列推荐语言模型生成一段回答

5. OneRec-V1 架构：Encoder-Decoder 生成推荐

OneRec-V1 使用 encoder-decoder 思路：encoder 编码用户历史行为序列，decoder 逐步生成用户可能感兴趣的视频。论文摘要还强调使用 Sparse-MoE 来扩展模型容量，同时避免计算 FLOPs 等比例增长。

图 3：OneRec-V1 encoder-decoder 架构示意。
OneRec-V1 abstract view:

User behavior sequence
    ↓
Encoder: compress user interest state
    ↓
Decoder: autoregressively generate item tokens
    ↓
Session-wise recommendation list
    ↓
Preference alignment with DPO-like training

6. 工业场景里的 Item Token / SID

生成式推荐必须解决一个基础问题：item 怎么进入语言模型式的 token 空间？原始 item_id 太稀疏、太大、无语义，因此 Stage 2 已经讲过 Semantic ID。到了 OneRec / OpenOneRec，item tokenization 进一步变成连接推荐系统和 LLM 的接口。

表示方式例子优点问题
原始 item_idvideo_39847291唯一、简单无语义，词表巨大，泛化差
Semantic ID(23, 501, 77)压缩、可生成、相似 item 可能共享前缀需要构造 tokenizer / quantizer
Itemic Token<s_a_340><s_b_6566><s_c_5603>可以和 LLM token 序列拼接需要对齐文本、行为、多模态信息

OpenOneRec 的 GitHub 说明里把 item 当成一种独立模态，并使用来自 hierarchical vector quantization 的 Itemic Tokens，让 LLM 能把交互历史作为连续上下文序列处理。

\[ \operatorname{ItemTokenizer}(i)= \left(z_1,z_2,\ldots,z_m\right),\qquad z_t\in\mathcal{V}_{\operatorname{item}} \]

如果引入文本、多模态和行为信号，可以抽象成：

\[ \mathbf{h}_i=\operatorname{Encoder}\left(x_i^{\operatorname{text}},x_i^{\operatorname{image}},x_i^{\operatorname{behavior}}\right) \]
\[ (z_1,\ldots,z_m)=\operatorname{Quantize}(\mathbf{h}_i) \]
理解重点：SID / Itemic Token 的作用不是“压缩编号”这么简单，而是给生成模型提供一个可学习、可泛化、可约束解码的 item 语言。

7. 训练目标：从 next item 到整页生成

如果只做 next-item，训练目标通常是：

\[ \mathcal{L}_{\operatorname{next}}(\theta)= -\sum_{u}\sum_{t=1}^{m}\log p_{\theta}\left(c_{n+1,t}\mid H_u,c_{n+1,\lt t}\right) \]

OneRec 的 session-wise 思路更接近生成一段目标序列：

\[ \mathcal{L}_{\operatorname{session}}(\theta)= -\sum_{u}\sum_{j=1}^{K}\sum_{t=1}^{m} \log p_{\theta}\left(c_{j,t}\mid H_u,c_{1:m}^{(1:j-1)},c_{j,\lt t}\right) \]

其中：

符号含义
\(H_u\)用户历史行为序列，包括观看、点击、停留、互动等
\(K\)本次请求生成的 item 个数
\(m\)每个 item 的 token 数，例如 3 层或 4 层 SID
\(c_{j,t}\)第 \(j\) 个 item 的第 \(t\) 个 token
\(c_{j,\lt t}\)同一个 item 中已经生成的前缀 token
\(c_{1:m}^{(1:j-1)}\)前面已经生成的 item token 序列
为什么这很重要：模型不只是学“用户喜欢什么 item”，还在学“这一页里前后 item 怎么搭配”。这和传统 ranker 独立打分 item 完全不同。

8. 推理与约束解码：生成出来的 token 必须是合法 item

生成式推荐的推理不是普通文本生成。文本生成时出现一个不常见的词也许可以接受，但推荐生成时，如果 SID 不对应任何真实 item，就无法展示。因此需要约束解码。

Naive decoding problem:

model outputs: (12, 87, 999)
item vocabulary: no item has this SID
  ↓
invalid recommendation

一种常见做法是用 SID Trie 或前缀树约束解码：

图 4：SID Trie 约束解码示意。

推理时通常还要处理：

问题处理方式
Invalid SID用 Trie mask 只允许合法 token
重复 item对已生成 item 做去重约束
不适合展示结合审核、库存、广告预算、曝光频控等过滤
探索不足温度、top-k、beam search、多样性奖励
延迟压力KV cache、批处理、短 token SID、量化推理

9. Sparse-MoE：如何扩大容量但不等比例增加计算

OneRec-V1 摘要明确提到 sparse Mixture-of-Experts：用来扩大模型容量，同时避免计算 FLOPs 按参数量等比例增长。这个思想和大语言模型中的 MoE 一样：总参数很多，但每个 token 只激活一小部分专家。

\[ \operatorname{MoE}(x)= \sum_{k\in\operatorname{TopK}(g(x))}g_k(x)E_k(x) \]

其中：

符号含义
\(g(x)\)router / gate，决定 token 分配给哪些专家
\(E_k(x)\)第 \(k\) 个专家网络
\(\operatorname{TopK}(g(x))\)只选得分最高的少数专家
MoE 在推荐里的直觉：不同用户、不同兴趣簇、不同内容类型、不同商业目标，可能需要不同专家来建模。Sparse-MoE 允许模型容量变大，但每次请求只走少数专家。

10. Iterative Preference Alignment 与 DPO

OneRec-V1 的另一条核心线是 Iterative Preference Alignment，并结合 Direct Preference Optimization（DPO）提升生成结果质量。DPO 在 NLP 里通常依赖成对偏好数据：同一个 prompt 下，一个回答更好，一个回答更差。推荐场景难点是：用户每次请求通常只看到一组结果，系统很难同时得到同一请求下正负两组完整展示结果。

DPO 的标准形式可以抽象为：

\[ \mathcal{L}_{\operatorname{DPO}}(\theta)= -\mathbb{E}_{(x,y^+,y^-)} \left[ \log\sigma\left( \beta\left( \log\frac{\pi_{\theta}(y^+\mid x)}{\pi_{\operatorname{ref}}(y^+\mid x)} - \log\frac{\pi_{\theta}(y^-\mid x)}{\pi_{\operatorname{ref}}(y^-\mid x)} \right) \right) \right] \]
在 NLP 中在推荐中
同一个 prompt 可以离线收集多个回答的偏好同一个用户请求通常只展示一次，很难同时得到正负列表
偏好可能来自人工标注偏好更多来自点击、观看时长、停留、转化等隐式反馈
回答质量相对稳定推荐反馈受位置、时间、上下文、曝光机会影响
奖励偏主观质量奖励多目标：时长、点击、留存、商业价值、生态约束

OneRec-V1 的处理思路是设计 reward model 来模拟用户反馈，并定制采样策略构造 DPO 样本。你可以把它理解成：

当前 OneRec 生成列表 A
        ↓
采样 / 扰动 / 生成候选列表 B
        ↓
Reward model 估计 A 和 B 谁更好
        ↓
构造 preference pair
        ↓
DPO 更新生成模型
        ↓
重新生成更符合用户偏好的列表
注意：推荐里的 DPO 不是把 NLP 的 DPO 直接套过来。它必须解决“没有同请求正负展示对”的问题，所以 reward model 和采样策略非常关键。

11. 为什么 OneRec 强调 MFU 和系统效率

Stage 1 的 RankMixer 已经讲过 MFU：Model FLOPs Utilization。传统推荐模型经常有大量稀疏 embedding lookup、特征拼接、规则模块和不规则计算，导致硬件利用率不高。OneRec 把推荐改成更像 LLM 的自回归序列建模后，理论上更容易利用大矩阵乘法、KV cache、MoE、量化等大模型系统优化。

\[ \operatorname{MFU}=\frac{\text{实际有效模型计算吞吐}}{\text{硬件理论峰值计算吞吐}} \]
系统计算形态硬件友好性
传统多阶段推荐召回、排序、规则、特征服务混合链路碎片化，GPU 利用不稳定
RankMixer / TokenMixer更统一的 ranking backbone比传统模块更适合 scaling
OneRec自回归生成 + Transformer / MoE更接近 LLM serving，可借鉴大模型系统优化

12. OneRec-V2：Lazy Decoder-Only 架构

OneRec-V2 解决的是 V1 继续放大时的两个关键问题：第一，计算分配不合理，论文摘要称 V1 中 97.66% 的资源被 sequence encoding 消耗，而不是 generation；第二，偏好对齐如果只依赖 reward model 也有局限。V2 提出 Lazy Decoder-Only Architecture，把总计算减少 94%、训练资源减少 90%，并支持扩展到 8B 参数。

图 5：OneRec-V1 到 OneRec-V2 的架构升级。
OneRec-V2 点具体解决什么
Lazy Decoder-Only去掉 V1 encoder bottleneck，降低历史编码计算压力
94% total computation reduction让大模型推荐在工业场景更可服务
90% training resources reduction降低训练成本，支撑更大参数规模
8B parameters说明生成式推荐进入 billion-level foundation recommender 阶段
Duration-Aware Reward Shaping把观看时长等真实用户行为反馈纳入偏好优化
Adaptive Ratio Clipping稳定策略更新，避免偏好优化破坏推荐质量
Stage 3 的关键理解：OneRec-V1 是“统一召回和排序”的突破；OneRec-V2 是“把这个范式做大、做快、做稳”的工程化升级。

13. OpenOneRec：从工业闭源到开源 Foundation Model

OpenOneRec 的意义是把 OneRec 系列从工业系统报告推向公开研究框架。它的目标不是只开源一个模型，而是提供 benchmark、训练数据、模型、训练 pipeline 和后训练流程。

模块OpenOneRec 提供什么学习价值
RecIF-Bench覆盖 8 类任务，从 item understanding 到 recommendation explanation不再只看 Recall / NDCG，而是看推荐、指令跟随、推理解释
Open Data论文摘要称释放大规模交互数据，GitHub 说明有跨短视频、广告、电商的数据组织解决生成式推荐公开复现数据不足
OneRec-Foundation1.7B 和 8B 模型族，基于 Qwen 架构说明推荐 foundation model 开始和通用 LLM 融合
Itemic Tokens用分层向量量化把 item 变成 token连接 item 模态和 LLM token 序列
Training Pipelinedata processing、co-pretraining、post-training提供从预训练到 RL 的完整路线

OpenOneRec 的 GitHub README 把训练流程概括为：

Pre-training:
  Itemic-Text Alignment
  Full-Parameter Co-Pretraining

Post-training:
  Stage 1: Multi-task SFT
  Stage 2: On-policy Distillation
  Stage 3: Reinforcement Learning

这个流程对你后续做项目非常重要，因为它把生成式推荐拆成了可落地的三件事：

  1. 先让模型认识 item token 和文本/行为之间的关系；
  2. 再用多任务 SFT 学会推荐指令和基本任务；
  3. 最后用蒸馏或 RL 强化推荐指标和偏好对齐。

14. MiniOneRec：公开复现路线

MiniOneRec 的价值在于把 OneRec 这类工业范式变成公开数据上可跑的框架。其摘要提出两个问题：生成式推荐在公开 benchmark 上是否存在 scaling law，以及最小后训练 recipe 是什么。MiniOneRec 提供 SID construction、SFT 和 recommendation-oriented RL 的端到端流程，并在 Amazon Review 数据上对 Qwen 0.5B 到 7B 进行后训练。

步骤MiniOneRec 式复现应该做什么你能学到什么
SID construction用 RQ-VAE 或量化方法把 item 变成 SID理解 tokenizer 对生成式推荐的影响
SFT用用户历史 → 目标 SID 的格式训练模型理解推荐任务如何变成语言模型任务
Constrained decoding只允许生成合法 SID理解生成式推荐 serving 的关键约束
RL / preference optimization用 recall、rank、diversity 等 hybrid reward 训练理解推荐后训练和普通 LLM RL 的区别
Scaling analysis比较 0.5B、1.5B、3B、7B 等模型观察 loss 和推荐指标是否随参数规模改善
对你最有用的地方：如果你要做一个能讲得出去的项目，不建议从 OneRec 工业系统复刻开始，而应该从 MiniOneRec 式公开数据流程开始：SID 构造 + SFT + 约束解码 + 小型 RL。

16. 读 OneRec 论文时应该抓什么

  1. 它到底统一了什么？不是统一所有推荐工程，而是用生成模型承担 retrieve + rank 的核心决策。
  2. session-wise generation 为什么重要？它让模型生成一组结果，而不是生成单点 item 后再规则拼接。
  3. item token 如何设计？原始 item_id、SID、Itemic Token 的差异会直接影响泛化和解码。
  4. MoE 放在哪里？它怎样增加容量，又怎样控制 FLOPs。
  5. DPO 样本怎么来？推荐系统没有天然的同请求正负回答，所以 reward model 和采样策略是重点。
  6. 线上指标是什么？OneRec-V1 报告快手主场景 watch-time 提升；V2 报告 App Stay Time 提升。
  7. V2 为什么改成 Lazy Decoder-Only？因为 V1 历史编码计算占比太高，继续 scaling 会被 encoder bottleneck 卡住。
  8. OpenOneRec 如何公开复现？看数据、Itemic Tokens、RecIF-Bench、SFT、distillation、RL。
最关键三问：
OneRec 和 TIGER 的区别是什么？
Session-wise generation 解决了传统推荐的什么问题？
推荐 DPO 为什么不能直接照搬 NLP DPO？

17. 可以做的实验项目

17.1 Toy OneRec：公开数据上的 session-wise generative recommender

Dataset:
Amazon Review / MovieLens / KuaiRec if available

Pipeline:
1. Build item content embeddings
2. Train RQ-VAE or simple hierarchical k-means tokenizer
3. Convert item_id to SID sequence
4. Format user history as token sequence
5. SFT a small decoder-only LM or Transformer
6. Use Trie-constrained decoding to generate legal SID
7. Evaluate Recall@K / NDCG@K / diversity

17.2 从 next-item 到 session-wise 的 ablation

实验目的
Next-item only验证基础生成式推荐能力
Generate Top-K independently观察重复和多样性问题
Session-wise sequence generation观察列表整体建模是否提升多样性和 Recall
Trie-constrained decoding验证 invalid SID 是否显著下降
Reward reranking / RL尝试把多样性、点击概率、长尾覆盖合成 reward

17.3 推荐后训练小方向

如果你后续想转成论文或面试项目，可以考虑：

SID-aware SFT
  ↓
Constrained decoding
  ↓
Hybrid reward:
  Recall reward
  Diversity reward
  Popularity penalty
  Invalid SID penalty
  User history consistency reward
  ↓
GRPO / DPO / pairwise preference optimization

这个方向比单纯复现 TIGER 更有含金量，因为它贴近 OneRec / MiniOneRec 的后训练路线。

18. 一页总结

概念核心理解
TIGER用 Semantic ID 把推荐召回改成自回归生成问题
OneRec-V1用生成式模型统一 retrieve 和 rank，输入用户历史，输出 session-wise 推荐结果
Session-wise Generation生成一组 item，而不是只预测下一个 item；列表内部关系由模型学习
Sparse-MoE扩大模型容量，但每个 token 只激活部分专家，控制计算成本
Iterative Preference Alignment用 reward model 和采样策略构造偏好样本，再用 DPO 类方法优化生成质量
OneRec-V2用 Lazy Decoder-Only 解决 V1 encoder bottleneck，并引入真实用户交互偏好对齐
OpenOneRec开放 benchmark、模型、数据和训练流程，把生成式推荐推向 foundation model 研究
MiniOneRec公开复现路线：SID 构造、SFT、约束解码、推荐 RL、scaling analysis
核心风险invalid SID、重复生成、延迟、线上反馈偏差、多目标冲突、item tokenizer 质量
学习重点把“推荐系统工程问题”翻译成“序列生成 + 约束解码 + 偏好对齐问题”
Stage 3 结束标准：
  1. 能讲清 TIGER 和 OneRec 的区别；
  2. 能解释 OneRec 为什么说自己统一 retrieve 和 rank；
  3. 能写出 session-wise generation 的训练目标；
  4. 能解释推荐场景里 DPO 为什么需要 reward model 和采样策略；
  5. 能讲清 OneRec-V2 为什么从 encoder-decoder 转向 Lazy Decoder-Only；
  6. 能设计一个公开数据上的 Toy OneRec 实验流程。
