A. 推荐系统基础与工业链路

1. 推荐系统通常分为哪些阶段？为什么不是一个模型直接做完？R13

工业推荐一般是多阶段级联系统：召回、粗排、精排、重排、混排和策略层。原因是候选规模从千万级到百级逐步缩小，不同阶段的目标、延迟预算、特征复杂度完全不同。

全量 item 池
  ↓ 召回：千万 → 千/万
  ↓ 粗排：万 → 千
  ↓ 精排：千 → 百
  ↓ 重排/混排：百 → 最终曝光列表

面试要强调：多阶段不是学术优雅，而是工业约束下的结果。召回追求高覆盖和低延迟，排序追求高精度，重排追求整体体验、约束和多样性。

2. 召回、粗排、精排、重排分别优化什么？
  • 召回：从全库快速找回可能相关的 item，核心是 Recall、覆盖率、延迟。
  • 粗排：用较轻模型过滤召回候选，兼顾速度和一定准确性。
  • 精排：使用最丰富特征和复杂模型，预测 CTR、CVR、时长、GMV 等。
  • 重排：考虑多样性、去重、频控、业务规则、探索、列表级体验。

标准回答里可以加一句：越靠前越看重效率和覆盖，越靠后越看重精度和策略一致性。

3. 推荐系统和搜索系统最大的区别是什么？R23

搜索有显式 query，用户主动表达意图；推荐通常没有显式 query，需要从历史行为、上下文、实时兴趣中推断意图。搜索更强调 query-item 语义匹配，推荐更强调用户长期兴趣、短期兴趣、探索和业务目标。

生成式推荐出来后，两者边界变模糊：Semantic ID 可以把搜索和推荐都改造成生成式检索任务。

4. CTR、CVR、CTCVR、GMV、WatchTime 分别是什么？
  • CTR：点击率，通常是点击 / 曝光。
  • CVR：转化率，通常是转化 / 点击，或者转化 / 曝光，取决于定义。
  • CTCVR：点击且转化概率，常写作 pCTR × pCVR。
  • GMV：成交金额，更接近商业收益。
  • WatchTime / StayTime：内容平台常用目标，衡量用户停留或观看时长。

面试追问通常是：为什么不能只优化 CTR？因为标题党、低质点击、短期刺激会损害留存、转化和长期体验。

5. 隐式反馈和显式反馈有什么区别？

显式反馈是评分、点赞、收藏等明确偏好；隐式反馈是点击、浏览、停留、购买、跳出等行为信号。推荐系统大多数真实数据是隐式反馈，因为规模大、实时性强，但噪声也更大。

  • 点击不一定喜欢，可能是误点。
  • 未点击不一定不喜欢，可能没看到。
  • 停留长不一定满意，可能内容难懂或页面卡顿。
  • 购买强度高，但稀疏且延迟。
6. 曝光、点击、转化三者在样本构造中有什么关系？R9

曝光是推荐系统真正给用户看到的 item，点击是在曝光基础上的行为，转化通常又发生在点击之后。因此训练 CTR 应该以曝光样本为主，训练 CVR 如果只用点击样本会产生样本选择偏差。

曝光 impression → 点击 click → 转化 conversion

ESMM 的经典动机就是：CVR 训练只看点击空间，但推理面对全曝光空间，会造成 sample selection bias 和数据稀疏问题。

7. 为什么推荐系统里负样本很难定义？

因为用户没有点击某个 item，不代表一定不喜欢。可能它没有曝光，可能位置不好，可能用户当时没时间。严格来说，只有曝光未点击更接近负样本，但也仍然受位置、时间、上下文影响。

面试回答要补一句：负样本采样会改变训练分布，所以要关注采样比例、采样来源、样本权重和线上校准。

8. 推荐系统中“候选集偏差”是什么？R15

候选集偏差是指排序模型只在上游召回给出的候选上训练和推理，导致它无法感知被召回漏掉的 item。上游召回越偏，排序模型越可能在局部候选中学到偏差。

生成式推荐试图缓解这一点：模型直接生成 item 标识符，不完全受固定召回候选集限制，但会引入 invalid ID、解码成本和可控性问题。

9. 推荐系统为什么要区分长期兴趣和短期兴趣？R3R4R5R6

长期兴趣稳定，例如用户长期喜欢数码、健身；短期兴趣受当前会话影响，例如最近正在看手机壳。只建模长期兴趣会不够敏感，只建模短期兴趣会被偶然行为带偏。

常见做法是长期画像 + session 序列 + target attention，DIN、DIEN、SASRec、BERT4Rec 都是在不同角度处理兴趣表示。

10. 推荐系统里的 personalization 和 relevance 有什么区别？

Relevance 是 item 与当前需求是否相关；personalization 是是否符合这个用户的个人偏好。同一个 item 可以对大多数人相关，但对某个用户不个性化；也可能对用户兴趣高度匹配，但当前场景不相关。

面试里可以举例：搜索“耳机”时相关性很强，但推荐首页没有 query，更依赖 personalization。

11. 为什么推荐系统容易形成信息茧房？

因为模型根据历史行为强化已有偏好，高点击类别获得更多曝光，更多曝光又产生更多点击，形成 feedback loop。热门 item 和已知兴趣会被反复放大，长尾和新兴趣难以进入。

  • 解决思路：探索流量、多样性约束、去重、类目打散、新颖性指标、长期收益建模。
  • 注意：多样性不能粗暴牺牲相关性，否则用户体验会下降。
12. 推荐系统为什么要做实时推荐？

用户兴趣有强实时性，尤其在短视频、新闻、电商场景。用户刚搜索、点击、加购、停留，可能反映当前即时意图。如果只用离线画像，响应会滞后。

实时推荐通常依赖实时特征、实时行为序列、在线 embedding 更新、实时召回，以及流式样本回流。

13. 推荐模型为什么不能只看离线 AUC？R20

AUC 衡量正负样本排序能力，但线上收益受候选集、策略、延迟、校准、多目标、用户体验和流量分布影响。离线 AUC 提升不一定带来 CTR 或留存提升。

面试标准句：离线指标用于筛选模型，线上 A/B 才是最终裁判。离线要同时看 AUC、LogLoss、NDCG、Recall、Calibration、分桶效果和业务指标。

14. 冷启动分为哪几类？怎么解决？R15
  • 新用户冷启动：没有历史行为，依赖注册信息、上下文、热门、问卷、探索。
  • 新 item 冷启动：没有交互，依赖内容特征、类目、图文 embedding、商家/作者信息。
  • 新场景冷启动：业务刚上线，依赖迁移学习、相似场景、多任务共享。

生成式推荐里的 Semantic ID 对新 item 有潜在优势，因为它可以从 item 内容构造 ID，而不完全依赖行为历史。

15. 推荐系统中“记忆”和“泛化”怎么理解？R1

记忆是模型学到历史上出现过的强组合，例如用户喜欢某品牌；泛化是模型能处理未见过的新组合，例如新品牌但同类风格。Wide & Deep 的经典解释就是 Wide 侧负责 memorization，Deep 侧负责 generalization。

B. 召回与匹配八股

16. UserCF 和 ItemCF 的区别是什么？

UserCF 根据相似用户推荐物品，ItemCF 根据相似物品推荐物品。UserCF 更适合用户规模较小、兴趣相似明显的场景；ItemCF 在工业中更常用，因为 item 相似关系相对稳定，解释性也强。

UserCF：喜欢相同 item 的用户相似
ItemCF：被相同用户喜欢的 item 相似
17. ItemCF 的相似度怎么计算？

最常见是共现相似度、余弦相似度、Jaccard、修正余弦等。以余弦为例：

\[\operatorname{sim}(i,j)=\frac{|U_i \cap U_j|}{\sqrt{|U_i|\,|U_j|}}\]

工业里还会加入时间衰减、行为权重、类目约束、用户活跃度惩罚，避免超级活跃用户和热门 item 主导相似度。

18. 为什么 ItemCF 要惩罚热门 item？

热门 item 和很多 item 都共现，如果不惩罚，会导致所有 item 都和热门 item 相似，推荐结果变得同质化。常见方式是对用户贡献做活跃度降权，对 item 共现做 IDF 或 popularity penalty。

\[w_{u}=\frac{1}{\log(1+|I_u|)}\]
19. Swing 召回的核心思想是什么？

Swing 是一种基于用户共同兴趣的 item 相似度方法。它不仅看 item 是否被同一批用户点击，还惩罚用户对之间兴趣集合过大的情况。直觉是：两个小众兴趣相近的用户共同点击的 item，更能说明 item 相似；两个超级活跃用户共同点击，信息量较低。

面试里可以说：Swing 是比普通共现更注重用户对贡献质量的 ItemCF 变体。

20. 矩阵分解 MF 的基本思想是什么？

矩阵分解把用户和物品映射到同一个隐向量空间，用内积表示偏好。

\[\hat r_{ui}=p_u^{\top}q_i\]

它的核心假设是：用户偏好和 item 属性可以用少量 latent factors 表示。优点是简单有效，缺点是难以利用复杂上下文和高阶特征。

21. BPR Loss 是什么？适合什么场景？

BPR 是 pairwise 排序损失，适合隐式反馈推荐。它希望用户真实交互 item 的分数高于未交互 item。

\[\mathcal{L}_{\operatorname{BPR}}=-\sum_{(u,i,j)}\log \sigma(\hat y_{ui}-\hat y_{uj})\]

其中 i 是正样本，j 是负样本。面试追问通常是负样本如何采样：随机负样本简单但太容易，hard negative 更有效但可能引入噪声。

22. DSSM / 双塔召回的结构是什么？R24

DSSM 的核心是把 user/query 和 item/doc 分别编码到同一向量空间，然后用点积或 cosine 计算相似度。双塔结构适合大规模召回，因为 item 向量可以离线预计算，线上只算 user 向量再做 ANN 检索。

User Tower        Item Tower
   ↓                 ↓
user embedding   item embedding
        \      /
       dot / cosine

DSSM 最早用于 query-document 语义匹配，后来双塔思想广泛用于推荐召回。

23. 双塔模型为什么容易“弱交互”？

双塔模型在 user 和 item 分别编码后只用点积交互，表达能力受限。它无法像精排模型那样使用丰富的交叉特征，例如用户年龄 × item 类目 × 当前时间。

解决办法包括：增强 tower 特征、引入多兴趣向量、hard negative、蒸馏精排模型、late interaction、召回后接更强排序。

24. ANN 召回是什么？为什么需要它？

ANN 是近似最近邻搜索。召回时需要从百万/亿级 item embedding 中快速找 top-K，如果精确暴力计算点积成本太高，就使用 HNSW、IVF、PQ 等近似方法。

面试要讲 trade-off：ANN 牺牲一点召回精度，换取巨大速度提升。

25. 多路召回为什么重要？

单一路召回容易覆盖不足。工业系统通常会同时使用 ItemCF、双塔、热门、实时行为、同类目、内容相似、社交关系、地理位置、探索召回等多路结果。

多路召回 → 合并 → 去重 → 截断 → 粗排

追问：多路召回怎么融合？可以按通道配额、分数归一化、学习融合、召回来源作为特征给排序模型。

26. 为什么召回要关注覆盖率？

召回的上限决定了后续排序能做到多好。如果好 item 没被召回，排序再强也无能为力。覆盖率包括用户覆盖、item 覆盖、类目覆盖和长尾覆盖。

常见指标有 Recall@K、Coverage、命中精排正样本比例、各召回通道贡献。

27. 召回阶段如何处理新 item？R15

新 item 没有行为 embedding，常用内容召回、类目召回、作者/商家召回、冷启动池、探索流量、内容模型生成 embedding。生成式推荐中，Semantic ID 可以由内容 embedding 量化得到，因此对新 item 更友好。

28. MIND 为什么提出多兴趣向量？R12

一个用户可能同时喜欢数码、运动、母婴、电影等多个兴趣，用单个 user embedding 会混在一起。MIND 用 capsule routing 从用户行为中提取多个兴趣向量，在召回阶段可以用多个兴趣向量分别检索。

user history → multi-interest extractor → interest_1, interest_2, ..., interest_k

面试回答重点：MIND 是为召回阶段服务的多兴趣用户表示。

29. 多兴趣召回怎么避免结果重复？

不同兴趣向量可能召回相同或相似 item，需要在融合时做去重、类目打散、通道配额和多样性约束。否则多兴趣只是增加召回量，并不一定改善最终体验。

30. 召回模型如何做负样本？

常见负样本包括随机负样本、曝光未点、同 batch 负样本、hard negative、跨类目负样本。召回模型如果只用随机负样本，训练任务太容易，线上区分能力弱；hard negative 能提升边界，但要防止把潜在正样本当负样本。

C. 排序、CTR 与特征交叉八股

31. CTR 预估的基本任务是什么？

CTR 预估是预测用户在某个上下文下点击某个 item/ad 的概率。通常是二分类问题，常用 sigmoid 输出和交叉熵损失。

\[\hat y=\sigma(f_\theta(x)),\quad \mathcal{L}=-y\log \hat y-(1-y)\log(1-\hat y)\]

工业里 CTR 不是唯一目标，还要联合 CVR、时长、收益、负反馈等。

32. LR 在推荐排序里为什么仍然重要？

LR 简单、可解释、训练快、线上稳定，适合作为 baseline 或策略特征的校准模型。早期广告 CTR 大量依赖 LR + 人工交叉特征。

它的限制是表达能力弱，需要大量人工特征工程，难以自动学习高阶非线性交互。

33. FM 解决了什么问题？R7

FM 用隐向量分解参数来建模二阶特征交叉，尤其适合稀疏高维特征。

\[\hat y=w_0+\sum_i w_i x_i+\sum_{i=1}^n\sum_{j=i+1}^n \langle v_i,v_j\rangle x_i x_j\]

关键点：即使某个特征组合没出现过，也能通过各自 embedding 的内积估计交互。

34. FM 和矩阵分解有什么关系？

矩阵分解可以看作 FM 在只有 user_id 和 item_id 两个特征域时的特例。FM 更一般，可以处理任意稀疏特征，包括用户、物品、上下文、广告、设备等。

35. FFM 相比 FM 改进在哪里？

FFM 是 Field-aware Factorization Machine。同一个特征在和不同 field 的特征交互时使用不同 embedding。比如 user_id 与 category 交互、user_id 与 shop 交互，可以用不同向量。

缺点是参数量更大，线上成本更高。

36. Wide & Deep 的核心思想是什么？R1

Wide & Deep 联合训练 wide 线性部分和 deep 神经网络部分。Wide 侧记忆强规则和共现组合，Deep 侧通过 embedding 泛化到未见组合。

Wide: sparse feature crosses → memorization
Deep: embeddings + MLP → generalization

面试一句话：Wide & Deep 解决的是记忆与泛化的平衡。

37. DeepFM 相比 Wide & Deep 有什么优点？R2

DeepFM 把 FM 和 DNN 结合起来，并共享同一套 embedding 输入。FM 负责低阶特征交叉，DNN 负责高阶非线性交互。相比 Wide & Deep，它不需要额外手工设计 wide 侧交叉特征。

raw features → shared embeddings → FM part + Deep part → output

面试重点：DeepFM 是自动学习低阶和高阶特征交互。

38. xDeepFM 的 CIN 解决什么问题？R25

普通 DNN 学习高阶交互是隐式的、bit-wise 的；xDeepFM 的 CIN 显式学习 vector-wise 的高阶特征交互。它把显式交叉和隐式 DNN 结合。

标准回答：xDeepFM 希望同时保留显式可控交叉和深层泛化能力。

39. DCN 的 Cross Layer 是什么？R8

DCN 用 Cross Network 显式构造有界阶数的特征交叉。典型 cross layer 形式为：

\[x_{l+1}=x_0 x_l^{\top} w_l+b_l+x_l\]

它的特点是参数效率高、交叉阶数随层数增加。DCN-V2 进一步增强 cross network 的表达能力，使其更适合大规模工业排序。

40. AutoInt 为什么用 Self-Attention 做特征交叉？R26

AutoInt 把每个特征 field 的 embedding 看作 token，用多头自注意力显式学习高阶特征交互。多层 attention 后，不同 field 的信息可以多次交互。

它的优势是自动学习重要交叉并具有一定解释性；问题是 attention 成本和工业延迟要控制。

41. DIN 的核心创新是什么？R3

DIN 不把用户历史压缩成固定向量，而是针对当前候选 item，用 activation unit 从历史行为里挑出相关兴趣。

候选广告 / item
      ↓
对历史行为做 target attention
      ↓
得到与当前候选相关的兴趣表示

面试重点：DIN 的用户兴趣表示是 target-aware 的，同一个用户面对不同候选会有不同兴趣向量。

42. DIN 为什么比固定用户向量更适合广告 CTR？R3

广告候选差异很大，用户历史兴趣也多样。如果用固定用户向量，会把多个兴趣混合在一起。DIN 对每个候选动态激活相关历史行为，可以更好捕捉“用户当前对这个 item 是否感兴趣”。

43. DIEN 相比 DIN 改进在哪里？R4

DIN 关注 target-aware 的历史行为加权，但没有显式建模兴趣随时间演化。DIEN 引入兴趣抽取层和兴趣演化层，通常使用 GRU/AUGRU 来捕捉用户兴趣变化趋势。

一句话：DIN 建模兴趣相关性，DIEN 进一步建模兴趣演化。

44. DLRM 的结构是什么？R14

DLRM 是 Meta 提出的深度推荐模型，典型结构是稀疏类别特征通过 embedding table，稠密特征通过 bottom MLP，然后做 feature interaction，再接 top MLP 输出预测。

dense features → bottom MLP ┐
sparse ids → embeddings ─────┤→ interaction → top MLP → prediction

它代表了工业 CTR 模型的经典形态：大规模 embedding + dense MLP + 特征交互。

45. 排序模型为什么离不开 embedding？

推荐特征大量是高基数类别特征，例如 user_id、item_id、author_id、query、shop_id。One-hot 太稀疏，embedding 可以把离散 ID 映射到低维稠密向量，便于学习相似性和交互。

46. Embedding 维度怎么选？

维度太小表达不足，太大容易过拟合、成本高、训练慢。常见做法是按特征基数、频次、业务重要性设维度；高频高价值特征给更大维度，低频特征可以共享、哈希或降维。

追问：embedding 维度不是越大越好，线上内存和延迟往往是硬约束。

47. Hash Trick 在推荐里有什么作用？

Hash Trick 把大量稀疏 ID 映射到固定桶，降低存储成本并支持 OOV。缺点是 hash collision：不同 ID 可能共享同一 embedding，造成信息混淆。

工业里会对核心特征保留独立 embedding，对长尾特征使用 hashing 或分桶。

48. 特征交叉为什么重要？

用户点击往往不是单个特征决定，而是组合决定，例如“年轻用户 × 夜间 × 短视频 × 美妆类目”。LR 需要人工交叉，FM/DCN/DeepFM/AutoInt 等模型都在尝试自动学习有效交叉。

面试可以说：推荐排序的核心之一就是高维稀疏特征下的有效交互建模。

49. 显式特征交叉和隐式特征交叉区别？
  • 显式交叉：模型结构明确构造交互，如 FM、DCN、CIN。
  • 隐式交叉：通过 MLP 等非线性网络隐式学习交互。

显式交叉可控、参数效率高；隐式交叉表达能力强但可解释性弱，可能需要更多数据。

50. 为什么精排模型不能无限变大？R21

精排面对高 QPS 和严格延迟，模型变大直接增加推理成本。大 embedding、复杂 attention、MoE routing、特征读取都会增加延迟和资源。

RankMixer 这类工作正是从硬件效率和 MFU 角度重新设计 ranking backbone。

51. RankMixer 想解决什么问题？R21

RankMixer 面向工业排序模型 scaling，认为传统手工 feature-crossing 模块低 MFU、难放大、GPU 不友好。它用 token mixing、per-token FFN、Sparse-MoE 等结构构造更统一、硬件友好的 ranking backbone。

52. Wukong 和 RankMixer 的区别是什么？R22R21

Wukong 重点证明推荐模型也可以出现 scaling law，并用 stacked FM 和 upscaling 策略验证规模扩大带来的收益；RankMixer 更偏工业排序架构，强调硬件友好、MFU、低延迟和大模型部署。

53. 为什么排序模型要做校准？

模型输出分数如果作为概率使用，需要和真实点击/转化率匹配。AUC 高只能说明排序好，不代表概率准。校准影响出价、流量分配、多目标融合和业务策略。

常见方法：Platt scaling、isotonic regression、分桶校准、温度缩放、后验修正。

54. LogLoss 和 AUC 有什么区别？R20

AUC 看排序能力，对概率绝对值不敏感；LogLoss 看概率预测质量，错误且自信会被重罚。一个模型可能 AUC 高但校准差，也可能 LogLoss 好但 top 排序改善不大。

工业里通常同时看 AUC、LogLoss、Calibration 和业务指标。

55. 为什么模型离线提升很小也可能值得上线？

在大流量推荐系统中，0.1% 的线上指标提升可能带来很大收益。尤其是精排主模型，baseline 已经很强，离线小幅提升如果稳定、成本可控、分桶无损，仍然值得 A/B。

D. 序列推荐与兴趣建模八股

56. 序列推荐和普通协同过滤有什么不同？

普通协同过滤更多关注用户整体偏好，序列推荐关注行为顺序和时间动态，预测用户下一步可能交互什么。

\[p(i_{t+1}\mid i_1,i_2,\ldots,i_t)\]

短视频、新闻、电商会话、音乐播放都非常依赖序列推荐。

57. GRU4Rec 的思想是什么？

GRU4Rec 用 RNN/GRU 建模 session 内行为序列，根据前面的点击预测下一个 item。它是 session-based recommendation 的经典神经方法之一。

它的优势是自然处理顺序；缺点是训练和推理不如 Transformer 并行，长程依赖捕捉有限。

58. SASRec 的核心思想是什么？R5

SASRec 使用 self-attention 做序列推荐，用用户历史序列中的 item 互相注意来预测下一个 item。它结合了 RNN 捕捉长期语义和 Markov 方法关注少数关键行为的优势。

面试重点：SASRec 是单向自回归式 sequential recommendation。

59. SASRec 为什么需要 causal mask？

因为预测第 t 个位置时不能看到未来 item，否则会信息泄漏。causal mask 保证每个位置只 attend 到过去位置。

\[\operatorname{Attention}(Q,K,V)=\operatorname{softmax}\left(\frac{QK^{\top}}{\sqrt d}+M\right)V\]

其中 M 对未来位置加负无穷。注意公式里用 mask 防止未来信息泄漏。

60. BERT4Rec 和 SASRec 最大区别是什么？R6

SASRec 是从左到右预测下一项，BERT4Rec 用双向 Transformer 和 Cloze/masked item prediction。BERT4Rec 可以同时利用左右上下文学习序列表示，但训练时要通过 mask 防止直接看到目标 item。

一句话：SASRec 是 next-item autoregressive，BERT4Rec 是 masked-item bidirectional。

61. 序列推荐里位置编码有什么作用？

Transformer 本身不感知顺序，位置编码告诉模型行为发生的顺序。推荐序列里顺序很重要：最近点击通常比很久以前点击更能反映短期兴趣。

62. 为什么用户历史序列要截断？

历史太长会增加计算成本，也可能引入过时兴趣。通常保留最近 N 个行为，或者按行为类型/时间衰减筛选。长序列建模要考虑效率和兴趣漂移。

63. 序列推荐中时间间隔怎么建模？

行为间隔本身有信息。连续点击同类 item 说明短期强兴趣；很久以前的行为权重应降低。可以加入时间差 embedding、时间衰减、session 切分、相对时间位置编码。

64. DIN/DIEN 和 SASRec 的区别？R3R4R5

DIN/DIEN 多用于 CTR 精排，围绕候选 item 对历史行为做 target-aware 建模；SASRec 多用于序列推荐或召回，直接预测下一个 item。

  • DIN：候选 item 触发历史兴趣。
  • DIEN：兴趣抽取 + 兴趣演化。
  • SASRec：self-attention 建模行为序列。
65. 序列推荐中的训练目标有哪些？
  • Next-item prediction：给历史预测下一个 item。
  • Masked item prediction：随机 mask 序列中的 item 并恢复。
  • Pairwise ranking：正 item 分数高于负 item。
  • Contrastive learning：增强序列表示鲁棒性。

生成式推荐通常也是 next-token / next-item 的形式，只是 item 被表示为 token 或 Semantic ID。

66. 为什么序列推荐会有 exposure bias？

训练时模型看到真实历史，推理时模型可能基于自己前一步推荐导致的新行为分布继续预测。训练分布和线上反馈分布不一致，就会有 exposure bias。生成式推荐也有类似问题：训练用真实 SID，推理用模型生成 SID。

67. 长序列推荐的难点是什么？

长序列会带来 O(n²) attention 成本、噪声行为、兴趣漂移、多兴趣混合和实时更新问题。常见解决思路包括分层建模、兴趣聚类、时间衰减、稀疏 attention、记忆压缩。

68. 为什么行为类型要区分点击、收藏、加购、购买？

不同行为强度不同。点击弱、收藏中等、加购和购买更强，但也更稀疏。模型可以用行为类型 embedding、行为权重、多任务目标或序列事件类型建模不同信号。

E. 多目标、多任务、样本偏置八股

69. 多目标推荐为什么困难？

推荐目标往往冲突：CTR 高不一定转化高，时长高不一定满意，收入高可能损害体验。多目标推荐要平衡短期点击、长期留存、商业收入、内容生态和用户满意度。

70. CTR 和 CVR 为什么不能简单分别训练？R9

CVR 样本通常只来自点击后的用户，样本稀疏且有选择偏差。单独训练 CVR 可能在全曝光空间推理时失真。ESMM 通过联合建模 CTR、CVR、CTCVR，把 CVR 学习放到整个曝光空间。

71. ESMM 的核心公式是什么？R9

ESMM 利用用户行为链路：曝光 → 点击 → 转化。它建模：

\[pCTCVR=pCTR\times pCVR\]

其中 CTR 和 CTCVR 都可以在全曝光空间监督，CVR 通过关系间接学习，从而缓解样本选择偏差和数据稀疏。

72. MMoE 解决什么问题？R10

MMoE 用多个 expert 和多个 task-specific gate 学习不同任务之间的共享与差异。相比所有任务共享一个底层网络，MMoE 可以让不同任务选择不同专家组合，缓解任务冲突。

\[y_k=h_k\left(\sum_{i=1}^{n}g_{k,i}(x)f_i(x)\right)\]

其中 k 是任务，f_i 是专家，g_{k,i} 是第 k 个任务对专家 i 的门控权重。

73. PLE 相比 MMoE 改进在哪里？R11

PLE 将 shared experts 和 task-specific experts 分层组织，逐层抽取共享信息和任务专属信息，重点缓解多任务中的负迁移和 seesaw 现象。

一句话：MMoE 是多门控专家，PLE 进一步把共享与专属专家分离并逐层提取。

74. 什么是负迁移？

负迁移指多任务共享参数后，一个任务的学习损害另一个任务。例如优化点击可能让模型偏向标题党，从而损害转化或留存。

解决思路：任务权重调节、MMoE/PLE、梯度冲突处理、任务分层、样本分桶、目标解耦。

75. 多任务 loss 权重怎么设？

可以手工设定，也可以动态调整。手工权重依赖业务经验；动态方法包括 uncertainty weighting、GradNorm、按指标敏感性调节、按训练阶段调节。

面试里要强调：loss 权重不是纯技术问题，它反映业务目标优先级。

76. 样本选择偏差是什么？R9

训练样本分布和线上推理分布不一致。例如 CVR 只在点击样本上训练，但线上要对所有曝光 item 估计转化概率。这样训练空间比推理空间更窄，产生偏差。

77. 位置偏差是什么？怎么处理？

排名靠前的 item 更容易被看到和点击，因此点击不完全代表偏好。处理方式包括随机化实验、IPS 反偏、position feature、点击模型、探索流量、counterfactual learning。

\[\mathcal{L}_{IPS}=\sum_i \frac{o_i}{p_i}\,\ell_i\]

其中 p_i 是曝光/观察概率，o_i 表示是否被观察。

78. 延迟反馈问题是什么？

转化、留存、复购等反馈可能在曝光后很久才发生。如果过早把未转化样本当负样本，会污染训练。解决方式包括延迟窗口、survival modeling、样本回补、正负样本状态更新。

79. 数据穿越/信息泄漏有哪些常见情况？
  • 用到了曝光之后才产生的特征。
  • 训练集和测试集按随机切分导致未来行为进入训练。
  • 负样本从未曝光 item 中采样，却当作真实负反馈。
  • 特征统计使用了全量数据，没有按时间截断。

推荐系统面试很喜欢问这个，因为离线实验容易虚高。

80. Hard Negative 是什么？

Hard negative 是模型容易混淆的负样本，例如同类目、相似 embedding、曝光未点击、高分但未转化的 item。它比随机负样本更能训练决策边界。

风险是可能包含潜在正样本，尤其是用户没看到或未来会喜欢的 item。

81. 如何解决热门偏置？

热门偏置来自曝光、点击和训练数据的正反馈循环。可以通过重采样、样本降权、长尾指标、探索流量、去流行度正则、重排多样性来缓解。

但不能简单压热门，因为热门本身往往质量高，要平衡准确性和长尾覆盖。

82. 推荐系统中的因果问题是什么？

用户点击某个 item，并不一定是模型推荐造成的，也可能是 item 本身热门、位置靠前、用户已有强需求。因果推荐关注“如果不推荐这个 item，用户是否仍然会点击/转化”。

常见关键词：counterfactual、uplift、IPS、propensity score、A/B randomized exposure。

F. 评估、A/B 与工程八股

83. Recall@K 怎么定义？R20

Recall@K 衡量真实相关 item 是否出现在前 K 个推荐中。

\[\operatorname{Recall@K}=\frac{|R_u\cap \hat R_u^K|}{|R_u|}\]

召回阶段尤其看 Recall@K，但排序阶段还要看位置质量。

84. NDCG@K 为什么比 Recall@K 更细？R20

NDCG 考虑命中 item 的位置，越靠前权重越高。推荐列表的前几位影响最大，所以 NDCG 更符合排序体验。

\[\operatorname{DCG@K}=\sum_{i=1}^{K}\frac{rel_i}{\log_2(i+1)}\]

NDCG 是 DCG 归一化后的结果。

85. MRR@K 适合什么场景？

MRR 关注第一个相关结果出现的位置，适合用户只会点击一个结果或第一个命中非常重要的场景。

\[\operatorname{MRR}=\frac{1}{|U|}\sum_{u\in U}\frac{1}{\operatorname{rank}_u}\]
86. AUC 的直观含义是什么？

AUC 可以理解为随机抽一个正样本和一个负样本，模型把正样本排在负样本前面的概率。AUC 对阈值不敏感，适合二分类排序能力评估。

87. GAUC 为什么比 AUC 更适合推荐？

普通 AUC 混合所有用户，活跃用户可能主导指标。GAUC 先按用户计算 AUC，再按用户权重加权，更能反映个性化排序效果。

常见权重是用户曝光数或点击数。

88. 离线评估为什么要按时间切分？

推荐系统是时间序列问题。如果随机切分，未来行为会进入训练，导致信息泄漏。按时间切分或 leave-one-out 更接近真实线上：用过去预测未来。

89. A/B 实验怎么设计？

基本原则是随机分流、实验组和对照组只改一个关键变量、样本量足够、周期覆盖业务波动、同时看核心指标和护栏指标。

  • 核心指标：CTR、CVR、GMV、WatchTime、留存。
  • 护栏指标：延迟、投诉、负反馈、内容生态、收入风险。
90. 为什么 A/B 要看分桶效果？

总体提升可能掩盖局部伤害。例如新模型对高活用户提升、对新用户下降；对热门类目提升、对小众类目下降。分桶能发现模型是否稳定。

常见分桶：新老用户、活跃度、地域、设备、类目、流量来源、冷启动 item、长尾 item。

91. 推荐服务延迟主要花在哪里？
  • 特征读取：在线 KV、特征服务、实时特征拼接。
  • Embedding lookup：大表查询、跨机通信。
  • 模型推理：MLP、attention、MoE、batching。
  • 召回检索：ANN 查询、多路召回合并。
  • 策略层：去重、过滤、重排、多样性。

工业模型不能只看 FLOPs，还要看内存访问、通信、batch size、cache 命中。

92. 什么是特征穿越和训练服务不一致？

训练时使用的特征和线上服务时拿到的特征不一致，会导致模型线上效果下降。原因可能是统计窗口不同、实时特征延迟、缺失值处理不同、特征版本不一致。

解决方式：统一 feature pipeline、特征回放、训练服务一致性校验、监控特征分布漂移。

93. 推荐系统怎么做模型上线灰度？

通常先离线验证，再小流量 A/B，再扩大流量，最后全量。过程中监控指标、延迟、错误率、特征缺失、分桶效果和异常报警。

offline → shadow / replay → 1% A/B → 5% → 20% → full traffic
94. 推荐系统中的探索与利用怎么平衡？

利用是推荐模型认为最可能成功的 item，探索是给不确定 item 一些机会。只利用会强化历史偏见，只探索会损害短期体验。

常见方法：epsilon-greedy、UCB、Thompson sampling、探索流量池、新品扶持、重排注入。

G. 生成式推荐与 Semantic ID 八股

95. 什么是生成式推荐？R15

生成式推荐把推荐从“对候选 item 打分排序”改造成“根据用户历史直接生成 item 标识符或推荐列表”。它的关键不是生成推荐理由，而是生成推荐对象本身。

\[p(i_{t+1}\mid H_u)\]

如果 item 被表示为 token 序列，就可以像语言模型一样自回归生成。

96. 生成式推荐和传统 Retrieve + Rank 的区别？R15R17

传统系统先召回候选，再排序；生成式推荐试图直接从用户历史生成候选或列表。

传统：User → Recall candidates → Rank → Re-rank
生成式：User history → Generator → item IDs / Semantic IDs

但工业上不一定完全替代所有策略，仍需要过滤、约束解码、重排和安全控制。

97. 为什么不能直接生成原始 item_id？R15

原始 item_id 没有语义结构，item_123 和 item_124 不一定相似；大规模 item 词表巨大；长尾 item 学习困难；新 item 没历史行为难生成。

因此很多生成式推荐工作使用 Semantic ID，把 item 映射成语义上有结构的离散 token 序列。

98. Semantic ID 是什么？R15

Semantic ID 是 item 的离散语义标识符，通常由 item 内容 embedding 或行为 embedding 经过量化得到。相似 item 可能共享部分 codeword。

item embedding → quantization → Semantic ID = (c1, c2, c3, c4)

TIGER 的核心就是构造 semantically meaningful codeword tuple 作为 item 标识，并训练模型预测下一个 item 的 Semantic ID。

99. RQ-VAE / 残差量化怎么生成 Semantic ID？

残差量化逐层从 codebook 中选择最接近当前残差的 codeword，每层量化后更新残差。

\[c_m=\arg\min_{k}\left\|r_{m-1}-e_{m,k}\right\|_2^2,\quad r_m=r_{m-1}-e_{m,c_m}\]

最终 item embedding 被表示为多个 codeword 的组合，形成 Semantic ID。

100. TIGER 的流程是什么？R15
  1. 用 item 内容信息训练 tokenizer / quantizer，得到每个 item 的 Semantic ID。
  2. 把用户历史 item 序列替换成 Semantic ID 序列。
  3. 训练 Transformer seq2seq 模型，根据历史 SID 预测下一个 item 的 SID。
  4. 推理时自回归生成 SID，再映射回真实 item。

它是 Semantic ID-based generative recommendation 的经典起点。

101. 生成式推荐的训练目标怎么写？

如果下一个 item 的 Semantic ID 是 \((c_1,c_2,\ldots,c_M)\)，模型按 token 自回归预测：

\[p(c_1,c_2,\ldots,c_M\mid H_u)=\prod_{m=1}^{M}p(c_m\mid c_{\lt m},H_u)\]
\[\mathcal{L}_{NTP}=-\sum_{m=1}^{M}\log p_\theta(c_m\mid c_{\lt m},H_u)\]
102. 什么是 constrained decoding？

生成式推荐可能生成不存在的 SID。Constrained decoding 用 Trie 或合法前缀表限制每一步只能生成仍可能构成真实 item 的 token。

prefix = (12, 7)
合法下一 token = {3, 9, 20}
模型只能在这些 token 中选择

这是生成式推荐落地非常关键的问题。

103. Beam Search 在生成式推荐里怎么用？

Beam Search 保留多个高概率 SID 前缀，逐步扩展成 top-K item。它比 greedy 更能找到多个候选，但 beam 太大会增加延迟。

面试要讲 trade-off：beam size 越大，召回多样性和准确率可能提升，但推理成本增加。

104. 生成式推荐中的 invalid ID 问题是什么？

模型可能生成一个不存在的 Semantic ID，无法映射到真实 item。这类似 LLM 幻觉，但对象是 item 标识符。

解决方式包括 constrained decoding、SID Trie、训练时加入合法性约束、后处理过滤、改进 tokenizer。

105. Semantic ID 的好坏如何影响推荐？R23

SID 决定了生成任务的难度和泛化能力。好的 SID 应该语义紧凑、冲突少、层级结构合理、支持冷启动和长尾泛化；差的 SID 可能导致相似 item 被分散，或者无关 item 共享前缀。

Spotify 的 joint generative search/recommendation 工作也强调 SID 构造方式是统一生成模型效果的关键。

106. Random ID、Atomic ID、Semantic ID 有什么区别？
  • Atomic ID：每个 item 一个独立 token，词表巨大，缺少结构。
  • Random ID：随机离散编码，可能降低词表但无语义。
  • Semantic ID：由内容或行为 embedding 量化得到，有语义结构和泛化能力。

面试重点：Semantic ID 的价值在于把 item 空间变成可生成、可共享、可泛化的 token 空间。

107. 生成式推荐如何处理冷启动 item？R15

如果 SID 来自 item 内容 embedding，那么新 item 即使没有交互历史，也可以通过标题、类目、文本、图像等内容生成 SID，进入生成空间。

这并不等于完全解决冷启动，因为模型是否会生成该 SID 还取决于训练分布和解码约束。

108. 生成式推荐如何做 Top-K 推荐？

模型不是输出每个 item 的分数，而是生成多个高概率 SID。常见流程是 beam search 生成候选 SID，过滤非法/重复/不可曝光 item，再按生成概率或融合分排序。

history → beam generate SID candidates → item lookup → filter → rank
109. 生成式推荐和 LLM 推荐理由生成有什么区别？

LLM 推荐理由生成是解释层或交互层，输出文本；ID-based generative recommendation 输出 item ID 或 SID，直接参与推荐候选生成。二者可以结合，但不是一回事。

面试中要避免把“让大模型说推荐理由”误当成生成式推荐本体。

110. 生成式推荐如何评估？R20

仍然可以用 Recall@K、NDCG@K、MRR@K、Coverage、TailRatio、Cold-start Recall 等指标。还要额外看 invalid SID rate、decoding latency、SID collision、合法前缀命中率。

生成式推荐不是只看语言模型 loss，最终仍要看推荐列表质量。

111. 生成式推荐有哪些风险？
  • 生成非法 item。
  • 解码成本高，难满足线上延迟。
  • SID tokenizer 训练复杂。
  • 模型可能过度生成热门模式。
  • 业务过滤和策略约束不如传统系统自然。
  • 离线好不代表线上稳定。

标准回答：生成式推荐是新范式，但工程落地必须解决合法性、效率、可控性和对齐问题。

112. Semantic ID 和 Tokenizer-first 推荐有什么关系？

Tokenizer-first 指先设计/学习 item tokenizer，把 item 转成离散 token，再训练生成模型。很多早期 GenRec 工作的瓶颈就在 tokenizer：如果 token 与推荐目标不一致，生成模型再强也会受限。

113. 生成式推荐为什么可能提升长尾覆盖？

传统召回容易偏向热门 item；Semantic ID 可以让长尾 item 与相似热门 item 共享部分 token，从而获得泛化机会。模型生成语义簇后，有机会落到长尾 item。

但这取决于 SID 构造和解码策略，不能简单说一定提升。

114. 生成式推荐和向量召回的关系是什么？

向量召回是先学 user/item embedding，再 ANN 检索；生成式推荐是直接生成 item 的离散标识。两者都在解决大规模候选检索，但一个是 nearest neighbor search，一个是 autoregressive decoding。

实际系统也可以混用：生成式召回作为一路召回，和 ANN、ItemCF、热门召回融合。

115. 生成式推荐适合什么场景？

适合 item 规模大、内容语义丰富、用户行为序列强、希望统一召回排序或提升冷启动/长尾的场景。短视频、电商、音乐、内容推荐都比较适合。

如果场景 item 很少、强规则约束、实时价格库存复杂，传统系统可能更稳。

H. OneRec、后训练与项目八股

116. OneRec 解决了什么问题？R16R17

OneRec 试图用端到端生成式推荐统一 retrieve 和 rank，减少多阶段级联系统中的计算碎片和优化不一致。它把推荐改造成自回归生成任务，并在工业场景中验证。

117. OneRec 和 TIGER 的关系是什么？R15R17

TIGER 证明 Semantic ID + generative retrieval 可行，主要偏生成式召回；OneRec 进一步走向工业级统一推荐，把 retrieve 和 rank 合到一个生成模型，并加入 session-wise generation、MoE 和偏好对齐。

118. OneRec 为什么要 session-wise generation？R17

传统 next-item prediction 一次生成一个 item，列表还要靠规则组合。session-wise generation 直接生成一组更连贯的推荐结果，更接近用户一次请求下的列表级体验。

它的目标从单点预测升级到列表/会话级生成。

119. OneRec 中 Sparse-MoE 的作用是什么？

Sparse-MoE 用大量专家扩大模型容量，但每个样本只激活部分专家，从而避免计算量和参数量同步线性增长。推荐系统中用户兴趣和场景高度异质，MoE 有助于不同专家学习不同模式。

\[\operatorname{MoE}(x)=\sum_{k\in \operatorname{TopK}(g(x))}g_k(x)E_k(x)\]
120. DPO 在推荐里怎么理解？R19R17

DPO 用 chosen/rejected 偏好对直接优化策略模型，使模型更倾向生成用户更喜欢的结果。原始 DPO 来自 LLM 对齐；OneRec 将偏好对齐思想用于推荐生成结果质量提升。

\[\mathcal{L}_{DPO}=-\log \sigma\left(\beta\log\frac{\pi_\theta(y_w\mid x)}{\pi_{ref}(y_w\mid x)}-\beta\log\frac{\pi_\theta(y_l\mid x)}{\pi_{ref}(y_l\mid x)}\right)\]

推荐里的难点是：同一次请求通常只展示一个结果列表，很难天然同时得到 chosen 和 rejected。

121. 推荐系统里的偏好对怎么构造？

可以用点击/停留/转化/负反馈构造，也可以用 reward model 或在线采样构造。比如高停留列表作为 chosen，低停留或跳出列表作为 rejected。

关键是偏好对要可靠，否则 DPO 会对齐到噪声。

122. Rec-RL 和传统监督学习推荐有什么区别？

监督学习通常拟合历史点击或转化；RL 更关注策略执行后的长期奖励，如停留、留存、复访。推荐的 RL 难点是反馈延迟、离线评估困难、策略安全和探索成本。

生成式推荐提供了更统一的 action space，因此 RL/DPO 更容易接入，但仍需要严格线上控制。

123. OpenOneRec 的意义是什么？R18

OpenOneRec 试图提供开源框架、foundation model 和 benchmark，降低生成式推荐复现门槛。它把传统推荐和 LLM 式训练流程连接起来，包含 tokenizer、pretrain、SFT、distillation、RL 等模块。

124. 生成式推荐项目怎么设计最稳？

建议从小项目做起：在 Amazon Reviews 或 MovieLens 上复现 TIGER-style Semantic-ID GenRec，对比 Random ID、Category ID、Content SID、Behavior SID，再看 Recall@K、NDCG@K、Coverage、TailRatio、Invalid SID Rate。

这比一上来复现 OneRec 更适合个人项目，因为 OneRec 需要工业数据和巨大算力。

125. 生成式推荐面试最核心的一句话是什么？

推荐可以不只是“候选打分排序”，当 item 被离散 token 化后，推荐可以被建模成条件生成任务。Semantic ID 决定可生成性和泛化能力，约束解码决定合法性，偏好对齐决定生成结果是否符合用户和业务目标。

126. 如果面试官问“生成式推荐是不是噱头”，怎么答？

可以回答：不是完全替代传统系统的银弹，但它提供了新的统一建模方式。它的价值在于端到端优化、突破固定候选集、利用生成式模型 scaling 和对齐技术；难点是合法解码、延迟、可控性、业务约束和线上稳定性。

这样的回答既不盲目吹，也能体现你理解工程现实。

127. 生成式推荐和 RankMixer 这条 ranking scaling 线有什么关系？R21R15

它们是两条不同但相关的路线。RankMixer 关注传统排序模型如何做大、做统一、做硬件友好；生成式推荐关注能否把推荐改造成 item/SID 生成任务，甚至统一召回和排序。

面试里可以说：一个是 ranking model scaling，一个是 recommendation paradigm shift。

128. 为什么生成式推荐也需要业务过滤？

模型生成的 item 可能库存不可用、地域不可投、未过审、已看过、重复、价格不合适或触发频控。推荐系统不是纯模型问题，必须接业务约束。

所以生成式推荐线上通常还要接 filter、policy、dedup、rerank。

129. 推荐算法岗面试如何讲自己的生成式推荐项目？

可以按五段讲：问题背景、方法选择、数据与任务、实验结果、失败分析。

背景：传统 next-item 推荐候选受限
方法：构造 Semantic ID + Transformer 生成
数据：Amazon Reviews / MovieLens
对比：SASRec、BERT4Rec、Random-ID GenRec
指标：Recall@K、NDCG@K、Coverage、Invalid SID
分析：冷启动、长尾、beam size、SID 长度
130. 生成式推荐还有哪些开放问题？
  • 更好的 item tokenizer：让 SID 同时适合推荐、搜索、冷启动和多模态。
  • 高效解码：低延迟生成 top-K。
  • 列表级对齐：从 next item 到 session/list generation。
  • 在线反馈 RL：安全利用真实用户反馈。
  • 可解释与推理：推荐理由、兴趣链路、可控生成。
  • 工业部署：MFU、成本、监控、回滚。

这也是后续写论文或项目 proposal 的切入点。

I. 高频追问速记

131. 召回看什么？

看覆盖、速度、多样性、各通道贡献和后续精排可用性，不只看离线 Recall。

132. 精排看什么？

看 AUC/LogLoss/校准/分桶效果/线上业务指标，还要看延迟和资源。

133. 重排解决什么？

解决列表级体验，包括去重、多样性、频控、业务规则、探索和公平。

134. 冷启动最常用信号？

内容、类目、上下文、热门、相似用户/物品、探索流量。

135. 为什么多任务会 seesaw？

目标冲突或梯度冲突，一个任务提升可能牺牲另一个任务。

136. 为什么要蒸馏？

用大模型/精排模型指导小模型或召回模型，提升效果同时控制线上成本。

137. 负反馈有哪些？

不感兴趣、跳出、短停留、拉黑、举报、快速划走、取消关注。

138. 为什么点击是弱标签？

点击可能来自标题党、位置偏差、误触，不一定代表满意。

139. WatchTime 有什么风险？

可能鼓励冗长、沉迷或低质量内容，需要满意度和长期指标约束。

140. Semantic ID 最怕什么？

语义簇不合理、冲突高、合法 SID 稀疏、与推荐目标不一致。

141. Beam Search 最怕什么？

延迟高、结果同质化、生成重复，需要约束和去重。

142. DPO 最怕什么？

偏好对噪声大或 chosen/rejected 构造不可靠。

143. 推荐大模型 Scaling 难点？

高 QPS、低延迟、巨大 embedding、特征稀疏、MFU 低、线上成本高。

144. 为什么说推荐不是纯 NLP？

推荐有曝光机制、反馈偏差、实时性、多目标和线上策略约束。

145. 面试讲生成式推荐先讲什么？

先讲范式变化：从 score candidates 到 generate item/Semantic ID。
