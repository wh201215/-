2023：TIGER / Recommender Systems with Generative Retrieval
核心贡献：给 item 分配 Semantic ID，让模型自回归生成目标 item 的标识符，推荐从“向量召回”变成“序列生成”。
2024：LLM for Generative Recommendation Survey / LETTER / ETEGRec
核心贡献：综述层面定义生成式推荐；方法层面开始认真处理 item tokenizer 的质量、协同信号和端到端对齐。

核心贡献：OneRec 推进工业级统一 retrieve + rank；GRID 解决实验不可比；OneRec-V2 解决架构效率和真实反馈对齐；Spotify 等工作探索搜索和推荐共用 Semantic ID。

4. TIGER 线：生成式推荐的起点

TIGER 的关键不是“用了 Transformer”，而是它把推荐检索写成了一个自回归生成问题。传统大规模检索通常是把用户和 item 放进同一个向量空间，再做 ANN 搜索；TIGER 则让模型直接生成目标 item 的 Semantic ID。

\[ \operatorname{Rec}(u)=\operatorname{Decode}_{\theta}\big(H_u\big) \]

如果 item \(i\) 的 Semantic ID 是：

\[ \operatorname{SID}(i)=\left(c_{i,1},c_{i,2},\ldots,c_{i,L}\right) \]

那么 next-item recommendation 可以写成：

\[ P_{\theta}\left(\operatorname{SID}(i_{n+1})\mid \operatorname{SID}(i_1),\ldots,\operatorname{SID}(i_n)\right) =\prod_{t=1}^{L}P_{\theta}\left(c_{n+1,t}\mid c_{n+1,\lt t},H_u\right) \]
TIGER 的真正创新：它不是让模型从百万 item 里分类，而是把 item 表示成短 token 序列；模型每一步只在 codebook token 上预测，从而把 item 空间转换成 token 组合空间。

问题为什么重要后续论文如何接

5. Semantic ID 构造线：这是当前最核心的研究点

Semantic ID 是生成式推荐的“词表设计”。如果 ID 只是随机编号，生成模型很难泛化；如果 ID 有语义结构，相似 item 可以共享前缀或局部 code，模型就可能通过 token 组合泛化到长尾和冷启动 item。

图 2：Semantic ID 构造流程。自绘示意图。

5.1 RQ-VAE Semantic ID

Stage 2 已经讲过，RQ-VAE 的直觉是多层残差量化：

\[ \mathbf{r}^{(0)}=\mathbf{z},\quad c_l=\arg\min_{k}\left\|\mathbf{r}^{(l-1)}-\mathbf{e}^{(l)}_k\right\|_2^2,\quad \mathbf{r}^{(l)}=\mathbf{r}^{(l-1)}-\mathbf{e}^{(l)}_{c_l} \]

最终：

\[ \operatorname{SID}(i)=\left(c_1,c_2,\ldots,c_L\right) \]

5.2 LETTER：让 tokenizer 吃到协同信号

LETTER 的动机是：仅靠文本或内容语义生成 ID 不够，因为推荐不是纯语义匹配，还需要 collaborative filtering 信号。它将层级语义、协同信号和 code assignment diversity 结合起来，试图让 item token 既有内容语义，又能反映用户行为共现关系。

普通 SID：
item content embedding → RQ-VAE → SID

LETTER：
item semantics + collaborative signal + diversity regularization → learnable tokenizer → better SID

5.3 ETEGRec：把 tokenizer 和 recommender 端到端耦合

ETEGRec 的问题意识更明确：很多方法先单独训练 item tokenizer，再训练生成式 recommender，这会造成目标错位。ETEGRec 使用 item tokenizer + generative recommender 的双 encoder-decoder 架构，并通过 sequence-item alignment 和 preference-semantic alignment 让两者互相增强。

ETEGRec 的论文价值：它把“SID 是预处理产物”推进到“SID 是推荐目标的一部分，可以和 recommender 联合优化”。

5.4 UniSID / Differentiable SID：继续解决两阶段压缩问题

2026 年后，SID 方向继续往端到端、可微、广告场景细粒度语义走。核心批评是：RQ-VAE 两阶段压缩可能带来 objective misalignment、semantic degradation 和 error accumulation。因此新的 SID 论文倾向于直接把推荐目标、广告语义、重构目标、对比学习目标一起放进 SID 学习里。

SID 方法核心思路适合回答的问题
Random ID随机给 item 编号生成模型是否只靠行为记忆？
Content SID用文本/图像 embedding 量化语义信息是否提升冷启动？
Behavior SID用协同过滤 embedding 量化行为相似性是否更重要？
Hybrid SID融合内容和行为内容语义和协同信号如何平衡？
Learnable SIDtokenizer 与推荐任务对齐SID 是否可以端到端优化？
Purely Semantic Indexing避免冲突时追加随机非语义 token如何保持唯一性又不破坏语义？

6. GRID 与 Benchmark 线：为什么需要统一框架

生成式推荐论文越来越多后，一个新问题出现了：大家的数据处理、SID 长度、codebook size、Transformer 配置、beam search、负采样、过滤逻辑都不一样，导致论文之间很难公平比较。GRID 的价值就是把 Semantic-ID GenRec 做成一个模块化框架，让 SID 构造、生成模型、训练和推理都可以替换。

GRID-style pipeline:
Dataset
  ↓
Representation Layer
  ↓
SID Tokenizer
  ↓
Generative Backbone
  ↓
Training Objective
  ↓
Constrained Decoding
  ↓
Evaluation
GRID 的意义：它不是单纯提出一个更强模型，而是把这个方向变成可以系统 ablation 的实验平台。对你做项目很重要，因为你需要一个“可比较”的实验框架，而不是只跑一个 Toy TIGER。

7. OneRec 工业系统线：从生成式召回到统一 Retrieve + Rank

TIGER 更像是证明“生成式召回可行”。OneRec 的野心更大：不只是生成候选，而是用端到端生成式模型重塑传统多阶段推荐系统。

传统系统：
Recall → Pre-rank → Rank → Re-rank

OneRec 思路：
User History → Generative Model → Session-wise Recommended Items

7.1 OneRec-V1：encoder-decoder + session-wise generation

OneRec-V1 的核心组件可以概括为：

组件作用
Encoder编码用户历史行为序列
Decoder逐步生成用户可能感兴趣的视频/item
Sparse-MoE扩大模型容量，但控制每次推理激活计算量
Session-wise Generation不只预测一个 item，而是直接生成一组 session 推荐
Iterative Preference Alignment用线上反馈迭代对齐用户偏好
\[ P_{\theta}(Y\mid H_u)=\prod_{j=1}^{M}P_{\theta}(y_j\mid y_{\lt j},H_u) \]

这里 \(Y=(y_1,y_2,\ldots,y_M)\) 是一个推荐 session，而不是单个 next item。

7.2 OneRec-V2：为什么要 Lazy Decoder-Only

OneRec-V2 的问题意识是：encoder-decoder 结构在大规模工业场景下会有 encoder bottleneck 和 cross-attention 复杂度。V2 改成 Lazy Decoder-Only Architecture，目标是减少计算、简化结构，并让模型可以扩到更大参数规模。

OneRec-V2 的关键词：Lazy Decoder-Only、real-world user interaction feedback、Duration-Aware Reward Shaping、Adaptive Ratio Clipping、8B scaling。

7.3 工业系统线真正要学什么

读 OneRec 不要只看模型结构，还要看三个工程问题：

  1. 候选生成是否合法：是否需要 Trie、mask、业务过滤。
  2. 线上指标如何定义：watch time、app stay time、互动率、负反馈、商业目标。
  3. 反馈如何进入后训练：DPO、RL、reward shaping、ratio clipping。

8. OpenOneRec / MiniOneRec 开源线：从封闭工业系统到可复现 Foundation Rec

OneRec 很强，但如果没有开源框架，学术界和个人项目很难复现。OpenOneRec / MiniOneRec 这条线的意义是：把生成式推荐从“工业黑盒系统”变成公开模型、公开数据、公开 benchmark、公开后训练流程。

8.1 OpenOneRec：桥接推荐系统和 LLM

OpenOneRec 的核心问题是：传统推荐系统往往缺少世界知识、推理能力和 instruction following；而通用 LLM 又缺少 itemic token、协同过滤和大规模推荐行为建模。因此它试图做一个更像 foundation model 的推荐框架。

能力传统推荐OpenOneRec 想补什么
Item 行为建模强保留 itemic token 和行为序列
世界知识弱接入 LLM backbone
推理解释弱支持 instruction / reasoning / explanation
公开复现不足公开模型、数据和 benchmark

8.2 MiniOneRec：验证公开 benchmark 上的 scaling 和后训练

MiniOneRec 更适合作为你后面项目的参考，因为它强调端到端开源流程：SID construction、SFT、recommendation-oriented RL，并在公开 Amazon Review 数据上验证模型规模扩大和后训练的效果。

对你项目最有用的点：MiniOneRec 把“生成式推荐项目”拆成了 SID 构造、SFT、RL、评估四步，非常适合改造成公开数据实验。

9. Reasoning Recommendation 线：推荐到底需不需要 think？

OneRec-Think 的出发点是：OneRec 这类生成式推荐模型虽然可以直接生成 item，但仍然像黑盒隐式预测器，缺少 LLM 的显式、可控推理能力。因此它提出把 dialogue、reasoning 和 personalized recommendation 结合起来。

Implicit GenRec:
User history → item tokens

Reasoning GenRec:
User history + instruction
  ↓
reasoning trace / preference analysis
  ↓
item tokens / recommendation answer
\[ P_{\theta}(r,y\mid H_u,q)=P_{\theta}(r\mid H_u,q)\cdot P_{\theta}(y\mid r,H_u,q) \]

其中 \(r\) 表示推理过程或偏好分析，\(y\) 表示推荐结果。

9.1 但 OneReason 提出了一个反直觉现象

OneReason 的技术报告里提到一个值得注意的现象：在生成式推荐中，thinking mode 不一定优于 non-thinking mode。这说明推荐推理不同于数学推理或代码推理；推荐偏好可能更依赖历史行为压缩、itemic token 语义、候选约束和在线反馈，而不是越长的 chain-of-thought 越好。

重要判断：推荐场景的 reasoning 不是简单让模型“多想几步”。它需要回答：推理过程是否真的提升点击、时长、满意度、覆盖率和可控性？如果只增加推理文本但不提升推荐行为指标，就可能只是增加成本。

9.2 Reasoning Rec 的三类价值

价值说明适合场景
可解释性告诉用户为什么推荐内容推荐、知识推荐、理财知识文章
可控性根据用户指令调整偏好“不要太贵”“更适合新手”“偏保守”
多约束决策同时考虑兴趣、时间、预算、风险偏好旅游、课程、内容包、权益推荐

10. Search + Recommendation 统一线

另一个重要趋势是：Semantic ID 不只用于推荐，也用于搜索和 generative retrieval。搜索和推荐本质上都是 matching，只是搜索有显式 query，推荐主要来自隐式用户历史。

任务输入输出SID 难点
Recommendation用户历史 \(H_u\)用户可能喜欢的 itemSID 要捕捉协同过滤和偏好迁移
Searchquery \(q\)相关 item / documentSID 要捕捉语义相关性和意图匹配
Joint Search + Recquery + history个性化搜索结果搜索最优 SID 和推荐最优 SID 可能冲突

Spotify 的相关研究指出，面向搜索调优的 Semantic ID 会提升搜索检索效果，但可能损害推荐表现；推荐调优的 ID 也可能反过来影响搜索效果。这说明统一 search + rec 不是简单共用一个 tokenizer，而是要设计 task-aware 或 multi-objective Semantic ID。

\[ \mathcal{L}_{\operatorname{joint}}=\lambda_{\operatorname{rec}}\mathcal{L}_{\operatorname{rec}}+\lambda_{\operatorname{search}}\mathcal{L}_{\operatorname{search}}+\lambda_{\operatorname{align}}\mathcal{L}_{\operatorname{align}} \]

11. Scaling 争议：SID-based GenRec 是否真的能像 LLM 一样 scaling？

前面几篇我们一直在说生成式推荐像 LLM 一样生成 token，但这不代表它一定具有 LLM 那样平滑的 scaling law。2025 年的一些研究开始质疑 SID-based GenRec 的 scaling 上限：当模型、tokenizer、modality encoder 都变大时，性能可能快速饱和，瓶颈可能出现在 SID 对 item 语义和协同信息的压缩能力上。

关键争议：如果 Semantic ID 本身只能表达有限信息，那么继续把 Transformer 做大，未必能持续提升推荐效果。这和 Stage 1 的 Ranking Model Scaling 是一个很好的对照：推荐大模型化不只是 backbone scaling，还包括 item representation scaling。

11.1 两种可能路线

路线代表基本假设风险
SID-based GRTIGER、GRID、OneRec把 item 离散成短 token 序列，生成效率高，可统一检索排序SID 信息瓶颈、冲突、语义压缩损失
LLM-as-RSOpenOneRec、OneRec-Think、Reasoning Rec直接利用 LLM 的世界知识、语义理解和指令能力成本高、行为建模弱、item grounding 难

实际未来可能不是二选一，而是混合路线：

LLM backbone + Itemic Tokens + Semantic IDs + Recommendation RL + Constrained Decoding

12. 统一数学框架：把这些论文放到一个公式里

为了避免论文越读越散，可以用下面这个统一框架理解。

12.1 Item 表示层

\[ \mathbf{z}_i=f_{\phi}(x_i,b_i) \]

其中 \(x_i\) 是 item 内容信息，\(b_i\) 是行为协同信息。

12.2 Tokenizer 层

\[ \operatorname{SID}(i)=Q_{\psi}(\mathbf{z}_i)=\left(c_{i,1},c_{i,2},\ldots,c_{i,L}\right) \]

12.3 生成模型层

\[ P_{\theta}\left(\operatorname{SID}(i_{n+1})\mid H_u\right)=\prod_{t=1}^{L}P_{\theta}\left(c_{n+1,t}\mid c_{n+1,\lt t},H_u\right) \]

12.4 约束解码层

\[ \hat{i}=\operatorname{Lookup}\left(\operatorname{Decode}_{\operatorname{Trie}}\left(P_{\theta}(\cdot\mid H_u)\right)\right) \]

12.5 偏好对齐层

\[ \max_{\theta}\;\mathbb{E}_{(H_u,y^+,y^-)}\left[\log\sigma\left(\beta\left(\log\frac{P_{\theta}(y^+\mid H_u)}{P_{\operatorname{ref}}(y^+\mid H_u)}-\log\frac{P_{\theta}(y^-\mid H_u)}{P_{\operatorname{ref}}(y^-\mid H_u)}\right)\right)\right] \]
你讲论文时可以这样总结：不同论文其实是在改 \(f_{\phi}\)、\(Q_{\psi}\)、\(P_{\theta}\)、\(\operatorname{Decode}\) 或偏好对齐目标。这样就不会被论文名绕晕。

14.2 方向 B：长尾发现与冷启动

项目题目：Generative Recommendation for Long-tail and Cold-start Item Discovery

核心问题：Semantic ID 是否真的能让模型泛化到低频 item？如果 item 的文本和图像语义足够强，生成式推荐是否比 SASRec / BERT4Rec 更容易推荐长尾 item？

14.3 方向 C：推荐后训练小项目

项目题目：Preference-Aligned Generative Recommendation with Constrained Decoding

做法：先训练一个小型 Semantic-ID GenRec，再构造 chosen / rejected pair，用 DPO 或轻量 RL 做偏好对齐。重点不是追求工业指标，而是展示后训练如何改变推荐多样性、长尾比例和热门偏置。

14.4 方向 D：银行内容推荐迁移

不碰风控、不碰授信，可以做：

用户历史浏览文章 / 活动 / 功能入口
        ↓
生成式推荐模型
        ↓
推荐下一篇金融知识文章、权益活动、App 功能入口

这个方向适合面试讲，因为场景明确、风险较低、公开数据可替代、方法和业务能结合。

15. 一页总结

问题结论
生成式推荐主线是什么？从 item scoring 变成 item token generation，核心是 Semantic ID 和自回归生成。
TIGER 的意义是什么？证明 Semantic ID + generative retrieval 可以作为推荐新范式。
后续为什么都在改 SID？因为 SID 是生成目标，ID 质量直接决定泛化、冷启动、长尾和 scaling 上限。
OneRec 解决什么？把生成式推荐推进工业级统一 retrieve + rank，并引入 MoE、session-wise generation、偏好对齐。
GRID / OpenOneRec 解决什么？让生成式推荐从封闭系统走向可比较、可复现、可开源。
Reasoning Rec 的问题是什么？推荐是否需要显式推理还没有完全定论，关键看推理是否真实提升行为指标和可控性。
现在最适合做什么项目？做 Semantic ID 质量比较、长尾发现、轻量偏好对齐，最容易用公开数据落地。
Stage 6 结束标准：
  1. 能把 TIGER、LETTER、ETEGRec、GRID、OneRec、OpenOneRec、OneRec-Think 放进一张谱系图；
  2. 能解释为什么 Semantic ID 是生成式推荐的核心瓶颈；
  3. 能区分 SID-based GenRec、LLM-as-RS、Reasoning Rec、Search+Rec 统一；
  4. 能讲清楚当前前沿争议：SID scaling 是否受限、推荐是否需要显式 thinking；
  5. 能基于公开数据提出一个可做项目。
