    RLVRreward_func.py、math checker、unit test、JSON validator、tool success用可验证任务结果替代主观人类偏好，尤其适合数学、代码和工具调用。

    1.2 为什么后训练工程比普通微调复杂？

    普通 SFT 的训练链路很短：读数据 → tokenize → forward → loss → backward。 但是 RLHF / RLVR 的链路中间多了一段“模型自己生成数据”的过程，这就使工程复杂度暴涨。 

    普通 SFT：
数据已经写好 → 模型模仿 → 计算 cross entropy loss

Online RL：
当前模型先生成 → 评估器打分 → 计算 advantage → 约束 KL → 再更新当前模型

    这也是为什么 vLLM、SGLang、Ray、DeepSpeed 会同时出现。它们不是“炫技组件”，而是为了解决在线 RL 的三个现实瓶颈：生成慢、模型多、显存大。

    瓶颈为什么出现常见解决组件
    生成慢RL 每一步都需要当前模型生成大量 samples；reasoning 任务还会生成很长答案。vLLM、SGLang、continuous batching、KV cache 管理。
    模型多PPO 至少涉及 actor、critic、reward、reference；再加 rollout engine，调度复杂。Ray、placement group、worker、resource pool。
    显存大大模型训练要存参数、梯度、优化器状态；rollout 还要用 KV cache。DeepSpeed ZeRO-3、FSDP、Megatron-LM、offload、sleep mode。

    2.2 SFTTrainer：工程中的“监督微调”

    SFTTrainer 的核心作用是把指令数据变成语言模型训练任务。TRL 官方文档说明 SFTTrainer 支持 standard 和 conversational 两类数据格式；如果给的是 conversational dataset，它会自动应用 chat template。 

    from trl import SFTTrainer
from datasets import load_dataset

trainer = SFTTrainer(
    model="Qwen/Qwen3-0.6B",
    train_dataset=load_dataset("trl-lib/Capybara", split="train"),
)
trainer.train()

    这段代码表面很短，但背后做了很多事：

    代码背后含义
    model="Qwen/Qwen3-0.6B"加载一个待微调的 causal language model。SFT 阶段训练的是“在给定 prompt 后生成 assistant response”。
    load_dataset(...)读取指令数据。数据可能是普通 text，也可能是 messages 格式。
    SFTTrainer(...)把数据预处理、tokenization、collator、loss computation 封装成 Trainer。
    trainer.train()执行训练循环：forward、loss、backward、optimizer step、save checkpoint。

    2.2.1 SFT 数据格式应该怎么看？

    TRL 支持的 SFT 数据大体可以分成四类。你需要重点记住：普通文本训练和对话训练不是一回事；prompt-completion 和 full text 也不是一回事。 

    # 1. Standard language modeling
{"text": "The sky is blue."}

# 2. Conversational language modeling
{"messages": [
  {"role": "user", "content": "What color is the sky?"},
  {"role": "assistant", "content": "It is blue."}
]}

# 3. Standard prompt-completion
{"prompt": "The sky is", "completion": " blue."}

# 4. Conversational prompt-completion
{"prompt": [{"role": "user", "content": "What color is the sky?"}],
 "completion": [{"role": "assistant", "content": "It is blue."}]}
    常见坑： 很多 SFT 训练效果差，不是模型不行，而是 chat template 错了。比如训练时用了 A 模板，推理时用了 B 模板；或者把 user token 也算进 loss；或者 assistant 结束符没有处理好。后训练里，格式错误会直接变成行为错误。

    2.2.2 SFT 在 Agent 里意味着什么？

    Agent SFT 不是让模型“自由探索”，而是先让模型学会稳定的轨迹格式。 例如一个工具调用样本可以写成： 

    {
  "messages": [
    {"role": "user", "content": "请计算 27 * 19。"},
    {"role": "assistant", "content": "Thought: 这是精确计算问题，需要调用 calculator。\nAction: calculator({\"expression\": \"27 * 19\"})"},
    {"role": "tool", "content": "513"},
    {"role": "assistant", "content": "Final Answer: 27 * 19 = 513。"}
  ]
}
    Agent SFT 学到什么为什么重要
    什么时候调用工具避免模型明明需要计算却直接瞎猜。
    工具调用格式Action 必须能被程序解析，不然无法执行。
    如何使用 Observation工具返回后要基于结果回答，不能忽略工具。
    什么时候结束避免无意义循环调用工具。

    2.3 DPOTrainer：工程中的“偏好优化”

    DPOTrainer 的输入不是单个标准答案，而是偏好数据。TRL 官方文档说明 DPO 需要 preference dataset，常见字段是 prompt、chosen、rejected。 

    from trl import DPOTrainer
from datasets import load_dataset

trainer = DPOTrainer(
    model="Qwen/Qwen3-0.6B",
    train_dataset=load_dataset("trl-lib/ultrafeedback_binarized", split="train"),
)
trainer.train()

    DPO 的工程重点不是“怎么写一个好答案”，而是“怎么构造好坏对”。

    DPO 字段含义Agent 场景示例
    prompt同一个用户任务“查找文档中 RLHF 的定义。”
    chosen更好的回答或轨迹调用 search_doc，找到证据后回答。
    rejected更差的回答或轨迹不调用工具，直接编造定义。

    DPO 的核心理解：它不是简单地让 chosen 概率变大，而是让 policy 相对于 reference model 更偏向 chosen、远离 rejected。reference model 的作用是保留原模型分布作为“锚点”，避免模型为了偏好数据过度偏移。

    2.3.1 Agent DPO 的三类好坏对

    偏好类型ChosenRejected为什么适合 DPO
    工具选择计算题调用 calculator计算题调用 search差异清楚，容易自动构造。
    参数合法性{"query":"RLHF definition"}{"keyword":"RLHF definition"}schema 对错明确。
    轨迹效率2 步完成任务8 步绕远路都可能成功，但成本不同。

{
  "prompt": "计算 125 * 37。",
  "chosen": "Action: calculator({\"expression\": \"125 * 37\"})\nFinal Answer: 4625",
  "rejected": "Action: search({\"query\": \"125 * 37\"})\nFinal Answer: 我查到结果是 4625"
}

    2.4 GRPOTrainer：工程中的“组内相对奖励”

    GRPOTrainer 是理解 RLVR 的关键入口。它适合数学、代码、格式检查、工具调用等“可以自动判断好坏”的任务。和 DPO 不同，GRPO 不只吃固定 chosen / rejected 数据，而是让当前模型对同一个 prompt 生成多条回答，再用 reward function 给每条回答打分。

    图 4：GRPO 的组内比较
    同一个 Prompt→Response AResponse BResponse CResponse D
    Reward→1.00.00.80.2
    Group Baseline→比组内平均更好的回答被强化，比组内平均更差的回答被压低。
    图 4：本文自绘。依据 GRPO 的 group sampling 与 relative rewards 思路绘制。 
    from trl import GRPOTrainer, GRPOConfig
from datasets import load_dataset

# 一个极简奖励函数：答案中出现正确结果就给 1，否则 0
def accuracy_reward(completions, answer, **kwargs):
    rewards = []
    for completion, gold in zip(completions, answer):
        text = completion[0]["content"] if isinstance(completion, list) else completion
        rewards.append(1.0 if str(gold) in text else 0.0)
    return rewards

config = GRPOConfig(
    output_dir="./grpo_math_demo",
    num_generations=4,
    per_device_train_batch_size=2,
    max_completion_length=512,
)

trainer = GRPOTrainer(
    model="Qwen/Qwen2.5-1.5B-Instruct",
    args=config,
    train_dataset=load_dataset("your_math_dataset", split="train"),
    reward_funcs=[accuracy_reward],
)
trainer.train()
    参数 / 代码作用为什么重要
    num_generations=4每个 prompt 生成 4 条回答GRPO 要有“组”，否则没法做组内比较。
    reward_funcs=[...]把回答变成奖励分数RLVR 的关键。奖励函数质量决定训练方向。
    max_completion_length限制回答长度reasoning 任务很容易越写越长，必须控制显存和上下文。
    KL penalty约束 policy 不要偏离 reference没有约束时，模型可能为了 reward 变得格式怪异或语言退化。
    GRPO 最大的坑： 如果同一组里所有回答 reward 都一样，例如全是 0 或全是 1，那么组内相对优势几乎没有信息。训练会变慢，甚至没有有效学习信号。所以 RLVR 数据要有难度梯度，不能太容易，也不能太难。

    3. OpenRLHF：理解大规模 RLHF / RLVR 系统

    如果说 TRL 是入门 Trainer，那么 OpenRLHF 更像是真实训练系统。它关注的不是“十几行代码跑通”，而是如何在多 GPU / 多节点环境中协调 actor、critic、reward、reference、vLLM、DeepSpeed、Ray。

    图 5：OpenRLHF 官方架构图。来源：OpenRLHF Documentation，Architecture Foundation: Ray + vLLM Distribution。原图链接：https://openrlhf.readthedocs.io/en/latest/_images/openrlhf-arch.png

    3.1 这张官方架构图应该怎么看？

    这张图的关键不是 logo，而是它把 RLHF 的计算拆成了几个不同角色：

    图中组件中文解释后训练中的作用
    Ray分布式调度器和控制器决定 actor、critic、reward、reference、vLLM 等组件跑在哪些 GPU 上，如何提交任务、同步数据。
    vLLM高吞吐推理引擎负责 rollout generation。RL 训练中大量时间花在生成样本上，所以需要专门推理引擎。
    Transformers模型权重格式与状态提供 Hugging Face 模型加载、权重管理、tokenizer、checkpoint 接口。
    DeepSpeed显存高效训练框架用 ZeRO / AutoTP / RingAttention 等方式训练 actor、critic、reward、reference。
    NCCL / CUDA IPCGPU 间通信用于权重同步、梯度通信、跨 GPU 数据传输。
    关键理解： OpenRLHF 不是只实现一个 PPO loss。它实现的是一个“训练系统”：谁负责生成，谁负责打分，谁负责训练，谁负责同步权重，谁负责释放显存，谁负责调度 GPU。

    3.2 为什么 OpenRLHF 强调 Ray + vLLM + DeepSpeed？

    LLM 的 RLHF 训练不是单模型训练，而是多模型协作。以 PPO 为例，常见组件包括：

    模型 / 组件作用是否训练显存压力
    Actor / Policy当前要优化的模型，生成回答训练高
    Critic / Value Model估计 value，用于计算 advantage训练高
    Reward Model / Reward Function给回答或轨迹打分可能训练，也可能固定中到高
    Reference Model提供 KL 参照，约束 actor 不偏离太远固定中
    vLLM Engine快速生成 rollout不训练，但要加载权重高，尤其 KV cache

    如果这些东西都朴素地各占一套 GPU，会非常浪费。OpenRLHF 的重点就是调度和复用。

    3.3 Hybrid Engine：为什么要 sleep mode？

    OpenRLHF 文档里非常关键的概念是 Hybrid Engine。直觉上，它解决的是“生成和训练不能同时充分利用同一批 GPU”的问题。

    图 6：Hybrid Engine 的时间片复用
    阶段 A：生成 rollout→vLLM awakeDeepSpeed sleep
    权重同步→NCCL 把训练后的 actor 权重同步给 vLLM
    阶段 B：训练更新→vLLM sleepDeepSpeed awake
    图 6：本文自绘。依据 OpenRLHF 官方文档中 Hybrid Engine / sleep-mode time-sharing 的说明整理。

    这件事非常工程化，但你要理解它背后的训练逻辑： 

    1. 当前 actor 权重交给 vLLM。
2. vLLM 批量生成 responses / trajectories。
3. reward function / RM 给这些 responses 打分。
4. DeepSpeed 唤醒，使用这些样本更新 actor / critic。
5. 更新后的 actor 权重再同步给 vLLM。
6. 进入下一轮 rollout。
    为什么重要： 大规模 RLHF 最耗时的部分往往不是 backward，而是 sample generation。OpenRLHF 把 vLLM 放进训练系统，就是为了加速 rollout；把 sleep mode 放进系统，是为了让生成和训练共享 GPU 资源。

    3.4 OpenRLHF 的 RL Training Guide 怎么读？

    OpenRLHF 文档把 RL 训练拆成三个相互独立的选择轴：execution mode、RL algorithm、pipeline。 这比背命令更重要。

    选择轴可选项你应该怎么理解
    Execution ModeSingle-turn / Multi-turn一次回答给一次 reward，还是一条多步交互轨迹给 reward。
    RL AlgorithmPPO / REINFORCE++ / RLOO / GRPO / Dr. GRPOreward 如何变成 advantage，是否需要 critic，是否组内归一化。
    PipelineSync Hybrid Engine / Async / Partial Rollout生成和训练是交替进行，还是并行重叠。

    3.4.1 Single-turn Mode

    Single-turn 是最常见的 RLVR 形式：一个 prompt，模型生成一个 response，然后 reward function 给出一个分数。 

    Prompt: 计算 13 * 17
Response: 13 * 17 = 221
Reward: 1.0

    OpenRLHF 支持三种 reward 来源：训练好的 reward model、远程 HTTP reward server、本地 Python reward function。对于数学、代码、格式检查，最适合先用本地 Python reward function。 

    # reward_func.py 示例：OpenRLHF 风格的本地奖励函数
import torch

def reward_func(queries, prompts, labels):
    rewards = torch.zeros(len(queries))
    for i, (query, label) in enumerate(zip(queries, labels)):
        # query 通常是 prompt + response 的完整文本
        pred_ok = extract_final_answer(query) == str(label)
        rewards[i] = 1.0 if pred_ok else 0.0
    return {
        "rewards": rewards,
        "scores": rewards,
        "extra_logs": {"accuracy": rewards.mean().item()},
    }
    返回字段作用
    rewards真正用于 advantage 计算的奖励。
    scores可用于动态过滤、日志或额外指标。
    extra_logs写入 WandB / TensorBoard，便于观察准确率、格式通过率等。

    3.4.2 Multi-turn Mode

    Multi-turn 是 Agent 后训练的入口。它不再是“一个 prompt → 一个 answer”，而是： 

    reset environment
  ↓
model generates action
  ↓
environment returns feedback / observation / reward / done
  ↓
model continues
  ↓
until done=True

    这非常适合工具调用、代码调试、网页操作、数据库查询等任务。

    任务Multi-turn 中的 environmentreward 可以怎么设计
    代码 Agent代码沙箱、单元测试器测试通过给 1，编译错误给 0，部分测试通过给中间分。
    搜索 Agent搜索 API、文档检索器找到证据、引用正确、回答准确给分。
    工具调用 Agent函数执行器工具选择正确、参数合法、执行成功、最终答案正确。
    网页 Agent浏览器环境最终页面状态达到目标，步数越少奖励越高。

    4. verl：把 RL 后训练看成可组合的数据流

    verl 的定位更偏 RL post-training infrastructure。它强调把复杂 RL 数据流拆成模块：actor、rollout、reference、reward、trainer、worker、config。 如果你想理解更复杂的推理 RL、Agent RL、多轮工具调用和大规模集群训练，verl 很值得读。

    4.1 verl 最应该先看什么？

    学习入口你要看什么解决什么问题

    verl 配置里常见 actor_rollout_ref 这个名字。它看起来有点长，但其实很直白：把 actor、rollout、reference 放在同一个配置域里，因为它们共享很多模型和数据流关系。

    4.3 verl 的 GRPO 怎么理解？

    verl 官方 GRPO 文档把 GRPO 拆成四步：group sampling、reward assignment、baseline calculation、policy update。这个解释非常适合初学者。 

    同一个问题 q
  ↓
模型生成 G 个回答：o1, o2, ..., oG
  ↓
每个回答得到 reward：r1, r2, ..., rG
  ↓
计算组内平均 reward 作为 baseline
  ↓
ri 高于 baseline 的回答被强化；低于 baseline 的回答被削弱
    GRPO 环节中文解释工程上要注意什么
    Group Sampling同一 prompt 采多条采样参数必须有多样性；batch size 和显存要能承受。
    Reward Assignment每条 completion 打分reward function 要稳定，不能大量解析失败。
    Baseline Calculation组内平均或归一化同一组全 0 或全 1 会导致信号弱。
    Policy Update用相对优势更新 actor仍要 KL 约束，避免模型跑飞。

    4.4 Reward Loop：reward 不是一个小函数那么简单

    在玩具实验里，reward 可能只是一个 Python 函数。但在真实系统里，reward 可能很重：

    • 数学任务：需要调用 math-verify 或符号计算。
    • 代码任务：需要运行单元测试，可能很慢。
    • 对话偏好：需要 reward model 推理。
    • 复杂任务：可能需要 rule-based reward + generative reward model 混合。

    verl 的 Reward Loop 文档强调 distributed reward manager、rule-based reward、discriminative reward model、generative reward model、hybrid reward。你要把它理解成：reward 本身也可以是一套并行计算系统。

    reward 类型例子优点风险
    Rule-based Reward答案字符串匹配、JSON schema 检查快、可控、成本低容易被钻漏洞；语义等价难处理。
    Discriminative RM输入 prompt + response，输出分数适合偏好判断RM 本身可能有偏差，会被 reward hacking。
    Generative RM让另一个模型生成评价再解析分数更灵活，能解释复杂标准慢、成本高、解析不稳定。
    Hybrid Reward规则 + RM + 长度惩罚 + 工具成功率更贴近真实目标权重难调，容易奖励冲突。

    4.5 Agent Loop：Agent RL 的工程入口

    verl 的 Agent Loop 文档把 agent loop 设计成一个通用接口：给定 prompt，用户自己定义循环，里面可以调用 LLM generate API、调用工具、与环境交互，最后返回输出和轨迹。

    图 8：Agent Loop 的训练直觉
    Prompt→LLM generate→Tool / Environment
    Observation→LLM continue→Final Output
    Reward→把整条轨迹变成训练信号，用于 RL update。

    Agent Loop 的重点不是“工具在哪里定义”，而是“多轮轨迹如何被训练系统接收”。文档中有一个关键概念：response_mask。

    字段含义Agent 训练中的作用
    prompt_idsprompt token ids环境初始输入。
    response_idsresponse token ids，包括模型生成和工具返回完整轨迹。

    5. Rollout：后训练工程里最容易被低估的概念

    Rollout 可以翻译为“轨迹采样”或“生成过程”。在普通 LLM 里，它就是模型对 prompt 生成一段回答；在 Agent 里，它是一整条多步交互轨迹。

    5.1 普通回答的 rollout

    Prompt: 请解方程 2x + 3 = 9
Policy generates:
  Step 1: 2x = 6
  Step 2: x = 3
Final Answer: 3
Reward: 1.0
    记录内容为什么要记录
    生成 token训练时要知道 policy 生成了什么。
    log probabilitiesPPO / GRPO loss 需要用当前策略概率。
    reference log probabilities计算 KL，限制 policy 不偏离原模型太远。
    reward判断这条回答好不好。
    attention / response mask区分 prompt token、response token、padding token、tool token。

    5.2 Agent 的 rollout

    User: 帮我查文档里 GRPO 和 PPO 的区别

Assistant: Action: search_doc({"query":"GRPO PPO difference"})
Tool: 检索结果：GRPO 不需要 critic，PPO 需要 value model ...
Assistant: Action: search_doc({"query":"PPO critic value function"})
Tool: 检索结果：PPO 使用 critic 估计 value ...
Assistant: Final Answer: GRPO 和 PPO 的核心区别是 ...
Reward: 0.85

    这里的 rollout 不只是 final answer，而是整条动作轨迹。训练系统要知道哪些 token 是模型生成的，哪些 token 是工具返回的；reward 也可能来自最终答案、工具使用效率、引用正确性、步数成本等多个维度。

    5.3 为什么 rollout 需要 vLLM / SGLang？

    在 RLVR 中，同一个 prompt 可能要生成 4、8、16 条回答；每条回答可能很长。若仍用普通训练框架逐条生成，GPU 利用率会很低。

    后端适合场景原因
    Transformers generate小实验、调试简单，但吞吐低。
    vLLM大批量单轮 / 长文本 rollout高吞吐、KV cache 管理、continuous batching。
    SGLang多轮、工具调用、结构化交互适合复杂 generation workflow 和多轮 rollout。
    工程理解： vLLM / SGLang 在这里不是“部署模型给用户用”，而是“给训练过程快速生产新样本”。在线 RL 的数据不是静态写好的，而是当前 policy 一边训练一边生成出来的。

    6. Reward Function：RLVR 的真正核心

    很多人学 RLVR 会先看 GRPO 公式，但真正决定训练效果的是 reward function。 算法只是把 reward 变成梯度；reward 才定义了模型到底应该学什么。

    6.1 一个数学 reward 不应该只做字符串包含

    最简单的 reward 是答案包含正确数字。但这很脆弱：模型可能在错误推理中偶然包含正确答案，也可能正确答案格式不同导致解析失败。 更稳的数学 reward 一般需要三个步骤：抽取、规范化、验证。 

    import re


def extract_answer(text: str):
    """从模型输出中抽取最终答案。"""
    patterns = [
        r"Final Answer\s*[:：]\s*([^\n]+)",
        r"答案是\s*([^\n。]+)",
        r"\\boxed\{([^}]+)\}",
    ]
    for p in patterns:
        m = re.search(p, text)
        if m:
            return m.group(1).strip()
    return None


def normalize_answer(ans: str):
    if ans is None:
        return None
    ans = ans.strip()
    ans = ans.replace("，", ",").replace("。", "")
    ans = ans.replace(" ", "")
    return ans


def math_reward(text: str, gold: str):
    pred = normalize_answer(extract_answer(text))
    gold = normalize_answer(str(gold))
    return 1.0 if pred == gold else 0.0
    步骤目的常见错误
    抽取答案找到 final answer没有强制输出格式，导致抽不到。
    规范化处理空格、标点、分数、小数“1/2”和“0.5”等价但被判错。
    验证判断是否正确只做字符串包含，导致假阳性。

    6.2 Agent reward 应该拆成多项

    Agent 任务不能只看最终答案。因为一个 Agent 可能最终答对，但过程很危险、很低效，或者根本没有真正使用工具。

    奖励项含义示例
    格式奖励Action 是否可解析JSON 合法 +0.1
    工具选择奖励是否调用了正确工具计算题调用 calculator +0.2
    参数奖励参数是否完整准确query 字段存在且非空 +0.2
    执行奖励工具是否成功执行工具无报错 +0.2
    最终答案奖励最终任务是否完成答案正确 +1.0
    成本惩罚是否过多调用工具每多一步 -0.02
    def tool_agent_reward(trace, gold_answer):
    reward = 0.0

    if trace.action_json_valid:
        reward += 0.1
    if trace.tool_name == trace.expected_tool:
        reward += 0.2
    if trace.tool_args_valid:
        reward += 0.2
    if trace.tool_execution_success:
        reward += 0.2
    if normalize_answer(trace.final_answer) == normalize_answer(gold_answer):
        reward += 1.0

    reward -= 0.02 * max(0, trace.num_tool_calls - 2)
    return max(reward, 0.0)
    Reward Hacking 警告： 如果你只奖励“出现 Final Answer”，模型就会学会每次都输出 Final Answer；如果你只奖励“调用工具”，模型就会疯狂调用工具；如果你只奖励“答案包含关键词”，模型就会堆关键词。reward 必须同时约束结果、格式、成本和安全边界。

    7. Advantage：为什么 reward 不能直接拿来训练？

    Reward 是结果分数，Advantage 是“这次输出比基线好多少”。强化学习更新的不是绝对 reward，而是相对表现。

    7.1 PPO 的 advantage：critic / value / GAE

    PPO 通常有 critic。critic 的作用是估计某个状态下“本来大概能拿多少分”。如果实际 reward 比 critic 估计高，advantage 为正；反之为负。 

    Advantage = 实际回报 - Critic 估计值
    组件作用
    Actor生成动作 / token。
    Critic估计 value，提供 baseline。
    Reward实际反馈。
    Advantage告诉 actor 哪些动作比预期好，哪些比预期差。

    7.2 GRPO 的 advantage：组内平均就是 baseline

    GRPO 不训练 critic。它用同一个 prompt 的多条回答形成一个 group，然后用组内平均 reward 当 baseline。

    ResponseReward组内平均Advantage 直觉
    A1.00.5+0.5，强化
    B0.00.5-0.5，削弱
    C0.80.5+0.3，强化
    D0.20.5-0.3，削弱

    如果进一步做标准差归一化，advantage 还会除以组内或 batch 的 reward 标准差。不同实现对 reward scaling 有不同处理，TRL 和 verl 文档都提醒过 reward scaling / std normalization 可能影响训练偏差。

    关键理解： GRPO 的“聪明”之处在于：它不问“这个回答绝对好不好”，而问“同一个问题下，这条回答比模型自己生成的其他回答好不好”。这让它不需要单独训练 critic，但要求每个 prompt 有多条有差异的 samples。

    8. KL：为什么要把模型拴在 reference model 附近？

    在 RL 训练中，模型会朝 reward 高的方向移动。但 reward 往往不完美。如果没有约束，模型可能学会奇怪的输出格式、重复文本、过度迎合 reward function，甚至语言能力退化。 KL 的作用就是把 policy 拴在 reference model 附近。 

    Policy objective = 提高 reward / advantage
                  - KL penalty × 偏离 reference 的程度
    KL 太小KL 太大
    模型可能为了 reward 跑飞，输出越来越奇怪。模型被 reference 锁太死，学不动，reward 上不去。
    可能出现 reward hacking、重复、格式坍塌。训练很稳定，但提升很小。
    调参直觉： 如果 reward 很快上升但输出质量变怪，检查 KL 是否太弱；如果输出稳定但 reward 不动，可能 KL 太强、学习率太低、reward 信号太弱或组内差异不足。

    9. Mini Math RLVR：从玩具实验到真实框架

    学工程框架不能一上来就训练 7B / 70B。最稳的方式是先做一个 Mini Math RLVR：数据小、奖励明确、失败可观察。

    9.1 数据格式

    {"prompt": "计算 17 * 23，并把最终答案写成 Final Answer: xxx", "answer": "391"}
{"prompt": "如果 x + 8 = 15，x 等于多少？Final Answer 格式回答。", "answer": "7"}
{"prompt": "计算 144 / 12。Final Answer 格式回答。", "answer": "12"}

    9.2 输出格式约束

    为什么要强制 Final Answer:？因为 reward function 需要稳定解析。格式越稳定，reward 越可靠；reward 越可靠，RL 越容易收敛。 

    推荐输出：
推理过程：...
Final Answer: 391

不推荐输出：
应该是三百九十一吧。

    9.3 TRL 版最小 GRPO 实验

    from datasets import Dataset
from trl import GRPOConfig, GRPOTrainer
import re

samples = [
    {"prompt": "计算 17 * 23，并把最终答案写成 Final Answer: xxx", "answer": "391"},
    {"prompt": "如果 x + 8 = 15，x 等于多少？Final Answer 格式回答。", "answer": "7"},
]
dataset = Dataset.from_list(samples)


def extract_final(text):
    m = re.search(r"Final Answer\s*[:：]\s*([^\n]+)", text)
    return m.group(1).strip() if m else None


def reward_func(completions, answer, **kwargs):
    rewards = []
    for completion, gold in zip(completions, answer):
        text = completion[0]["content"] if isinstance(completion, list) else completion
        pred = extract_final(text)
        rewards.append(1.0 if pred == str(gold) else 0.0)
    return rewards

config = GRPOConfig(
    output_dir="./mini_math_grpo",
    num_generations=4,
    per_device_train_batch_size=2,
    max_completion_length=256,
    logging_steps=1,
)

trainer = GRPOTrainer(
    model="Qwen/Qwen2.5-0.5B-Instruct",
    args=config,
    train_dataset=dataset,
    reward_funcs=[reward_func],
)
trainer.train()

    9.4 观察什么日志？

    指标正常现象异常信号
    平均 reward慢慢上升一直 0：解析失败 / 题太难 / 输出格式不对。
    reward 方差同组内有差异全 0 或全 1：advantage 信号弱。
    completion length合理范围内波动越来越长：可能 reward 没有长度约束。
    KL稳定在可控范围过高：policy 跑飞；过低：学不动。
    格式通过率逐渐提高仍然乱：SFT 冷启动不足。

    10. Mini Tool-use Agent：从 SFT 到 DPO 再到 RLVR

    Agent 后训练建议不要一开始就做网页浏览或复杂代码仓库。最稳的第一个实验是 Function Picking：给一个任务，让模型选择正确工具并输出合法参数。

    10.1 工具集合

    TOOLS = {
    "calculator": {
        "description": "用于精确数学计算",
        "schema": {"expression": "string"}
    },
    "search_doc": {
        "description": "用于检索本地文档",
        "schema": {"query": "string"}
    },
    "python": {
        "description": "用于运行简单 Python 代码",
        "schema": {"code": "string"}
    }
}

    10.2 SFT 阶段：先学会格式

    {
  "messages": [
    {"role": "user", "content": "计算 88 * 17。"},
    {"role": "assistant", "content": "Thought: 这是精确计算问题，应该使用 calculator。\nAction: calculator({\"expression\": \"88 * 17\"})"},
    {"role": "tool", "content": "1496"},
    {"role": "assistant", "content": "Final Answer: 88 * 17 = 1496。"}
  ]
}

    10.3 DPO 阶段：比较好坏轨迹

    {
  "prompt": "计算 88 * 17。",
  "chosen": "Thought: 需要精确计算。\nAction: calculator({\"expression\": \"88 * 17\"})",
  "rejected": "Thought: 我可以搜索一下。\nAction: search_doc({\"query\": \"88 * 17\"})"
}

    10.4 RLVR 阶段：用执行结果给 reward

    检查项reward
    Action JSON 可解析+0.1
    工具名正确+0.2
    参数字段符合 schema+0.2
    工具执行成功+0.2
    最终答案正确+1.0
    超过 3 次工具调用-0.1
    Agent 后训练最小闭环： 先用 SFT 学格式，再用 DPO 学偏好，最后用 RLVR 优化执行成功率。不要一开始就指望 RL 自动学会格式；格式不稳时，RL 会把 reward 噪声放大。

    11. 三个框架怎么选？

    你的条件 / 目标推荐框架原因
    刚入门，想看懂 SFT / DPO / GRPOTRL代码短，Hugging Face 生态友好，适合小模型和 LoRA。
    想用 1-8 张卡跑 RLVRTRL 或 OpenRLHFTRL 适合小实验；OpenRLHF 更适合 vLLM + Ray 的高吞吐训练。
    想理解生产级 RLHF 系统OpenRLHF架构清楚，强调 Ray + vLLM + DeepSpeed、Hybrid Engine、Async Training。
    想研究 Agent RL / 多轮工具调用verl / OpenRLHF都有 multi-turn / agent 相关设计；verl 的 Agent Loop / Reward Loop 很适合读工程数据流。
    想复现 reasoning RL 论文路线verl / OpenRLHF更接近大规模 RL post-training 的真实配置。
    只有消费级显卡TRL + LoRA / QLoRA先跑 0.5B / 1.5B 小模型，验证数据和 reward。

    12. 常见报错和实践坑

    12.1 数据格式类

    问题表现解决
    chat template 不匹配模型输出奇怪角色标记，或者推理时格式崩训练和推理使用同一 tokenizer / chat template。
    assistant loss mask 错误模型学会生成 user / tool 内容检查只对 assistant 生成 token 计算 loss。
    答案抽取失败reward 一直 0强制 Final Answer 格式，增强 parser。
    chosen / rejected 质量差DPO 后模型变差偏好对必须差异明确，避免 chosen 本身也错。

    12.2 RL 训练类

    问题表现解决
    组内 reward 全相同GRPO 学不动调整题目难度、采样温度、num_generations。
    KL 过大输出跑飞、重复、格式怪增大 KL 系数，降低学习率，缩短 rollout。
    KL 过小但 reward 不涨模型几乎没变化降低 KL 约束、增大学习率、检查 reward 信号。
    completion 越来越长reasoning 膨胀，显存爆加入长度惩罚，设置 max_completion_length。
    reward hacking指标涨但人工看变差加入多目标 reward 和人工抽查。

    12.3 工程环境类

    问题表现解决
    Ray worker 找不到依赖本地能跑，Ray job 报 import error使用 runtime env 或在所有节点安装依赖。
    vLLM / CUDA 版本不兼容启动 rollout engine 失败固定 CUDA、PyTorch、vLLM、flash-attn 版本。
    权重同步慢rollout 和 training 间等待很久检查 NCCL、tensor parallel、网络带宽。
    OOM训练或 rollout 爆显存降低 batch、降低 max_new_tokens、启用 ZeRO / FSDP / gradient checkpointing。

    13. 读源码时应该抓哪几条线？

    真正读框架源码时，不要从文件树第一行开始看。你要按后训练闭环追踪。

    追踪线索要找什么你会理解什么
    数据线dataset → tokenizer → collator → batchprompt / response / mask 如何进入模型。
    生成线rollout engine → sampling params → generated sequences当前 policy 如何产生新样本。
    奖励线reward function / RM → reward tensor任务目标如何变成标量分数。
    优势线reward → baseline → advantage为什么某些 token 被强化，某些被削弱。
    KL 线policy logprob vs reference logprob模型如何被约束在 reference 附近。
    更新线loss → backward → optimizer → checkpoint最终参数如何改变。

    14. 下一步学习路线

    这篇读完以后，不建议立刻上大模型。下一步应该做两个小实验。

    14.1 实验一：Mini Math GRPO

    目标：让 0.5B / 1.5B 模型学会固定格式回答简单数学题。

步骤：
1. 构造 100 条小学数学题。
2. 强制 Final Answer 格式。
3. 写 extract_answer + normalize_answer + reward_func。
4. 用 TRL GRPOTrainer 跑通。
5. 观察 reward、KL、completion length、格式通过率。
6. 把同样 reward 思路迁移到 OpenRLHF 或 verl。

    14.2 实验二：Function Picking Agent

    目标：让模型在 calculator / search_doc / python 三个工具中选择正确工具。

步骤：
1. 写 100 条 SFT 工具调用轨迹。
2. 构造 chosen / rejected 偏好对。
3. 用 SFT 稳定格式。
4. 用 DPO 改善工具选择偏好。
5. 用 RLVR 奖励 JSON 合法、工具正确、执行成功、最终答案正确。
6. 观察是否出现伪造工具结果、乱调用工具、过长轨迹。
    最终建议： 你的学习路线应该从“读懂概念”转成“跑通最小闭环”。先用 TRL 验证数据和 reward，再读 OpenRLHF / verl 的系统设计。这样你不会被复杂工程吓住，也不会只停在理论总结。

    15. 本篇总结：工程框架到底帮你做了什么？

    框架一句话总结最值得学的点
    TRL把后训练方法封装成 Hugging Face 风格 Trainer。数据格式、SFT/DPO/GRPO 最小实验、reward function。
    OpenRLHF把 RLHF / RLVR 做成 Ray + vLLM + DeepSpeed 的大规模训练系统。Actor/Critic/Reward/Reference 调度、Hybrid Engine、Async rollout。
    verl把复杂 RL post-training 组织成可组合的数据流。actor_rollout_ref、Reward Loop、Agent Loop、multi-turn rollout。

    到这里，你应该能把理论词汇和工程词汇对应起来： 

    SFT → SFTTrainer / supervised dataset / chat template
DPO → DPOTrainer / preference dataset / chosen-rejected
RLVR → reward_func.py / verifier / math checker / unit test
GRPO → group sampling / relative reward / critic-less advantage
PPO → actor + critic + reward + reference + clipped update
Rollout → vLLM / SGLang generation / trajectories
Agent RL → agent loop / tool feedback / response mask / environment reward
    这篇的核心收获： 后训练工程不是“换一个 Trainer 名字”。它是一条闭环：当前模型生成 → 奖励系统评价 → advantage 计算 → KL 约束 → policy 更新 → 再生成。TRL、OpenRLHF、verl 只是对这条闭环的不同工程实现。
