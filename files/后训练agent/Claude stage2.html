    0. 这一篇要解决什么问题？

    第一部分我们已经把 Claude / Claude Code 理解成一个 Agent Loop：模型看到上下文，决定回答或调用工具；工具执行后把结果返回给模型；模型继续判断下一步。 这一篇继续往下拆：Claude Code 的工具系统到底由哪些部件组成？为什么 Read / Edit / Bash / Grep / Glob / Todo 不是随便堆出来的工具，而是一套 Agent-Computer Interface？ 

    用户任务
  ↓
Claude Code Agent Loop
  ↓
工具选择：Read / Edit / Bash / Glob / Grep / Todo / MCP
  ↓
权限检查：是否允许读、写、执行、联网、调用外部服务
  ↓
工具执行：文件系统 / shell / 搜索 / 后台任务 / MCP server
  ↓
工具结果：Observation 返回给 Claude
  ↓
Claude 继续计划、修改、验证、总结
    一句话理解： Claude Code 的关键不是“模型会写代码”这么简单，而是把模型放进一个受控的开发环境中：模型负责决策，工具负责执行，权限系统负责边界，Hooks 负责拦截和自动化，Agent Loop 负责多轮推进。

    0.1 为什么工具系统比模型回答更重要？

    普通聊天模型只能说“你可以修改这个文件”。Coding Agent 不一样，它必须真的去读文件、定位代码、修改代码、运行测试、根据报错再修复。因此，代码 Agent 的能力来自两部分：

    能力来源具体含义Claude Code 中的体现
    模型能力理解任务、规划步骤、选择工具、解释结果。Claude 判断应该读哪个文件、改哪里、跑什么命令。
    工具能力把模型意图变成真实环境动作。Read 读文件，Edit 改文件，Bash 跑测试，Grep 搜索代码。
    权限能力限制工具动作，避免越权或危险操作。读写路径规则、Bash 命令规则、Hooks 审批。
    反馈能力把环境结果返回给模型，形成闭环。工具结果、测试输出、报错、diff、日志。

    图里最关键的不是“LLM”三个字，而是右侧环境反馈循环：Search files → Return paths → Write code → Status → Test → Results。也就是说，代码 Agent 的真正训练和工程重点是： 

    不是一次性生成完整代码
而是：
搜索 → 阅读 → 修改 → 测试 → 看结果 → 再修改

    2. Claude Code 工具总览：不是只有 Read / Edit / Bash

    Claude Code 官方 Tools reference 列出了内置工具，并说明这些工具名会出现在 permission rules、subagent tool lists 和 hook matchers 中。也就是说，工具不只是“能不能调用”，还会进入权限、子代理、Hooks、技能系统。

    2.1 按功能分类理解 Claude Code 工具

    工具类别典型工具用途是否高风险
    文件读取Read读取文件内容，是后续 Edit 的基础。低到中，取决于路径。
    文件创建/修改Write、Edit、NotebookEdit创建文件、精确替换、修改 Notebook。高，需要权限控制。
    代码搜索Glob、Grep、LSP按文件名、内容、符号搜索项目。低到中。
    命令执行Bash、PowerShell、Monitor运行测试、构建、git、脚本、后台监控。最高，需要强权限边界。
    任务管理TodoWrite、TaskCreate、TaskList、TaskUpdate维护多步任务计划和状态。低。
    子代理Agent、TeamCreate、SendMessage把任务分给独立上下文或 agent team。中到高，取决于子代理工具权限。
    外部连接MCP tools、WebFetch、WebSearch接外部数据、API、文档、网页。中到高。
    自动化扩展Skill、Workflow、Hooks封装重复流程，拦截工具调用，自动执行脚本。取决于具体动作。
    关键理解： Claude Code 的工具系统不是一个“函数列表”，而是一套可治理的执行环境。工具名同时服务于三个系统：模型选择工具、权限系统限制工具、Hooks 系统监听工具。

    3. 源码复现一：工具基类与统一返回格式

    为了写清楚工具系统，先不要急着写 Read / Edit / Bash。我们先设计一个统一接口：每个工具都要有名字、描述、输入 schema、权限需求、执行函数、返回结果。这样才能对应 Claude Code 的真实工程逻辑。 

    from __future__ import annotations

from dataclasses import dataclass, field
from pathlib import Path
from typing import Any, Dict, Optional, Protocol, List
import json
import time


@dataclass
class ToolResult:
    """
    所有工具统一返回这个结构。

    为什么要统一？
    1. Claude 需要看到稳定的 Observation。
    2. Agent Loop 需要知道工具是否成功。
    3. 权限系统 / 日志系统 / Hooks 都要读这个结构。
    """
    ok: bool
    tool_name: str
    content: Any = None
    error: Optional[str] = None
    metadata: Dict[str, Any] = field(default_factory=dict)

    def to_text(self, max_chars: int = 8000) -> str:
        payload = {
            "ok": self.ok,
            "tool_name": self.tool_name,
            "content": self.content,
            "error": self.error,
            "metadata": self.metadata,
        }
        text = json.dumps(payload, ensure_ascii=False, indent=2)
        if len(text) > max_chars:
            return text[:max_chars] + "\n... [truncated]"
        return text


class Tool(Protocol):
    name: str
    description: str
    input_schema: Dict[str, Any]
    requires_permission: bool

    def run(self, tool_input: Dict[str, Any]) -> ToolResult:
        ...

    3.1 这段源码对应 Claude Code 哪一层？

    源码结构对应 Claude Code 概念为什么重要
    ToolResulttool_result / Observation工具执行结果必须结构化返回给模型。
    ok工具是否成功模型下一步要根据成功/失败选择继续、修复或放弃。
    metadata路径、耗时、截断标记、权限来源帮助 Agent 理解环境状态。
    requires_permissionPermission required有些工具低风险，有些工具必须询问或检查规则。
    input_schematool schema告诉模型这个工具需要哪些字段。
    实践启发： 你自己写 Agent 时，不要让工具随便返回字符串。最好所有工具都返回统一 JSON：ok、content、error、metadata。这样 reward、日志、错误恢复都会简单很多。

    4. 源码复现二：Read 工具

    Read 是 Claude Code 最基础的工具。没有 Read，模型不知道项目真实内容；没有稳定的 Read，后续 Edit 就容易乱改。官方工具参考中也把 Read 作为读取文件内容的工具。 

    class ReadTool:
    name = "Read"
    description = "Read the content of a UTF-8 text file."
    requires_permission = False
    input_schema = {
        "type": "object",
        "properties": {
            "path": {"type": "string", "description": "File path to read"},
            "offset": {"type": "integer", "description": "Start line, optional"},
            "limit": {"type": "integer", "description": "Maximum number of lines, optional"},
        },
        "required": ["path"],
    }

    def __init__(self, project_root: str):
        self.project_root = Path(project_root).resolve()

    def _resolve_path(self, path: str) -> Path:
        target = Path(path).expanduser()
        if not target.is_absolute():
            target = self.project_root / target
        return target.resolve()

    def run(self, tool_input: Dict[str, Any]) -> ToolResult:
        path = tool_input.get("path")
        offset = int(tool_input.get("offset", 1))
        limit = int(tool_input.get("limit", 200))

        if not path:
            return ToolResult(False, self.name, error="Missing required field: path")

        target = self._resolve_path(path)

        if not target.exists():
            return ToolResult(False, self.name, error=f"File does not exist: {target}")
        if not target.is_file():
            return ToolResult(False, self.name, error=f"Path is not a file: {target}")

        try:
            lines = target.read_text(encoding="utf-8").splitlines()
        except UnicodeDecodeError:
            return ToolResult(False, self.name, error="Only UTF-8 text files are supported.")

        start = max(offset - 1, 0)
        end = min(start + limit, len(lines))
        selected = lines[start:end]

        numbered = [f"{i + 1}: {line}" for i, line in enumerate(selected, start=start)]

        return ToolResult(
            ok=True,
            tool_name=self.name,
            content="\n".join(numbered),
            metadata={
                "path": str(target),
                "total_lines": len(lines),
                "returned_lines": [start + 1, end],
                "truncated": end < len(lines),
            },
        )

    4.1 Read 工具为什么要支持 offset / limit？

    设计点原因不这样做的后果
    行号让模型能精确定位代码位置。模型只能模糊说“上面那里”。
    offset读取长文件的一部分。一次塞入太多 token，浪费上下文。
    limit控制返回长度。工具结果过长，冲掉对话上下文。
    metadata告诉模型是否被截断。模型可能误以为文件已经读完。
    UTF-8 检查避免二进制文件污染上下文。模型看到乱码，后续判断失真。
    一句话理解： Read 不是简单 cat file。一个好的 Read 工具要服务于模型推理：行号、截断、路径、错误信息都要清楚。

    5. 源码复现三：Glob 与 Grep 搜索工具

    Claude Code 不能一上来就把整个仓库读进上下文。它通常先用 Glob 找文件，再用 Grep 找内容，再用 Read 读目标文件。官方工具参考中，Glob 用于按文件名 pattern 找文件，Grep 用于按内容 pattern 找匹配。

    5.1 Glob：按文件名找文件

    class GlobTool:
    name = "Glob"
    description = "Find files by glob pattern, such as **/*.py or src/**/*.ts."
    requires_permission = False
    input_schema = {
        "type": "object",
        "properties": {
            "pattern": {"type": "string"},
            "path": {"type": "string", "description": "Base directory, optional"},
            "limit": {"type": "integer", "description": "Maximum results"},
        },
        "required": ["pattern"],
    }

    def __init__(self, project_root: str):
        self.project_root = Path(project_root).resolve()

    def run(self, tool_input: Dict[str, Any]) -> ToolResult:
        pattern = tool_input.get("pattern")
        base = Path(tool_input.get("path", "."))
        limit = int(tool_input.get("limit", 100))

        if not pattern:
            return ToolResult(False, self.name, error="Missing required field: pattern")

        if not base.is_absolute():
            base = self.project_root / base
        base = base.resolve()

        matches = []
        for item in base.glob(pattern):
            if item.is_file():
                matches.append(item)

        matches.sort(key=lambda p: p.stat().st_mtime, reverse=True)
        shown = matches[:limit]

        return ToolResult(
            ok=True,
            tool_name=self.name,
            content=[str(p.relative_to(self.project_root)) for p in shown],
            metadata={
                "pattern": pattern,
                "base": str(base),
                "total_matches": len(matches),
                "returned": len(shown),
                "truncated": len(matches) > limit,
            },
        )

    5.2 Grep：按内容找匹配

    import re

class GrepTool:
    name = "Grep"
    description = "Search file contents with a regex pattern."
    requires_permission = False
    input_schema = {
        "type": "object",
        "properties": {
            "pattern": {"type": "string"},
            "glob": {"type": "string", "description": "File glob, e.g. **/*.py"},
            "output_mode": {"type": "string", "enum": ["files_with_matches", "content", "count"]},
            "limit": {"type": "integer"},
        },
        "required": ["pattern"],
    }

    def __init__(self, project_root: str):
        self.project_root = Path(project_root).resolve()

    def run(self, tool_input: Dict[str, Any]) -> ToolResult:
        pattern = tool_input.get("pattern")
        file_glob = tool_input.get("glob", "**/*")
        output_mode = tool_input.get("output_mode", "files_with_matches")
        limit = int(tool_input.get("limit", 100))

        if not pattern:
            return ToolResult(False, self.name, error="Missing required field: pattern")

        try:
            regex = re.compile(pattern)
        except re.error as e:
            return ToolResult(False, self.name, error=f"Invalid regex: {e}")

        results = []
        for path in self.project_root.glob(file_glob):
            if not path.is_file():
                continue
            if ".git" in path.parts or "node_modules" in path.parts:
                continue

            try:
                lines = path.read_text(encoding="utf-8", errors="ignore").splitlines()
            except Exception:
                continue

            hits = []
            for line_no, line in enumerate(lines, start=1):
                if regex.search(line):
                    hits.append({"line": line_no, "text": line[:300]})

            if hits:
                rel = str(path.relative_to(self.project_root))
                if output_mode == "files_with_matches":
                    results.append(rel)
                elif output_mode == "count":
                    results.append({"path": rel, "count": len(hits)})
                else:
                    for hit in hits[:5]:
                        results.append({"path": rel, **hit})

            if len(results) >= limit:
                break

        return ToolResult(
            ok=True,
            tool_name=self.name,
            content=results[:limit],
            metadata={
                "pattern": pattern,
                "glob": file_glob,
                "output_mode": output_mode,
                "returned": min(len(results), limit),
                "truncated": len(results) >= limit,
            },
        )

    5.3 Glob / Grep / Read 的协作关系

    任务：修复登录模块 bug
  ↓
Glob("**/*auth*.ts")
  ↓
Grep("login|token|session", glob="src/**/*.ts")
  ↓
Read("src/auth/session.ts")
  ↓
Edit("src/auth/session.ts", old_string=..., new_string=...)
  ↓
Bash("npm test")
    工具解决的问题模型什么时候用
    Glob文件在哪里？不知道具体文件名，但知道大概 pattern。
    Grep代码片段在哪里？知道关键词、函数名、错误信息。
    Read目标文件具体内容是什么？准备分析或修改文件之前。
    Edit如何精确修改？已经读过文件，并定位到唯一片段。
    Bash修改是否有效？需要测试、构建、lint、git。

    6. 源码复现四：Edit 工具与 read-before-edit

    Edit 是代码 Agent 中最容易出事故的工具，所以它不能像普通字符串替换那样随便执行。Claude Code 官方工具参考强调，Edit 是精确字符串替换，并要求读取过文件、匹配旧字符串、旧字符串唯一等检查。 

    class EditTool:
    name = "Edit"
    description = "Make a precise edit by replacing an exact old_string with new_string."
    requires_permission = True
    input_schema = {
        "type": "object",
        "properties": {
            "path": {"type": "string"},
            "old_string": {"type": "string"},
            "new_string": {"type": "string"},
            "replace_all": {"type": "boolean"},
        },
        "required": ["path", "old_string", "new_string"],
    }

    def __init__(self, project_root: str, read_cache: Dict[str, float]):
        self.project_root = Path(project_root).resolve()
        self.read_cache = read_cache

    def _resolve_path(self, path: str) -> Path:
        target = Path(path).expanduser()
        if not target.is_absolute():
            target = self.project_root / target
        return target.resolve()

    def run(self, tool_input: Dict[str, Any]) -> ToolResult:
        path = tool_input.get("path")
        old = tool_input.get("old_string")
        new = tool_input.get("new_string")
        replace_all = bool(tool_input.get("replace_all", False))

        if not path or old is None or new is None:
            return ToolResult(False, self.name, error="Missing path, old_string, or new_string")

        target = self._resolve_path(path)
        if not target.exists() or not target.is_file():
            return ToolResult(False, self.name, error=f"File not found: {target}")

        cache_key = str(target)
        last_read_mtime = self.read_cache.get(cache_key)
        current_mtime = target.stat().st_mtime

        if last_read_mtime is None:
            return ToolResult(False, self.name, error="Read-before-edit check failed: file was not read in this session.")
        if abs(last_read_mtime - current_mtime) > 1e-6:
            return ToolResult(False, self.name, error="Read-before-edit check failed: file changed after it was read.")

        text = target.read_text(encoding="utf-8")
        count = text.count(old)

        if count == 0:
            return ToolResult(False, self.name, error="old_string not found exactly. Whitespace must match.")
        if count > 1 and not replace_all:
            return ToolResult(False, self.name, error=f"old_string appears {count} times. Provide more context or set replace_all=true.")

        updated = text.replace(old, new) if replace_all else text.replace(old, new, 1)
        target.write_text(updated, encoding="utf-8")

        return ToolResult(
            ok=True,
            tool_name=self.name,
            content="Edit applied successfully.",
            metadata={
                "path": str(target),
                "replacements": count if replace_all else 1,
                "old_length": len(old),
                "new_length": len(new),
            },
        )

    6.1 为什么 Edit 要这么严格？

    检查作用没有它会怎样
    read-before-edit确保模型看过最新文件内容。模型根据旧上下文乱改。
    old_string 精确匹配避免模糊替换。误伤相似代码。
    唯一性检查确保只改目标位置。把多个重复片段都改坏。
    replace_all 显式开关全局替换必须明确。模型无意中大范围修改。
    权限检查写操作必须可审计。Agent 可以随意改项目文件。
    重要： Edit 工具的设计思路不是“让模型更自由”，而是“让模型难以犯低级破坏性错误”。这就是 Anthropic 博客里说的 Agent-Computer Interface：给模型设计工具时，要像给人设计 HCI 一样认真。

    7. 源码复现五：Bash 工具与命令权限

    Bash 是 Claude Code 中风险最高但也最有价值的工具。它能跑测试、构建、git、脚本、安装依赖，也可能删除文件、泄露环境变量、执行危险命令。因此 Bash 必须接权限系统。

    7.1 先写一个危险命令分类器

def classify_bash_command(command: str) -> Dict[str, Any]:
    """
    简化版命令分类器。
    真实系统可以更复杂：规则 + LLM classifier + user permission + hooks。
    """
    normalized = command.strip().lower()

    for pattern in DANGEROUS_PATTERNS:
        if pattern in normalized:
            return {
                "risk": "deny",
                "reason": f"Command contains dangerous pattern: {pattern}",
            }

    try:
        tokens = shlex.split(command)
    except ValueError as e:
        return {"risk": "deny", "reason": f"Shell parse error: {e}"}

    if not tokens:
        return {"risk": "allow", "reason": "Empty command"}

    first = tokens[0]

    if first in ["ls", "pwd", "cat", "head", "tail", "grep", "rg", "find"]:
        return {"risk": "allow", "reason": "Read-only inspection command"}

    if first in WRITE_COMMANDS:
        return {"risk": "ask", "reason": "Command may modify files or environment"}

    return {"risk": "ask", "reason": "Unknown command, ask for permission"}

    7.2 Bash 执行器

    class BashTool:
    name = "Bash"
    description = "Run a shell command in the project directory."
    requires_permission = True
    input_schema = {
        "type": "object",
        "properties": {
            "command": {"type": "string"},
            "timeout": {"type": "integer", "description": "Timeout in seconds"},
        },
        "required": ["command"],
    }

    def __init__(self, project_root: str, permission_engine: "PermissionEngine"):
        self.project_root = Path(project_root).resolve()
        self.permission_engine = permission_engine

    def run(self, tool_input: Dict[str, Any]) -> ToolResult:
        command = tool_input.get("command")
        timeout = int(tool_input.get("timeout", 120))

        if not command:
            return ToolResult(False, self.name, error="Missing required field: command")

        decision = self.permission_engine.check(tool_name="Bash", specifier=command)
        if decision["behavior"] == "deny":
            return ToolResult(False, self.name, error=f"Permission denied: {decision['reason']}")
        if decision["behavior"] == "ask":
            return ToolResult(False, self.name, error=f"Permission required: {decision['reason']}")

        started = time.time()
        try:
            completed = subprocess.run(
                command,
                cwd=self.project_root,
                shell=True,
                capture_output=True,
                text=True,
                timeout=timeout,
            )
        except subprocess.TimeoutExpired:
            return ToolResult(False, self.name, error=f"Command timed out after {timeout}s")

        stdout = completed.stdout[-12000:]
        stderr = completed.stderr[-12000:]

        return ToolResult(
            ok=completed.returncode == 0,
            tool_name=self.name,
            content={
                "stdout": stdout,
                "stderr": stderr,
                "returncode": completed.returncode,
            },
            error=None if completed.returncode == 0 else "Command exited with non-zero status.",
            metadata={
                "command": command,
                "cwd": str(self.project_root),
                "duration_sec": round(time.time() - started, 3),
                "timeout_sec": timeout,
                "output_truncated": len(completed.stdout) > 12000 or len(completed.stderr) > 12000,
            },
        )

    7.3 Bash 工具为什么要返回 returncode？

    返回字段模型如何使用
    stdout看测试输出、构建结果、命令正常信息。
    stderr定位报错、栈信息、依赖缺失、类型错误。
    returncode判断命令是否成功。
    duration_sec判断命令是否异常慢。
    output_truncated提醒模型必要时读取完整日志。
    一句话理解： Bash 不是“给模型一个终端”。Bash 是一个可审计、可截断、可超时、可权限控制的执行环境。

    8. 源码复现六：权限系统 PermissionEngine

    官方 Tools reference 说明，工具可以通过 permission rules、CLI flags、Agent SDK allowedTools / disallowedTools、subagent frontmatter、skill frontmatter 和 hook matcher 进行控制。我们可以写一个简化版权限引擎来理解这套机制。 

    import fnmatch

@dataclass
class PermissionRule:
    behavior: str   # allow / deny / ask
    tool_name: str
    pattern: str = "*"
    reason: str = ""


class PermissionEngine:
    """
    简化版权限系统。

    支持：
    - deny 优先
    - allow 其次
    - 未命中则 ask
    - Bash 按命令 pattern
    - Read/Edit 按路径 pattern
    """
    def __init__(self, rules: List[PermissionRule]):
        self.rules = rules

    def check(self, tool_name: str, specifier: str) -> Dict[str, str]:
        matched_allow = None

        for rule in self.rules:
            if rule.tool_name != tool_name:
                continue
            if fnmatch.fnmatch(specifier, rule.pattern):
                if rule.behavior == "deny":
                    return {
                        "behavior": "deny",
                        "reason": rule.reason or f"Denied by rule: {tool_name}({rule.pattern})",
                    }
                if rule.behavior == "allow":
                    matched_allow = rule

        if matched_allow:
            return {
                "behavior": "allow",
                "reason": matched_allow.reason or "Allowed by matching rule",
            }

        # Bash 再走命令风险分类
        if tool_name == "Bash":
            classified = classify_bash_command(specifier)
            if classified["risk"] == "deny":
                return {"behavior": "deny", "reason": classified["reason"]}
            if classified["risk"] == "allow":
                return {"behavior": "allow", "reason": classified["reason"]}
            return {"behavior": "ask", "reason": classified["reason"]}

        return {
            "behavior": "ask",
            "reason": f"No explicit allow rule for {tool_name}({specifier})",
        }


# 示例权限配置
rules = [
    PermissionRule("allow", "Read", "src/**", "Can read project source files"),
    PermissionRule("allow", "Edit", "src/**", "Can edit source files"),
    PermissionRule("deny", "Read", "**/.env", "Never read secrets"),
    PermissionRule("deny", "Edit", "**/.env", "Never modify secrets"),
    PermissionRule("allow", "Bash", "npm test*", "Tests are safe to run"),
    PermissionRule("allow", "Bash", "git diff*", "Read-only git inspection"),
    PermissionRule("deny", "Bash", "rm -rf*", "Dangerous deletion command"),
]

permission_engine = PermissionEngine(rules)

    8.1 权限系统的层级

    控制入口作用适合场景
    settings.json permissions项目或用户级默认权限。长期项目规则。
    CLI flags启动会话时临时限制工具。一次性任务。
    Agent SDK allowed_tools在代码里限制可用工具。自己写 Agent 应用。
    Subagent tools限制子代理工具范围。代码审查 agent 只能 Read/Grep，不能 Edit/Bash。
    Hooks在工具调用前后拦截。自动格式化、安全检查、审批流。
    实践建议： 如果你自己做 Code Agent，默认不要给 Bash 全权限。更稳的方式是白名单：允许 pytest、npm test、git diff、ruff check 等低风险命令；其他命令进入 ask 或 deny。

    9. 源码复现七：Todo / Task 工具

    很多人忽视 Todo，但它是 Agent 长任务稳定性的关键。Agent 如果没有任务列表，很容易做着做着忘记目标。Claude Code 旧版常见 TodoWrite，新版工具参考里也出现 TaskCreate、TaskList、TaskUpdate 等任务工具。 

    @dataclass
class TaskItem:
    id: int
    content: str
    status: str = "pending"  # pending / in_progress / done / blocked
    notes: str = ""


class TaskManager:
    def __init__(self):
        self.items: List[TaskItem] = []
        self.next_id = 1

    def create(self, content: str) -> TaskItem:
        item = TaskItem(id=self.next_id, content=content)
        self.next_id += 1
        self.items.append(item)
        return item

    def update(self, task_id: int, status: Optional[str] = None, notes: Optional[str] = None) -> ToolResult:
        for item in self.items:
            if item.id == task_id:
                if status:
                    item.status = status
                if notes is not None:
                    item.notes = notes
                return ToolResult(True, "TaskUpdate", content=item.__dict__)
        return ToolResult(False, "TaskUpdate", error=f"Task not found: {task_id}")

    def list(self) -> ToolResult:
        return ToolResult(
            ok=True,
            tool_name="TaskList",
            content=[item.__dict__ for item in self.items],
            metadata={"count": len(self.items)},
        )


tasks = TaskManager()
tasks.create("Locate the auth module")
tasks.create("Read the failing test")
tasks.create("Patch session validation")
tasks.create("Run unit tests")
print(tasks.list().to_text())

    9.1 Todo 对 Agent 的价值

    没有 Todo有 Todo / Task
    Agent 可能读了几个文件后忘记目标。每一步都能回到任务列表。
    用户不知道 Agent 进展。用户能看到 pending / in_progress / done。
    失败后不知道卡在哪里。blocked 任务记录原因。
    长任务容易循环。任务完成条件更明确。
    后训练启发： 如果你做 Agent RLVR，Todo 也可以进入 reward：是否建立了合理任务列表、是否按顺序推进、是否完成后更新状态、是否避免无意义循环。

    10.2 一个简化版 Hook 管线

    class Hook:
    def before_tool(self, tool_name: str, tool_input: Dict[str, Any]) -> Optional[ToolResult]:
        return None

    def after_tool(self, tool_name: str, tool_input: Dict[str, Any], result: ToolResult) -> ToolResult:
        return result


class BlockEnvReadHook(Hook):
    """阻止读取 .env 文件。"""
    def before_tool(self, tool_name: str, tool_input: Dict[str, Any]) -> Optional[ToolResult]:
        if tool_name in ["Read", "Edit", "Write"]:
            path = str(tool_input.get("path", ""))
            if path.endswith(".env") or "/.env" in path:
                return ToolResult(False, tool_name, error="Hook blocked access to .env files.")
        return None


class FormatAfterEditHook(Hook):
    """Edit 后自动提示应该运行 formatter。教学版只记录，不真实执行。"""
    def after_tool(self, tool_name: str, tool_input: Dict[str, Any], result: ToolResult) -> ToolResult:
        if tool_name == "Edit" and result.ok:
            result.metadata["suggested_next_action"] = "Run formatter or tests after editing."
        return result


class ToolRunner:
    def __init__(self, tools: Dict[str, Tool], hooks: List[Hook]):
        self.tools = tools
        self.hooks = hooks

    def run(self, tool_name: str, tool_input: Dict[str, Any]) -> ToolResult:
        if tool_name not in self.tools:
            return ToolResult(False, tool_name, error="Unknown tool")

        for hook in self.hooks:
            blocked = hook.before_tool(tool_name, tool_input)
            if blocked is not None:
                return blocked

        result = self.tools[tool_name].run(tool_input)

        for hook in self.hooks:
            result = hook.after_tool(tool_name, tool_input, result)

        return result

    10.3 Hooks 适合做什么？

    Hook 类型用途例子
    安全拦截阻止危险工具调用。禁止读取 .env、禁止删除目录、禁止外发 token。
    自动格式化在 Edit 后运行 formatter。Prettier、Black、Ruff format。
    审计日志记录每次工具调用。写入 session log。
    权限增强动态允许或拒绝某些动作。测试命令自动允许，部署命令必须询问。
    质量检查工具执行后自动触发检查。Edit 后自动提示运行 test。
    注意： Hooks 很强，但也可能增加复杂度。对学习者来说，先理解 PreToolUse / PostToolUse 两个位置就够了：一个在工具执行前拦截，一个在工具执行后处理结果。

    11. 把所有工具接进一个最小 Claude Code Runtime

    现在我们把 Read、Glob、Grep、Edit、Bash、Task、Permission、Hooks 接到一起，形成一个最小 Claude Code Runtime。真实 Claude Code 当然更复杂，但核心思路就是这样。 

    class MiniClaudeCodeRuntime:
    def __init__(self, project_root: str):
        self.project_root = Path(project_root).resolve()
        self.read_cache: Dict[str, float] = {}

        self.permission_engine = PermissionEngine([
            PermissionRule("allow", "Read", "*"),
            PermissionRule("allow", "Glob", "*"),
            PermissionRule("allow", "Grep", "*"),
            PermissionRule("allow", "Edit", "src/**"),
            PermissionRule("allow", "Bash", "pytest*"),
            PermissionRule("allow", "Bash", "npm test*"),
            PermissionRule("allow", "Bash", "git diff*"),
            PermissionRule("deny", "Bash", "rm -rf*"),
        ])

        self.read_tool = ReadTool(project_root)

        # 给 Read 增加 read_cache 记录
        original_read_run = self.read_tool.run
        def cached_read(tool_input: Dict[str, Any]) -> ToolResult:
            result = original_read_run(tool_input)
            if result.ok and result.metadata.get("path"):
                path = result.metadata["path"]
                self.read_cache[path] = Path(path).stat().st_mtime
            return result
        self.read_tool.run = cached_read

        tools = {
            "Read": self.read_tool,
            "Glob": GlobTool(project_root),
            "Grep": GrepTool(project_root),
            "Edit": EditTool(project_root, self.read_cache),
            "Bash": BashTool(project_root, self.permission_engine),
        }

        self.runner = ToolRunner(
            tools=tools,
            hooks=[BlockEnvReadHook(), FormatAfterEditHook()],
        )

    def run_tool(self, tool_name: str, tool_input: Dict[str, Any]) -> str:
        result = self.runner.run(tool_name, tool_input)
        return result.to_text()


# 使用示例
runtime = MiniClaudeCodeRuntime(project_root=".")
print(runtime.run_tool("Glob", {"pattern": "**/*.py", "limit": 20}))
print(runtime.run_tool("Grep", {"pattern": "class .*Tool", "glob": "**/*.py", "output_mode": "content"}))

    11.1 这个 Runtime 和真实 Claude Code 的差距

    模块教学版真实 Claude Code 可能更复杂的地方
    工具执行Python 函数直接执行。跨平台 shell、后台任务、IDE、MCP、LSP、Notebook、Web。
    权限系统简单 allow / deny / ask。多来源 settings、session permission、managed settings、hooks、subagent 权限。
    Bash 安全规则匹配。命令分类、用户确认、沙箱、输出截断、工作目录策略。

    12. 为什么 Claude Code 工具系统适合后训练 Agent 学习？

    你前面学的 Agent RLVR，本质上就是把工具调用轨迹变成可奖励、可筛选、可优化的数据。Claude Code 工具系统刚好是最好的样例，因为每一步都可以转成 reward。

    Claude Code 行为Agent RLVR 奖励设计示例
    是否选择正确工具tool selection reward查代码应该先 Grep / Glob，而不是直接 Bash。
    参数是否完整parameter completeness rewardRead 必须有 path，Edit 必须有 old_string / new_string。
    是否先读后改read-before-edit rewardEdit 前必须 Read 当前文件。
    是否执行成功execution success rewardBash returncode == 0。
    是否利用结果observation usage reward根据 pytest 报错修改，而不是忽略报错。
    轨迹是否高效trajectory length penalty避免无意义重复 Grep 或反复读同一文件。
    是否遵守权限safety reward不读取 .env，不运行危险命令。
    迁移到你自己的学习路线： Claude Code 工具系统可以作为 Agent RLVR 的样板。你可以先做一个 mini coding agent，然后给每条轨迹打分：工具选对了吗？参数合法了吗？有没有 read-before-edit？测试过了吗？有没有根据测试失败修复？这比抽象讲 Agent 更有用。

    13. 可直接写进 HTML 总结的核心句

    第一，Claude Code 的工具系统不是工具列表，而是 Agent 执行环境。Read、Glob、Grep 负责感知项目；Edit、Write 负责修改环境；Bash、PowerShell、Monitor 负责执行和验证；Todo / Task 负责维持长任务状态；MCP 负责扩展外部工具。

    第二，工具系统必须和权限系统绑定。如果没有权限控制，Bash 和 Edit 会变成高风险能力；如果权限太死，Agent 又无法完成任务。Claude Code 的设计重点就在于把工具名、路径、命令、settings、hooks、subagent tools 统一纳入可治理机制。

    第三，Code Agent 的能力来自环境反馈循环。它不是一次性写答案，而是不断搜索、读取、编辑、运行测试、根据结果修复。这个循环正是 Agent RLVR 可以训练和奖励的对象。
